# Go Code Generation

> `go generate` turns your development tools into first-class citizens of the build pipeline — generating type-safe, zero-overhead code that reflection would compute at runtime at 100x the cost.

## Mental Model

Code generation in Go is the philosophical bet that **explicitness beats magic**. When `protoc-gen-go` generates 10,000 lines of Go from a `.proto` file, those 10,000 lines are committed to the repository, reviewed by humans, stepped through by debuggers, and analyzed by every static analysis tool in the ecosystem. They are just Go code. There is no special runtime behavior, no reflection, no JIT. The generated code is indistinguishable from hand-written code to the compiler.

The `//go:generate` directive is the wiring mechanism. It is a comment in your source file that the `go generate` tool reads and executes as a shell command. It has no special relationship to the build system — `go build` ignores it. The intent is that developers run `go generate` after changing the source artifacts (proto files, SQL schema files, enum type definitions) and commit the resulting Go files.

The distinction between code generation and reflection comes down to *when* the work happens. Reflection does at runtime what code generation does at development time. If you can enumerate all the types your code will process at development time — if the schema is fixed, if the enum values are known — code generation always wins on performance, type safety, and debuggability.

Three tools compose into most Go code generation pipelines:

1. **A source file** (proto schema, SQL query, Go source with type definitions)
2. **A generator** (a Go program that reads the source and writes `.go` files)
3. **`//go:generate`** (a directive in a `.go` file that invokes the generator)

## Core Concepts

### `//go:generate` Directive

```go
//go:generate stringer -type=Direction -output=direction_string.go
//go:generate go run ./cmd/gen/main.go -input=schema.sql -output=./db/queries.go
//go:generate protoc --go_out=. --go_opt=paths=source_relative api/v1/service.proto
```

The directive is any shell command. `go generate ./...` walks all `.go` files in the package tree, finds lines matching `//go:generate`, and runs them. Within the command, several environment variables are available: `$GOPACKAGE` (current package name), `$GOFILE` (current file name), `$GOLINE` (line number of the directive).

Generated files must begin with a header identifying them as generated:
```go
// Code generated by stringer -type=Direction; DO NOT EDIT.
```
This header allows tools (IDEs, linters, coverage reporters) to skip generated files. The exact format is standardized.

### `text/template` for Code Generation

Go's `text/template` package is a logic-capable template engine that outputs text. For code generation, it is the standard tool for the "fill in this pattern for each item" class of problems.

Key template constructs for code generation:
```
{{ range .Fields }}   ... {{ .Name }} ... {{ end }}   ← iteration
{{ if .IsPointer }}   ...                 {{ end }}   ← condition  
{{ .Name | title }}                                   ← function pipeline
{{- ... -}}                                           ← trim surrounding whitespace
{{ printf "%q" .Name }}                               ← Go-style formatting
```

The generated code must be passed through `go/format.Source()` before writing to disk. This normalizes whitespace, indentation, and formatting to `gofmt` standards — which is important because templates produce inconsistent whitespace that makes diffs unreadable.

### `go/ast` for Source Analysis

When the generator needs to read existing Go source (not just an external schema), it uses the `go/ast` and `go/parser` packages:

```go
fset := token.NewFileSet()
f, err := parser.ParseFile(fset, "input.go", nil, parser.ParseComments)
// f is a *ast.File with the full parsed AST
```

The AST is a tree of `ast.Node` values. The `ast.Inspect` function walks it:

```go
ast.Inspect(f, func(n ast.Node) bool {
    if typeSpec, ok := n.(*ast.TypeSpec); ok {
        // found a type declaration
    }
    return true // continue walking
})
```

## Implementation: Go

### A `stringer`-like Generator for Typed Enums

This generator reads Go source files, finds types annotated with a `//go:enum` comment, and generates a `String()` method with a lookup table.

**Generator program (`cmd/enumgen/main.go`):**

```go
package main

import (
	"bytes"
	"flag"
	"fmt"
	"go/ast"
	"go/format"
	"go/parser"
	"go/token"
	"os"
	"strings"
	"text/template"
	"unicode"
)

func main() {
	inputFile := flag.String("input", "", "Go source file to parse")
	outputFile := flag.String("output", "", "Output file (default: <input>_string.go)")
	flag.Parse()

	if *inputFile == "" {
		fmt.Fprintln(os.Stderr, "usage: enumgen -input=<file.go> [-output=<file_string.go>]")
		os.Exit(1)
	}

	if *outputFile == "" {
		base := strings.TrimSuffix(*inputFile, ".go")
		*outputFile = base + "_string.go"
	}

	enums, pkgName, err := extractEnums(*inputFile)
	if err != nil {
		fmt.Fprintf(os.Stderr, "parse error: %v\n", err)
		os.Exit(1)
	}

	if len(enums) == 0 {
		fmt.Printf("no //go:enum types found in %s\n", *inputFile)
		return
	}

	code, err := generateEnumFile(pkgName, enums)
	if err != nil {
		fmt.Fprintf(os.Stderr, "generate error: %v\n", err)
		os.Exit(1)
	}

	if err := os.WriteFile(*outputFile, code, 0o644); err != nil {
		fmt.Fprintf(os.Stderr, "write error: %v\n", err)
		os.Exit(1)
	}
	fmt.Printf("wrote %s\n", *outputFile)
}

type EnumInfo struct {
	TypeName string
	Values   []EnumValue
}

type EnumValue struct {
	Name  string   // identifier: DirectionNorth
	Label string   // display label: "North" (derived from name by stripping type prefix)
}

// extractEnums parses a Go source file and finds all const blocks whose type
// has a preceding "//go:enum" comment.
func extractEnums(filename string) ([]EnumInfo, string, error) {
	fset := token.NewFileSet()
	f, err := parser.ParseFile(fset, filename, nil, parser.ParseComments)
	if err != nil {
		return nil, "", err
	}

	// Collect type names marked with //go:enum
	markedTypes := map[string]bool{}
	for _, decl := range f.Decls {
		genDecl, ok := decl.(*ast.GenDecl)
		if !ok || genDecl.Tok != token.TYPE {
			continue
		}
		if genDecl.Doc != nil {
			for _, comment := range genDecl.Doc.List {
				if strings.TrimSpace(comment.Text) == "//go:enum" {
					for _, spec := range genDecl.Specs {
						if ts, ok := spec.(*ast.TypeSpec); ok {
							markedTypes[ts.Name.Name] = true
						}
					}
				}
			}
		}
	}

	// Collect const values grouped by type
	enumMap := map[string]*EnumInfo{}
	for _, decl := range f.Decls {
		genDecl, ok := decl.(*ast.GenDecl)
		if !ok || genDecl.Tok != token.CONST {
			continue
		}
		for _, spec := range genDecl.Specs {
			vs, ok := spec.(*ast.ValueSpec)
			if !ok || vs.Type == nil {
				continue
			}
			ident, ok := vs.Type.(*ast.Ident)
			if !ok || !markedTypes[ident.Name] {
				continue
			}
			typeName := ident.Name
			if _, exists := enumMap[typeName]; !exists {
				enumMap[typeName] = &EnumInfo{TypeName: typeName}
			}
			for _, name := range vs.Names {
				label := deriveLabel(typeName, name.Name)
				enumMap[typeName].Values = append(enumMap[typeName].Values, EnumValue{
					Name:  name.Name,
					Label: label,
				})
			}
		}
	}

	var result []EnumInfo
	for _, info := range enumMap {
		result = append(result, *info)
	}
	return result, f.Name.Name, nil
}

// deriveLabel strips the type name prefix from a const name and converts to Title Case.
// DirectionNorth → "North", StatusHTTPNotFound → "HTTPNotFound"
func deriveLabel(typeName, constName string) string {
	stripped := strings.TrimPrefix(constName, typeName)
	if stripped == "" || stripped == constName {
		return constName
	}
	// Insert spaces before uppercase runs: "HTTPNotFound" → "HTTP Not Found"
	var b strings.Builder
	for i, r := range stripped {
		if i > 0 && unicode.IsUpper(r) && !unicode.IsUpper(rune(stripped[i-1])) {
			b.WriteRune(' ')
		}
		b.WriteRune(r)
	}
	return b.String()
}

var enumTemplate = template.Must(template.New("enum").Parse(`// Code generated by enumgen; DO NOT EDIT.

package {{ .PackageName }}

import "fmt"
{{ range .Enums }}
var _{{ .TypeName }}Strings = map[{{ .TypeName }}]string{
{{ range .Values }}	{{ .Name }}: {{ printf "%q" .Label }},
{{ end }}}

func (v {{ .TypeName }}) String() string {
	if s, ok := _{{ .TypeName }}Strings[v]; ok {
		return s
	}
	return fmt.Sprintf("{{ .TypeName }}(%d)", int(v))
}
{{ end }}`))

type templateData struct {
	PackageName string
	Enums       []EnumInfo
}

func generateEnumFile(pkgName string, enums []EnumInfo) ([]byte, error) {
	var buf bytes.Buffer
	if err := enumTemplate.Execute(&buf, templateData{PackageName: pkgName, Enums: enums}); err != nil {
		return nil, err
	}
	// Always run through gofmt — templates produce messy whitespace
	formatted, err := format.Source(buf.Bytes())
	if err != nil {
		// Return the unformatted source so the caller can see what went wrong
		return buf.Bytes(), fmt.Errorf("format error (raw output preserved): %w", err)
	}
	return formatted, nil
}
```

**Input file (`direction.go`):**
```go
package shapes

//go:enum
type Direction int

const (
	DirectionNorth Direction = iota
	DirectionSouth
	DirectionEast
	DirectionWest
)
```

**`//go:generate` directive (at top of `direction.go`):**
```go
//go:generate go run ../../cmd/enumgen/main.go -input=direction.go -output=direction_string.go
```

**Generated output (`direction_string.go`):**
```go
// Code generated by enumgen; DO NOT EDIT.

package shapes

import "fmt"

var _DirectionStrings = map[Direction]string{
	DirectionNorth: "North",
	DirectionSouth: "South",
	DirectionEast:  "East",
	DirectionWest:  "West",
}

func (v Direction) String() string {
	if s, ok := _DirectionStrings[v]; ok {
		return s
	}
	return fmt.Sprintf("Direction(%d)", int(v))
}
```

**Main program demonstrating the generated code:**
```go
package main

import (
	"fmt"
	"shapes"
)

func main() {
	d := shapes.DirectionNorth
	fmt.Println(d)           // "North"
	fmt.Printf("%v\n", d)   // "North"

	unknown := shapes.Direction(99)
	fmt.Println(unknown)     // "Direction(99)"
}
```

### Go-specific considerations

**Generator architecture patterns**: Generators are normal Go programs. They follow two patterns:

1. *Schema-driven*: Read a non-Go source (`.proto`, `.sql`, `.graphql`) and emit `.go` files. `protoc-gen-go` and `sqlc` work this way. The generator is invoked per-schema.

2. *Source-driven*: Read existing `.go` files using `go/ast` (or `golang.org/x/tools/go/packages` for full type information) and emit more `.go` files. `stringer`, `mockgen`, and `Wire` work this way. The generator is invoked per-package.

**`golang.org/x/tools/go/packages`**: For source-driven generators that need type information (not just syntax), `go/ast` alone is insufficient — it does not resolve identifiers or understand types. Use `golang.org/x/tools/go/packages` with `NeedTypes | NeedTypesInfo` to load packages with full type resolution. This is significantly more expensive but enables analysis like "which types implement this interface."

**Generated file conventions**: Every generated file must: (1) start with `// Code generated ... DO NOT EDIT.` on the first line; (2) have a `package` declaration matching the package; (3) never be edited by hand (the `DO NOT EDIT` comment is a strong convention that IDEs and code review tools respect). Teams should add generated files to `.gitattributes` with `linguist-generated=true` to exclude them from GitHub's language stats and diffs.

**When code generation beats reflection**: Use code generation when: types are known at development time, the generated code is called in a hot path, type safety errors should be caught at compile time (not runtime), or the generated implementation is complex enough that reflection would be hard to debug.

## Go vs Rust: Direct Comparison

| Aspect | Go | Rust |
|--------|-----|------|
| Code generation mechanism | `go:generate` + external tool | `build.rs` + `proc_macro` |
| Source analysis | `go/ast`, `golang.org/x/tools/go/packages` | `syn` crate |
| Template engine | `text/template` (stdlib) | `quote!` macro |
| Generated code visibility | Committed to repo | In-memory (use `cargo expand`) |
| Format normalization | `go/format.Source()` → `gofmt` | `prettyplease` crate |
| Proto codegen | `protoc-gen-go` | `prost` (uses `build.rs` + `prost-build`) |
| SQL codegen | `sqlc` | `sqlx` (compile-time macros), `sea-orm-cli` |
| Enum stringification | `stringer` | `strum` proc macro |
| Schema → code round-trip | Development time | Build time (`build.rs`) |

## Production War Stories

**protoc-gen-go: the canonical generator**: Every Google-style microservice in Go uses `protoc-gen-go` to generate its networking layer from `.proto` files. The generated code is ~3x more verbose than the proto schema and would be tedious to maintain by hand — but it compiles, is fully type-safe, and is directly debuggable. The `protoc-gen-go-grpc` plugin adds gRPC client/server stubs on top. Together they generate all the serialization, client code, and server interface — typically thousands of lines per service. The commit history of the generated files tracks schema evolution without any special tools.

**sqlc: type-safe SQL without ORM reflection**: `sqlc` reads a SQL schema and annotated query files, and generates Go structs for each table and a function for each query. A query like `-- name: GetUser :one\nSELECT * FROM users WHERE id = $1` becomes a `GetUser(ctx context.Context, id int64) (User, error)` function with direct `database/sql` calls. There is no reflection, no query building at runtime, no ORM magic. Type errors (passing a string where an int is expected) are caught at compile time. The Stripe engineering team published a case study showing `sqlc` eliminated an entire class of production bugs that GORM's reflection-based API could not catch.

**`stringer` — the simplest useful generator**: The `stringer` tool ships with `golang.org/x/tools` and has been running in production Go codebases since 2014. Its implementation is under 200 lines. It parses `const` blocks using `go/ast`, extracts the values and names, and generates a `String()` method using a lookup table. The table is an array indexed by the const value (when values are dense) or a `map` (when sparse). This is the "hello world" of Go code generation and a good starting point for writing custom generators.

## Complexity Analysis

| Dimension | Cost |
|-----------|------|
| Generator run time | O(schema size); typically 0.1-5s; run manually, not in `go build` |
| Generated code runtime | Identical to hand-written code — zero overhead |
| Maintenance | Medium — must re-run generator when schema changes |
| CI integration | `go generate ./...` + `git diff --exit-code` in CI to catch stale generated files |
| Debuggability | Full — generated code is real Go, debugger steps through it |
| Test coverage | Generated code can (and should) be unit tested like any other code |

## Common Pitfalls

**1. Not running `gofmt` on generated output.** Templates produce inconsistent whitespace. Always pass generated source through `go/format.Source()`. This also validates that the output is syntactically valid Go — if `format.Source` fails, the generator has a bug.

**2. Stale generated files in CI.** The most common code generation problem: a developer changes the schema, forgets to re-run the generator, and commits. The CI pipeline passes because it just builds the committed `.go` files. Add a CI step: `go generate ./... && git diff --exit-code -- '*.go'` to catch stale files.

**3. Generator not being reproducible.** Generators that use `map` iteration for output ordering produce non-deterministic `.go` files (different field order on every run). This creates noisy diffs. Always sort keys before iterating in generators.

**4. Embedding the generator invocation in the wrong file.** `//go:generate` directives should be in the file that declares the types being generated, not in a separate `generate.go` file. This makes it obvious what generates what. The exception is project-level generation scripts that combine multiple generators.

**5. Not handling the `//go:build ignore` pattern for generator programs.** Generator programs in `cmd/gen/` are often Go files that should not be compiled as part of the main package. Use `//go:build ignore` at the top of the generator's `main.go` to prevent the build tool from compiling it except when explicitly invoked.

## Exercises

**Exercise 1** (30 min): Write a generator that reads a `.go` file and produces a `_mock.go` file with a simple mock struct for each exported interface it finds. The mock should implement all interface methods by panicking with `"not implemented"`. Use `go/ast` and `text/template`.

**Exercise 2** (2-4h): Extend the `enumgen` generator from this document to support: (1) `--prefix=false` flag that disables stripping the type name prefix from const names; (2) generating a `Values() []TypeName` function that returns all valid enum values; (3) generating a `Parse(s string) (TypeName, error)` function that parses a string to an enum value. Test with an enum that has non-contiguous values.

**Exercise 3** (4-8h): Build a minimal `mockgen`-style interface mock generator. Given a Go source file with interface definitions, generate a `Mock<InterfaceName>` struct for each interface where: (a) each method records its call arguments to a `[]<MethodName>Call` slice; (b) each method's return values are configurable via `On<MethodName>(returns...)` setter; (c) a `AssertExpectations(t *testing.T)` method fails the test if any method was not called. Use `golang.org/x/tools/go/packages` for type resolution so you can handle interfaces with types from other packages.

**Exercise 4** (8-15h): Build a minimal `sqlc`-like generator. Accept a SQL CREATE TABLE statement and a file of annotated SQL queries. For each table, generate a Go struct. For each annotated query (`:one`, `:many`, `:exec`), generate a typed Go function that calls `database/sql`. Handle `NULL` columns as `sql.NullString` / `sql.NullInt64`. Generate a `Queries` struct that wraps `*sql.DB`. Write integration tests against an in-memory SQLite database.
