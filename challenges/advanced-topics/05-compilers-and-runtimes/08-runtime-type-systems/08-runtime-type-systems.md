# Runtime Type Systems

> The cost of `obj.method()` is not the cost of the method body — it is the cost of deciding which method body to call, and that decision happens at runtime in ways that have significant and measurable impact on performance.

## Mental Model

In statically-typed languages, method dispatch can happen in two fundamentally different ways:

**Static dispatch (monomorphization)**: The compiler knows the exact type at compile time and generates a direct call to the concrete method. There is no runtime overhead — the call is just a `CALL` instruction to a fixed address. Rust's generics use this: `fn foo<T: Display>(v: T)` generates one copy of `foo` per concrete type `T` used.

**Dynamic dispatch**: The receiver's concrete type is not known at compile time. The call is resolved at runtime by looking up the method in a dispatch table. Go's interfaces and Rust's `dyn Trait` both use dynamic dispatch. The overhead is one or two memory indirections (load vtable pointer, load method pointer from vtable, call).

The performance difference between static and dynamic dispatch is not the dispatch mechanism itself — on modern CPUs with branch prediction and instruction caches, a correctly predicted indirect call costs ~4–5 cycles vs ~1 cycle for a direct call. The real cost is **inlining**: a direct call can be inlined (the callee's body is copied to the call site, eliminating the call overhead and enabling optimizations). An indirect call almost never gets inlined, because the compiler does not know which function is being called.

This is the fundamental tradeoff: **dynamic dispatch enables runtime polymorphism; monomorphization enables inlining and optimization**. The choice between `dyn Trait` and generics in Rust, and between interface values and concrete types in Go, is a choice between these two properties.

V8's hidden class optimization is the JIT-world equivalent: instead of falling back to dictionary lookup for every property access on a dynamic JavaScript object, V8 uses "hidden classes" (also called shapes or maps) to represent the structure of objects with the same property layout. As long as objects of the same "class" keep the same property order, V8 generates fast inline cache code equivalent to static dispatch.

## Core Concepts

### Go's Interface Representation (`iface`)

A Go interface value is a 2-word struct:
```
+------------------+
| *itab            |  → type descriptor + method table
+------------------+
| *data            |  → pointer to the concrete value (or value if ≤ pointer-sized)
+------------------+
```

The `itab` (interface table) contains:
- A pointer to the concrete type's `_type` descriptor (used for type assertions)
- The interface's type descriptor
- A hash of the type (for fast type assertion comparison)
- A table of function pointers — one per method in the interface

When you call `iface.Method()`, the compiler generates:
1. Load the `itab` pointer from the interface value (1 memory load)
2. Load the method pointer from the `itab` at the method's offset (1 memory load)
3. Call the method pointer with the data pointer as the first argument (1 indirect call)

The `itab` is generated by the runtime once per (concrete type, interface) pair and cached. If the same concrete type implements multiple interfaces, there is one `itab` per interface.

**Scalar storage optimization**: If the concrete value fits in a pointer (integer, pointer types), it is stored directly in the data word — no heap allocation. If it is larger than a pointer (struct with multiple fields), the value is boxed: allocated on the heap, and the data word holds a pointer to it. This means converting a `struct` to an interface can cause a heap allocation — a performance footgun in hot paths.

### Rust's `dyn Trait` (Fat Pointer)

A `dyn Trait` value (trait object) is a 2-word fat pointer:
```
+------------------+
| *data            |  → pointer to the concrete value on the heap
+------------------+
| *vtable          |  → pointer to the vtable for this (type, trait) pair
+------------------+
```

The vtable contains:
- The size and alignment of the concrete type (for `drop_in_place`)
- A pointer to `drop_in_place` (the destructor)
- Function pointers for each method in the trait

When you call `trait_obj.method()`, the compiler generates:
1. Load the vtable pointer from the fat pointer (1 memory load)
2. Load the method pointer from the vtable at the fixed offset (1 memory load)
3. Call the method with the data pointer as the first argument (1 indirect call)

This is structurally identical to Go's interface dispatch. The difference: Rust makes this explicit (`dyn Trait` vs `T: Trait`), while Go's interface always does dynamic dispatch.

**`dyn Trait` vs `impl Trait` vs generics**: 
- `fn foo(v: dyn Trait)`: dynamic dispatch, one copy of `foo`, heap-allocated argument
- `fn foo(v: impl Trait)`: static dispatch, syntactic sugar for the generic version
- `fn foo<T: Trait>(v: T)`: static dispatch, monomorphization, one copy per type

### Monomorphization and Code Size

Rust's generics generate one copy of a function per unique set of type parameters. `HashMap<String, Vec<i32>>` generates specialized code for the String key and Vec<i32> value type. This is called monomorphization and it is the mechanism behind Rust's "zero-cost abstractions": the abstract generic code compiles to code as efficient as if you had written the concrete version by hand.

The cost: binary size. A generic function used with 10 different type combinations generates 10 copies. In large Rust codebases, monomorphization is a significant contributor to binary size and compile times. The `cargo bloat` tool can show which monomorphized functions are taking the most space.

Go does not monomorphize generic functions (added in Go 1.18). Instead, Go uses a GC-shape-based approach: all types with the same "GC shape" (same pointer layout) share a single implementation. For pointer types, they all share one implementation (since all pointers have the same size and layout). For value types, separate implementations are generated. This is a middle ground between full monomorphization and full dynamic dispatch.

### V8 Hidden Classes

In JavaScript, objects are dictionaries — you can add and remove properties at any time. Dictionary lookup for every property access would be O(log n) or O(1) with hash, but with constant factors that make it impractical for tight loops.

V8 assigns a "hidden class" (also called a Shape or Map) to each object based on the set and order of properties it has. All objects with the same properties in the same order share a hidden class. V8 generates "inline cache" code that, for a given call site with an observed hidden class, directly computes the property offset — turning the dictionary lookup into a constant-offset memory access.

The critical implication: creating JavaScript objects with properties in different orders creates different hidden classes, defeating the inline cache:

```javascript
// Good: both objects get the same hidden class
function makeGood(x, y) { return { x, y }; }

// Bad: two different hidden classes (different insertion order)
function makeBad1(x, y) { const o = {}; o.x = x; o.y = y; return o; }
function makeBad2(x, y) { const o = {}; o.y = y; o.x = x; return o; }
```

## Implementation: Go

```go
package main

import (
	"fmt"
	"reflect"
	"unsafe"
)

// --- Inspecting Go Interface Internals ---
// We use unsafe to look inside an interface value's representation.

// An interface value in Go is represented as:
//   type iface struct {
//       tab  *itab         // type + method table
//       data unsafe.Pointer // pointer to concrete value
//   }
// This matches the internal layout in go/src/runtime/iface.go

// interfaceLayout mirrors the internal representation of an interface value.
// WARNING: This is implementation-specific and may break between Go versions.
type interfaceLayout struct {
	tab  uintptr        // pointer to itab
	data unsafe.Pointer // pointer to concrete data
}

// Animal is a typical interface with two methods.
type Animal interface {
	Sound() string
	Move() string
}

type Dog struct {
	Name string
	Age  int
}
func (d *Dog) Sound() string { return "Woof" }
func (d *Dog) Move() string  { return "Run" }

type Cat struct {
	Name string
}
func (c *Cat) Sound() string { return "Meow" }
func (c *Cat) Move() string  { return "Slink" }

// Small value type: stored directly in the interface data word (no heap alloc).
type Point struct {
	X, Y int32
}
func (p Point) Sound() string { return "silence" }
func (p Point) Move() string  { return "teleport" }

func inspectInterface() {
	fmt.Println("=== Interface Layout Inspection ===")

	// --- Dog (pointer type) ---
	dog := &Dog{Name: "Rex", Age: 3}
	var a Animal = dog

	// Use reflect to inspect the interface without raw unsafe (safer).
	at := reflect.TypeOf(a)
	fmt.Printf("Type:          %v\n", at)
	fmt.Printf("Kind:          %v\n", at.Kind())
	fmt.Printf("Implements Animal: %v\n", at.Implements(reflect.TypeOf((*Animal)(nil)).Elem()))

	// Interface method call: triggers itab lookup + indirect call.
	fmt.Printf("Sound():       %s\n", a.Sound())
	fmt.Printf("Move():        %s\n", a.Move())

	// --- Inspecting the raw layout (implementation-specific) ---
	// This reads the tab and data pointers from the interface value.
	layout := (*interfaceLayout)(unsafe.Pointer(&a))
	fmt.Printf("\nRaw interface layout:\n")
	fmt.Printf("  itab ptr:  0x%x\n", layout.tab)
	fmt.Printf("  data ptr:  0x%p\n", layout.data)
	fmt.Printf("  dog ptr:   %p (should match data ptr)\n", dog)

	// --- Type assertions: use the itab hash for fast comparison ---
	fmt.Println("\n=== Type Assertions ===")
	animals := []Animal{&Dog{Name: "Buddy"}, &Cat{Name: "Whiskers"}, &Dog{Name: "Max"}}
	for _, a := range animals {
		// Type switch: compiles to comparison of itab pointers (O(1) per case).
		switch v := a.(type) {
		case *Dog:
			fmt.Printf("  Dog: %s\n", v.Name)
		case *Cat:
			fmt.Printf("  Cat: %s\n", v.Name)
		}
	}
}

func interfaceAllocationDemo() {
	fmt.Println("\n=== Interface Allocation Costs ===")

	// Converting a large struct to interface{} causes heap allocation.
	type LargeStruct struct {
		A, B, C, D int64 // 32 bytes — too large for interface data word
	}

	ls := LargeStruct{1, 2, 3, 4}

	// This line causes a heap allocation (ls is boxed into the interface).
	var i interface{} = ls
	_ = i

	fmt.Println("LargeStruct to interface{} = heap allocation (value is boxed)")

	// Pointer types do NOT cause extra allocation (the pointer is stored directly).
	lsPtr := &LargeStruct{1, 2, 3, 4}
	var j interface{} = lsPtr // No extra allocation: pointer stored in data word.
	_ = j
	fmt.Println("*LargeStruct to interface{} = no allocation (pointer fits in data word)")

	// Small integer: no allocation (fits in data word via scalar optimization).
	var k interface{} = 42
	_ = k
	fmt.Println("int(42) to interface{} = no allocation (fits in data word)")
}

// --- Performance: concrete type vs interface dispatch ---

type Adder interface {
	Add(x, y int) int
}

type ConcreteAdder struct{}
func (a ConcreteAdder) Add(x, y int) int { return x + y }

// benchmarkDirect: compiler can inline ConcreteAdder.Add (static dispatch)
func benchmarkDirect(n int) int {
	a := ConcreteAdder{}
	sum := 0
	for i := 0; i < n; i++ {
		sum += a.Add(i, 1) // direct call: inlined to `sum += i + 1`
	}
	return sum
}

// benchmarkInterface: cannot inline (interface dispatch, indirect call)
func benchmarkInterface(n int) int {
	var a Adder = ConcreteAdder{}
	sum := 0
	for i := 0; i < n; i++ {
		sum += a.Add(i, 1) // indirect call: load itab, load method, call
	}
	return sum
}

// --- GC Shape and Go Generics ---

// Go 1.18+ generics use GC-shape-based instantiation.
// For pointer types, all share a single instantiation.
// For value types with different sizes, separate instantiations are generated.
func identity[T any](v T) T { return v }

func demoGoGenerics() {
	fmt.Println("\n=== Go Generics and GC Shapes ===")

	// These calls all use the SAME underlying instantiation
	// because *Dog and *Cat have the same GC shape (both are pointers).
	d := identity(&Dog{Name: "Rex"})
	c := identity(&Cat{Name: "Whiskers"})
	fmt.Printf("identity(*Dog): %v\n", d.Name)
	fmt.Printf("identity(*Cat): %v\n", c.Name)
	fmt.Println("Both use the same GC-shape instantiation (pointer shape)")

	// int and string have different GC shapes → separate instantiations.
	n := identity(42)
	s := identity("hello")
	fmt.Printf("identity(int): %v\n", n)
	fmt.Printf("identity(string): %v\n", s)
}

func main() {
	inspectInterface()
	interfaceAllocationDemo()

	// Demonstrate dispatch cost (in a real benchmark, use `go test -bench`)
	n := 10_000_000
	direct := benchmarkDirect(n)
	iface := benchmarkInterface(n)
	fmt.Printf("\n=== Dispatch Demo (n=%d) ===\n", n)
	fmt.Printf("Direct:    result=%d (compiler inlines the call)\n", direct)
	fmt.Printf("Interface: result=%d (indirect call, no inlining)\n", iface)
	fmt.Println("Use 'go test -bench=.' with -gcflags='-m' to see inlining decisions")

	demoGoGenerics()
}
```

### Go-specific considerations

**`go tool compile -S` and interface dispatch**: Compile with `go tool compile -S file.go` and search for `CALL`. Direct method calls on concrete types produce `CALL package.Type.Method(SB)` — a direct call instruction. Interface method calls produce `CALL <indirect>` — an indirect call through the itab. Each has a distinctly different cost profile under the CPU's branch predictor.

**`devirtualization`**: Go's compiler can sometimes devirtualize an interface call — if it can prove at compile time that only one concrete type flows to an interface variable in a specific scope, it replaces the indirect call with a direct call. This is called "devirtualization" and is more common than people expect. The `GOSSAFUNC` SSA dump shows `devirtualize` as a pass. You can force this by keeping interfaces local: `var a Animal = &Dog{}` in a narrow scope where the compiler can see the concrete type assignment.

**Interface boxing and `sync.Pool`**: If you frequently convert the same struct type to an interface, and the struct is larger than a pointer, each conversion causes a heap allocation. `sync.Pool` can cache the boxed values, avoiding repeated allocation. Alternatively, pass `*Struct` (pointer) to the interface — no boxing needed since a pointer fits in the data word.

## Implementation: Rust

```rust
// Runtime type systems in Rust.
// Demonstrates:
// 1. dyn Trait fat pointer layout
// 2. Monomorphization vs dynamic dispatch
// 3. vtable inspection via std::mem::size_of and trait objects
// 4. Performance comparison of generic vs dyn Trait

use std::fmt;

// --- Trait and Implementations ---

trait Animal: fmt::Debug {
    fn sound(&self) -> &'static str;
    fn move_style(&self) -> &'static str;
    fn describe(&self) -> String {
        format!("{} says {} and moves by {}", self.name(), self.sound(), self.move_style())
    }
    fn name(&self) -> &str;
}

#[derive(Debug)]
struct Dog { name: String, age: u32 }
#[derive(Debug)]
struct Cat { name: String }
#[derive(Debug)]
struct Fish { name: String }

impl Animal for Dog {
    fn sound(&self) -> &'static str { "Woof" }
    fn move_style(&self) -> &'static str { "run" }
    fn name(&self) -> &str { &self.name }
}
impl Animal for Cat {
    fn sound(&self) -> &'static str { "Meow" }
    fn move_style(&self) -> &'static str { "slink" }
    fn name(&self) -> &str { &self.name }
}
impl Animal for Fish {
    fn sound(&self) -> &'static str { "(silence)" }
    fn move_style(&self) -> &'static str { "swim" }
    fn name(&self) -> &str { &self.name }
}

// --- Fat Pointer Layout Inspection ---

fn inspect_fat_pointer() {
    println!("=== Fat Pointer Layout ===");

    let dog = Dog { name: "Rex".into(), age: 3 };

    // `dyn Animal` is a trait object: fat pointer = (data ptr, vtable ptr)
    // Size of dyn Trait reference = 2 × pointer size = 16 bytes on 64-bit
    println!("size_of::<Box<Dog>>()          = {} bytes (single pointer)", std::mem::size_of::<Box<Dog>>());
    println!("size_of::<Box<dyn Animal>>()   = {} bytes (fat pointer)", std::mem::size_of::<Box<dyn Animal>>());
    println!("size_of::<&dyn Animal>()       = {} bytes (fat pointer ref)", std::mem::size_of::<&dyn Animal>());

    // Transmute the fat pointer to inspect its two words.
    // Layout: (data_ptr: *const (), vtable_ptr: *const ())
    let boxed: Box<dyn Animal> = Box::new(Dog { name: "Buddy".into(), age: 2 });
    let fat_ptr: (*const (), *const ()) = unsafe { std::mem::transmute(boxed) };
    println!("\nFat pointer internals:");
    println!("  data  ptr: {:p}", fat_ptr.0);
    println!("  vtable ptr: {:p}", fat_ptr.1);

    // The vtable is a static data structure generated by the compiler.
    // Reading vtable entries (unsafe, implementation-specific):
    // vtable layout: [drop_in_place, size, align, method1, method2, ...]
    let vtable: *const usize = fat_ptr.1 as *const usize;
    unsafe {
        println!("  vtable[0] (drop_in_place): 0x{:x}", *vtable.add(0));   // fn ptr
        println!("  vtable[1] (size of Dog):   {}", *vtable.add(1));        // size
        println!("  vtable[2] (align of Dog):  {}", *vtable.add(2));        // align
        println!("  vtable[3] (sound fn ptr):  0x{:x}", *vtable.add(3));   // method
        println!("  vtable[4] (move fn ptr):   0x{:x}", *vtable.add(4));   // method
    }

    // Verify sizes
    println!("\nDog: size={}, align={}", std::mem::size_of::<Dog>(), std::mem::align_of::<Dog>());
    println!("Cat: size={}, align={}", std::mem::size_of::<Cat>(), std::mem::align_of::<Cat>());

    let _ = dog;
}

// --- Monomorphization vs Dynamic Dispatch ---

// Generic (monomorphization): one copy of this function per concrete T.
fn process_generic<A: Animal>(animals: &[A]) -> String {
    animals.iter().map(|a| a.describe()).collect::<Vec<_>>().join("; ")
}

// Dynamic dispatch: one copy of this function, but indirect calls.
fn process_dynamic(animals: &[Box<dyn Animal>]) -> String {
    animals.iter().map(|a| a.describe()).collect::<Vec<_>>().join("; ")
}

fn demo_dispatch() {
    println!("\n=== Monomorphization vs Dynamic Dispatch ===");

    let dogs = vec![
        Dog { name: "Rex".into(), age: 3 },
        Dog { name: "Buddy".into(), age: 5 },
    ];

    // Generic: process_generic::<Dog> is generated at compile time.
    // The compiler knows exactly which `describe()` to call → can inline.
    let result = process_generic(&dogs);
    println!("Generic:  {}", result);

    // Dynamic: heterogeneous collection (different concrete types).
    // This is impossible with the generic version (requires all same type).
    let mixed: Vec<Box<dyn Animal>> = vec![
        Box::new(Dog { name: "Max".into(), age: 2 }),
        Box::new(Cat { name: "Whiskers".into() }),
        Box::new(Fish { name: "Nemo".into() }),
    ];
    let result = process_dynamic(&mixed);
    println!("Dynamic:  {}", result);

    // The tradeoff:
    // Generic: homogeneous collection only, but inlining + optimization enabled.
    // Dynamic: heterogeneous collection, heap allocation for each element, no inlining.
}

// --- Enum as a Manual Dispatch Table ---
// Sometimes you know all concrete types at compile time.
// An enum is a "sealed" dispatch table: no vtable, inline data, can be matched exhaustively.

#[derive(Debug)]
enum AnyAnimal {
    Dog(Dog),
    Cat(Cat),
    Fish(Fish),
}

impl AnyAnimal {
    fn sound(&self) -> &'static str {
        match self {
            AnyAnimal::Dog(d) => d.sound(),
            AnyAnimal::Cat(c) => c.sound(),
            AnyAnimal::Fish(f) => f.sound(),
        }
    }
    fn name(&self) -> &str {
        match self {
            AnyAnimal::Dog(d) => &d.name,
            AnyAnimal::Cat(c) => &c.name,
            AnyAnimal::Fish(f) => &f.name,
        }
    }
}

fn demo_enum_dispatch() {
    println!("\n=== Enum as Sealed Dispatch Table ===");

    // AnyAnimal uses no vtable — the `match` in each method is a branch table.
    // Size: max(Dog, Cat, Fish) + discriminant tag.
    println!("size_of::<Dog>()       = {}", std::mem::size_of::<Dog>());
    println!("size_of::<Cat>()       = {}", std::mem::size_of::<Cat>());
    println!("size_of::<AnyAnimal>() = {}", std::mem::size_of::<AnyAnimal>());

    let animals = vec![
        AnyAnimal::Dog(Dog { name: "Rex".into(), age: 3 }),
        AnyAnimal::Cat(Cat { name: "Whiskers".into() }),
        AnyAnimal::Fish(Fish { name: "Nemo".into() }),
    ];

    for a in &animals {
        println!("  {} says {}", a.name(), a.sound());
    }

    println!("No heap allocation per element (unlike Vec<Box<dyn Animal>>)");
    println!("No vtable lookup (unlike dyn Trait calls)");
    println!("Tradeoff: closed set — adding a new variant requires changing the enum");
}

// --- Code Size: Monomorphization Explosion ---

// This generic function will be instantiated for each unique T used.
fn identity<T>(x: T) -> T { x }

fn demo_monomorphization_cost() {
    println!("\n=== Monomorphization and Binary Size ===");

    // Each call instantiates identity<T> for a new T.
    // In a large codebase, this multiplies binary size.
    let _i:   i32     = identity(42_i32);
    let _f:   f64     = identity(3.14_f64);
    let _s:   String  = identity("hello".to_string());
    let _v:   Vec<i32>= identity(vec![1, 2, 3]);

    // The compiler generates 4 separate copies of `identity`.
    // For a 10-line function used with 100 type combinations: 100 copies in the binary.
    // Use `cargo bloat --release` to measure this in real codebases.
    println!("identity instantiated for: i32, f64, String, Vec<i32>");
    println!("Use 'cargo bloat --release --crates' to measure monomorphization overhead");

    // Mitigation: extract the non-generic part of the function into a helper
    // that takes a raw pointer or erased type, reducing duplication.
}

fn main() {
    inspect_fat_pointer();
    demo_dispatch();
    demo_enum_dispatch();
    demo_monomorphization_cost();
}
```

### Rust-specific considerations

**vtable layout is not stable**: The memory layout of vtables in Rust is not specified by the language and can change between compiler versions. Do not rely on vtable offsets in production code — the `inspect_fat_pointer` example is for educational purposes only. Use `std::any::TypeId` for runtime type identification in production.

**`dyn Trait` and object safety**: Not all traits can be made into trait objects. A trait is "object safe" if:
1. It has no associated functions without `self` (no static methods)
2. It has no generic methods (monomorphization is incompatible with dynamic dispatch)
3. It does not require `Sized` on `Self`

Attempting to use `dyn Trait` for an object-unsafe trait is a compile error. This is Rust's way of preventing you from accidentally creating a trait object for a trait that cannot dispatch dynamically.

**`enum_dispatch` crate**: The `enum_dispatch` crate automates the "enum as sealed dispatch table" pattern. It generates the `match` boilerplate and implements the trait for the enum automatically. This gives you the performance of enum-based dispatch with the ergonomics of trait objects.

**The `Sized` bound and fat pointers**: When Rust knows the size of a type at compile time, it is `Sized`. When it does not (e.g., `dyn Trait`, `[T]`), it is `!Sized`. `!Sized` types can only appear behind pointers (`&dyn Trait`, `Box<dyn Trait>`), which is why they are fat pointers — the pointer must carry the size/vtable information that the type itself does not have.

## Go vs Rust: Direct Comparison

| Aspect | Go | Rust |
|--------|----|------|
| Interface/dispatch syntax | `var a Animal = &Dog{}` (always dynamic) | `dyn Animal` (explicit dynamic), `impl Animal` / generics (static) |
| Representation | 2-word: (itab ptr, data ptr) | 2-word fat pointer: (data ptr, vtable ptr) |
| Static dispatch | Only for concrete types, not interfaces | Generics + monomorphization |
| Dynamic dispatch overhead | 2 loads + indirect call | 2 loads + indirect call (identical) |
| Inlining through dispatch | No (interface calls never inlined) | No for `dyn Trait`; yes for generics |
| Heterogeneous collections | `[]Animal` (interface slice) | `Vec<Box<dyn Animal>>` (heap allocation per element) |
| Binary size | GC-shape based: shared implementations for same-shape types | Full monomorphization: one copy per type instantiation |
| Type erasure | `interface{}` / `any` | `Box<dyn Any>` |
| Type assertion | `a.(ConcreteType)` (itab comparison) | `downcast_ref::<T>()` from `Any` trait |
| Enum-based dispatch | No direct equivalent (manual type switch) | `enum` variants — idiomatic sealed dispatch |

## Production War Stories

**Discord's interface dispatch bottleneck**: An early version of Discord's Go voice server used interfaces extensively in the hot path for audio packet processing. Profiling showed the interface dispatch overhead was measurable — not because of the indirect call itself, but because the interface calls prevented inlining of small, trivial methods (type conversion, header parsing). Refactoring to use concrete types in the hot path with interfaces only at the package boundary restored inlining and improved throughput by 15%.

**Rust monomorphization and compile times**: The `serde` library is notorious for long compile times due to monomorphization. `serde::Serialize` and `serde::Deserialize` are implemented generically over the serialization format. Each combination of (data type, format) generates a separate instantiation. A project using `serde_json`, `serde_yaml`, and `serde_toml` with 20 custom types generates 60+ serialization implementations. Build tools like `cargo-timing` and `cargo-llvm-lines` can show which monomorphizations dominate compile time. The mitigation: use `Box<dyn Serializer>` for less-critical paths.

**V8 hidden class transitions and React performance**: Facebook's React team discovered that certain patterns in React component creation caused "hidden class transitions" in V8 — objects changing shape after creation (adding properties dynamically), destroying inline cache effectiveness. The optimization that fixed this: using object spread syntax `{...defaults, ...overrides}` instead of mutating objects in place, ensuring all objects of a given component type have the same property order. After this change, V8's inline caches hit consistently and several React benchmarks improved 10–20%.

**Go interface boxing and `encoding/json`**: `encoding/json` uses `interface{}` / `any` extensively for its generic JSON handling. Every value passed to `json.Marshal` or `json.Unmarshal` is either an interface value or accessed via reflection. This causes:
1. Interface boxing: every value converted to `interface{}` may allocate if the value is larger than a pointer
2. Reflect-based dispatch: reflection is slower than direct method calls

Libraries like `easyjson` and `go-json` generate type-specific serialization code at code-generation time, eliminating interface dispatch entirely. `go-json` reports 3–6x faster serialization compared to `encoding/json` for typical workloads.

## Complexity Analysis

| Operation | Go | Rust |
|-----------|----|----|
| Static method call | O(1) direct call, inlineable | O(1) direct call, inlineable |
| Interface/dyn Trait call | O(1) indirect call (2 loads + indirect call, ~4–8 cycles), not inlineable | Same |
| Type assertion | O(1) itab pointer comparison | O(1) TypeId comparison |
| Monomorphization compile cost | O(types × function size) — GC-shape limited | O(types × function size) — full monomorphization |
| Fat pointer size | 2 words (16 bytes) | 2 words (16 bytes) |

## Common Pitfalls

**1. Interface boxing in tight loops.** Converting a non-pointer value to an interface in a loop may allocate on each iteration. Profile with `go test -memprofile` and look for unexpected allocations. Fix: pass pointer to interface, or restructure to use concrete types in the hot path.

**2. Using `dyn Trait` when generics would serve.** If your function is called with only one or two concrete types, a generic function with monomorphization is strictly better (enables inlining). Use `dyn Trait` only when the concrete type is genuinely not known at compile time (e.g., plugin systems, heterogeneous collections).

**3. Enum variant size inflation.** When using enums as dispatch tables, `enum AnyAnimal` is sized to the largest variant. If one variant is much larger than others, all enum values waste memory. Fix: box the large variant: `Large(Box<LargeVariant>)`.

**4. Forgetting to seal interfaces in Go.** Go interfaces are structurally typed — any type that implements the methods satisfies the interface. This is powerful but means you cannot prevent external packages from implementing your interface. If you need a "sealed" set of implementations (for performance optimization or logical correctness), use an unexported marker method in the interface or an enum-equivalent struct.

**5. Indirect function calls and branch misprediction.** An indirect call (`CALL [rax]`) is harder for the branch predictor to predict than a direct call (`CALL address`). In a polymorphic dispatch loop (many different concrete types in sequence), the indirect call will mispredict frequently, causing 15–20 cycle penalties per misprediction. Sorting a collection by concrete type before processing it in a loop (layout optimization) can dramatically improve branch prediction accuracy.
