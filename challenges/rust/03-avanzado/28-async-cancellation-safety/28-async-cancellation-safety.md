## Difficulty: Avanzado

## Introduction

In async Rust, any future can be **dropped** at any point where it yields (`.await`). This is fundamentally different from synchronous code, where a function either runs to completion or panics. Cancellation safety is the discipline of writing async code that behaves correctly even when futures are abandoned mid-execution.

This is not an academic concern. Every `tokio::select!` branch, every timeout, every connection drop in a production system is a cancellation event. If your async code is cancellation-unsafe, you will lose data, corrupt state, or create resource leaks -- silently and intermittently.

This exercise covers the mechanics of cancellation, how to identify unsafe patterns, and how to design APIs that are resilient to cancellation.

---

## What Happens When a Future Is Dropped

A future in Rust is a state machine generated by the compiler. Each `.await` point is a yield boundary. When you drop a future, Rust runs the `Drop` implementation of whatever state the future currently holds, but **it does not resume execution** of the future.

```rust
use tokio::time::{sleep, Duration};

async fn do_work() {
    println!("Step 1: acquiring resource");
    sleep(Duration::from_secs(1)).await;

    // If the future is dropped while sleeping above,
    // this line NEVER executes.
    println!("Step 2: releasing resource");
}

#[tokio::main]
async fn main() {
    let handle = tokio::spawn(do_work());

    // Give it a moment to start, then drop it
    sleep(Duration::from_millis(100)).await;
    handle.abort();

    // Wait for the abort to take effect
    let result = handle.await;
    println!("Task result: {:?}", result);
    // Output: Task result: Err(JoinError::Cancelled)
}
```

**Key insight**: Step 2 never prints. The resource acquired in Step 1 is never explicitly released. The `Drop` implementations of local variables still run, but any *logical* cleanup that requires further async execution is skipped.

## Cancellation-Safe vs Cancellation-Unsafe Operations

An async operation is **cancellation-safe** if dropping the future at any `.await` point does not lose data or corrupt state.

### Cancellation-Safe Operations

These operations can be safely used in `select!` branches because dropping them loses nothing:

```rust
use tokio::sync::mpsc;

async fn cancellation_safe_examples() {
    let (tx, mut rx) = mpsc::channel::<String>(32);

    // SAFE: recv() is cancellation-safe.
    // If dropped before a message arrives, no message is lost.
    // The message stays in the channel.
    let msg = rx.recv().await;

    // SAFE: TcpListener::accept() is cancellation-safe.
    // If dropped, the connection stays in the OS backlog.
    let listener = tokio::net::TcpListener::bind("127.0.0.1:0").await.unwrap();
    let (socket, addr) = listener.accept().await.unwrap();

    // SAFE: sleep() is cancellation-safe.
    // If dropped, nothing happens -- time just passes.
    tokio::time::sleep(std::time::Duration::from_secs(1)).await;
}
```

### Cancellation-Unsafe Operations

These operations can lose data if the future is dropped mid-execution:

```rust
use tokio::io::{AsyncReadExt, AsyncWriteExt};

async fn cancellation_unsafe_examples() {
    // UNSAFE: read_exact() is cancellation-unsafe.
    // If dropped after reading some bytes but before reading all,
    // those partial bytes are lost -- they're consumed from the stream
    // but never returned to the caller.
    let mut buf = vec![0u8; 1024];
    let mut stream = tokio::net::TcpStream::connect("127.0.0.1:8080")
        .await
        .unwrap();
    stream.read_exact(&mut buf).await.unwrap();

    // UNSAFE: read_to_string() is cancellation-unsafe.
    // Same problem: partial reads are consumed but lost on drop.
    let mut contents = String::new();
    stream.read_to_string(&mut contents).await.unwrap();
}
```

### The Fundamental Rule

An operation is cancellation-unsafe when it has **already performed an irrecoverable side effect** at a yield point. The side effect is committed, but the future hasn't yet returned it to the caller.

```rust
// This function is cancellation-unsafe:
async fn fetch_and_process(url: &str) -> Result<String, Box<dyn std::error::Error>> {
    // Side effect 1: HTTP request sent, response received
    let response = reqwest::get(url).await?;

    // If dropped HERE, we've consumed the response but never
    // returned the body to the caller. The HTTP request was wasted.

    let body = response.text().await?;

    // Side effect 2: database write
    save_to_database(&body).await?;

    // If dropped HERE, we saved to DB but the caller never gets
    // confirmation. They might retry, causing a duplicate write.

    Ok(body)
}

async fn save_to_database(data: &str) -> Result<(), Box<dyn std::error::Error>> {
    // placeholder
    Ok(())
}
```

## The `tokio::select!` Pitfall

`select!` is the most common source of accidental cancellation. When one branch completes, **all other branches are dropped**.

### The Classic Bug

```rust
use tokio::sync::mpsc;
use tokio::io::{AsyncReadExt, BufReader};

async fn broken_relay(
    mut rx: mpsc::Receiver<Vec<u8>>,
    mut socket: tokio::net::TcpStream,
) {
    let mut buf = vec![0u8; 4096];

    loop {
        tokio::select! {
            // Branch A: read from socket
            result = socket.read(&mut buf) => {
                match result {
                    Ok(0) => break,
                    Ok(n) => process_bytes(&buf[..n]),
                    Err(e) => {
                        eprintln!("Read error: {e}");
                        break;
                    }
                }
            }
            // Branch B: receive from channel
            msg = rx.recv() => {
                match msg {
                    Some(data) => send_response(&mut socket, &data).await,
                    None => break,
                }
            }
        }
    }
}

fn process_bytes(data: &[u8]) {
    println!("Processing {} bytes", data.len());
}

async fn send_response(socket: &mut tokio::net::TcpStream, data: &[u8]) {
    use tokio::io::AsyncWriteExt;
    let _ = socket.write_all(data).await;
}
```

This looks reasonable, but `socket.read()` is actually cancellation-safe (a single `read` does not do partial buffering). The real danger is when you use `read_exact` or multi-step reads:

### The Actual Bug with read_exact

```rust
use tokio::sync::mpsc;
use tokio::io::AsyncReadExt;

async fn broken_framed_reader(
    mut rx: mpsc::Receiver<Vec<u8>>,
    mut socket: tokio::net::TcpStream,
) {
    loop {
        let mut len_buf = [0u8; 4];

        tokio::select! {
            // BUG: read_exact is cancellation-unsafe!
            // If Branch B wins, and read_exact has read 2 of 4 bytes,
            // those 2 bytes are GONE. Next iteration reads from byte 3,
            // interpreting data bytes as length bytes.
            result = socket.read_exact(&mut len_buf) => {
                let len = u32::from_be_bytes(len_buf) as usize;
                let mut payload = vec![0u8; len];
                socket.read_exact(&mut payload).await.unwrap();
                process_frame(&payload);
            }
            msg = rx.recv() => {
                if msg.is_none() { break; }
            }
        }
    }
}

fn process_frame(data: &[u8]) {
    println!("Frame: {} bytes", data.len());
}
```

### The Fix: Move Cancellation-Unsafe Code Outside select!

```rust
use tokio::sync::mpsc;
use tokio::io::AsyncReadExt;

async fn fixed_framed_reader(
    mut rx: mpsc::Receiver<Vec<u8>>,
    mut socket: tokio::net::TcpStream,
) {
    // Use a cancellation-safe wrapper: read into a persistent buffer
    // that survives across select! iterations.
    let mut read_buf = Vec::with_capacity(4096);

    loop {
        // Only use cancellation-safe operations in select!
        tokio::select! {
            // read() is cancellation-safe: it returns whatever
            // bytes are available, or waits for at least 1 byte.
            result = async {
                let mut tmp = [0u8; 4096];
                let n = socket.read(&mut tmp).await?;
                Ok::<_, std::io::Error>(n)
            } => {
                match result {
                    Ok(0) => break,
                    Ok(n) => {
                        // Accumulate in our persistent buffer
                        // Process complete frames outside select!
                        read_buf.extend_from_slice(&[0u8; 4096][..n]);
                        while let Some(frame) = try_extract_frame(&mut read_buf) {
                            process_frame(&frame);
                        }
                    }
                    Err(e) => {
                        eprintln!("Read error: {e}");
                        break;
                    }
                }
            }
            msg = rx.recv() => {
                if msg.is_none() { break; }
            }
        }
    }
}

fn try_extract_frame(buf: &mut Vec<u8>) -> Option<Vec<u8>> {
    if buf.len() < 4 {
        return None;
    }
    let len = u32::from_be_bytes([buf[0], buf[1], buf[2], buf[3]]) as usize;
    if buf.len() < 4 + len {
        return None;
    }
    let frame = buf[4..4 + len].to_vec();
    buf.drain(..4 + len);
    Some(frame)
}

fn process_frame(data: &[u8]) {
    println!("Frame: {} bytes", data.len());
}
```

## Designing Cancellation-Safe APIs

### Principle 1: Do One Thing Per Await

Each `.await` should perform at most one irrecoverable side effect. If the future is dropped, the caller has either received the result of that effect or it never happened.

```rust
use std::collections::VecDeque;

struct MessageBatch {
    messages: VecDeque<String>,
}

impl MessageBatch {
    // BAD: Multiple side effects across awaits
    async fn process_all_bad(&mut self) -> Result<(), String> {
        while let Some(msg) = self.messages.pop_front() {
            // If dropped here after processing some messages,
            // caller doesn't know which were processed.
            self.send_message(&msg).await?;
        }
        Ok(())
    }

    // GOOD: Process one at a time, let caller control the loop
    async fn process_next(&mut self) -> Option<Result<(), String>> {
        let msg = self.messages.front()?;
        let result = self.send_message(msg).await;
        if result.is_ok() {
            self.messages.pop_front();
        }
        Some(result)
    }

    async fn send_message(&self, msg: &str) -> Result<(), String> {
        // Simulated send
        println!("Sending: {msg}");
        Ok(())
    }
}
```

### Principle 2: Make Progress Visible

Store intermediate state so that if the future is dropped, the next call can resume from where it left off.

```rust
use std::collections::VecDeque;

struct StreamProcessor {
    buffer: VecDeque<u8>,
    state: ProcessState,
}

enum ProcessState {
    ReadingHeader,
    ReadingBody { expected_len: usize },
}

impl StreamProcessor {
    fn new() -> Self {
        Self {
            buffer: VecDeque::new(),
            state: ProcessState::ReadingHeader,
        }
    }

    // Cancellation-safe: all state is stored in self.
    // If dropped, next call resumes from stored state.
    async fn poll_next(&mut self, reader: &mut (impl tokio::io::AsyncRead + Unpin))
        -> Result<Option<Vec<u8>>, std::io::Error>
    {
        use tokio::io::AsyncReadExt;

        let mut tmp = [0u8; 1024];
        let n = reader.read(&mut tmp).await?;
        if n == 0 {
            return Ok(None);
        }
        self.buffer.extend(&tmp[..n]);

        // Try to extract a complete message from the buffer
        loop {
            match &self.state {
                ProcessState::ReadingHeader => {
                    if self.buffer.len() < 4 {
                        return Ok(None); // Need more data
                    }
                    let header: Vec<u8> = self.buffer.drain(..4).collect();
                    let len = u32::from_be_bytes([
                        header[0], header[1], header[2], header[3],
                    ]) as usize;
                    self.state = ProcessState::ReadingBody { expected_len: len };
                }
                ProcessState::ReadingBody { expected_len } => {
                    if self.buffer.len() < *expected_len {
                        return Ok(None); // Need more data
                    }
                    let body: Vec<u8> = self.buffer.drain(..*expected_len).collect();
                    self.state = ProcessState::ReadingHeader;
                    return Ok(Some(body));
                }
            }
        }
    }
}
```

### Principle 3: Use `tokio::pin!` with Fused Futures

When you need to poll a future across multiple `select!` iterations without restarting it:

```rust
use tokio::time::{sleep, Duration};
use tokio::sync::mpsc;

async fn fused_select_example(mut rx: mpsc::Receiver<String>) {
    // Create the future once, pin it, reuse across select! calls
    let timeout = sleep(Duration::from_secs(30));
    tokio::pin!(timeout);

    loop {
        tokio::select! {
            // The timeout future is NOT recreated each iteration.
            // It counts down continuously across all loop iterations.
            _ = &mut timeout => {
                println!("Global timeout reached");
                break;
            }
            msg = rx.recv() => {
                match msg {
                    Some(m) => println!("Got: {m}"),
                    None => break,
                }
            }
        }
    }
}
```

Without `tokio::pin!` and `&mut timeout`, you would create a new 30-second timer on each loop iteration, effectively never timing out.

## The CancellationToken Pattern

`tokio_util::sync::CancellationToken` provides a structured way to propagate cancellation through a tree of tasks.

```toml
# Cargo.toml
[dependencies]
tokio = { version = "1", features = ["full"] }
tokio-util = "0.7"
```

```rust
use tokio_util::sync::CancellationToken;
use tokio::time::{sleep, Duration};

struct WorkerPool {
    token: CancellationToken,
    handles: Vec<tokio::task::JoinHandle<()>>,
}

impl WorkerPool {
    fn new() -> Self {
        Self {
            token: CancellationToken::new(),
            handles: Vec::new(),
        }
    }

    fn spawn_worker(&mut self, id: usize) {
        // Each worker gets a child token.
        // Cancelling the parent cancels all children.
        let token = self.token.child_token();

        let handle = tokio::spawn(async move {
            // Use select! with the token to make the worker
            // respond to cancellation at every await point.
            tokio::select! {
                _ = token.cancelled() => {
                    println!("Worker {id}: cancelled, cleaning up");
                }
                _ = worker_loop(id) => {
                    println!("Worker {id}: finished naturally");
                }
            }
        });

        self.handles.push(handle);
    }

    async fn shutdown(self) {
        println!("Initiating shutdown...");

        // Signal all workers to stop
        self.token.cancel();

        // Wait for all workers to finish cleanup
        for handle in self.handles {
            let _ = handle.await;
        }

        println!("All workers shut down.");
    }
}

async fn worker_loop(id: usize) {
    loop {
        println!("Worker {id}: doing work");
        sleep(Duration::from_secs(1)).await;
    }
}

#[tokio::main]
async fn main() {
    let mut pool = WorkerPool::new();

    pool.spawn_worker(1);
    pool.spawn_worker(2);
    pool.spawn_worker(3);

    // Let workers run for a bit
    sleep(Duration::from_secs(3)).await;

    // Graceful shutdown
    pool.shutdown().await;
}
```

### Hierarchical Cancellation

Child tokens allow fine-grained cancellation control:

```rust
use tokio_util::sync::CancellationToken;
use tokio::time::{sleep, Duration};

async fn hierarchical_cancellation() {
    let root = CancellationToken::new();

    // Service-level tokens
    let http_token = root.child_token();
    let grpc_token = root.child_token();

    // Per-request tokens (children of service tokens)
    let request_1 = http_token.child_token();
    let request_2 = http_token.child_token();

    // Cancel a single request -- other requests unaffected
    request_1.cancel();
    assert!(request_1.is_cancelled());
    assert!(!request_2.is_cancelled());
    assert!(!http_token.is_cancelled());

    // Cancel the HTTP service -- all HTTP requests cancelled,
    // gRPC unaffected
    http_token.cancel();
    assert!(request_2.is_cancelled());
    assert!(!grpc_token.is_cancelled());

    // Cancel root -- everything cancelled
    root.cancel();
    assert!(grpc_token.is_cancelled());
}
```

## Graceful Shutdown

A production async application needs to handle shutdown signals, drain in-flight work, and release resources in order.

```rust
use tokio::sync::mpsc;
use tokio::time::{sleep, Duration, timeout};
use tokio_util::sync::CancellationToken;
use std::sync::Arc;
use std::sync::atomic::{AtomicUsize, Ordering};

struct Server {
    token: CancellationToken,
    in_flight: Arc<AtomicUsize>,
}

impl Server {
    fn new() -> Self {
        Self {
            token: CancellationToken::new(),
            in_flight: Arc::new(AtomicUsize::new(0)),
        }
    }

    async fn handle_request(&self, request_id: usize) {
        let _guard = InFlightGuard::new(self.in_flight.clone());
        let token = self.token.child_token();

        tokio::select! {
            _ = token.cancelled() => {
                println!("Request {request_id}: cancelled during processing");
            }
            _ = async {
                println!("Request {request_id}: processing");
                sleep(Duration::from_secs(2)).await;
                println!("Request {request_id}: done");
            } => {}
        }
    }

    async fn graceful_shutdown(&self, drain_timeout: Duration) {
        println!("Shutdown: stopping new request acceptance");

        // Signal all tasks to stop
        self.token.cancel();

        // Wait for in-flight requests to complete, with a timeout
        let drain_result = timeout(drain_timeout, async {
            while self.in_flight.load(Ordering::Relaxed) > 0 {
                sleep(Duration::from_millis(100)).await;
            }
        }).await;

        match drain_result {
            Ok(_) => println!("Shutdown: all requests drained cleanly"),
            Err(_) => {
                let remaining = self.in_flight.load(Ordering::Relaxed);
                println!("Shutdown: timed out with {remaining} requests in flight");
            }
        }
    }
}

struct InFlightGuard {
    counter: Arc<AtomicUsize>,
}

impl InFlightGuard {
    fn new(counter: Arc<AtomicUsize>) -> Self {
        counter.fetch_add(1, Ordering::Relaxed);
        Self { counter }
    }
}

impl Drop for InFlightGuard {
    fn drop(&mut self) {
        self.counter.fetch_sub(1, Ordering::Relaxed);
    }
}
```

## Drop Guards for Async Cleanup

Since `Drop` is synchronous, you cannot run async cleanup code in a destructor. A common pattern is to use a channel-based guard:

```rust
use tokio::sync::oneshot;

struct AsyncDropGuard {
    _sender: oneshot::Sender<()>,
}

impl AsyncDropGuard {
    fn new(cleanup: impl std::future::Future<Output = ()> + Send + 'static) -> Self {
        let (tx, rx) = oneshot::channel();

        tokio::spawn(async move {
            // Wait for the guard to be dropped (sender dropped = channel closed)
            let _ = rx.await;
            // Run async cleanup
            cleanup.await;
        });

        Self { _sender: tx }
    }
}

async fn example_with_async_drop() {
    let _guard = AsyncDropGuard::new(async {
        println!("Async cleanup running!");
        tokio::time::sleep(std::time::Duration::from_millis(100)).await;
        println!("Async cleanup done!");
    });

    // Do work...
    println!("Working...");

    // When _guard is dropped (here, at end of scope),
    // the cleanup future runs.
}
```

**Trade-off**: This pattern spawns a background task, so cleanup is not guaranteed to complete before the program exits. For critical cleanup, combine with graceful shutdown.

## Common Cancellation Pitfalls and Fixes

### Pitfall 1: Mutex Guards Across Await Points

```rust
use tokio::sync::Mutex;
use std::sync::Arc;

struct SharedState {
    data: Vec<String>,
}

// BAD: holding a mutex guard across an await point
async fn bad_update(state: Arc<Mutex<SharedState>>) {
    let mut guard = state.lock().await;
    // If this future is cancelled HERE, the mutex is released
    // (MutexGuard's Drop runs), but the data is in an
    // inconsistent state.
    let value = fetch_value().await; // <-- cancel point!
    guard.data.push(value);
}

// GOOD: minimize the critical section
async fn good_update(state: Arc<Mutex<SharedState>>) {
    // Do async work FIRST
    let value = fetch_value().await;

    // Then take the lock briefly, no await inside
    let mut guard = state.lock().await;
    guard.data.push(value);
    // Guard dropped immediately
}

async fn fetch_value() -> String {
    "hello".to_string()
}
```

### Pitfall 2: Partial Writes in select!

```rust
use tokio::io::AsyncWriteExt;
use tokio::net::TcpStream;

// BAD: write_all inside select! can be cancelled mid-write
async fn bad_writer(
    socket: &mut TcpStream,
    mut shutdown: tokio::sync::watch::Receiver<bool>,
) {
    let data = vec![0u8; 1_000_000]; // 1MB

    tokio::select! {
        result = socket.write_all(&data) => {
            // write_all is cancellation-unsafe:
            // if shutdown fires mid-write, some bytes are sent
            // but we don't know how many.
            println!("Write result: {:?}", result);
        }
        _ = shutdown.changed() => {
            println!("Shutdown during write");
        }
    }
}

// GOOD: track write progress externally
async fn good_writer(
    socket: &mut TcpStream,
    mut shutdown: tokio::sync::watch::Receiver<bool>,
) {
    let data = vec![0u8; 1_000_000];
    let mut written = 0;

    while written < data.len() {
        let chunk_end = (written + 8192).min(data.len());
        let chunk = &data[written..chunk_end];

        tokio::select! {
            result = socket.write(chunk) => {
                match result {
                    Ok(n) => written += n,
                    Err(e) => {
                        eprintln!("Write error: {e}");
                        break;
                    }
                }
            }
            _ = shutdown.changed() => {
                println!("Shutdown after writing {written} bytes");
                break;
            }
        }
    }
}
```

### Pitfall 3: Channel Send in Multi-Step Operations

```rust
use tokio::sync::mpsc;

struct Processor {
    tx: mpsc::Sender<ProcessedItem>,
}

struct ProcessedItem {
    id: u64,
    data: String,
}

impl Processor {
    // BAD: fetch + send as one future
    // If cancelled between fetch and send, the fetched data is lost
    async fn process_bad(&self, id: u64) -> Result<(), Box<dyn std::error::Error>> {
        let raw = fetch_from_api(id).await?;
        let processed = transform(raw);
        self.tx.send(processed).await.map_err(|e| e.to_string())?;
        Ok(())
    }

    // GOOD: return the result, let the caller decide when to send
    async fn process_good(&self, id: u64) -> Result<ProcessedItem, Box<dyn std::error::Error>> {
        let raw = fetch_from_api(id).await?;
        Ok(transform(raw))
    }
}

async fn fetch_from_api(id: u64) -> Result<String, Box<dyn std::error::Error>> {
    Ok(format!("data-{id}"))
}

fn transform(raw: String) -> ProcessedItem {
    ProcessedItem {
        id: 0,
        data: raw.to_uppercase(),
    }
}
```

## Testing for Cancellation Safety

You can write tests that explicitly cancel futures at different points:

```rust
#[cfg(test)]
mod tests {
    use tokio::sync::mpsc;
    use tokio::time::{sleep, Duration, timeout};

    // A processor that should be cancellation-safe
    struct SafeProcessor {
        buffer: Vec<u8>,
    }

    impl SafeProcessor {
        fn new() -> Self {
            Self { buffer: Vec::new() }
        }

        // Feed data, returns complete messages
        fn feed(&mut self, data: &[u8]) -> Vec<Vec<u8>> {
            self.buffer.extend_from_slice(data);
            let mut messages = Vec::new();

            while self.buffer.len() >= 4 {
                let len = u32::from_be_bytes([
                    self.buffer[0],
                    self.buffer[1],
                    self.buffer[2],
                    self.buffer[3],
                ]) as usize;

                if self.buffer.len() < 4 + len {
                    break;
                }

                let msg = self.buffer[4..4 + len].to_vec();
                self.buffer.drain(..4 + len);
                messages.push(msg);
            }

            messages
        }
    }

    #[tokio::test]
    async fn test_cancellation_preserves_buffer() {
        let mut processor = SafeProcessor::new();

        // Feed partial message (header only, no body)
        let header = 10u32.to_be_bytes();
        processor.feed(&header);

        // Simulate a select! cancellation by timing out
        let result = timeout(Duration::from_millis(10), async {
            // Simulate work that gets cancelled
            sleep(Duration::from_secs(10)).await;
        })
        .await;

        assert!(result.is_err()); // Timed out = cancelled

        // Buffer should still have our partial data
        assert_eq!(processor.buffer.len(), 4);

        // Feed the body
        let body = vec![0u8; 10];
        let messages = processor.feed(&body);

        // Should now get the complete message
        assert_eq!(messages.len(), 1);
        assert_eq!(messages[0].len(), 10);
    }

    #[tokio::test]
    async fn test_channel_recv_cancellation_safe() {
        let (tx, mut rx) = mpsc::channel::<u32>(10);

        // Send a message
        tx.send(42).await.unwrap();

        // Start recv, then cancel it before it completes
        // (it would complete immediately here, so use a different approach)
        tx.send(43).await.unwrap();

        // Create a recv future and drop it
        let recv_future = rx.recv();
        drop(recv_future);

        // Message should still be in the channel
        let msg = rx.recv().await.unwrap();
        assert_eq!(msg, 42); // Not lost!

        let msg = rx.recv().await.unwrap();
        assert_eq!(msg, 43);
    }
}
```

## Trade-Off Analysis

| Approach | Pros | Cons |
|----------|------|------|
| **Avoid select! entirely** | Simplest, no cancellation issues | Cannot react to multiple event sources |
| **Only cancel-safe ops in select!** | Correct, uses select! safely | Requires knowing which ops are safe |
| **CancellationToken** | Structured, hierarchical | Adds dependency, more boilerplate |
| **Persistent buffer / state machine** | Fully cancel-safe by design | More complex code, manual state management |
| **Spawn + JoinHandle::abort()** | Clean cancellation boundary | Spawning has overhead, loses borrowing |
| **AsyncDropGuard** | Async cleanup on drop | Cleanup not guaranteed to finish |

### When Is Cancellation Safety Worth the Complexity?

- **Always worth it**: Data processing pipelines, network protocols, anything involving partial reads/writes.
- **Often worth it**: Request handlers in servers (graceful shutdown), background job processors.
- **Rarely worth it**: Short-lived CLI tools, one-shot scripts, prototypes.

## What You Learned

- **Futures can be dropped at any `.await`**: This is the fundamental async cancellation mechanism in Rust. Unlike Go's goroutines or Java's threads, Rust futures are cancelled by being dropped.
- **Cancellation-safe operations** lose no data on drop; **cancellation-unsafe operations** (like `read_exact`) consume data that is lost if the future is abandoned.
- **`tokio::select!` is the primary source of accidental cancellation**: Every branch that does not win is dropped. Only use cancellation-safe operations directly in select! branches.
- **CancellationToken** provides structured, hierarchical cancellation propagation with child tokens, replacing ad-hoc boolean flags or channel-based signaling.
- **Graceful shutdown** combines signal handling, cancellation tokens, in-flight tracking, and drain timeouts to cleanly stop a production system.
- **State machines with persistent buffers** are the most robust pattern for cancellation-safe stream processing, at the cost of more explicit state management.
- **Testing cancellation** requires deliberately dropping futures mid-execution and verifying that state remains consistent.
