## Learning Objectives

- **Design** a traffic management layer implementing retries, timeouts, and circuit breaking as composable middleware
- **Create** a retry policy engine with retry budgets, exponential backoff with jitter, and retriable status code classification
- **Evaluate** the interaction between retries, timeouts, and circuit breakers to prevent retry storms and cascading failures

## The Challenge

Retries, timeouts, and circuit breakers are the three pillars of resilient service-to-service communication. A service mesh data plane applies these policies transparently, so application code does not need to implement them. But getting these right is deceptively difficult: naive retries can cause retry storms that amplify failures, timeouts must propagate across service boundaries to prevent wasted work, and circuit breakers must transition through states correctly to allow recovery.

You will implement all three as composable middleware that wraps the proxy's request forwarding path. The retry policy must support configurable retry counts, retriable status codes (e.g., 502, 503, 504), retriable request methods (idempotent only by default), exponential backoff with jitter, and a retry budget that limits the percentage of additional requests generated by retries. The timeout system must support per-route timeouts with deadline propagation via the `grpc-timeout` header pattern. The circuit breaker must implement the standard three-state model (closed, open, half-open) with configurable failure thresholds and recovery probes.

The critical design challenge is the interaction between these systems: a retry should not be attempted if the circuit is open, a timeout should cancel pending retries, and the retry budget should prevent retries from overwhelming the system during widespread failures.

## Requirements

1. Implement a `RetryPolicy` with configurable: max retries, retriable status codes, retriable methods, backoff base duration, backoff max duration, and jitter factor
2. Implement exponential backoff: `min(base * 2^attempt, max)` with added random jitter of `[0, backoff * jitter_factor]`
3. Implement a retry budget that tracks the ratio of retries to original requests over a sliding window and stops retrying when the budget is exceeded (e.g., 20% additional traffic from retries)
4. Implement per-route request timeouts that wrap the forwarded request's context with `context.WithTimeout`
5. Implement deadline propagation: if an incoming request carries a deadline (via header), subtract elapsed time and propagate the remaining deadline to the upstream request
6. Implement a circuit breaker with three states: Closed (normal), Open (failing, reject immediately), and Half-Open (allow one probe request to test recovery)
7. The circuit breaker transitions from Closed to Open when the failure rate exceeds a configurable threshold within a sliding window
8. The circuit breaker transitions from Open to Half-Open after a configurable cooldown period, and from Half-Open to Closed on a successful probe or back to Open on a failed probe
9. When the circuit is Open, return 503 Service Unavailable immediately without forwarding the request
10. Compose all three systems: check circuit breaker first, then apply timeout, then execute with retry policy
11. Expose metrics: retry counts per route, circuit breaker state transitions, timeout occurrences, and retry budget utilization

## Hints

- Use `time.After` in a select statement for backoff delays, combined with context cancellation to abort waiting when the parent context is cancelled
- For the retry budget, maintain a ring buffer of timestamped request counts (original + retry) and compute the ratio over the window on each retry decision
- Implement the circuit breaker as a state machine with a mutex-protected state field and a `sync.Once`-guarded timer for the Open-to-HalfOpen transition
- For deadline propagation, parse a custom header (e.g., `X-Request-Deadline` as Unix timestamp) and compute remaining time as `deadline - time.Now()`
- Make each policy a `func(next http.Handler) http.Handler` middleware to enable composition via chaining
- For jitter, use `rand.Float64() * backoff * jitterFactor` added to the computed backoff

## Success Criteria

1. Retries are attempted for configured retriable status codes and methods, up to the max retry count
2. Exponential backoff with jitter produces increasing delays with randomized spread
3. The retry budget stops retries when the additional traffic ratio exceeds the configured threshold
4. Per-route timeouts cancel requests that exceed the configured duration
5. Deadline propagation correctly reduces the timeout based on elapsed time from the incoming deadline
6. The circuit breaker opens after the failure threshold is exceeded and rejects requests immediately
7. The circuit breaker transitions to half-open after the cooldown and recovers on a successful probe
8. All tests pass with the `-race` flag enabled

## Summary

- Retries with exponential backoff and jitter recover from transient failures without amplifying load
- Retry budgets cap the additional traffic from retries to prevent retry storms during widespread outages
- Deadline propagation ensures that retries do not waste work when the original caller has already timed out
- Circuit breakers provide fast failure when backends are down, preventing resource exhaustion from pending requests
- Composing retries, timeouts, and circuit breakers in the correct order prevents pathological interactions
- All three mechanisms work together to provide transparent resilience at the data plane layer
