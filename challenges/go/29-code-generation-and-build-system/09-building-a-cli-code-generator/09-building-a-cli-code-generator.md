# Exercise 29.9: Building a CLI Code Generator

## Prerequisites

Before starting this exercise, you should be comfortable with:

- AST parsing with `go/ast` and `go/parser` (Exercise 29.4)
- Template-based code generation (Exercise 29.5)
- Writing custom code generators (Exercise 29.3)

## Learning Objectives

By the end of this exercise, you will be able to:

1. Build a production-quality CLI code generator that discovers types from source
2. Support multiple generation targets (constructors, validators, JSON methods) via subcommands or flags
3. Handle multi-file and multi-package scanning
4. Produce idiomatic, well-tested generated output with proper error handling

## Why This Matters

Real-world Go projects rely on CLI code generators that combine AST analysis, template rendering, and robust CLI design. Tools like `protoc-gen-go`, `mockgen`, `ent`, and `sqlc` all follow this architecture. Building your own teaches you the patterns that underpin the entire Go tooling ecosystem.

## Challenge

Build a CLI tool called `gogen` that generates boilerplate code from Go struct definitions. The tool must support three generation modes, operate on real source files, and produce correct, formatted Go output.

### Requirements

1. **CLI Interface**: The tool accepts these flags:
   - `-input` -- path to the Go source file to analyze (required)
   - `-output` -- path to the generated output file (default: `<input>_gen.go`)
   - `-type` -- comma-separated list of struct names to process (default: all exported structs)
   - `-mode` -- generation mode: `constructor`, `validator`, or `json` (required)

2. **Constructor mode** (`-mode=constructor`):
   - Generate a `New<Struct>(...)` function for each target struct
   - Parameters: all exported fields in declaration order
   - Returns a pointer to the struct

3. **Validator mode** (`-mode=validator`):
   - Generate a `Validate() error` method for each target struct
   - String fields: check for empty string
   - Numeric fields (`int`, `int64`, `float64`): check for zero value
   - Pointer fields: check for nil
   - Collect all errors into a slice and return a combined error or nil
   - Read a `validate:"required"` struct tag to determine which fields are required

4. **JSON mode** (`-mode=json`):
   - Generate `MarshalJSON` and `UnmarshalJSON` methods
   - Respect `json` struct tags for field naming
   - Skip fields tagged `json:"-"`
   - Handle `omitempty` semantics correctly in marshaling

5. **Output quality**:
   - Every generated file starts with `// Code generated by gogen; DO NOT EDIT.`
   - Output passes `go/format.Source` (i.e., is valid, formatted Go)
   - If formatting fails, dump the raw template output to stderr for debugging

6. **Error handling**:
   - Invalid input path: clear error message and exit code 1
   - Parse errors in source: report filename and position
   - Unknown type name in `-type`: warn on stderr but continue with valid types
   - No matching types found: exit with a clear message

### Success Criteria

- [ ] `gogen -input=models.go -mode=constructor` produces a valid `models_gen.go` with `New<Struct>` functions for all exported structs
- [ ] `gogen -input=models.go -mode=validator -type=User` generates a `Validate()` method only for `User`, checking fields tagged `validate:"required"`
- [ ] `gogen -input=models.go -mode=json -type=Config,Endpoint` generates `MarshalJSON`/`UnmarshalJSON` for both types
- [ ] Generated files compile without errors when placed alongside the source
- [ ] Running `go vet` on the generated output produces no warnings
- [ ] The tool exits with code 1 and a clear message when given a nonexistent input file
- [ ] The tool warns but continues when `-type` includes a name that does not exist in the source

### Test Input

Create `models.go` to test against:

```go
package models

import "time"

type User struct {
	ID        int       `json:"id" validate:"required"`
	Name      string    `json:"name" validate:"required"`
	Email     string    `json:"email" validate:"required"`
	Age       int       `json:"age,omitempty"`
	Bio       string    `json:"bio,omitempty"`
	CreatedAt time.Time `json:"created_at"`
	Manager   *User     `json:"manager,omitempty"`
	password  string    // unexported, should be ignored
}

type Config struct {
	Host    string `json:"host" validate:"required"`
	Port    int    `json:"port" validate:"required"`
	Debug   bool   `json:"debug,omitempty"`
	Timeout int    `json:"timeout"`
}

type Endpoint struct {
	Path   string `json:"path" validate:"required"`
	Method string `json:"method" validate:"required"`
	Auth   bool   `json:"auth,omitempty"`
	Hidden bool   `json:"-"`
}
```

### Suggested Architecture

```
gogen/
  main.go              -- flag parsing, orchestration
  internal/
    parser/
      parser.go        -- AST parsing, struct/field extraction
    generator/
      constructor.go   -- constructor template + logic
      validator.go     -- validator template + logic
      json.go          -- JSON template + logic
      generator.go     -- shared output/formatting logic
    model/
      types.go         -- StructInfo, FieldInfo data types
```

### Key Design Decisions

- Parse struct tags using `reflect.StructTag` on the raw AST tag value (strip backticks first)
- Use a central `model.FieldInfo` type that carries name, Go type string, JSON name, whether it is required, and whether it is a pointer
- Each generator mode is a function that takes `[]model.StructInfo` and returns `[]byte`
- The orchestrator in `main.go` calls parser, filters by `-type`, calls the generator, formats, and writes

## Verify

Run all three modes and check the output compiles:

```bash
go run ./gogen -input=models.go -mode=constructor -output=models_constructor_gen.go
go run ./gogen -input=models.go -mode=validator -output=models_validator_gen.go
go run ./gogen -input=models.go -mode=json -type=Config,Endpoint -output=models_json_gen.go

go vet ./...
go build ./...
```

All commands should succeed with exit code 0 and no warnings.

## What's Next

## Summary

- Production code generators combine AST parsing for discovery with templates for output
- A clean internal architecture (parser / model / generator) keeps each concern isolated
- `reflect.StructTag` bridges AST-level tag strings and structured metadata
- Always format output with `go/format.Source` and include the `DO NOT EDIT` header
- Robust CLI tools handle errors gracefully, warn on non-fatal issues, and exit cleanly
