# Exercise 29.1: go generate Basics

## Prerequisites

Before starting this exercise, you should be comfortable with:

- Go packages and modules
- Writing and using interfaces
- Running command-line tools from the shell

## Learning Objectives

By the end of this exercise, you will be able to:

1. Understand the purpose and mechanics of `//go:generate` directives
2. Write `go:generate` comments that invoke external tools
3. Run `go generate` on packages and entire modules
4. Manage generated files with proper naming conventions

## Why This Matters

Real Go projects frequently use code generation to eliminate boilerplate, produce type-safe code from schemas, and automate repetitive patterns. The `go generate` command is the standard mechanism that ties generation into the Go toolchain. Understanding it is the gateway to all advanced code generation techniques.

## Steps

### Step 1: Create a project with a simple generator

Create a new module and a file that uses `//go:generate` to run a basic command.

```bash
mkdir -p gogen-basics && cd gogen-basics
go mod init gogen-basics
```

Create `version.go`:

```go
package main

//go:generate sh -c "echo 'package main\n\nconst generatedVersion = \"1.0.0-generated\"' > version_gen.go"

import "fmt"

func main() {
	fmt.Println("Version:", generatedVersion)
}
```

Run the generator:

```bash
go generate ./...
```

#### Intermediate Verification

```bash
cat version_gen.go
```

Expected output:

```
package main

const generatedVersion = "1.0.0-generated"
```

```bash
go run .
```

Expected output:

```
Version: 1.0.0-generated
```

---

### Step 2: Use go generate with a Go program as the generator

Create a `gen/` directory with a generator program.

```bash
mkdir -p gen
```

Create `gen/main.go`:

```go
//go:build ignore

package main

import (
	"fmt"
	"os"
	"time"
)

func main() {
	f, err := os.Create("buildinfo_gen.go")
	if err != nil {
		fmt.Fprintf(os.Stderr, "error: %v\n", err)
		os.Exit(1)
	}
	defer f.Close()

	fmt.Fprintf(f, "// Code generated by gen/main.go; DO NOT EDIT.\n\n")
	fmt.Fprintf(f, "package main\n\n")
	fmt.Fprintf(f, "const buildTimestamp = %q\n", time.Now().UTC().Format(time.RFC3339))
}
```

Update `version.go` to add a second generate directive:

```go
package main

//go:generate sh -c "echo 'package main\n\nconst generatedVersion = \"1.0.0-generated\"' > version_gen.go"
//go:generate go run gen/main.go

import "fmt"

func main() {
	fmt.Println("Version:", generatedVersion)
	fmt.Println("Built at:", buildTimestamp)
}
```

Run generation and build:

```bash
go generate ./...
go run .
```

#### Intermediate Verification

```bash
cat buildinfo_gen.go
```

You should see a file containing a `buildTimestamp` constant with the current time. The first line should be the `DO NOT EDIT` comment, which is the standard marker for generated files.

---

### Step 3: Understand go generate environment variables

Create `gen/envdemo.go`:

```go
//go:build ignore

package main

import (
	"fmt"
	"os"
)

func main() {
	fmt.Println("GOPACKAGE:", os.Getenv("GOPACKAGE"))
	fmt.Println("GOFILE:", os.Getenv("GOFILE"))
	fmt.Println("GOLINE:", os.Getenv("GOLINE"))
	fmt.Println("GOARCH:", os.Getenv("GOARCH"))
	fmt.Println("GOOS:", os.Getenv("GOOS"))
}
```

Add a directive in `version.go`:

```go
//go:generate go run gen/envdemo.go
```

```bash
go generate -v ./...
```

#### Intermediate Verification

You should see output showing the environment variables that `go generate` sets for the generator:

- `GOPACKAGE` = `main`
- `GOFILE` = `version.go`
- `GOLINE` = the line number of the directive
- `GOARCH` and `GOOS` = your system's architecture and OS

---

### Step 4: Use the -run flag to selectively run generators

```bash
go generate -run "envdemo" -v ./...
```

#### Intermediate Verification

Only the `envdemo` generator runs. The `-run` flag accepts a regular expression that matches against the generator command. This is useful when a file has multiple `//go:generate` directives and you only want to run one.

## Common Mistakes

1. **Forgetting the `//go:build ignore` tag on generator programs** -- Without it, the generator code becomes part of your package and causes compilation errors.
2. **Missing the space between `//` and `go:generate`** -- The directive must be exactly `//go:generate` with no space after `//`.
3. **Not committing generated files** -- Generated files should typically be committed to version control so consumers can `go get` your package without running generators.
4. **Missing the `DO NOT EDIT` marker** -- The standard comment `// Code generated ... DO NOT EDIT.` tells tools that a file is generated and should not be modified by hand.

---

## Verify

Run all generators and confirm the program compiles and runs:

```bash
rm -f *_gen.go
go generate ./...
go run .
```

You should see version and build timestamp output. Confirm that both `version_gen.go` and `buildinfo_gen.go` exist and contain the `DO NOT EDIT` marker (for `buildinfo_gen.go`).

## Summary

- `//go:generate` directives embed shell commands in Go source files
- `go generate ./...` runs all directives across the module
- Generators receive context via environment variables (`GOPACKAGE`, `GOFILE`, `GOLINE`)
- The `-run` flag filters which generators execute
- Generated files should follow the `_gen.go` suffix convention and include the `DO NOT EDIT` marker
