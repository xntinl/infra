# Ports with Timeout Handling and Backpressure

**Project**: `pdf_renderer` — invoke headless Chromium for PDF generation via a `Port`, with strict per-request timeouts and bounded concurrency.

## Project context

Invoices for a SaaS are generated by rendering HTML to PDF. The service runs `chromium --headless
--print-to-pdf=/dev/stdout` as a subprocess. Each render takes 300–2000ms and may hang indefinitely
if JavaScript on the page does. Hangs cannot propagate into the BEAM — the rendering process must
be killable, observable, and bounded.

`Port.open/2` spawns and monitors an OS subprocess. Unlike a port driver, the subprocess has its
own address space — a crash does not touch the VM. The price is IPC cost (pipes), which for
PDFs in the hundreds of KB range is negligible.

## Why `Port.open` and not `System.cmd`

`System.cmd/3` is blocking: the caller process waits until the subprocess exits. No way to
kill a hung render, no way to stream stdout as it arrives, no monitoring. For an occasional
script this is fine. For an always-on service with per-request timeouts, it is not.

`Port.open/2` gives:
- Async I/O (messages to the owning process: `{port, {:data, bytes}}`, `{port, {:exit_status, n}}`).
- `Port.close/1` and the ability to kill via `:os.cmd("kill -9 ...")` using the OS pid.
- A monitor: if the owning process dies, the VM closes the port — the OS signals SIGPIPE/SIGHUP.

## Why a guard process per render and not a global worker pool

A single worker pool would need internal queueing, timeout handling, and worker resurrection
— all things the BEAM supervision tree does for free if each render is its own supervised
process. Using `Task.Supervisor` + `Task.async_stream` gives us per-job timeouts and
automatic cleanup.

**Why this matters:**
These concepts form the foundation of production Elixir systems. Understanding them deeply allows you to build fault-tolerant, scalable applications that operate correctly under load and failure.

**Real-world use case:**
This pattern appears in systems like:
- Phoenix applications handling thousands of concurrent connections
- Distributed data processing pipelines
- Financial transaction systems requiring consistency and fault tolerance
- Microservices communicating over unreliable networks

**Common pitfall:**
Many developers overlook that Elixir's concurrency model differs fundamentally from threads. Processes are isolated; shared mutable state does not exist. Trying to force shared-memory patterns leads to deadlocks, race conditions, or silently incorrect behavior. Always think in terms of message passing and immutability.
### 1. Port options that matter

- **`:binary`** — stdout arrives as binaries, not charlists. Always set this.
- **`:exit_status`** — send `{port, {:exit_status, n}}` on exit. Needed to detect crashes.
- **`{:packet, 2}` or `{:packet, 4}`** — length-prefixed framing. Useful when the subprocess
  emits multiple discrete responses. For one-shot commands, `stream_to_stdout` is simpler.
- **`:stderr_to_stdout`** — merge stderr. Convenient for logging; avoid if you need to
  distinguish.
- **`:nouse_stdio`** / **`:use_stdio`** — pipes vs 3,4 fds. Default stdio is fine.

### 2. OS pid and forced kill

`Port.info(port, :os_pid)` returns the subprocess pid. If the port refuses to close (the
child ignored SIGTERM), `System.cmd("kill", ["-9", "#{pid}"])` is the escape hatch.

### 3. The zombie problem

When a port closes but the subprocess has forked grandchildren (chromium does this — renderer
processes), those grandchildren can outlive the port. Use `kill -9` on the process group, or
run subprocesses with `setsid` so you can kill the whole group.

### 4. Backpressure on stdin

`Port.command/2` blocks if the subprocess's stdin pipe is full (~64KB on Linux). For large
inputs (HTML can be 1MB+), that is a feature: the BEAM applies backpressure automatically.

## Design decisions

- **Option A — one persistent chromium per render**: spawn, render, kill. Pros: clean
  isolation, no shared state between renders. Cons: 200ms spawn cost per request.
- **Option B — chromium-as-a-service talking DevTools over WebSocket**: pay spawn cost once.
  Pros: fast per-render. Cons: stateful, connection pooling, much harder crash recovery.

→ We pick **Option A** for the exercise. Option B is the next evolution once p99 becomes
dominated by spawn.

- **Option A — timeout via `Task.async_stream` `:timeout`**: kills the task but leaks the
  port unless trapping exits. Cons: subtle, race-prone.
- **Option B — explicit `receive ... after timeout_ms`** in a guard process that owns the
  port: on timeout, `Port.close` + `kill -9` + return `{:error, :timeout}`. Clean shutdown.

→ We use **Option B**. Explicit is safer for process-and-pid hybrid state.

### Step 1: The guard (`lib/pdf_renderer/port_guard.ex`)

**Objective**: Enforce hard timeout with SIGKILL fallback so subprocess hangs never block the caller indefinitely.

```elixir
defmodule PdfRenderer.PortGuard do
  @moduledoc """
  Runs a single external command with:
    - hard per-request timeout
    - stdin push
    - stdout + exit status capture
    - forced SIGKILL of the OS pid on timeout

  Callers receive `{:ok, stdout_binary}` or `{:error, reason}`.
  """

  require Logger

  @spec run(String.t(), [String.t()], iodata() | nil, keyword()) ::
          {:ok, binary()} | {:error, term()}
  def run(cmd, args, stdin \\ nil, opts \\ []) do
    timeout_ms = Keyword.get(opts, :timeout_ms, 5_000)
    max_output = Keyword.get(opts, :max_output_bytes, 50_000_000)

    case System.find_executable(cmd) do
      nil  -> {:error, {:executable_not_found, cmd}}
      path -> do_run(path, args, stdin, timeout_ms, max_output)
    end
  end

  defp do_run(path, args, stdin, timeout_ms, max_output) do
    port_opts = [
      :binary,
      :exit_status,
      :use_stdio,
      :stderr_to_stdout,
      args: args,
      line: false
    ]

    port = Port.open({:spawn_executable, path}, port_opts)
    os_pid = Port.info(port, :os_pid) |> elem(1)

    if stdin, do: Port.command(port, stdin)
    # Closing our end of stdin tells chromium "input is done".
    send_eof(port)

    collect(port, os_pid, <<>>, timeout_ms, max_output)
  end

  defp send_eof(port) do
    # Port.close on Elixir's side would terminate the subprocess.
    # Instead, simulate EOF by sending an empty write and keeping the port open.
    # For commands that need explicit close-stdin, caller must do it externally.
    :ok
  end

  defp collect(port, os_pid, acc, timeout_ms, max_output) do
    receive do
      {^port, {:data, chunk}} ->
        new_acc = acc <> chunk
        if byte_size(new_acc) > max_output do
          kill_hard(port, os_pid)
          {:error, {:output_too_large, byte_size(new_acc)}}
        else
          collect(port, os_pid, new_acc, timeout_ms, max_output)
        end

      {^port, {:exit_status, 0}} ->
        {:ok, acc}

      {^port, {:exit_status, code}} ->
        {:error, {:exit_status, code, acc}}
    after
      timeout_ms ->
        kill_hard(port, os_pid)
        {:error, :timeout}
    end
  end

  defp kill_hard(port, os_pid) do
    # Port.close sends SIGTERM; give no mercy on timeout path.
    try do
      Port.close(port)
    rescue
      ArgumentError -> :ok  # already closed
    end
    System.cmd("kill", ["-9", "#{os_pid}"], stderr_to_stdout: true)
    :ok
  end
end
```

### Step 2: The renderer (`lib/pdf_renderer/renderer.ex`)

**Objective**: Wrap headless chromium invocation so HTML-to-PDF conversion stays timeout-protected and memory-safe.

```elixir
defmodule PdfRenderer.Renderer do
  @moduledoc """
  Renders HTML to PDF by invoking headless chromium via PortGuard.
  Each call is a fresh subprocess — no shared state.
  """

  alias PdfRenderer.PortGuard

  @chromium System.get_env("CHROMIUM_BIN") || "chromium"

  @spec html_to_pdf(String.t(), keyword()) ::
          {:ok, binary()} | {:error, term()}
  def html_to_pdf(html, opts \\ []) when is_binary(html) do
    timeout_ms = Keyword.get(opts, :timeout_ms, 5_000)

    args = [
      "--headless=new",
      "--disable-gpu",
      "--no-sandbox",
      "--print-to-pdf=/dev/stdout",
      "--virtual-time-budget=10000",
      "data:text/html;base64," <> Base.encode64(html)
    ]

    PortGuard.run(@chromium, args, nil, timeout_ms: timeout_ms)
  end
end
```

### Step 3: Supervision

**Objective**: Boot TaskSupervisor so port guard tasks run isolated from the caller and crash-safe.

```elixir
defmodule PdfRenderer.Application do
  use Application

  @impl true
  def start(_, _) do
    children = [
      {Task.Supervisor, name: PdfRenderer.TaskSup}
    ]
    Supervisor.start_link(children, strategy: :one_for_one, name: PdfRenderer.Supervisor)
  end
end
```

## Why this works

```
caller ──Task.Supervisor──▶ task process
                                  │
                                  ▼
                         PortGuard.run(...)
                                  │
                       Port.open(chromium)
                                  │
              ┌──── receive {:data, _}  ────┐
              │                              │
              │      (accumulates, honors)   │
              │       max_output bytes       │
              │                              │
              └──── receive {:exit_status, ──┘
                                  │
                     timeout? ─── yes ──▶ Port.close + kill -9 os_pid
                       │
                      no
                       ▼
                   {:ok, pdf_bytes}
```

- Timeout is enforced in a dedicated `receive ... after`: the guard cannot miss it.
- `kill -9` on the OS pid is the safety net for chromium processes that ignore SIGTERM.
- The guard runs inside a short-lived task supervised by `Task.Supervisor`, so a crash
  in the guard itself does not affect the caller's tree.
- Memory cap (`max_output_bytes`) prevents a misbehaving subprocess from flooding the
  BEAM heap.

```elixir
defmodule PdfRenderer.RendererTest do
  use ExUnit.Case, async: true
  alias PdfRenderer.PortGuard

  describe "PortGuard.run/4 — happy path" do
    test "captures stdout of a short-lived command" do
      {:ok, out} = PortGuard.run("printf", ["hello"], nil, timeout_ms: 1_000)
      assert out == "hello"
    end

    test "captures exit code of a failing command" do
      assert {:error, {:exit_status, code, _}} =
               PortGuard.run("sh", ["-c", "exit 42"], nil, timeout_ms: 1_000)
      assert code == 42
    end
  end

  describe "PortGuard.run/4 — timeout handling" do
    test "kills a hung subprocess after timeout" do
      start_ms = System.monotonic_time(:millisecond)
      assert {:error, :timeout} =
               PortGuard.run("sleep", ["30"], nil, timeout_ms: 300)
      elapsed = System.monotonic_time(:millisecond) - start_ms
      # Should return close to timeout_ms, nowhere near 30s.
      assert elapsed < 1_500
    end
  end

  describe "PortGuard.run/4 — output limits" do
    test "refuses output larger than max_output_bytes" do
      # yes produces infinite "y\n".
      assert {:error, {:output_too_large, _}} =
               PortGuard.run("yes", [], nil, timeout_ms: 2_000, max_output_bytes: 1_000)
    end
  end

  describe "PortGuard.run/4 — executable not found" do
    test "returns descriptive error" do
      assert {:error, {:executable_not_found, "nope-not-a-command-xyz"}} =
               PortGuard.run("nope-not-a-command-xyz", [], nil)
    end
  end
end
```

## Advanced Considerations: NIF Isolation and Scheduler Integration

NIF calls run atomically on a scheduler thread, blocking all other processes on that scheduler until the function returns. For operations exceeding ~1 millisecond, this starvation becomes visible: heartbeat processes delay, ETS owner replies hang, supervision timeouts fire. The BEAM's dirty scheduler pool (8 CPU + 10 IO by default) isolates long NIFs from the main scheduler ring, but they're still a finite resource.

Understanding scheduler capacity is critical. Each dirty CPU scheduler can run ~1,000 100-microsecond operations per second, or ~5 100-millisecond operations. Beyond that, callers queue. A GenServer pool capping concurrency and applying backpressure prevents cascade failures: if the dirty pool saturates, reject new work immediately instead of queuing unboundedly.

Resource management inside NIFs differs from pure Elixir. A `Binary<'a>` is a borrow tied to the NIF call; it cannot escape to threads or be stored in resources. An `OwnedBinary` allocation isn't visible to BEAM's garbage collector, so memory limits must be enforced in the Elixir layer. Hybrid architectures (Port processes for I/O, NIFs for CPU work) offer better observability and failure isolation than trying to do everything in a single NIF crate.

## Deep Dive: Interop Patterns and Production Implications

Interop with native code (NIFs, ports, C extensions) introduces failure modes that pure Elixir code doesn't have: segfaults, memory leaks, deadlocks with the Erlang emulator. Testing interop requires separate test suites for the native layer and integration tests that exercise the boundary.

---

## Trade-offs and production gotchas

**1. Orphan processes on BEAM crash.** If the BEAM is SIGKILLed, children stay alive (ports
close but subprocesses are not killed). Use `setsid` + `trap exit` wrappers, or `epmd`-style
monitor scripts. On Linux, `prctl(PR_SET_PDEATHSIG, SIGKILL)` in a wrapper binary is the
cleanest fix.

**2. `System.cmd` vs Port.open for one-shots.** If you do not need a timeout and output is
bounded, `System.cmd/3` is simpler. Reach for `Port.open` when you need any of: timeout,
streaming, kill-on-owner-death.

**3. stderr_to_stdout hides failures.** If you merge stderr into stdout, your PDF binary
contains log lines. Keep streams separate in production and log stderr explicitly.

**4. Pipe buffering gotcha.** Subprocesses using libc stdout see full-buffering on pipes.
Your chromium may have 4KB sitting in its buffer on exit — check that logs flush before exit,
or set `stdbuf -oL` if you need line buffering.

**5. `max_output_bytes` is memory protection, not throttling.** It kills the job past the
limit. For streaming workloads, parse-and-discard in the receive loop, do not accumulate.

**6. When NOT to use a Port.** If you need sub-millisecond latency and are willing to trade
isolation for speed, a NIF or driver is better. Ports are the right tool when the subprocess
is expensive, opaque, or untrusted.
