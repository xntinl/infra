# Behaviour-Based Mocking with Mox (Deep Dive)

**Project**: `mox_testing_deep` — a payments processor that talks to an external gateway.
---

## Project context

You're the tech lead of a payments service that integrates with a third-party card processor
(think Stripe / Adyen). Every call costs real money, has rate limits, and cannot be exercised
from unit tests. The team started with `:meck` and quickly fell into the classic trap: brittle
tests, runtime redefinitions of modules, and flakiness when `async: true` tests collided on
the same global function.

The solution is to introduce [Mox](https://hexdocs.pm/mox) — a behaviour-based mocking library
by José Valim that enforces contracts at compile time and supports concurrent tests via a
per-test expectation allowance model. The challenge is not "how do I use Mox" (that's a
5-line tutorial), it's how to structure the application so Mox expectations live only in
tests and the real HTTP client lives only in production, with zero runtime magic.

This exercise covers **private mocks**, **dispatch mode** (global vs private), **`stub_with`
for wiring shared stubs**, and how to allow expectations to cross process boundaries
(`Mox.allow/3`) — the single most common production-grade Mox pitfall.

Project structure:

```
mox_testing_deep/
├── lib/
│   └── payments/
│       ├── application.ex
│       ├── gateway.ex              # behaviour + runtime dispatch
│       ├── gateway/

│       │   └── in_memory.ex        # dev/staging stub with explicit :ok/:error
│       ├── charge.ex               # domain logic, depends on Gateway behaviour
│       └── worker.ex               # Task that invokes Gateway from another process
├── test/

│   └── payments/

│       └── worker_test.exs
└── mix.exs
```

## Why this approach and not alternatives

Alternatives considered and discarded:

- **Hand-rolled equivalent**: reinvents primitives the BEAM/ecosystem already provides; high risk of subtle bugs around concurrency, timeouts, or failure propagation.
- **External service (e.g. Redis, sidecar)**: adds a network hop and an extra failure domain for a problem the VM can solve in-process with lower latency.
- **Heavier framework abstraction**: couples the module to a framework lifecycle and makes local reasoning/testing harder.

The chosen approach stays inside the BEAM, uses idiomatic OTP primitives, and keeps the contract small.

## Core concepts

### 1. Behaviours are the contract, Mox enforces it

Mox does NOT mock arbitrary modules. It generates a module that implements a **behaviour**.
If you add a function to the behaviour and the test mock doesn't expect it, Mox warns you.
If you expect a function that isn't in the behaviour, Mox rejects it at test compile time.

```
+-----------------------+           +-----------------------+
|  Payments.Gateway     |  behaves  |  Payments.GatewayMock |
|  (behaviour/callback) |<--------->|  (generated by Mox)   |
+-----------------------+           +-----------------------+
           ^                                     ^
           |                                     |
     implements                            expect(:charge)
           |                                     |
+-----------------------+           +-----------------------+
| HttpClient (prod)     |           |   charge_test.exs     |
| InMemory (dev/staging)|           +-----------------------+
+-----------------------+
```

This is dependency inversion: `Charge` depends on the `Gateway` behaviour (abstract), not
on a concrete module. Production wires `HttpClient`, tests wire the mock.

### 2. Runtime dispatch via `Application.get_env/3`

The cleanest way to swap implementations between environments is a dispatcher module that
reads the implementation from config at runtime:

```elixir
defmodule Payments.Gateway do
  @callback charge(amount :: integer, token :: String.t()) ::
              {:ok, String.t()} | {:error, atom()}

  def charge(amount, token), do: impl().charge(amount, token)
  defp impl, do: Application.get_env(:payments, :gateway_impl, Payments.Gateway.HttpClient)
end
```

Using a module attribute (`@impl Application.compile_env(...)`) is faster but re-compiles
are required if config changes. In tests we want zero recompile overhead, so use
`Application.get_env/3` (runtime) — the micro-overhead is invisible in practice.

### 3. Private mocks vs global mocks

Mox has two dispatch modes:

| Mode | Scope | `async: true`? | Use case |
|------|-------|----------------|----------|
| `:private` (default) | Only the current test process + allowed processes | yes | 99% of tests |
| `:global` | All processes in the VM | no (use `async: false`) | When allowance is impossible |

Set via `Mox.set_mox_private/1` or `Mox.set_mox_global/1` in `setup`. Mixing modes across
tests is fine as long as `async: true` tests are all `:private`.

### 4. Expectation allowance across processes

By default, only the test process can consume expectations. If your code under test
spawns a Task or sends a message to a GenServer, that other process CANNOT see the
expectations — it will get `(Mox.UnexpectedCallError)`.

Solution: `Mox.allow(Mock, owner_pid, allowed_pid)`. Or use `:global` mode and pay the
`async: false` cost.

```
Test process (owner)
     |
     |  expect(Mock, :charge, fn ... end)
     v
+-------------+   allow(Mock, self(), task_pid)   +-----------+
| Mock state  |<------------------------------------| Task pid  |
+-------------+                                     +-----------+
```

### 5. `stub_with/2` for wiring a pre-made stub

When the same test suite needs dozens of mock calls with identical behaviour, defining
expectations per test becomes noisy. Instead:

```elixir
Mox.stub_with(Payments.GatewayMock, Payments.Gateway.InMemory)
```

Every function not explicitly `expect`ed falls through to `InMemory`. Tests become
declarative: "I expect this specific interaction; for everything else, behave like the
in-memory fake".

## Design decisions

**Option A — naive/simple approach**
- Pros: minimal code, easy to reason about.
- Cons: breaks under load, lacks observability, hard to evolve.

**Option B — the approach used here** (chosen)
- Pros: production-grade, handles edge cases, testable boundaries.
- Cons: more moving parts, requires understanding of the BEAM primitives involved.

→ Chose **B** because correctness under concurrency and failure modes outweighs the extra surface area.

### Step 2: Behaviour and dispatcher

```elixir
# lib/payments/gateway.ex
defmodule Payments.Gateway do
  @moduledoc """
  Contract for card-processor integrations.

  Production wiring: `Payments.Gateway.HttpClient` (Finch + JSON).
  Test wiring: `Payments.GatewayMock` (generated by Mox in test/support).
  """

  @type amount :: pos_integer()
  @type token :: String.t()
  @type auth_code :: String.t()
  @type reason :: :declined | :network | :rate_limited | :invalid_token

  @callback charge(amount, token) :: {:ok, auth_code} | {:error, reason}
  @callback refund(auth_code) :: :ok | {:error, reason}

  @spec charge(amount, token) :: {:ok, auth_code} | {:error, reason}
  def charge(amount, token), do: impl().charge(amount, token)

  @spec refund(auth_code) :: :ok | {:error, reason}
  def refund(auth_code), do: impl().refund(auth_code)

  defp impl, do: Application.fetch_env!(:payments, :gateway_impl)
end
```

### Step 3: Production implementation

```elixir
# lib/payments/gateway/http_client.ex
defmodule Payments.Gateway.HttpClient do
  @behaviour Payments.Gateway

  @base_url "https://api.example-processor.com"

  @impl true
  def charge(amount, token) do
    body = Jason.encode!(%{amount: amount, token: token})
    req = Finch.build(:post, @base_url <> "/charge", headers(), body)

    case Finch.request(req, Payments.Finch, receive_timeout: 5_000) do
      {:ok, %Finch.Response{status: 200, body: b}} ->
        %{"auth_code" => code} = Jason.decode!(b)
        {:ok, code}

      {:ok, %Finch.Response{status: 402}} -> {:error, :declined}
      {:ok, %Finch.Response{status: 429}} -> {:error, :rate_limited}
      {:ok, %Finch.Response{status: 422}} -> {:error, :invalid_token}
      {:error, _} -> {:error, :network}
    end
  end

  @impl true
  def refund(auth_code) do
    req = Finch.build(:post, @base_url <> "/refund/" <> auth_code, headers(), "")
    case Finch.request(req, Payments.Finch, receive_timeout: 5_000) do
      {:ok, %Finch.Response{status: 200}} -> :ok
      {:ok, %Finch.Response{status: 429}} -> {:error, :rate_limited}
      {:error, _} -> {:error, :network}
    end
  end

  defp headers do
    [{"content-type", "application/json"}, {"authorization", "Bearer " <> token()}]
  end

  defp token, do: Application.fetch_env!(:payments, :api_token)
end
```

### Step 4: Dev/staging stub (used with `stub_with`)

```elixir
# test/support/in_memory_gateway.ex
defmodule Payments.Gateway.InMemory do
  @moduledoc "Deterministic fake. Pattern on token prefix to force outcomes."
  @behaviour Payments.Gateway

  @impl true
  def charge(_, "decline_" <> _), do: {:error, :declined}
  def charge(_, "net_" <> _), do: {:error, :network}
  def charge(_, "rl_" <> _), do: {:error, :rate_limited}
  def charge(_, token), do: {:ok, "auth_" <> token}

  @impl true
  def refund("auth_" <> _ = _code), do: :ok
  def refund(_), do: {:error, :invalid_token}
end
```

### Step 5: Domain logic

```elixir
# lib/payments/charge.ex
defmodule Payments.Charge do
  @moduledoc "Business rules wrapped around Payments.Gateway."

  alias Payments.Gateway

  @spec run(pos_integer(), String.t(), keyword()) ::
          {:ok, String.t()} | {:error, term()}
  def run(amount, token, opts \\ []) do
    max_retries = Keyword.get(opts, :retries, 2)
    attempt(amount, token, max_retries)
  end

  defp attempt(amount, token, retries_left) do
    case Gateway.charge(amount, token) do
      {:ok, code} -> {:ok, code}
      {:error, :rate_limited} when retries_left > 0 ->
        Process.sleep(10)
        attempt(amount, token, retries_left - 1)
      {:error, :network} when retries_left > 0 ->
        Process.sleep(10)
        attempt(amount, token, retries_left - 1)
      {:error, reason} -> {:error, reason}
    end
  end
end
```

### Step 6: Worker (spawns tasks — triggers allowance concerns)

```elixir
# lib/payments/worker.ex
defmodule Payments.Worker do
  @moduledoc "Runs charges in parallel tasks."

  def batch_charge(items) when is_list(items) do
    items
    |> Task.async_stream(
      fn {amount, token} -> Payments.Charge.run(amount, token) end,
      max_concurrency: 8,
      timeout: 10_000
    )
    |> Enum.map(fn {:ok, r} -> r end)
  end
end
```

### Step 9: Charge tests (private mode, async)

```elixir
# test/payments/charge_test.exs
defmodule Payments.ChargeTest do
  use ExUnit.Case, async: true
  import Mox

  setup :verify_on_exit!
  setup :set_mox_from_context

  describe "run/3 — happy path" do
    test "returns auth code on first try" do
      expect(Payments.GatewayMock, :charge, fn 100, "tok_ok" ->
        {:ok, "auth_123"}
      end)

      assert {:ok, "auth_123"} = Payments.Charge.run(100, "tok_ok")
    end
  end

  describe "run/3 — retry behaviour" do
    test "retries on :rate_limited up to the configured max" do
      Payments.GatewayMock
      |> expect(:charge, fn _, _ -> {:error, :rate_limited} end)
      |> expect(:charge, fn _, _ -> {:error, :rate_limited} end)
      |> expect(:charge, fn _, _ -> {:ok, "auth_retry"} end)

      assert {:ok, "auth_retry"} = Payments.Charge.run(50, "tok_rl", retries: 2)
    end

    test "propagates :declined without retrying" do
      expect(Payments.GatewayMock, :charge, 1, fn _, _ -> {:error, :declined} end)
      assert {:error, :declined} = Payments.Charge.run(50, "tok_dec")
    end
  end

  describe "stub_with/2" do
    test "falls through to InMemory for unexpected calls" do
      stub_with(Payments.GatewayMock, Payments.Gateway.InMemory)
      assert {:ok, "auth_foo"} = Payments.Charge.run(10, "foo")
      assert {:error, :declined} = Payments.Charge.run(10, "decline_x")
    end
  end
end
```

### Step 10: Worker tests (cross-process allowance)

```elixir
# test/payments/worker_test.exs
defmodule Payments.WorkerTest do
  use ExUnit.Case, async: true
  import Mox

  setup :verify_on_exit!

  test "expect across processes — explicit allow" do
    parent = self()

    expect(Payments.GatewayMock, :charge, fn _, _ -> {:ok, "auth_async"} end)

    {:ok, task_sup} = Task.Supervisor.start_link()
    task =
      Task.Supervisor.async(task_sup, fn ->
        Payments.Charge.run(10, "tok_async")
      end)

    # Without this line, the test FAILS with UnexpectedCallError
    # because task.pid is not the owner and not allowed.
    allow(Payments.GatewayMock, parent, task.pid)

    assert {:ok, "auth_async"} = Task.await(task)
  end

  test "stub_with works across tasks without explicit allow because stubs are looked up via $callers" do
    stub_with(Payments.GatewayMock, Payments.Gateway.InMemory)
    items = for i <- 1..20, do: {100, "tok_#{i}"}
    results = Payments.Worker.batch_charge(items)
    assert length(results) == 20
  end
end
```

### Why this works

The design leans on BEAM guarantees (process isolation, mailbox ordering, supervisor restarts) and pushes invariants to the boundaries of each module. State transitions are explicit, failure modes are declared rather than implicit, and each step is independently testable. That combination keeps the implementation correct under concurrent load and cheap to change later.

## Trade-offs and production gotchas

**1. Behaviour-first cost**
You must define a behaviour with `@callback`. Mox rejects mocking modules without behaviours
by design. If you're mocking a library that doesn't expose one, wrap it and behaviour the
wrapper (adapter pattern).

**2. `verify_on_exit!` vs `verify!`**
Always use `verify_on_exit!` in `setup`. It fires on test completion and fails the test if
any `expect/4` counts were not met. Without it, a test that expects 3 calls but gets 0 still
passes silently.

**3. Private mode requires explicit allowance for spawned processes**
If code under test spawns via `Task`, `GenServer`, `Oban`, or Broadway, you must `Mox.allow/3`
each spawned pid from the test process — unless Mox can follow `$callers` (works for
`Task.async_stream` via `Task.Supervisor` when you use `stub`). Missing allowance is the #1
source of "works locally, fails in CI" flakiness. Alternative: `set_mox_global` + `async: false`.

**4. `stub/3` never fails the test**
`stub/3` is "always allow". `expect/4` asserts the call happened (with `verify_on_exit!`).
Mixing them is legal but confusing. Rule of thumb: `expect` for behaviour you're asserting,
`stub` for irrelevant dependencies.

**5. `Application.get_env` at runtime vs `compile_env`**
`compile_env` is faster but bakes the value at compile time. If a developer changes
`:gateway_impl` in `config/test.exs` after compile, tests don't pick it up without `mix clean`.
`get_env` avoids this foot-gun at ~50ns cost — invisible except in tight loops.

**6. Don't mock what you own**
Mock at the infrastructure boundary (HTTP, DB, queues). Mocking your own domain couples
tests to implementation detail.

**7. When NOT to use Mox**
- Pure functions with no side effects — just call them.
- Integration tests that must verify real HTTP semantics — use Bypass.
- You don't own the behaviour contract AND it's unstable.
- Code where you spawn dozens of dynamic processes you can't reach — `:global` mode serializes
  the suite; consider a hand-written fake.

**8. Mox + Oban/Broadway**
Background processors run in their own processes with unpredictable lifecycle. Use
`set_mox_global` in their test modules and accept `async: false`. Or pre-spawn them in
`setup` and `allow/3` each one.

## Performance notes

Mox expectation lookup is O(log n) in expectations per mock per process. Typical overhead is
~2–5µs per call. It does not meaningfully affect a test suite — CPU is dominated by the mocked
function's own work.

```elixir
# bench/mox_overhead.exs
Mox.defmock(BenchMock, for: Payments.Gateway)
Mox.stub(BenchMock, :charge, fn _, _ -> {:ok, "x"} end)

Benchee.run(%{
  "stub call" => fn -> BenchMock.charge(1, "t") end,
  "direct module call" => fn -> Payments.Gateway.InMemory.charge(1, "t") end
}, time: 3, warmup: 1)
```

Expected: `stub call` ~3µs, `direct module call` ~200ns. A 15× factor — irrelevant for test
throughput, catastrophic if you were tempted to use Mox in production paths.
