# Hot upgrades — `appup`, `relup`, and why almost nobody uses them

**Project**: `hot_upgrade_intro` — a tiny release that generates an
`appup` and a `relup` to demonstrate the plumbing. It's a guided tour,
not a recommendation: the goal is to know what the feature is, when
it was useful, and why modern Elixir deployments almost always pick
rolling restarts instead.


---

## Project context

Erlang/OTP has a legendary feature: replace code in a running node
without restarting. Ericsson used it to keep telecom switches up for
years. In Elixir, with Mix releases, you can still technically do it —
there's `:appup` (per-app upgrade instructions) and `:relup` (per-
release upgrade plan) — but the ecosystem has largely moved on. Today,
99% of production Elixir deployments use **rolling restarts** behind a
load balancer.

This exercise shows what the hot-upgrade toolchain looks like so you
can recognize it, debug legacy systems, and make an informed choice.

## Why X and not Y

- **Why not blue/green?** Simpler and what most shops pick; hot upgrade is taught because understanding `code_change/3` informs design even when you don't use relups.

## Core concepts

### 1. `.appup` — per-application upgrade plan

```
{"0.2.0",
 [{"0.1.0", [{:load_module, MyMod}]}],
 [{"0.1.0", [{:load_module, MyMod}]}]}.
```

Lists the instructions to go from `0.1.0` to `0.2.0` and back:
load modules, update GenServers via `code_change/3`, start/stop
workers. One per OTP application.

### 2. `.relup` — per-release upgrade plan

Generated by Erlang's `systools` from the `.appup` files of every
app in the release. It sequences the per-app steps into a single
upgrade transaction.

### 3. `code_change/3` — GenServer state migration

When a running GenServer's module is replaced, OTP calls
`code_change(old_vsn, state, extra)` so you can reshape state.
Writing it correctly is hard — it runs inline in the live server
and must never crash or the upgrade aborts.

### 4. Why rolling restarts won

```
Upgrade strategy       Complexity  Rollback    Typical use today
─────────────────────  ──────────  ──────────  ─────────────────
hot upgrade (appup)    very high   fragile     telecom, embedded
rolling restart (k8s)  low         trivial     almost everything
blue/green             medium      trivial     large web services
```

Rolling restarts work at the *process orchestrator* level (Kubernetes,
systemd, ECS). They don't require code changes, they handle config
changes trivially, and if a new version crashes, the orchestrator
rolls back by redeploying the old image. Hot upgrades require
discipline at the *code* level (every state change must have a
`code_change/3`) and a bug in the upgrade code can corrupt live
state.

---

## Design decisions

**Option A — blue/green redeploy (simpler)**
- Pros: simpler upfront, fewer moving parts.
- Cons: hides the trade-off that this exercise exists to teach.

**Option B — relup hot upgrade (chosen here for learning)**
- Pros: explicit about the semantic that matters in production.
- Cons: one more concept to internalize.

→ Chose **B** because understanding `code_change/3` and appups is foundational even if most shops pick blue/green in practice.

### Step 2: `mix.exs` — enable appup generation

**Objective**: Edit `mix.exs` — enable appup generation, exposing release-time behavior that depends on application env resolution and runtime boot semantics.


```elixir
defmodule HotUpgradeIntro.MixProject do
  use Mix.Project

  def project do
    [
      app: :hot_upgrade_intro,
      version: "0.1.0",
      elixir: "~> 1.17",
      start_permanent: Mix.env() == :prod,
      deps: [],
      releases: releases()
    ]
  end

  def application do
    [extra_applications: [:logger], mod: {HotUpgradeIntro.Application, []}]
  end

  # NOTE: This is a DEMONSTRATION configuration. A real hot-upgrade flow
  # typically uses an external library such as `distillery` (legacy) or
  # a dedicated build script that invokes `systools` to generate the
  # .appup and .relup files. Mix releases DO NOT generate these
  # automatically — that is deliberate.
  defp releases do
    [
      hot_upgrade_intro: [
        include_executables_for: [:unix],
        applications: [hot_upgrade_intro: :permanent]
      ]
    ]
  end
end
```

### Step 3: `lib/hot_upgrade_intro.ex` and `lib/hot_upgrade_intro/application.ex`

**Objective**: Provide `lib/hot_upgrade_intro.ex` and `lib/hot_upgrade_intro/application.ex` — these are the supporting fixtures the main module depends on to make its concept demonstrable.


```elixir
defmodule HotUpgradeIntro.Application do
  @moduledoc false
  use Application

  @impl true
  def start(_type, _args) do
    children = [HotUpgradeIntro.Counter]
    Supervisor.start_link(children, strategy: :one_for_one, name: HotUpgradeIntro.Supervisor)
  end
end
```

```elixir
defmodule HotUpgradeIntro.Counter do
  @moduledoc """
  Version 0.1.0 state: a single integer counter.

  If we shipped 0.2.0 that changes state to `%{value: n, history: [...]}`,
  we'd need a `code_change/3` to transform the old integer state into
  the new map — during the hot upgrade, OTP would call that callback
  on every live instance.
  """

  use GenServer

  @vsn "0.1.0"

  def start_link(_), do: GenServer.start_link(__MODULE__, 0, name: __MODULE__)
  def value, do: GenServer.call(__MODULE__, :value)
  def bump, do: GenServer.call(__MODULE__, :bump)

  @impl true
  def init(n), do: {:ok, n}

  @impl true
  def handle_call(:value, _f, n), do: {:reply, n, n}
  def handle_call(:bump, _f, n), do: {:reply, n + 1, n + 1}

  @doc """
  `code_change/3` is OTP's hook for hot upgrades. `old_vsn` is the
  previous `@vsn` attribute; `state` is the live server's state; `extra`
  is whatever the appup instruction passed.

  Example — if 0.2.0's state were `%{value: n}`, we'd transform here:

      def code_change("0.1.0", n, _extra) when is_integer(n) do
        {:ok, %{value: n}}
      end

  Writing this wrong corrupts live state. That is the single biggest
  reason hot upgrades are risky in practice.
  """
  @impl true
  def code_change(_old_vsn, state, _extra), do: {:ok, state}
end
```

### Step 4: `test/hot_upgrade_intro_test.exs`

**Objective**: Write `hot_upgrade_intro_test.exs` — tests pin the behaviour so future refactors cannot silently regress the invariants established above.


```elixir
defmodule HotUpgradeIntroTest do
  use ExUnit.Case, async: false

  test "counter exists and increments" do
    assert HotUpgradeIntro.Counter.value() == 0
    assert HotUpgradeIntro.Counter.bump() == 1
  end

  test "code_change/3 is defined" do
    # Sanity check — the behaviour callback must be exported for
    # OTP to pick it up during a hot upgrade.
    assert function_exported?(HotUpgradeIntro.Counter, :code_change, 3)
  end
end
```

### Step 5: Sketch of the upgrade workflow

**Objective**: Sketch of the upgrade workflow.


A full hot-upgrade cycle looks roughly like this (not run here — use a
dedicated tool in practice):

```bash
# Build version 0.1.0
MIX_ENV=prod mix release
$REL/bin/hot_upgrade_intro daemon

# ... edit the code, bump @vsn on changed modules, bump version in mix.exs ...

# Build version 0.2.0 as an UPGRADE (not a fresh release)
# With a tool like distillery this would be:  MIX_ENV=prod mix release --upgrade
# With vanilla mix release you would run systools yourself to produce
# <app>-0.2.0.appup and hot_upgrade_intro-0.2.0.relup.

# Install on the live node
$REL/bin/hot_upgrade_intro install 0.2.0
```

For vanilla `mix release`, the upgrade path is deliberately manual —
which is Mix's polite way of steering you toward rolling restarts.

### Why this works

The design leans on OTP primitives that already encode the invariants we care about (supervision, back-pressure, explicit message semantics), so failure modes are visible at the right layer instead of being reinvented ad-hoc. Tests exercise the edges (timeouts, crashes, boundary states), which is where hand-rolled alternatives silently drift over time.



## Key Concepts

Hot upgrades (appups) allow code changes without stopping the VM—the OTP application code is reloaded in place. This requires precise tracking of code changes and careful state migration via `code_change/3` callbacks. Hot upgrades are powerful but fragile: a mistake can corrupt state or lose messages. Most production systems skip hot upgrades and instead redeploy, accepting brief downtime for simplicity and safety. Hot upgrades are valuable for critical infrastructure (telecom, financial) where downtime is unacceptable; for most CRUD apps, a restart is faster and safer. If you implement hot upgrades, invest heavily in testing—state migration bugs are subtle and hard to diagnose.

---

## Deep Dive: Compile-Time vs Runtime Configuration Boundaries

A release is a static artifact: code and compile-time config are baked in. Runtime config must be provided at boot via environment variables, config files, or config providers. Simple rule: if a value changes between dev and prod, it goes in `config/runtime.exs`, not `config/config.exs`.

Footgun: putting config in compile-time files and assuming environment variables work at runtime. Releases ignore env vars unless `config/runtime.exs` explicitly reads them. If you need env vars, fetch them in `config/runtime.exs` and store in application state.

For distributed systems, config providers (modules loading config from Consul, S3, etc.) are powerful but complex. Start with environment variables and `config/runtime.exs`; only reach for providers if you need dynamic reloading without downtime or multi-tenant config switching. Premature provider complexity is a mistake.

## Trade-offs and production gotchas

**1. `code_change/3` runs INSIDE the live server**
A crash there aborts the upgrade mid-way and can leave your fleet in
a mixed state. You need exhaustive tests of every possible old-state
shape — harder than it sounds.

**2. NIFs, ports, and external state don't hot-upgrade cleanly**
A hot upgrade swaps Erlang modules. Native code, open sockets with
changed protocols, external caches that now use a different key
format — none of this follows the upgrade.

**3. The tooling isn't in core Mix**
`mix release` deliberately does not generate appup/relup. You need
`distillery` (discontinued) or a custom `systools` driver. The
ecosystem signal is clear: this is not the recommended path.

**4. Rolling restarts get you 99% of the value**
If your deploy cadence is minutes (not microseconds) and your
service has more than one instance behind a load balancer, rolling
restarts give you zero-downtime deploys with 10% of the complexity.

**5. When NOT to use hot upgrades**
Almost always. Use them only if: you run a single-node system where
downtime is intolerable *and* you cannot add redundancy. In every
other case, add instances and roll restarts.

**6. When hot upgrades still make sense**
Embedded devices with no redundancy. Telecom switches that cost
millions to reboot. Latency-critical single-node systems. That's
about it.
