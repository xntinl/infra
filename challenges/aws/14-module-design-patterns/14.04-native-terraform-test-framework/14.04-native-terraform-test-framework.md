# 14.04 Framework Nativo de Testing en Terraform

## Metadata
| Campo | Valor |
|-------|-------|
| **Complejidad** | Intermedio |
| **Modalidad** | Terraform |
| **Features TF** | `terraform test`, `.tftest.hcl`, `mock_provider`, assertions |

## Descripcion

Usar el framework `terraform test` (1.6+) para escribir tests unitarios y de integracion para modulos. Crear archivos `.tftest.hcl` con run blocks, mock providers (1.7+), assertions y expected failures. Comparar con Terratest para evaluar trade-offs.

## Objetivos de aprendizaje

- Dominar el framework nativo de testing de Terraform
- Usar mock providers para tests unitarios sin despliegue real
- Escribir assertions y manejar expected failures
- Organizar test suites de forma efectiva

## Requisitos previos

- Conocimientos: Terraform modules, HCL avanzado, conceptos de testing
- Herramientas: Terraform >= 1.7 (para mock providers), AWS CLI (para tests de integracion)
- Features: `terraform test`, `.tftest.hcl`, `mock_provider`, `run` blocks

## Arquitectura

El framework nativo permite testing sin herramientas externas:

- **Archivos `.tftest.hcl`**: Ubicados en el directorio `tests/` del modulo, contienen los test cases
- **Run Blocks**: Cada `run` block ejecuta un plan o apply con variables especificas y assertions
- **Mock Providers** (1.7+): Proveedores simulados que permiten tests unitarios sin acceso real a AWS
- **Assertions**: Condiciones que verifican outputs, atributos de recursos y valores esperados
- **Expected Failures**: Tests que verifican que configuraciones invalidas son rechazadas correctamente
- **Comparacion**: Analisis de trade-offs entre `terraform test` (nativo, rapido, limitado) y Terratest (externo, completo, lento)

## Pasos sugeridos

1. Crear un modulo de Terraform de ejemplo con variables validadas y outputs definidos
2. Crear el directorio `tests/` y un primer archivo `.tftest.hcl`
3. Escribir un test unitario basico con un `run` block que valide outputs
4. Implementar mock providers para simular el proveedor AWS sin despliegue real
5. Escribir assertions que verifiquen atributos de recursos en el plan
6. Implementar un test de expected failure que verifique que inputs invalidos son rechazados
7. Crear tests de integracion (sin mocks) que desplieguen y verifiquen recursos reales
8. Organizar los tests en multiples archivos por area funcional
9. Ejecutar `terraform test` y analizar los resultados
10. Documentar una comparacion de trade-offs con Terratest (velocidad, cobertura, complejidad)

## Validacion

- [ ] Los archivos `.tftest.hcl` estan correctamente estructurados en el directorio `tests/`
- [ ] Los tests unitarios con mock providers se ejecutan sin acceso a AWS
- [ ] Las assertions validan correctamente outputs y atributos de recursos
- [ ] Los expected failures verifican que configuraciones invalidas son rechazadas
- [ ] Los tests de integracion despliegan y validan recursos reales
- [ ] `terraform test` ejecuta todos los tests y reporta resultados
- [ ] La comparacion con Terratest esta documentada con trade-offs claros
