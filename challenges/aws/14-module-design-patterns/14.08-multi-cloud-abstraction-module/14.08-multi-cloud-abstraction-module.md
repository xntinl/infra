# 14.08 Modulo de Abstraccion Multi-Cloud

## Metadata
| Campo | Valor |
|-------|-------|
| **Complejidad** | Avanzado |
| **Modalidad** | Terraform |
| **Features TF** | Provider aliases, `configuration_aliases`, conditional logic |

## Descripcion

Construir un modulo de abstraccion que despliegue una "compute instance" usando una interfaz generica (name, size, image). Internamente, usar provider aliases para soportar tanto AWS (EC2) como un proveedor mock/local. Demostrar como abstraer detalles especificos de cloud.

## Objetivos de aprendizaje

- Implementar abstraccion de proveedores en Terraform
- Disenar patrones multi-cloud con interfaces genericas
- Comprender los requisitos de providers en modulos
- Evaluar trade-offs de la abstraccion multi-cloud

## Requisitos previos

- Conocimientos: Terraform providers, modulos avanzados, EC2, provider configuration
- Herramientas: Terraform >= 1.0, AWS CLI, opcionalmente proveedor local/mock
- Features: Provider aliases, `configuration_aliases`, `required_providers`, conditional logic

## Arquitectura

El modulo abstrae detalles de cloud detras de una interfaz unificada:

- **Interfaz Generica**: Variables estandar independientes del cloud (name, size: small/medium/large, image, network)
- **Size Mapping**: Mapa interno que traduce tallas genericas a instance types especificos (small -> t3.micro en AWS)
- **Provider Abstraction**: El modulo declara `configuration_aliases` para recibir el proveedor configurado por el consumidor
- **AWS Implementation**: Recursos EC2 con translation de la interfaz generica a atributos de AWS
- **Mock Implementation**: Recursos locales (null_resource, local_file) que simulan el comportamiento para testing
- **Cloud Selection**: Logica condicional o modulos separados por proveedor, seleccionados por variable
- **Outputs Unificados**: IP, instance ID y status en formato generico independiente del cloud

## Pasos sugeridos

1. Definir la interfaz generica del modulo: variables de nombre, tamano, imagen y red
2. Crear un mapa de traduccion de tamanos genericos a instance types de AWS
3. Implementar el modulo para AWS: crear EC2 instance, security group, y elastic IP
4. Declarar `required_providers` con `configuration_aliases` en el modulo
5. Implementar una version mock usando null_resource o local_file para testing local
6. Crear logica de seleccion que determine cual implementacion usar (variable `cloud_provider`)
7. Definir outputs unificados que expongan informacion en formato cloud-agnostico
8. Probar el modulo con AWS: desplegar una instancia usando la interfaz generica
9. Probar el modulo con el proveedor mock para validar la abstraccion
10. Documentar los trade-offs: beneficios de la abstraccion vs perdida de funcionalidad especifica

## Validacion

- [ ] La interfaz del modulo es independiente del cloud (no hay variables especificas de AWS)
- [ ] El mapeo de tamanos traduce correctamente tallas genericas a instance types
- [ ] El modulo despliega recursos en AWS usando la interfaz generica
- [ ] El modulo funciona con el proveedor mock para testing local
- [ ] Los `configuration_aliases` estan correctamente declarados y configurados
- [ ] Los outputs son genericos y consistentes entre implementaciones
- [ ] Los trade-offs de la abstraccion estan documentados
