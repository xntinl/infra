# 14.02 Pipeline de Versionado de Modulos con Semantic Release

## Metadata
| Campo | Valor |
|-------|-------|
| **Complejidad** | Avanzado |
| **Modalidad** | Terraform + GitHub Actions |
| **Features TF** | Module sources (git tags, S3), version constraints |

## Descripcion

Configurar un repositorio de modulos con semantic versioning. Usar GitHub Actions para incrementar versiones automaticamente basado en mensajes de commit (conventional commits). Publicar en un registro privado (S3 o GitHub releases). Consumir modulos versionados desde una configuracion root.

## Objetivos de aprendizaje

- Implementar versionado semantico para modulos de Terraform
- Automatizar el proceso de release con CI/CD
- Fijar versiones de modulos y gestionar upgrades
- Disenar estrategias de actualizacion de modulos

## Requisitos previos

- Conocimientos: Terraform modules, git tags, semantic versioning, conventional commits
- Herramientas: Terraform >= 1.0, GitHub Actions, git, semantic-release tool
- Features: Module sources (git, S3), version constraints, GitHub releases

## Arquitectura

El sistema gestiona el ciclo de vida de versionado de modulos:

- **Module Repository**: Repositorio dedicado con uno o mas modulos de Terraform
- **Conventional Commits**: Mensajes de commit estandarizados (feat:, fix:, BREAKING CHANGE:) que determinan el bump de version
- **GitHub Actions Pipeline**: Workflow que analiza commits, determina la nueva version, crea el tag y publica el release
- **Registry**: GitHub Releases o bucket S3 como registro privado de modulos
- **Consumer Configuration**: Configuracion root que consume modulos con version constraints (`~>`, `>=`, etc.)
- **Upgrade Flow**: Proceso controlado para actualizar versiones de modulos en las configuraciones consumidoras

## Pasos sugeridos

1. Crear un repositorio para el modulo de Terraform con la estructura estandar
2. Configurar conventional commits con un linter de mensajes de commit (commitlint)
3. Implementar un workflow de GitHub Actions con semantic-release o similar
4. Configurar el pipeline para crear git tags con la version semantica (v1.0.0, v1.1.0, etc.)
5. Publicar el modulo como GitHub Release con los archivos del modulo
6. Opcionalmente, publicar el modulo en un bucket S3 como registro privado
7. Crear una configuracion root que consuma el modulo con version constraint `~> 1.0`
8. Simular un commit tipo `feat:` y verificar que se crea un minor version bump
9. Simular un commit tipo `fix:` y verificar el patch version bump
10. Probar la actualizacion del modulo en la configuracion consumidora

## Validacion

- [ ] Los conventional commits se validan en el pipeline
- [ ] GitHub Actions crea automaticamente tags y releases con la version correcta
- [ ] El modulo se puede consumir desde git tags con version constraints
- [ ] Un commit `feat:` genera un bump de minor version
- [ ] Un commit `fix:` genera un bump de patch version
- [ ] Un BREAKING CHANGE genera un bump de major version
- [ ] La configuracion consumidora puede pinear y actualizar versiones de forma controlada
