# 14.09 Jerarquias de Modulos Anidados

## Metadata
| Campo | Valor |
|-------|-------|
| **Complejidad** | Avanzado |
| **Modalidad** | Terraform |
| **Features TF** | Nested modules, output propagation, version constraints |

## Descripcion

Disenar una jerarquia de modulos de 3 niveles: modulos base (VPC, IAM) -> modulos de servicio (ECS service, Lambda function) -> modulos de aplicacion (stack de app completo). Gestionar versionado en cada nivel. Manejar inyeccion de dependencias entre niveles.

## Objetivos de aprendizaje

- Disenar jerarquias de modulos con niveles de abstraccion claros
- Gestionar versiones independientes en cada nivel de la jerarquia
- Implementar inyeccion de dependencias entre niveles de modulos
- Propagar outputs correctamente a traves de multiples niveles

## Requisitos previos

- Conocimientos: Terraform modules avanzado, patrones de diseno, AWS services (VPC, ECS, Lambda, IAM)
- Herramientas: Terraform >= 1.0, git (para versionado de modulos)
- Features: Nested modules, output propagation, version constraints, module sources

## Arquitectura

La jerarquia de tres niveles organiza la infraestructura por nivel de abstraccion:

- **Nivel 1 - Modulos Base**: Componentes fundamentales reutilizables
  - `base/vpc`: VPC con subnets y networking
  - `base/iam`: Roles y policies base
  - `base/kms`: Claves de encryption compartidas
- **Nivel 2 - Modulos de Servicio**: Servicios completos que consumen modulos base
  - `service/ecs-service`: Consume VPC e IAM. Crea cluster, task definition, service, ALB
  - `service/lambda-function`: Consume IAM y KMS. Crea funcion, log group, alarmas
- **Nivel 3 - Modulos de Aplicacion**: Stacks completos que orquestan modulos de servicio
  - `app/web-app`: Consume service/ecs-service + service/lambda-function + base/vpc
  - Interfaz simplificada: el consumidor solo necesita unas pocas variables de alto nivel
- **Dependency Injection**: Cada nivel recibe sus dependencias como inputs, no las crea internamente

## Pasos sugeridos

1. Disenar la estructura de directorios de la jerarquia de 3 niveles
2. Implementar los modulos base: VPC con outputs completos, IAM con roles parametrizables
3. Implementar los modulos de servicio: ECS service que consume VPC e IAM como inputs
4. Implementar Lambda function module que consume IAM y opcionalmente VPC
5. Crear el modulo de aplicacion que orquesta los modulos de servicio
6. Implementar inyeccion de dependencias: los modulos de servicio reciben IDs, no crean recursos base
7. Propagar outputs a traves de los niveles: base -> servicio -> aplicacion -> consumidor
8. Configurar versionado independiente para cada modulo (git tags por modulo)
9. Definir version constraints en cada nivel para pinear las dependencias
10. Probar el despliegue completo desde el modulo de aplicacion y verificar la propagacion de outputs

## Validacion

- [ ] La jerarquia de 3 niveles esta implementada con separacion clara de responsabilidades
- [ ] Los modulos base funcionan de forma independiente
- [ ] Los modulos de servicio consumen correctamente outputs de los modulos base
- [ ] Los modulos de aplicacion orquestan modulos de servicio con interfaz simplificada
- [ ] La inyeccion de dependencias funciona correctamente (no hay creacion interna de recursos base)
- [ ] Los outputs se propagan correctamente a traves de todos los niveles
- [ ] Cada nivel tiene versionado independiente con constraints definidos
