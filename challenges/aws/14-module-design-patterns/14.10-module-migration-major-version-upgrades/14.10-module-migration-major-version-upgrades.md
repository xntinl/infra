# 14.10 Migracion de Modulos y Upgrades de Major Version

## Metadata
| Campo | Valor |
|-------|-------|
| **Complejidad** | Avanzado |
| **Modalidad** | Terraform |
| **Features TF** | `moved` blocks, version constraints, `terraform plan` |

## Descripcion

Practicar el upgrade de un modulo de v1 a v2 con breaking changes (variables renombradas, recursos reestructurados). Usar bloques `moved` para evitar la recreacion de recursos. Escribir una guia de migracion. Probar el path de upgrade con validacion de `terraform plan`.

## Objetivos de aprendizaje

- Gestionar breaking changes en modulos de Terraform de forma segura
- Aplicar patrones de migracion con `moved` blocks
- Disenar estrategias de upgrade que eviten recreacion de recursos
- Validar la seguridad de upgrades con `terraform plan`

## Requisitos previos

- Conocimientos: Terraform modules, versionado semantico, `moved` blocks, resource addressing
- Herramientas: Terraform >= 1.1 (para moved blocks), git
- Features: `moved` blocks, version constraints, `terraform plan`, resource addressing

## Arquitectura

El ejercicio simula un upgrade de major version con breaking changes controlados:

- **Modulo v1**: Version actual con estructura original
  - Variables: `instance_type`, `vpc_id`, `subnet_ids`
  - Recursos: `aws_instance.main`, `aws_security_group.main`
  - Estructura plana sin sub-modulos
- **Modulo v2**: Nueva version con breaking changes
  - Variables renombradas: `compute_size`, `network_config` (objeto)
  - Recursos reestructurados: `aws_instance.this`, `aws_security_group.this` (dentro de sub-modulo `compute`)
  - Sub-modulos internos para mejor organizacion
- **Moved Blocks**: Mapean las direcciones antiguas a las nuevas para evitar destroy+create
  - `moved { from = aws_instance.main; to = module.compute.aws_instance.this }`
- **Migration Guide**: Documento que detalla cada cambio y los pasos de migracion
- **Validation**: `terraform plan` debe mostrar cero recreaciones de recursos

## Pasos sugeridos

1. Crear el modulo v1 con una estructura simple y desplegar recursos con el
2. Documentar los recursos actuales en el state (nombres, direcciones)
3. Disenar los breaking changes para v2: renombrar variables, reestructurar recursos
4. Implementar el modulo v2 con la nueva estructura y sub-modulos
5. Agregar bloques `moved` en v2 para cada recurso que cambia de direccion
6. Actualizar la configuracion consumidora para usar v2 con las nuevas variables
7. Ejecutar `terraform plan` y verificar que no se planifica destruccion de recursos
8. Si hay recreaciones inesperadas, ajustar los `moved` blocks y re-validar
9. Ejecutar `terraform apply` para confirmar la migracion exitosa
10. Escribir una guia de migracion con todos los cambios, moved blocks y pasos necesarios

## Validacion

- [ ] El modulo v1 esta desplegado con recursos en el state
- [ ] El modulo v2 tiene breaking changes claros (variables renombradas, recursos movidos)
- [ ] Los bloques `moved` mapean correctamente las direcciones antiguas a las nuevas
- [ ] `terraform plan` muestra cero recreaciones de recursos despues de la migracion
- [ ] La configuracion consumidora se actualizo para usar la interfaz v2
- [ ] `terraform apply` completa la migracion sin destruir/recrear recursos
- [ ] La guia de migracion esta documentada con pasos claros y reproducibles
