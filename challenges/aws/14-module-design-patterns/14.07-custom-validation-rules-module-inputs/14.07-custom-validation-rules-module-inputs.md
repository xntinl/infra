# 14.07 Reglas de Validacion Personalizadas para Inputs de Modulos

## Metadata
| Campo | Valor |
|-------|-------|
| **Complejidad** | Intermedio |
| **Modalidad** | Terraform |
| **Features TF** | `validation`, `precondition`, `postcondition`, `optional()`, type constraints |

## Descripcion

Implementar validacion exhaustiva de inputs: bloques `validation` con condiciones personalizadas, bloques `precondition`/`postcondition` en lifecycle, type constraints con `optional()` y defaults. Construir un modulo que rechace configuraciones invalidas en tiempo de plan.

## Objetivos de aprendizaje

- Implementar patrones de validacion de inputs robustos
- Disenar modulos con filosofia fail-fast
- Dominar el sistema de tipos de Terraform con `optional()` y defaults
- Usar lifecycle conditions para validar precondiciones y postcondiciones

## Requisitos previos

- Conocimientos: Terraform variables, tipos complejos, expresiones condicionales, modulos
- Herramientas: Terraform >= 1.3 (para optional()), IDE con soporte HCL
- Features: `validation` blocks, `precondition`, `postcondition`, `optional()`, type constraints

## Arquitectura

El modulo implementa validacion en multiples capas:

- **Variable Validation**: Bloques `validation` en las variables que verifican formato, rango y consistencia de los inputs antes del plan
- **Type Constraints**: Tipos complejos con `object()`, `optional()` y valores por defecto que guian al usuario
- **Preconditions**: Bloques `precondition` en recursos que validan condiciones que dependen de datos dinamicos (data sources)
- **Postconditions**: Bloques `postcondition` que verifican el estado de los recursos despues de su creacion
- **Error Messages**: Mensajes de error descriptivos que indican exactamente que esta mal y como corregirlo
- **Modulo de Ejemplo**: Un modulo de compute que valida instance type, CIDR ranges, tags, naming conventions

## Pasos sugeridos

1. Crear un modulo que acepte variables complejas (configuracion de instancia, networking, tags)
2. Implementar validaciones basicas: regex para nombres, rangos para numeros, listas de valores permitidos
3. Usar `optional()` con valores por defecto para campos opcionales en objetos complejos
4. Implementar validaciones cruzadas: verificar que combinaciones de variables son coherentes
5. Agregar `precondition` en recursos para validar datos de data sources (por ejemplo, que un AMI existe)
6. Agregar `postcondition` para verificar atributos del recurso despues de su creacion
7. Escribir mensajes de error claros y accionables para cada validacion
8. Probar con inputs validos e invalidos, verificando que los errores son descriptivos
9. Implementar validacion de CIDR ranges (formato, tamano de red, no-overlap)
10. Documentar todas las validaciones y sus mensajes de error

## Validacion

- [ ] Las variables tienen bloques `validation` con condiciones y mensajes claros
- [ ] Los tipos complejos usan `optional()` con valores por defecto apropiados
- [ ] Las validaciones rechazan configuraciones invalidas en tiempo de plan
- [ ] Los `precondition` validan datos dinamicos antes de crear recursos
- [ ] Los `postcondition` verifican el estado de recursos despues de la creacion
- [ ] Los mensajes de error son descriptivos e indican como corregir el problema
- [ ] Inputs validos pasan todas las validaciones sin errores
