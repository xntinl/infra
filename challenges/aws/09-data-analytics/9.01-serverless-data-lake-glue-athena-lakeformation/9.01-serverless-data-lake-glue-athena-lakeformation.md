## Why This Matters

Every organization that stores data in S3 eventually faces the same three problems: discovery ("what data do we have?"), access control ("who can see what?"), and query cost ("why did that Athena query cost $47?"). AWS Glue, Athena, and Lake Formation solve these problems as a unified serverless stack.

Glue Crawlers scan S3 paths, infer schemas, and register tables in the Glue Data Catalog -- a Hive-compatible metastore. Athena queries those tables using standard SQL without provisioning any servers. Lake Formation layers fine-grained access control (column-level, row-level, cell-level) on top of the catalog, replacing the brittle combination of S3 bucket policies and IAM policies that most teams start with.

The critical cost lever is **partitioning**. An unpartitioned 1 TB dataset costs $5.00 per full-table scan in Athena. The same dataset partitioned by `year/month/day` and queried for a single day scans ~2.7 GB -- costing $0.014. That is a 350x cost reduction from a file layout decision you make once. This exercise teaches you to build that foundation correctly from day one.

## The Challenge

Build a complete serverless data lake with the following components:

| Component | Resource | Purpose |
|-----------|----------|---------|
| Raw zone | S3 bucket | Landing zone for unprocessed data |
| Curated zone | S3 bucket | Cleaned, partitioned Parquet data |
| Data Catalog | Glue database + crawler | Schema discovery and metadata |
| Query engine | Athena workgroup | SQL queries with cost controls |
| Access control | Lake Formation | Catalog-level permissions |
| Results bucket | S3 bucket | Athena query results storage |

### Requirements

1. **Three S3 buckets** -- raw, curated, and Athena results -- all with `force_destroy = true` and `random_id` suffixes
2. **Glue Catalog Database** registered with Lake Formation
3. **Glue Crawler** that scans the raw zone and populates the catalog, configured for Hive-style partitioning (`year=YYYY/month=MM/day=DD/`)
4. **Athena Workgroup** with `bytes_scanned_cutoff_per_query` set to 1 GB (1073741824 bytes) to prevent runaway costs
5. **Lake Formation** settings that designate the Glue Crawler role as a data lake administrator
6. **IAM roles** using `data "aws_iam_policy_document"` for all policies (no `jsonencode`)
7. Tags on every taggable resource using `var.project_name`

## Best Practices

> **Best Practice:** Always partition S3 data using Hive-style paths (`year=YYYY/month=MM/day=DD/`) before running Crawlers. Athena leverages partition pruning to skip irrelevant data, reducing both scan volume and cost. A well-partitioned dataset can reduce query costs by 100x or more.

> **Best Practice:** Set `bytes_scanned_cutoff_per_query` on every Athena workgroup. Without this guard rail, a single `SELECT *` on an unpartitioned table can scan terabytes and generate a surprise bill. Start with 1 GB for development workgroups and increase for production as needed.

> **Best Practice:** Use Lake Formation permissions instead of S3 bucket policies for data access control. Lake Formation provides column-level and row-level security, integrates with the Glue Data Catalog, and gives you a single audit point for all data access.

<details>
<summary>Hint 3: Glue Crawler configuration</summary>

The Crawler needs an `s3_target` pointing to the raw bucket path. Configure `recrawl_policy` and `schema_change_policy` to handle evolving data:

```hcl
resource "aws_glue_crawler" "this" {
  database_name = aws_glue_catalog_database.this.name
  name          = "${var.project_name}-raw-crawler"
  role          = aws_iam_role.glue.arn

  s3_target {
    path = "s3://${aws_s3_bucket.raw.id}/"
  }

  recrawl_policy {
    recrawl_behavior = "CRAWL_EVERYTHING"
  }

  schema_change_policy {
    delete_behavior = "LOG"
    update_behavior = "UPDATE_IN_DATABASE"
  }

  tags = { Name = "${var.project_name}-raw-crawler" }
}
```

</details>

<details>
<summary>Hint 4: Athena workgroup with cost controls</summary>

The workgroup configuration block sets the scan limit and output location:

```hcl
resource "aws_athena_workgroup" "this" {
  name = "${var.project_name}-workgroup"

  configuration {
    bytes_scanned_cutoff_per_query = 1073741824  # 1 GB

    result_configuration {
      output_location = "s3://${aws_s3_bucket.results.id}/athena-results/"
    }
  }

  tags = { Name = "${var.project_name}-workgroup" }
}
```

1073741824 bytes = 1 GB. Any query that would scan more than this is cancelled automatically.

</details>

<details>
<summary>Hint 5: Lake Formation data lake settings</summary>

Register your IAM role as a Lake Formation admin. This replaces the default "IAMAllowedPrincipals" super-permission:

```hcl
resource "aws_lakeformation_data_lake_settings" "this" {
  admins = [aws_iam_role.glue.arn]
}

resource "aws_lakeformation_resource" "raw" {
  arn      = aws_s3_bucket.raw.arn
  role_arn = aws_iam_role.glue.arn
}
```

The `aws_lakeformation_resource` registration tells Lake Formation to manage access to this S3 location instead of relying on S3 bucket policies.

</details>

## Spot the Bug

A colleague wrote this Athena workgroup configuration. It deploys without errors, but analysts complain that query costs are not being controlled. **What is wrong?**

```hcl
resource "aws_athena_workgroup" "analytics" {
  name = "analytics-workgroup"

  configuration {
    result_configuration {
      output_location = "s3://${aws_s3_bucket.results.id}/output/"
    }
  }

  tags = { Name = "analytics-workgroup" }
}

resource "aws_athena_workgroup" "dev" {
  name = "dev-workgroup"

  configuration {
    bytes_scanned_cutoff_per_query = 1073741824

    result_configuration {
      output_location = "s3://${aws_s3_bucket.results.id}/output/"

      encryption_configuration {
        encryption_option = "SSE_S3"
      }
    }
  }

  tags = { Name = "dev-workgroup" }
}
```

<details>
<summary>Explain the bug</summary>

The `analytics-workgroup` is missing `bytes_scanned_cutoff_per_query`. Only the `dev-workgroup` has the 1 GB scan limit. Analysts using the `analytics-workgroup` can run `SELECT *` on multi-terabyte tables with no cost guardrail.

Additionally, both workgroups write results to the same S3 prefix (`/output/`), which means results from different teams are interleaved and there is no workgroup-level isolation of query results.

**The fix:**

```hcl
resource "aws_athena_workgroup" "analytics" {
  name = "analytics-workgroup"

  configuration {
    bytes_scanned_cutoff_per_query = 10737418240  # 10 GB for analytics

    result_configuration {
      output_location = "s3://${aws_s3_bucket.results.id}/analytics/"

      encryption_configuration {
        encryption_option = "SSE_S3"
      }
    }
  }

  tags = { Name = "analytics-workgroup" }
}
```

Every workgroup should have a scan cutoff and a unique output prefix.

</details>

resource "aws_glue_crawler" "raw" {
  database_name = aws_glue_catalog_database.raw.name
  name          = "${var.project_name}-raw-crawler"

  description   = "Crawls raw S3 data and discovers schemas"

  s3_target {
    path = "s3://${aws_s3_bucket.raw.id}/"

  recrawl_policy {
    recrawl_behavior = "CRAWL_EVERYTHING"
  }

  schema_change_policy {
    delete_behavior = "LOG"
    update_behavior = "UPDATE_IN_DATABASE"

  configuration = jsonencode({
    Version = 1.0
    Grouping = {
      TableGroupingPolicy = "CombineCompatibleSchemas"

    CrawlerOutput = {
      Partitions = {
        AddOrUpdateBehavior = "InheritFromTable"
      }
    }
  })

}

## Summary

- A **Glue Data Catalog** is a Hive-compatible metastore that stores table schemas, partition information, and connection details
- **Glue Crawlers** automatically discover schemas from S3 data and register tables in the catalog
- **Hive-style partitioning** (`year=YYYY/month=MM/day=DD/`) enables Athena partition pruning, reducing query costs by orders of magnitude
- **Athena workgroups** enforce `bytes_scanned_cutoff_per_query` to prevent runaway query costs
- **Lake Formation** replaces S3 bucket policies with catalog-level access control, supporting column-level and row-level security
- Always encrypt data at rest (SSE-KMS for S3, SSE-S3 for Athena results) and block public access on all buckets
