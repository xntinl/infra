5. **Write** federated queries that join S3 data with DynamoDB lookup data in a single SQL statement

## Why This Matters

Production data rarely lives in one place. Order records land in S3 as part of your data lake, but customer metadata lives in DynamoDB for low-latency application lookups. Business analysts need to join these sources -- "show me all orders from premium customers in the last 30 days" -- without building ETL pipelines to copy data between systems.

Athena federated queries solve this by pushing query execution to Lambda-based connectors. Each connector translates Athena's SQL into the native query language of the target data source. The connector reads data from DynamoDB, serializes it into Apache Arrow format, and returns it to Athena for joining with S3 data. This avoids data duplication, reduces pipeline complexity, and gives analysts a single SQL interface across all sources.

> **Best Practice:** Federated queries are slower than native S3 scans because every request goes through a Lambda function. Use CTAS (Create Table As Select) to materialize frequently-used federated results into S3 as Parquet files. This gives you the convenience of federated access for ad-hoc queries and the performance of native scans for dashboards and scheduled reports.

## Best Practices

1. **Athena workgroups enforce cost control** -- Always set `bytes_scanned_cutoff_per_query` on your workgroup. Without it, a single `SELECT *` on a multi-terabyte table costs hundreds of dollars. A 10 GB limit caps any individual query at ~$0.05.

2. **Federated queries are slower than native scans** -- The Lambda connector adds network round-trips and serialization overhead. For dashboards or scheduled queries that hit the same federated data repeatedly, use CTAS to materialize results into S3 as Parquet:
   ```sql
   CREATE TABLE materialized_customers
   WITH (format = 'PARQUET', external_location = 's3://bucket/materialized/')
   AS SELECT * FROM dynamodb_catalog.default.customers;
   ```

3. **Use spill buckets for large result sets** -- When federated query results exceed Lambda memory, the connector spills intermediate data to S3. Always configure a spill bucket and prefix to prevent connector failures on large tables.

4. **Partition S3 data for cost efficiency** -- Athena charges per byte scanned. Partitioning by date (e.g., `s3://bucket/orders/year=2024/month=01/`) lets Athena skip irrelevant partitions, dramatically reducing scan volume and cost.

5. **Cast CSV columns in queries** -- The Glue catalog defines CSV columns as strings. Always use `CAST(price AS double)` or `CAST(quantity AS integer)` in your queries to avoid type comparison bugs.

## Spot the Bug

A colleague deployed this Athena workgroup but query costs are unbounded. A developer ran a `SELECT *` on a 500 GB table and the bill spiked. **What is wrong?**

```hcl
resource "aws_athena_workgroup" "this" {
  name = "analytics-workgroup"

  configuration {
    result_configuration {
      output_location = "s3://results-bucket/results/"
    }

    enforce_workgroup_configuration = true
    # bytes_scanned_cutoff_per_query is missing  # <-- BUG
  }

  tags = { Name = "analytics-workgroup" }
}
```

<details>
<summary>Explain the bug</summary>

The workgroup is missing `bytes_scanned_cutoff_per_query`. Without this setting, Athena places no limit on how much data a single query can scan. Since Athena charges $5 per TB scanned, a `SELECT *` on a 500 GB table costs $2.50, and a careless `SELECT *` on a multi-terabyte table can cost hundreds of dollars in seconds.

**Wrong -- no scan limit:**

```hcl
configuration {
  result_configuration {
    output_location = "s3://results-bucket/results/"
  }
  enforce_workgroup_configuration = true
}
```

**Fix -- add byte scan cutoff:**

```hcl
configuration {
  result_configuration {
    output_location = "s3://results-bucket/results/"
  }
  enforce_workgroup_configuration    = true
  bytes_scanned_cutoff_per_query     = 10737418240  # 10 GB = ~$0.05/query max
}
```

Setting `enforce_workgroup_configuration = true` without a scan limit gives a false sense of security -- it only enforces the output location, not cost controls.

</details>

## Summary

- **Athena federated queries** let you join data across S3, DynamoDB, and other sources using a single SQL interface without ETL pipelines
- **Lambda-based connectors** translate Athena SQL into native data source queries and return results in Apache Arrow format
- **Workgroup scan limits** (`bytes_scanned_cutoff_per_query`) are essential cost controls -- without them, a single query can scan terabytes at $5/TB
- **CTAS materializes federated results** into S3 as optimized Parquet files for repeated queries, avoiding Lambda overhead on every execution
- **Glue catalog tables** map S3 file layouts to SQL schemas, enabling Athena to query CSV, JSON, and Parquet files as structured tables
