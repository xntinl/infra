# 9.09 Redshift Serverless y Spectrum

## Metadata
| Campo | Valor |
|-------|-------|
| **Complejidad** | Avanzado |
| **Modalidad** | Ambos |
| **Servicios AWS** | Redshift Serverless, Redshift Spectrum, S3, Glue Data Catalog |

## Descripcion

Configurar Redshift Serverless y Spectrum para consultar datos directamente en S3. Cargar datos en tablas locales de Redshift, crear esquemas externos apuntando a datos en S3, y escribir consultas que hagan JOIN entre tablas locales y externas. Optimizar con sort keys y distribution keys.

## Objetivos de aprendizaje

- Comprender los patrones de data warehouse y la arquitectura de Redshift
- Configurar Redshift Spectrum para federacion de consultas sobre S3
- Optimizar rendimiento con sort keys y distribution keys
- Crear y utilizar vistas materializadas para consultas frecuentes
- Gestionar compute serverless para cargas de trabajo variables

## Requisitos previos

- Conocimientos: SQL avanzado, conceptos de data warehouse, modelado dimensional
- Herramientas: AWS CLI, Terraform, cliente SQL (psql, DataGrip, o Query Editor v2)
- Servicios AWS: Cuenta con permisos para Redshift, S3, Glue, IAM

## Arquitectura

Redshift Serverless proporciona capacidad de computo elastica sin necesidad de gestionar clusters. Los datos frecuentemente consultados se cargan en tablas locales de Redshift con sort keys y distribution keys optimizadas. Los datos historicos o menos frecuentes residen en S3 en formato Parquet, accesibles via Redshift Spectrum a traves de esquemas externos registrados en el Glue Data Catalog. Las consultas pueden hacer JOIN entre tablas locales (datos calientes) y tablas externas en S3 (datos frios), implementando una arquitectura de almacenamiento por niveles. Vistas materializadas pre-computan JOINs frecuentes para mejorar rendimiento.

## Pasos sugeridos

1. Crear un namespace y workgroup de Redshift Serverless
2. Preparar datos en S3 en formato Parquet, particionados por fecha
3. Catalogar los datos de S3 con Glue Crawlers
4. Crear un esquema externo en Redshift apuntando al Glue Data Catalog
5. Verificar que las tablas externas son consultables desde Redshift via Spectrum
6. Cargar datos de alta frecuencia en tablas locales de Redshift con COPY
7. Definir sort keys y distribution keys apropiadas para las tablas locales
8. Escribir consultas que hagan JOIN entre tablas locales y externas
9. Crear vistas materializadas para consultas frecuentes y configurar auto-refresh
10. Comparar rendimiento entre consultas directas y vistas materializadas

## Validacion

- [ ] Redshift Serverless esta operativo y accesible via Query Editor v2 o cliente SQL
- [ ] Los esquemas externos apuntan correctamente al Glue Data Catalog
- [ ] Las consultas Spectrum retornan datos correctos desde S3
- [ ] Los datos locales estan cargados con sort keys y distribution keys configuradas
- [ ] Los JOINs entre tablas locales y externas funcionan correctamente
- [ ] Las vistas materializadas se crean y actualizan automaticamente
- [ ] El rendimiento con vistas materializadas es superior al de las consultas directas
