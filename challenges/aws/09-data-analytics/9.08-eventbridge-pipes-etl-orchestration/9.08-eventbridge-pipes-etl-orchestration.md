# 9.08 EventBridge Pipes para Orquestacion ETL

## Metadata
| Campo | Valor |
|-------|-------|
| **Complejidad** | Intermedio |

## Descripcion

Construir pipelines ETL ligeros usando EventBridge Pipes. Conectar fuentes (SQS, DynamoDB Streams, Kinesis) con destinos (Step Functions, Lambda, Firehose) con pasos de filtrado y enriquecimiento. Comparar con el procesamiento de eventos tradicional basado en Lambda.

## Objetivos de aprendizaje

- Comprender la arquitectura de EventBridge Pipes y su modelo source-filter-enrich-target
- Simplificar integraciones punto a punto sin codigo intermediario
- Configurar filtrado de eventos y enriquecimiento con Lambda o API destinations
- Comparar costos y complejidad con el enfoque tradicional basado en Lambda
- Identificar casos de uso ideales para Pipes vs Lambda

## Requisitos previos

- Conocimientos: EventBridge basico, SQS, DynamoDB Streams, conceptos de ETL
- Herramientas: AWS CLI, Terraform
- Servicios AWS: Cuenta con permisos para EventBridge, SQS, DynamoDB, Step Functions, Lambda

## Arquitectura

EventBridge Pipes conecta fuentes de eventos directamente con destinos a traves de un pipeline configurable de cuatro etapas: Source (origen del evento), Filter (filtrado por patron), Enrich (enriquecimiento opcional con Lambda, Step Functions o API), y Target (destino final). Se implementan tres pipes diferentes: (1) SQS a Lambda con filtrado por tipo de mensaje, (2) DynamoDB Streams a Step Functions con enriquecimiento Lambda, y (3) Kinesis a Firehose con transformacion. Esto se compara con la arquitectura equivalente usando Lambda como intermediario para cada integracion.

## Pasos sugeridos

1. Crear las fuentes de eventos: cola SQS, tabla DynamoDB con streams habilitados
2. Crear las funciones Lambda para enriquecimiento y como destinos
3. Crear un Pipe que conecte SQS a Lambda con filtrado por patron de evento
4. Crear un Pipe que conecte DynamoDB Streams a Step Functions con enriquecimiento Lambda
5. Crear un tercer Pipe conectando una fuente a Firehose con transformacion
6. Enviar eventos de prueba a cada fuente y verificar el procesamiento end-to-end
7. Implementar la misma logica con el enfoque tradicional (Lambda como glue)
8. Comparar la complejidad del codigo, cantidad de recursos y costos entre ambos enfoques
9. Monitorear las metricas de los Pipes en CloudWatch (invocaciones, errores, latencia)
10. Documentar los trade-offs y casos de uso ideales para cada enfoque

## Validacion

- [ ] El Pipe SQS-a-Lambda filtra correctamente los mensajes segun el patron
- [ ] El Pipe DynamoDB-a-Step Functions enriquece y enruta los eventos correctamente
- [ ] El tercer Pipe procesa y entrega datos al destino configurado
- [ ] Los eventos filtrados no llegan al destino (el filtro funciona)
- [ ] Las metricas de los Pipes son visibles en CloudWatch
- [ ] La comparacion con el enfoque Lambda-based muestra diferencias claras en complejidad
