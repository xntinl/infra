By the end of this exercise you will be able to:

2. **Deploy** a Glue Data Quality ruleset with completeness, uniqueness, and freshness checks

5. **Connect** data quality failures to SNS notifications for alerting data engineering teams

## Why This Matters

Bad data is worse than no data. A dashboard built on incomplete records leads to wrong decisions. A machine learning model trained on duplicated rows produces biased predictions. A billing pipeline processing records with null customer IDs generates invoices that cannot be delivered. These problems compound silently -- by the time someone notices, weeks of downstream artifacts are contaminated.

AWS Glue Data Quality catches these issues at ingestion time, before bad data propagates. You define rules in DQDL -- "the `email` column must be 100% complete," "the `order_id` column must be unique," "the table must have been updated in the last 24 hours" -- and Glue evaluates them against your catalog tables. When rules fail, you get a quality score and detailed per-rule results that you can route to CloudWatch for alerting.

> **Best Practice:** Define data quality rules before building downstream pipelines. Catching a null `customer_id` at ingestion costs minutes; catching it in a production dashboard costs hours of incident response and data backfills.

## Best Practices

1. **Define DQ rules before building pipelines** -- Data quality rules are cheapest to enforce at ingestion. A missing `customer_id` caught by a DQDL rule costs minutes; the same issue discovered in a production report costs hours of incident response and weeks of data backfill.

2. **Use DQDL for declarative rules** -- DQDL (Data Quality Definition Language) lets you express validation constraints without writing code. Rules like `Completeness "email" = 1.0` and `Uniqueness "order_id" = 1.0` are readable, auditable, and version-controllable in Terraform.

3. **Catch data issues at ingestion, not in dashboards** -- By the time a null value appears in a BI dashboard, it has already propagated through ETL pipelines, aggregate tables, and ML training datasets. Glue Data Quality rules act as a gate at the source.

4. **Alert on DQ failures via CloudWatch** -- Route `numFailedDataQualityRules` to a CloudWatch alarm connected to SNS. Data engineers should be paged when quality degrades, not when a stakeholder notices wrong numbers.

5. **Combine rule types for defense in depth** -- Completeness checks catch nulls. Uniqueness checks catch duplicates. ColumnValues checks catch invalid data. Freshness checks catch stale data. Use all four together for comprehensive coverage.

## Spot the Bug

A colleague deployed a Glue crawler, but the Data Quality rules always pass even though the data clearly has issues. The Glue catalog table shows zero rows. **What is wrong?**

```hcl
resource "aws_glue_crawler" "this" {
  name          = "transactions-crawler"
  database_name = aws_glue_catalog_database.this.name
  role          = aws_iam_role.crawler.arn

  s3_target {
    path = "s3://${aws_s3_bucket.this.id}/txns/"  # <-- BUG
  }

  table_prefix = "raw_"

  tags = { Name = "transactions-crawler" }
}
```

<details>
<summary>Explain the bug</summary>

The crawler's `s3_target.path` is `s3://bucket/txns/` but the data was uploaded to `s3://bucket/transactions/`. The crawler scans an empty prefix, creates a catalog table with no data (or fails to create a table at all), and the Data Quality rules evaluate against zero rows.

Most DQ rules pass vacuously on empty datasets -- `Completeness "col" = 1.0` is trivially true when there are no rows to check, because 0 out of 0 non-null values equals 100% completeness. This makes the bug invisible: everything reports green while no actual validation occurs.

**Wrong -- mismatched prefix:**

```hcl
s3_target {
  path = "s3://${aws_s3_bucket.this.id}/txns/"
}
```

**Fix -- use the correct prefix:**

```hcl
s3_target {
  path = "s3://${aws_s3_bucket.this.id}/transactions/"
}
```

To guard against this class of bug, add a `RowCount` rule to your DQDL: `RowCount > 0`. This fails when the table is empty, catching misconfigured crawlers and missing data uploads.

</details>

## Summary

- **Glue Data Quality** validates data against DQDL rules at ingestion time, before bad data propagates to downstream systems
- **DQDL** (Data Quality Definition Language) expresses validation constraints declaratively -- completeness, uniqueness, value ranges, and freshness
- **Glue crawlers** scan S3 data and populate the Glue Data Catalog with inferred schemas, enabling SQL queries via Athena
- **Empty-dataset DQ rules pass vacuously** -- always include a `RowCount > 0` rule to catch misconfigured crawlers and missing data
- **CloudWatch alarms on DQ metrics** close the feedback loop, alerting data engineers when quality degrades instead of waiting for stakeholders to report wrong numbers
