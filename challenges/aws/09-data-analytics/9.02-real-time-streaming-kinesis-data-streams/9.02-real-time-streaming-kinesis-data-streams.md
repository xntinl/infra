## Why This Matters

Batch pipelines (Glue crawlers, scheduled ETL) work when you can tolerate minutes or hours of delay. But many use cases demand real-time processing: fraud detection, live dashboards, IoT sensor aggregation, clickstream analysis. Kinesis Data Streams provides a durable, ordered log that multiple consumers can read from independently.

The architecture is simple: producers write records to a stream, the stream holds them for 24 hours (or up to 365 days with extended retention), and consumers read and process them. Lambda is the simplest consumer -- AWS manages the polling, batching, and checkpointing for you via an event source mapping. The consumer writes results to a sink (DynamoDB in this exercise), completing the real-time pipeline.

The key capacity concept is the **shard**. Each shard handles 1 MB/s (or 1,000 records/s) of write throughput and 2 MB/s of read throughput. If you need 5 MB/s ingress, you need at least 5 shards. ON_DEMAND mode handles this automatically (scaling up to 200 MB/s), but PROVISIONED mode gives you cost predictability. Understanding this trade-off is essential for production streaming systems.

## Best Practices

> **Best Practice:** Use enhanced fan-out for sub-200ms consumer latency. Standard consumers share the 2 MB/s per-shard read throughput and poll at 200ms intervals. Enhanced fan-out provides a dedicated 2 MB/s pipe per consumer with push-based delivery, reducing latency to under 70ms. Use it when you have multiple consumers on the same stream or strict latency requirements.

> **Best Practice:** Set `bisect_batch_on_function_error = true` on Lambda event source mappings for Kinesis. Without this, a single poison record blocks the entire shard until the record expires. Bisecting isolates the bad record by splitting the batch in half and retrying each half separately.

> **Best Practice:** Size your shard count using the formula: `ceil(max(ingress_MB_s / 1, egress_MB_s / 2))`. Each shard handles 1 MB/s ingress and 2 MB/s egress. Over-provisioning wastes money ($0.015/shard-hour); under-provisioning causes `ProvisionedThroughputExceededException` errors and data loss.

## Spot the Bug

A colleague wrote this Lambda event source mapping. It deploys successfully, but the Lambda function processes the same records repeatedly and never advances past the first batch. **What is wrong?**

```hcl
resource "aws_lambda_event_source_mapping" "kinesis" {
  event_source_arn  = aws_kinesis_stream.this.arn
  function_name     = aws_lambda_function.this.arn
  starting_position = "TRIM_HORIZON"
  batch_size        = 100

  maximum_retry_attempts = -1  # <-- BUG
}
```

<details>
<summary>Explain the bug</summary>

Setting `maximum_retry_attempts = -1` means **infinite retries**. If the Lambda function throws an error on any record in the batch, it will retry that same batch forever. Since Kinesis event source mappings process shards sequentially (each shard has one concurrent invocation), the entire shard is blocked until either:
1. The record expires from the stream (24 hours by default)
2. Someone manually advances the iterator

Combined with the absence of `bisect_batch_on_function_error`, one poison record blocks all records behind it on the shard for up to 24 hours.

**The fix:**

```hcl
resource "aws_lambda_event_source_mapping" "kinesis" {
  event_source_arn  = aws_kinesis_stream.this.arn
  function_name     = aws_lambda_function.this.arn
  starting_position = "TRIM_HORIZON"
  batch_size        = 100

  maximum_retry_attempts            = 3
  bisect_batch_on_function_error    = true
  maximum_record_age_in_seconds     = 3600  # skip records older than 1 hour

  destination_config {
    on_failure {
      destination_arn = aws_sqs_queue.dlq.arn
    }
  }
}
```

Always set a finite retry count, enable bisection, and configure a dead-letter queue for records that exhaust retries.

</details>

<summary>TODO 4 -- Lambda function (lambda.tf)</summary>

First, create the handler file `lambda/index.py`:

```python
import base64
import json
import os
import boto3

dynamodb = boto3.resource("dynamodb")
table = dynamodb.Table(os.environ["TABLE_NAME"])


def handler(event, context):
    records_written = 0

    for record in event["Records"]:
        payload = base64.b64decode(record["kinesis"]["data"]).decode("utf-8")
        data = json.loads(payload)

        table.put_item(
            Item={
                "event_id": data["event_id"],
                "timestamp": data["timestamp"],
                "event_type": data.get("event_type", "unknown"),
                "user_id": data.get("user_id", "unknown"),
                "raw_data": payload,
            }
        )
        records_written += 1

    return {"statusCode": 200, "recordsProcessed": records_written}
```


## Summary

- **Kinesis Data Streams** is a durable, ordered log with configurable retention (24 hours to 365 days)
- **Shards** are the unit of capacity: 1 MB/s ingress, 2 MB/s egress per shard
- **PROVISIONED** mode gives cost predictability; **ON_DEMAND** mode auto-scales up to 200 MB/s
- **Lambda event source mappings** handle polling, batching, and checkpointing automatically
- **`bisect_batch_on_function_error`** isolates poison records by splitting failed batches in half
- **Enhanced fan-out** provides dedicated per-consumer throughput with sub-70ms push-based delivery
- Always set `maximum_retry_attempts` to a finite number and configure a DLQ for unprocessable records
