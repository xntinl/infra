## Why Firehose Matters

In Exercise 9.02, you built a Kinesis Data Streams pipeline that required a Lambda consumer to process records and write them to DynamoDB. That pattern gives you full control over processing logic, but many use cases just need to get streaming data into S3 reliably. Kinesis Data Firehose does exactly this -- it is a fully managed delivery service that buffers incoming records and writes them to S3 in batches, with no consumer code to write or maintain.

Firehose handles three things automatically that you would otherwise build yourself: **batching** (accumulating records into larger files), **compression** (GZIP, Snappy, or uncompressed), and **partitioning** (organizing output by date or custom keys). The buffer interval controls how often Firehose flushes data to S3. A 60-second interval means data arrives in S3 within ~1 minute of ingestion. A 900-second (15-minute) interval produces fewer, larger files -- better for downstream query performance but worse for latency.

The trade-off is straightforward: smaller buffer interval = lower latency but more small files (which degrade Athena query performance); larger buffer interval = higher latency but fewer, larger files (which Athena handles efficiently). For most data lake use cases, a 300-second (5-minute) interval with a 5 MB buffer size strikes a good balance.

```hcl
resource "random_id" "this" {
  byte_length = 4
}

# S3 bucket for Firehose delivery -- the data lake landing zone.
# force_destroy = true allows terraform destroy to delete the
# bucket even if it contains objects (lab cleanup only).
resource "aws_s3_bucket" "lake" {
  bucket        = "${var.project_name}-lake-${random_id.this.hex}"
  force_destroy = true

  tags = { Name = "${var.project_name}-lake" }
}

# Block all public access -- data lake buckets should never be public.
resource "aws_s3_bucket_public_access_block" "lake" {
  bucket = aws_s3_bucket.lake.id

  block_public_acls       = true
  block_public_policy     = true
  ignore_public_acls      = true
  restrict_public_buckets = true
}

# Enable server-side encryption with S3-managed keys.
resource "aws_s3_bucket_server_side_encryption_configuration" "lake" {
  bucket = aws_s3_bucket.lake.id

  rule {
    apply_server_side_encryption_by_default {
      sse_algorithm = "AES256"
    }
  }
}

# S3 bucket for Firehose error logging.
resource "aws_s3_bucket" "errors" {
  bucket        = "${var.project_name}-errors-${random_id.this.hex}"
  force_destroy = true

  tags = { Name = "${var.project_name}-errors" }

resource "aws_s3_bucket_public_access_block" "errors" {
  bucket = aws_s3_bucket.errors.id

  block_public_acls       = true
  block_public_policy     = true
  ignore_public_acls      = true
  restrict_public_buckets = true
```

### `iam.tf`

```hcl
# Trust policy: allow Firehose to assume this role.
data "aws_iam_policy_document" "firehose_assume" {
  statement {
    actions = ["sts:AssumeRole"]

    principals {
      type        = "Service"
      identifiers = ["firehose.amazonaws.com"]
    }
  }
}

resource "aws_iam_role" "firehose" {
  name               = "${var.project_name}-firehose-role"
  assume_role_policy = data.aws_iam_policy_document.firehose_assume.json

  tags = { Name = "${var.project_name}-firehose-role" }
}

# Permissions: Firehose needs to write to the lake bucket
# and the error bucket, plus CloudWatch logging.
data "aws_iam_policy_document" "firehose_permissions" {
  statement {
    sid    = "S3LakeAccess"
    effect = "Allow"
    actions = [
      "s3:AbortMultipartUpload",
      "s3:GetBucketLocation",
      "s3:GetObject",
      "s3:ListBucket",
      "s3:ListBucketMultipartUploads",
      "s3:PutObject",
    ]
    resources = [
      aws_s3_bucket.lake.arn,
      "${aws_s3_bucket.lake.arn}/*",
      aws_s3_bucket.errors.arn,
      "${aws_s3_bucket.errors.arn}/*",
    ]
  }

  statement {
    sid    = "CloudWatchLogs"
    effect = "Allow"
    actions = [
      "logs:CreateLogGroup",
      "logs:CreateLogStream",
      "logs:PutLogEvents",
    ]
    resources = ["*"]
  }
}

resource "aws_iam_role_policy" "firehose_permissions" {
  name   = "${var.project_name}-firehose-policy"
  role   = aws_iam_role.firehose.id
  policy = data.aws_iam_policy_document.firehose_permissions.json
}
```

### `firehose.tf`

```hcl
# CloudWatch log group for Firehose delivery errors.
resource "aws_cloudwatch_log_group" "firehose" {
  name              = "/aws/firehose/${var.project_name}"
  retention_in_days = 7

  tags = { Name = "${var.project_name}-firehose-logs" }
}

resource "aws_cloudwatch_log_stream" "firehose_s3" {
  name           = "S3Delivery"
  log_group_name = aws_cloudwatch_log_group.firehose.name
}

# Firehose delivery stream: receives records and writes them
# to S3 in GZIP-compressed batches with date-based prefixes.
#
# Buffer settings:
#   - buffering_size = 5 MB: Firehose accumulates up to 5 MB
#     before writing a file to S3.
#   - buffering_interval = 300 seconds (5 minutes): Firehose
#     writes at least every 5 minutes, even if 5 MB hasn't
#     been reached.
#   - Whichever threshold is reached first triggers a write.
#
# The prefix uses Hive-style partitioning so Athena and Glue
# can leverage partition pruning:
#   events/year=2024/month=01/day=15/hour=10/
resource "aws_kinesis_firehose_delivery_stream" "this" {
  name        = "${var.project_name}-delivery"
  destination = "extended_s3"

  extended_s3_configuration {
    role_arn   = aws_iam_role.firehose.arn
    bucket_arn = aws_s3_bucket.lake.arn

    # Hive-style date partitioning prefix
    prefix              = "events/year=!{timestamp:yyyy}/month=!{timestamp:MM}/day=!{timestamp:dd}/hour=!{timestamp:HH}/"
    error_output_prefix = "errors/year=!{timestamp:yyyy}/month=!{timestamp:MM}/day=!{timestamp:dd}/!{firehose:error-output-type}/"

    buffering_size     = 5
    buffering_interval = 300

    # GZIP compression reduces storage cost by ~80% for JSON data.
    compression_format = "GZIP"

    cloudwatch_logging_options {
      enabled         = true
      log_group_name  = aws_cloudwatch_log_group.firehose.name
      log_stream_name = aws_cloudwatch_log_stream.firehose_s3.name
    }
  }

  tags = { Name = "${var.project_name}-delivery-stream" }
}
```

## Best Practices

> **Best Practice:** Always enable GZIP compression on Firehose delivery streams for JSON data. GZIP typically achieves 80-90% compression on JSON, reducing both S3 storage costs and Athena scan costs (Athena decompresses transparently). For columnar formats like Parquet, the benefit is smaller since Parquet has built-in compression.

> **Best Practice:** Use Hive-style partitioning prefixes (`year=YYYY/month=MM/day=DD/`) instead of plain date paths (`YYYY/MM/DD/`). Hive-style partitions are automatically recognized by Glue Crawlers and Athena's `MSCK REPAIR TABLE` command, eliminating manual partition management.

## Common Mistakes

### 1. Using a buffer interval that is too small

Setting `buffering_interval = 60` (the minimum) produces a new S3 object every minute. Over a day, that is 1,440 small files per partition. Athena and Glue perform poorly with thousands of small files because each file requires a separate S3 GET request.

**Wrong -- minimum buffer interval:**

```hcl
extended_s3_configuration {
  buffering_interval = 60   # new file every minute
  buffering_size     = 1    # or every 1 MB
}
```

**What happens:** Athena queries slow down dramatically as the number of small files grows. A query that scans 10,000 small files takes 10x longer than the same data in 100 larger files.

**Fix -- use a reasonable interval:**

```hcl
extended_s3_configuration {
  buffering_interval = 300  # 5 minutes
  buffering_size     = 5    # 5 MB
}
```

If you need near-real-time data in S3, set `buffering_interval = 60` but run a compaction job (Glue ETL or S3 Object Lambda) to merge small files into larger ones periodically.

### 2. Forgetting the error output prefix

If you omit `error_output_prefix`, failed records are silently dropped. You will never know that data was lost until someone queries the data lake and finds gaps.

**Wrong -- no error handling:**

```hcl
extended_s3_configuration {
  role_arn   = aws_iam_role.firehose.arn
  bucket_arn = aws_s3_bucket.lake.arn
  prefix     = "events/"
  # No error_output_prefix -- failed records vanish
}
```

**Fix -- always configure error output:**

```hcl
extended_s3_configuration {
  role_arn            = aws_iam_role.firehose.arn
  bucket_arn          = aws_s3_bucket.lake.arn
  prefix              = "events/year=!{timestamp:yyyy}/month=!{timestamp:MM}/day=!{timestamp:dd}/"
  error_output_prefix = "errors/year=!{timestamp:yyyy}/month=!{timestamp:MM}/!{firehose:error-output-type}/"
}
```

### 3. Missing S3 permissions on the Firehose role

Firehose requires `s3:AbortMultipartUpload` in addition to the obvious `s3:PutObject`. Without it, failed multipart uploads accumulate in the bucket, wasting storage, and Firehose retries cause delivery delays.

**Wrong -- incomplete permissions:**

```hcl
actions = [
  "s3:PutObject",
  "s3:GetBucketLocation",
]
```

**Fix -- include all required actions:**

```hcl
actions = [
  "s3:AbortMultipartUpload",
  "s3:GetBucketLocation",
  "s3:GetObject",
  "s3:ListBucket",
  "s3:ListBucketMultipartUploads",
  "s3:PutObject",
]
```

## Summary

- **Kinesis Data Firehose** is a fully managed delivery service that writes streaming data to S3 (or Redshift, OpenSearch, HTTP endpoints) with no consumer code
- **Buffer interval** and **buffer size** control delivery frequency: whichever threshold is reached first triggers a write to S3
- **GZIP compression** reduces storage and scan costs by ~80% for JSON data
- **Hive-style partitioning** (`year=YYYY/month=MM/day=DD/`) enables automatic partition discovery by Glue Crawlers and Athena
- Always configure an **error output prefix** to capture failed records instead of silently dropping them
- The Firehose IAM role requires `s3:AbortMultipartUpload` in addition to standard S3 write permissions
