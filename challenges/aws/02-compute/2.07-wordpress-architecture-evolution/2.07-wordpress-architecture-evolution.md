## Learning Objectives

By the end of this exercise you will be able to:

1. **Design** a multi-tier WordPress architecture with separation of compute, storage, and database
2. **Implement** an ALB distributing traffic to an Auto Scaling Group
3. **Configure** EFS as shared storage for WordPress uploads across multiple EC2 instances
4. **Deploy** RDS MySQL in a private data subnet for high availability

## Why Multi-Tier Architecture

A single EC2 instance running WordPress with a local MySQL database and local file storage is the simplest possible deployment -- and also the most fragile. If that instance fails, you lose everything. There is no way to scale horizontally because each instance has its own copy of the data.

The multi-tier pattern separates concerns: ALB in public subnets distributes traffic, EC2 ASG in private subnets provides stateless application servers, EFS provides shared storage for `wp-content/uploads`, and RDS in data subnets manages the database. Each tier can fail independently and recover without affecting the others.

> **Best Practice:** Never store stateful data (uploads, sessions, database) on an EC2 instance's local disk when running behind an Auto Scaling Group. Instance termination destroys local storage. Use EFS for shared files, RDS for databases, and ElastiCache or DynamoDB for sessions.

## Spot the Bug

A colleague deployed this EFS configuration but instances in the second AZ fail to start. The ASG keeps terminating and replacing them. **What is wrong?**

```hcl
resource "aws_efs_file_system" "this" {
  encrypted        = true
  performance_mode = "generalPurpose"
  tags             = { Name = "wp-multi-tier-lab-efs" }
}

resource "aws_efs_mount_target" "this" {
  file_system_id  = aws_efs_file_system.this.id
  subnet_id       = aws_subnet.private[local.azs[0]].id   # <-- only AZ-a!
  security_groups = [aws_security_group.efs.id]
}
```

<details>
<summary>Explain the bug</summary>

**The problem:** Only one EFS mount target exists, in AZ-a. EFS mount targets are AZ-specific -- an EC2 instance can only connect to a mount target in its own AZ.

1. Instances in AZ-a mount EFS successfully and pass health checks
2. Instances in AZ-b cannot find a mount target, `mount` times out
3. WordPress never starts, the ALB health check fails, ASG terminates the instance
4. The ASG tries again in AZ-b -- infinite loop of launch/terminate

**The fix:** Create a mount target in every AZ where ASG instances can run:

```hcl
resource "aws_efs_mount_target" "this" {
  for_each        = local.private_subnets
  file_system_id  = aws_efs_file_system.this.id
  subnet_id       = aws_subnet.private[each.key].id
  security_groups = [aws_security_group.efs.id]
}
```

</details>

## Summary

- **ALB in public subnets** distributes traffic and terminates SSL
- **ASG in private subnets** provides horizontal scaling and self-healing
- **EFS** provides shared storage for `wp-content/uploads` accessible from any instance in any AZ
- **RDS in data subnets** manages the MySQL database with automated backups
- **EFS mount targets** must exist in every AZ where instances run -- a missing mount target causes boot failures
- **Security group chaining** (ALB -> EC2 -> EFS/RDS) enforces least-privilege network access
