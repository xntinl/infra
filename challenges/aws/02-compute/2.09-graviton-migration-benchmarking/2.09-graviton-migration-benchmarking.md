> **AWS Cost Warning:** This exercise launches two EC2 instances: one t3.micro (~$0.0104/hr) and one t4g.micro (~$0.0084/hr). Remember to run `terraform destroy` when finished to avoid unexpected charges.

## Prerequisites

- Terraform >= 1.7 installed
- AWS CLI configured with a sandbox account
- Completed Exercise 01 (Your First VPC)

## Learning Objectives

After completing this exercise, you will be able to:

- **Deploy** both x86 (t3.micro) and Graviton (t4g.micro) EC2 instances side by side
- **Apply** architecture-specific AMI filters to select the correct image for each platform
- **Execute** identical benchmark workloads on both architectures for fair comparison
- **Compare** price-performance ratios between x86 and Graviton using CloudWatch metrics

## Why Graviton

AWS Graviton processors are custom ARM-based chips designed by AWS. Graviton3 instances offer up to 25% better compute performance than comparable x86 instances while costing approximately 20% less per hour. For workloads that do not depend on x86-specific binaries (like most web servers, containers, and interpreted languages), Graviton is a straightforward cost optimization.

The migration path is simple for stateless workloads: launch the same AMI family on the arm64 architecture, run the same workload, and compare. The key mistake teams make is using an x86 AMI on a Graviton instance -- the instance will fail to boot. Each architecture requires its own AMI, even when the underlying OS is identical.

## Step 1 -- Create the Project Files

Create the following files in your exercise directory:

### `providers.tf`

```hcl
terraform {
  required_providers {
    aws = {
      source  = "hashicorp/aws"

    }
    tls = {
      source  = "hashicorp/tls"

    }
  }

  region = var.region
}
```

### `variables.tf`

```hcl
variable "region" {
  description = "AWS region"
  type        = string
  default     = "us-east-1"
}

variable "project_name" {
  description = "Project name for resource naming and tagging"
  type        = string
  default     = "graviton-bench-lab"
}
```

### `vpc.tf`

```hcl
data "aws_availability_zones" "available" {
  state = "available"
}

# ------------------------------------------------------------------
# VPC: shared network for both x86 and Graviton instances.
# ------------------------------------------------------------------
resource "aws_vpc" "this" {
  cidr_block           = "10.0.0.0/16"
  enable_dns_support   = true
  enable_dns_hostnames = true

  tags = { Name = "${var.project_name}-vpc" }
}

# ------------------------------------------------------------------
# Public subnet: both instances get public IPs for SSH access
# and internet connectivity (to download benchmark tools).
# ------------------------------------------------------------------
resource "aws_subnet" "public" {
  vpc_id                  = aws_vpc.this.id
  cidr_block              = "10.0.1.0/24"
  availability_zone       = data.aws_availability_zones.available.names[0]
  map_public_ip_on_launch = true

  tags = { Name = "${var.project_name}-public-subnet" }
}

# ------------------------------------------------------------------
# Internet Gateway + route table for public subnet.
# ------------------------------------------------------------------
resource "aws_internet_gateway" "this" {
  vpc_id = aws_vpc.this.id

  tags = { Name = "${var.project_name}-igw" }
}

resource "aws_route_table" "public" {
  vpc_id = aws_vpc.this.id

  route {
    cidr_block = "0.0.0.0/0"
    gateway_id = aws_internet_gateway.this.id
  }

  tags = { Name = "${var.project_name}-public-rt" }
}

resource "aws_route_table_association" "public" {
  subnet_id      = aws_subnet.public.id
  route_table_id = aws_route_table.public.id
}
```

### `security.tf`

```hcl
# ------------------------------------------------------------------

resource "aws_security_group" "bench" {

```

### `compute.tf`

```hcl
# ------------------------------------------------------------------
# AMI lookups: separate data sources for x86 and arm64.
# The same Amazon Linux 2023 OS, but different architectures.
# This is the most common migration mistake -- using the wrong AMI.
# ------------------------------------------------------------------
data "aws_ami" "al2023_x86" {
  most_recent = true
  owners      = ["amazon"]

  filter {
    name   = "name"
    values = ["al2023-ami-2023.*-x86_64"]
  }

  filter {
    name   = "architecture"
    values = ["x86_64"]
  }

  filter {
    name   = "state"
    values = ["available"]
  }
}

data "aws_ami" "al2023_arm64" {
  most_recent = true
  owners      = ["amazon"]

  filter {
    name   = "name"
    values = ["al2023-ami-2023.*-arm64"]
  }

  filter {
    name   = "architecture"
    values = ["arm64"]
  }

  filter {
    name   = "state"
    values = ["available"]
  }
}

# ------------------------------------------------------------------
# SSH key pair: shared between both instances.
# ------------------------------------------------------------------
resource "tls_private_key" "this" {
  algorithm = "RSA"
  rsa_bits  = 4096
}

resource "aws_key_pair" "this" {
  key_name   = "${var.project_name}-key"
  public_key = tls_private_key.this.public_key_openssh
}

resource "local_file" "private_key" {
  content         = tls_private_key.this.private_key_pem
  filename        = "${path.module}/bench-key.pem"
  file_permission = "0400"
}

# ------------------------------------------------------------------
# Benchmark user_data: installs sysbench and runs CPU, memory,
# and fileio benchmarks. Results are written to /tmp/bench-*.txt.
# The same script runs on both architectures for a fair comparison.
# ------------------------------------------------------------------
locals {
  bench_user_data = <<-EOF
    #!/bin/bash
    set -e

    # Install benchmark tools
    yum install -y sysbench stress-ng jq

    # ------------------------------------------------------------------
    # CPU benchmark: single-threaded prime number calculation
    # ------------------------------------------------------------------
    echo "=== CPU Benchmark (single-thread) ===" > /tmp/bench-results.txt
    sysbench cpu --cpu-max-prime=20000 --threads=1 run >> /tmp/bench-results.txt 2>&1
    echo "" >> /tmp/bench-results.txt

    # ------------------------------------------------------------------
    # CPU benchmark: multi-threaded (2 vCPUs on t3/t4g.micro)
    # ------------------------------------------------------------------
    echo "=== CPU Benchmark (2 threads) ===" >> /tmp/bench-results.txt
    sysbench cpu --cpu-max-prime=20000 --threads=2 run >> /tmp/bench-results.txt 2>&1
    echo "" >> /tmp/bench-results.txt

    # ------------------------------------------------------------------
    # Memory benchmark: sequential read/write throughput
    # ------------------------------------------------------------------
    echo "=== Memory Benchmark ===" >> /tmp/bench-results.txt
    sysbench memory --memory-block-size=1M --memory-total-size=10G run >> /tmp/bench-results.txt 2>&1
    echo "" >> /tmp/bench-results.txt

    # ------------------------------------------------------------------
    # Record instance metadata for identification
    # ------------------------------------------------------------------
    echo "=== Instance Info ===" >> /tmp/bench-results.txt
    TOKEN=$(curl -s -X PUT "http://169.254.169.254/latest/api/token" -H "X-aws-ec2-metadata-token-ttl-seconds: 60")
    INSTANCE_TYPE=$(curl -s -H "X-aws-ec2-metadata-token: $TOKEN" http://169.254.169.254/latest/meta-data/instance-type)
    ARCH=$(uname -m)
    echo "Instance type: $INSTANCE_TYPE" >> /tmp/bench-results.txt
    echo "Architecture: $ARCH" >> /tmp/bench-results.txt
    echo "Date: $(date -u)" >> /tmp/bench-results.txt

    echo "Benchmarks complete" > /tmp/bench-done.txt
  EOF
}

# ------------------------------------------------------------------
# x86 instance: t3.micro (Intel/AMD x86_64)
# Baseline for price-performance comparison.
# ------------------------------------------------------------------
resource "aws_instance" "x86" {
  ami                    = data.aws_ami.al2023_x86.id
  instance_type          = "t3.micro"
  subnet_id              = aws_subnet.public.id
  vpc_security_group_ids = [aws_security_group.bench.id]
  key_name               = aws_key_pair.this.key_name

  user_data = locals.bench_user_data

  tags = {
    Name         = "${var.project_name}-x86"
    Architecture = "x86_64"
    InstanceType = "t3.micro"
  }
}

# ------------------------------------------------------------------
# Graviton instance: t4g.micro (AWS Graviton arm64)
# ~20% cheaper than t3.micro with comparable or better performance.
# ------------------------------------------------------------------
resource "aws_instance" "graviton" {
  ami                    = data.aws_ami.al2023_arm64.id
  instance_type          = "t4g.micro"
  subnet_id              = aws_subnet.public.id
  vpc_security_group_ids = [aws_security_group.bench.id]
  key_name               = aws_key_pair.this.key_name

  user_data = locals.bench_user_data

  tags = {
    Name         = "${var.project_name}-graviton"
    Architecture = "arm64"
    InstanceType = "t4g.micro"
  }
}
```

> **Best Practice:** When migrating to Graviton, start with stateless workloads (web servers, API servers, batch jobs) and always filter AMIs by architecture. Running a production Graviton migration? Use a canary deployment: route 5% of traffic to Graviton instances behind the same ALB and compare error rates and latency before shifting all traffic.

## Common Mistakes

### 1. Using an x86 AMI on a Graviton instance

The most common migration mistake: specifying an x86_64 AMI for a Graviton (arm64) instance type.

**Wrong -- x86 AMI on arm64 instance:**

```hcl
data "aws_ami" "al2023" {
  most_recent = true
  owners      = ["amazon"]

  filter {
    name   = "name"
    values = ["al2023-ami-2023.*-x86_64"]   # x86 AMI!
  }
}

resource "aws_instance" "graviton" {
  ami           = data.aws_ami.al2023.id
  instance_type = "t4g.micro"               # arm64 instance!
}
```

**What happens:** The instance fails to start. In the EC2 console, the system log shows:

```
Client.UnsupportedInstanceTypeForAmi: The instance type 't4g.micro'
is not supported by the AMI 'ami-0abc123...' with an architecture of 'x86_64'.
```

**Fix -- use architecture-specific AMI data sources:**

```hcl
data "aws_ami" "al2023_arm64" {
  most_recent = true
  owners      = ["amazon"]

  filter {
    name   = "name"
    values = ["al2023-ami-2023.*-arm64"]
  }

  filter {
    name   = "architecture"
    values = ["arm64"]
  }
}

resource "aws_instance" "graviton" {
  ami           = data.aws_ami.al2023_arm64.id
  instance_type = "t4g.micro"
}
```

### 2. Not filtering AMI by architecture

If you use a generic AMI filter without specifying the architecture, `most_recent = true` may return an x86 AMI even when you need arm64 (or vice versa), because the x86 AMI might have a newer publish date.

**Wrong -- no architecture filter:**

```hcl
data "aws_ami" "al2023" {
  most_recent = true
  owners      = ["amazon"]

  filter {
    name   = "name"
    values = ["al2023-ami-2023.*"]   # Matches both x86 and arm64!
  }
}
```

**What happens:** The data source returns whichever AMI was published most recently, regardless of architecture. If the x86 AMI was updated an hour after the arm64 AMI, you silently get the wrong architecture.

**Fix -- always include an explicit architecture filter:**

```hcl
data "aws_ami" "al2023_arm64" {
  most_recent = true
  owners      = ["amazon"]

  filter {
    name   = "name"
    values = ["al2023-ami-2023.*-arm64"]
  }

  filter {
    name   = "architecture"
    values = ["arm64"]
  }
}
```

### 3. Assuming all software is arm64-compatible

Some third-party binaries and packages are only available for x86_64. If your user_data script installs a binary that does not have an arm64 build, the installation will fail silently or the binary will not execute.

**What happens:** `yum install` succeeds for OS packages (Amazon Linux repos have arm64 builds), but a manually downloaded binary from GitHub or a vendor fails:

```bash
$ ./vendor-tool
bash: ./vendor-tool: cannot execute binary file: Exec format error
```

**Fix:** Before migrating, audit all binaries and dependencies for arm64 support. Check vendor documentation or use `file <binary>` to verify architecture. For containers, ensure your base images support multi-arch (e.g., `--platform linux/arm64`).

## Summary

- **Graviton** (arm64) instances offer ~20% lower cost and comparable or better performance than x86 equivalents
- **Always filter AMIs by architecture** -- using an x86 AMI on a Graviton instance will fail to boot
- **Use separate data sources** for x86 and arm64 AMIs to avoid architecture mismatches
- **Start migration with stateless workloads** (web servers, API servers, batch jobs) where rollback is simple
- **sysbench** provides consistent CPU, memory, and I/O benchmarks across architectures
- **Price-performance ratio** (events per second / cost per hour) is the key metric for migration decisions
- Not all third-party binaries support arm64 -- audit dependencies before migrating
