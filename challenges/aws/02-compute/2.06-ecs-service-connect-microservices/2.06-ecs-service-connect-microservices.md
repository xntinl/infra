# 2.06 ECS Service Connect para Microservicios

## Metadata
| Campo | Valor |
|-------|-------|
| **Complejidad** | Intermedio |
| **Modalidad** | Ambos |
| **Servicios AWS** | ECS, Service Connect, Cloud Map, CloudWatch, ECR |

## Descripcion

Desplegar 3 o mas microservicios en ECS Fargate usando ECS Service Connect para service discovery y load balancing. Configurar namespaces, health checks, y observar las metricas del proxy Envoy en CloudWatch.

## Objetivos de aprendizaje

- Comprender patrones de service mesh en ECS con Service Connect
- Implementar service discovery con Cloud Map
- Configurar y observar el sidecar Envoy automatico
- Dominar networking de microservicios en ECS

## Requisitos previos

- Conocimientos: Docker, ECS basico (tasks, services), conceptos de microservicios, networking basico
- Herramientas: AWS CLI, Docker, Terraform (opcional), curl para pruebas
- Servicios AWS: Cuenta AWS con permisos para ECS, ECR, Cloud Map, CloudWatch

## Arquitectura

Tres microservicios comunicandose a traves de ECS Service Connect:

- **ECS Cluster (Fargate):** Cluster unico que aloja los tres microservicios.
- **Cloud Map Namespace:** Namespace HTTP (ej: `myapp.local`) que registra los servicios para discovery.
- **Microservicio "API Gateway" (puerto 8080):** Punto de entrada. Recibe requests externos y los enruta al servicio apropiado. Configurado como client-server en Service Connect.
- **Microservicio "Orders" (puerto 8081):** Gestiona pedidos. Se comunica con el servicio "Inventory". Configurado como client-server en Service Connect.
- **Microservicio "Inventory" (puerto 8082):** Gestiona inventario. Solo recibe requests de otros servicios. Configurado como server en Service Connect.
- **Envoy Sidecar:** Inyectado automaticamente por ECS en cada task. Maneja el service discovery, load balancing, retries y circuit breaking entre servicios. Expone metricas a CloudWatch.
- **Flujo:** Cliente -> ALB -> API Gateway -> (via Service Connect/Envoy) -> Orders -> (via Service Connect/Envoy) -> Inventory.

## Pasos sugeridos

1. Crear repositorios ECR y hacer push de las imagenes de los tres microservicios
2. Crear la VPC, subnets y un ALB para exponer el servicio API Gateway
3. Crear el Cloud Map HTTP namespace para el service discovery
4. Crear el cluster ECS y habilitar Service Connect con el namespace creado
5. Crear la task definition del microservicio "Inventory" con la configuracion de Service Connect (portMappings con name y appProtocol)
6. Desplegar el servicio "Inventory" con Service Connect habilitado como server
7. Repetir para el microservicio "Orders", configurandolo como client-server para que pueda descubrir "Inventory"
8. Desplegar el microservicio "API Gateway" como client-server, con el ALB como target
9. Verificar que las llamadas entre servicios funcionan usando los nombres de Service Connect (ej: `http://inventory:8082`)
10. Observar las metricas de Envoy en CloudWatch: request count, latencia, errores, conexiones activas

## Validacion

- [ ] Los tres microservicios estan corriendo en ECS Fargate
- [ ] El Cloud Map namespace muestra los servicios registrados
- [ ] El servicio API Gateway puede llamar a Orders usando el nombre de Service Connect
- [ ] Orders puede llamar a Inventory usando el nombre de Service Connect
- [ ] Los Envoy sidecars estan inyectados automaticamente en cada task
- [ ] Las metricas de Envoy aparecen en CloudWatch (RequestCount, TargetResponseTime, etc.)
- [ ] Los health checks de Service Connect estan funcionando
- [ ] El flujo end-to-end funciona: cliente -> ALB -> API Gateway -> Orders -> Inventory
