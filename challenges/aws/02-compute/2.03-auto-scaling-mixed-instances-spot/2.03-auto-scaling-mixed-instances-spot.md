## Why This Matters

EC2 Spot instances cost up to 90% less than On-Demand, but AWS can reclaim them with two minutes' notice when capacity is needed elsewhere. Naive Spot usage -- picking a single instance type in a single AZ -- is fragile: when that specific pool runs low, all your instances get interrupted simultaneously.

Mixed instances policies solve this by spreading your fleet across multiple instance types, sizes, and AZs. Instead of requesting 10 `t3.medium` Spot instances, you request a mix of `t3.medium`, `t3a.medium`, `t2.medium`, and `m5.large` across three AZs. Each combination draws from a different Spot capacity pool, making it statistically unlikely that all pools experience interruptions at the same time. Combined with capacity rebalancing (which proactively launches replacements when AWS signals an upcoming interruption), you get near-On-Demand reliability at a fraction of the cost.

> **Best Practice:** Diversify across at least 3 instance families and 2 sizes per family for Spot workloads. A mixed instances policy with only one instance type is no better than a standard Spot request.

## Best Practices

1. **Use `capacity-optimized` Spot allocation.** This strategy selects Spot pools with the highest available capacity, reducing the chance of interruptions. The older `lowest-price` strategy saves a few cents but dramatically increases interruption rates.

2. **Set a non-zero `on_demand_base_capacity`.** A floor of 1-2 On-Demand instances ensures your service stays available even during a region-wide Spot capacity crunch. The remaining capacity fills with Spot at the configured percentage.

3. **Enable `capacity_rebalance`.** When AWS detects an instance is at elevated risk of interruption, capacity rebalancing proactively launches a replacement before the old instance is terminated. Without it, you rely on the two-minute warning, which may not be enough time for graceful draining.

4. **Spread across 3+ AZs.** ASG distributes instances evenly across the AZs you specify. More AZs means more Spot pools and better resilience. Never use fewer than 2 AZs for any production workload.

## Spot the Bug

A colleague wrote this mixed instances policy but Spot instances are never launched -- the ASG only runs On-Demand instances. **What is wrong?**

```hcl
resource "aws_autoscaling_group" "this" {
  name                = "my-asg"
  min_size            = 2
  max_size            = 8
  desired_capacity    = 4
  vpc_zone_identifier = [for s in aws_subnet.public : s.id]

  mixed_instances_policy {
    launch_template {
      launch_template_specification {
        launch_template_id = aws_launch_template.this.id
        version            = "$Latest"
      }

      override {
        instance_type = "t3.micro"
      }
    }

    instances_distribution {
      on_demand_base_capacity                  = 4    # <-- BUG
      on_demand_percentage_above_base_capacity = 100  # <-- BUG
      spot_allocation_strategy                 = "capacity-optimized"
    }
  }
}
```

<details>
<summary>Explain the bug</summary>

Two problems combine to eliminate all Spot capacity:

1. **`on_demand_base_capacity = 4`** -- this guarantees the first 4 instances are On-Demand. Since `desired_capacity` is also 4, the On-Demand base consumes the entire fleet.

2. **`on_demand_percentage_above_base_capacity = 100`** -- even if you scale beyond 4 instances, 100% of the additional capacity is also On-Demand. Spot is never used regardless of fleet size.

The fix depends on your cost/availability trade-off:

```hcl
instances_distribution {
  on_demand_base_capacity                  = 1   # 1 guaranteed On-Demand instance
  on_demand_percentage_above_base_capacity = 25  # 75% of remaining capacity is Spot
  spot_allocation_strategy                 = "capacity-optimized"
}
```

With `desired_capacity = 4`: 1 On-Demand (base) + 3 remaining, of which 75% (2.25, rounded to 2) are Spot and 25% (0.75, rounded to 1) is On-Demand. Result: 2 On-Demand + 2 Spot.

</details>

### Step 1 -- Confirm ASG is running with mixed instances

```bash
aws autoscaling describe-auto-scaling-groups \
  --auto-scaling-group-names asg-spot-lab-asg \
  --query "AutoScalingGroups[0].{Desired:DesiredCapacity,Min:MinSize,Max:MaxSize,AZs:AvailabilityZones}" \
  --output table
```

Expected: desired = 4, min = 2, max = 8, spanning 3 AZs.

### Step 2 -- Verify instance diversity

```bash
aws ec2 describe-instances \
  --filters "Name=tag:Name,Values=asg-spot-lab-instance" "Name=instance-state-name,Values=running" \
  --query "Reservations[].Instances[].{ID:InstanceId,Type:InstanceType,AZ:Placement.AvailabilityZone,Lifecycle:InstanceLifecycle}" \
  --output table
```

Expected: a mix of instance types across AZs with some showing `Lifecycle = spot` and others empty (On-Demand).

### Step 3 -- Verify capacity rebalancing is enabled

```bash

  --auto-scaling-group-names asg-spot-lab-asg \
```

## Summary

- **Mixed instances policies** spread Spot requests across multiple instance types and AZs, reducing correlated interruption risk
- **`on_demand_base_capacity`** guarantees a minimum number of On-Demand instances as a reliability floor
- **`capacity-optimized`** allocation selects Spot pools with the most available capacity, reducing interruptions compared to `lowest-price`
- **Capacity rebalancing** proactively replaces at-risk Spot instances before the two-minute termination warning
- **Target tracking policies** automatically adjust ASG size based on CloudWatch metrics without manual alarm management
