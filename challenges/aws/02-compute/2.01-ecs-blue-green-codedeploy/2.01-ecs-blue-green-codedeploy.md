## Why This Matters

Deploying new container versions to production is the most dangerous moment in a release cycle. A bad image can take down your service in seconds if you do a rolling update with no safety net. Blue/green deployment eliminates this risk: CodeDeploy launches a completely new set of tasks (the "green" target group), runs health checks against a test listener, and only shifts production traffic after the new tasks are healthy. If anything goes wrong, traffic shifts back to the original "blue" target group in seconds -- no manual intervention, no downtime.

ECS Fargate with CodeDeploy is AWS's recommended deployment strategy for production container workloads. The ALB runs two listeners: a production listener on port 80 that serves real users, and a test listener on port 8080 that lets you validate the green deployment before it goes live. Combined with `deployment_circuit_breaker`, which automatically rolls back if tasks fail to stabilize, you get a deployment pipeline that is both safe and fully automated.

> **Best Practice:** Always enable `deployment_circuit_breaker` with `rollback = true` on ECS services. Without it, a bad task definition will loop indefinitely trying to start unhealthy containers, burning Fargate compute costs while your service degrades.

## Best Practices

1. **Use `target_type = "ip"` for Fargate target groups.** Fargate tasks use `awsvpc` networking where each task gets its own ENI. The ALB must register tasks by IP address, not instance ID. Using `"instance"` causes silent registration failures.

2. **Set `ignore_changes` on task_definition and load_balancer in the ECS service.** After the initial `terraform apply`, CodeDeploy manages task definition updates and target group swaps. Without `ignore_changes`, Terraform will try to revert CodeDeploy's changes on the next plan.

3. **Always configure a termination wait time on blue instances.** Setting `termination_wait_time_in_minutes = 5` gives you a window to manually roll back after traffic shifts to green. Setting it to 0 terminates old tasks immediately with no safety net.

4. **Use `deployment_circuit_breaker` as a second safety net.** Even with CodeDeploy blue/green, the circuit breaker catches task-level failures (like OOM kills or missing environment variables) that might not trigger a CodeDeploy rollback.

## Spot the Bug

A colleague wrote this ECS task definition for Fargate but the tasks fail to start with error `CannotPullContainerError`. The ECR image exists and the execution role has the correct managed policy attached. **What is wrong?**

```hcl
resource "aws_ecs_task_definition" "this" {
  family                   = "my-app"
  network_mode             = "awsvpc"
  requires_compatibilities = ["FARGATE"]
  cpu                      = "256"
  memory                   = "512"
  execution_role_arn       = aws_iam_role.ecs_execution.arn

  container_definitions = jsonencode([{
    name  = "app"
    image = "${aws_ecr_repository.this.repository_url}:latest"
    portMappings = [{
      containerPort = 80
    }]
  }])
}

resource "aws_ecs_service" "this" {
  name            = "my-app"
  cluster         = aws_ecs_cluster.this.id
  task_definition = aws_ecs_task_definition.this.arn
  launch_type     = "FARGATE"
  desired_count   = 2

  network_configuration {
    subnets          = [for s in aws_subnet.public : s.id]   # <-- BUG
    assign_public_ip = false                                  # <-- BUG
    security_groups  = [aws_security_group.tasks.id]
  }
}
```

<details>
<summary>Explain the bug</summary>

The tasks are placed in **public subnets** but `assign_public_ip` is set to `false`. Without a public IP, Fargate tasks in a public subnet have no route to the internet (there is no NAT Gateway in a public subnet -- the IGW requires a public IP for outbound traffic). The tasks cannot pull the container image from ECR.

There are two correct approaches:

**Option A -- Public subnets with public IP (for development only):**

```hcl
network_configuration {
  subnets          = [for s in aws_subnet.public : s.id]
  assign_public_ip = true
  security_groups  = [aws_security_group.tasks.id]
}
```

**Option B -- Private subnets with NAT Gateway (recommended for production):**

```hcl
network_configuration {
  subnets          = [for s in aws_subnet.private : s.id]
  assign_public_ip = false
  security_groups  = [aws_security_group.tasks.id]
}
```

Option B is preferred because production tasks should not have public IPs. The NAT Gateway in the public subnet provides outbound internet access for image pulls and API calls.

## Summary

- **Blue/green deployment** launches a complete replacement task set before shifting traffic, enabling instant rollback
- **Dual ALB listeners** (prod on 80, test on 8080) let you validate green tasks before real users see them
- **`target_type = "ip"`** is mandatory for Fargate target groups because tasks use `awsvpc` networking
- **`deployment_circuit_breaker`** catches task-level failures that CodeDeploy might not detect
- **`ignore_changes`** on `task_definition` and `load_balancer` prevents Terraform from reverting CodeDeploy-managed state
- **CodeDeploy** orchestrates the full lifecycle: deploy green, health-check, shift traffic, terminate blue
