# 2.02 Lambda Custom Runtime con Rust o Go

## Metadata
| Campo | Valor |
|-------|-------|
| **Complejidad** | Avanzado |
| **Modalidad** | Ambos |
| **Servicios AWS** | Lambda, Lambda Layers, Lambda Extensions, API Gateway, CloudWatch |

## Descripcion

Construir una funcion Lambda usando un custom runtime en la base `provided.al2023`. Empaquetarla con un bootstrap binary, crear un Lambda Layer para librerias compartidas, implementar una Lambda Extension que envie telemetria a un endpoint externo, y probar el rendimiento de cold start.

## Objetivos de aprendizaje

- Comprender el ciclo de vida del entorno de ejecucion de Lambda (init, invoke, shutdown)
- Dominar la Lambda Runtime API para custom runtimes
- Implementar la Lambda Extension API para observabilidad
- Gestionar Lambda Layers y optimizar cold starts

## Requisitos previos

- Conocimientos: Rust o Go (compilacion a binarios nativos), conceptos de Lambda, HTTP APIs
- Herramientas: Rust toolchain (con target x86_64-unknown-linux-musl o aarch64) o Go compiler, AWS CLI, cargo-lambda (para Rust), SAM CLI (opcional)
- Servicios AWS: Cuenta AWS con permisos para Lambda, API Gateway, CloudWatch

## Arquitectura

Funcion Lambda con custom runtime, layers y extensions:

- **Lambda Function (provided.al2023):** Ejecuta un binario compilado llamado `bootstrap` que implementa el loop del Runtime API. El binario se compila estáticamente para Linux (x86_64 o arm64).
- **Lambda Layer:** Contiene librerias compartidas o archivos de configuracion que pueden ser reutilizados por multiples funciones. Se monta en `/opt/` dentro del entorno de ejecucion.
- **Lambda Extension:** Un proceso separado que corre junto a la funcion. Implementa la Extensions API para registrarse y recibir eventos del ciclo de vida (INVOKE, SHUTDOWN). Envia telemetria (metricas, logs) a un endpoint externo.
- **API Gateway:** Expone la funcion Lambda como un endpoint HTTP para pruebas de invocacion.
- **CloudWatch:** Captura logs y metricas para analisis de cold start y performance.
- **Flujo:** API Gateway -> Lambda (init: bootstrap + extension arrancan) -> Runtime API loop (poll for invocations -> process -> respond) -> Extension envia telemetria.

## Pasos sugeridos

1. Crear el proyecto en Rust (con cargo-lambda) o Go y escribir la logica de la funcion
2. Implementar el loop del Runtime API: registrarse, obtener invocacion, procesar, enviar respuesta
3. Compilar el binario como `bootstrap` para la arquitectura target (x86_64 o arm64 Linux)
4. Empaquetar y desplegar la funcion Lambda con runtime `provided.al2023`
5. Crear un Lambda Layer con archivos de configuracion o librerias compartidas
6. Asociar el Layer a la funcion y verificar que los archivos estan disponibles en `/opt/`
7. Implementar una Lambda Extension (como un binario separado en `/opt/extensions/`)
8. Configurar la Extension para registrarse con la Extensions API y capturar eventos de ciclo de vida
9. Crear un API Gateway HTTP API y conectarlo a la funcion Lambda
10. Ejecutar pruebas de rendimiento: medir cold starts con y sin Provisioned Concurrency, comparar con runtimes gestionados

## Validacion

- [ ] La funcion Lambda se ejecuta correctamente con el custom runtime provided.al2023
- [ ] El binario bootstrap implementa el Runtime API loop correctamente
- [ ] El Lambda Layer esta asociado y los archivos son accesibles en /opt/
- [ ] La Lambda Extension se registra y recibe eventos de ciclo de vida
- [ ] La Extension envia telemetria al endpoint configurado
- [ ] API Gateway invoca la funcion correctamente y retorna respuestas
- [ ] Los logs de CloudWatch muestran el ciclo de vida completo (init, invoke, shutdown)
- [ ] Las metricas de cold start son medibles y comparables
