# 2.08 Lambda con Container Images y Optimizacion

## Metadata
| Campo | Valor |
|-------|-------|
| **Complejidad** | Intermedio |
| **Modalidad** | Ambos |
| **Servicios AWS** | Lambda, ECR, Docker, CloudWatch |

## Descripcion

Empaquetar una funcion Lambda como una imagen de contenedor (hasta 10GB). Usar multi-stage Docker builds para minimizar el tamano de la imagen, implementar el Lambda Runtime Interface Client, desplegar via ECR, y hacer benchmarking de rendimiento comparado con equivalentes empaquetados como ZIP.

## Objetivos de aprendizaje

- Comprender Lambda basado en contenedores y sus casos de uso
- Optimizar imagenes Docker con multi-stage builds
- Implementar el Runtime Interface Client (RIC) y Runtime Interface Emulator (RIE)
- Medir el impacto del tamano de imagen en el cold start

## Requisitos previos

- Conocimientos: Docker (Dockerfile, multi-stage builds), Lambda basico, conceptos de registros de contenedores
- Herramientas: Docker Desktop, AWS CLI, Terraform (opcional)
- Servicios AWS: Cuenta AWS con permisos para Lambda, ECR, CloudWatch

## Arquitectura

Funcion Lambda empaquetada como imagen de contenedor con optimizacion de tamano:

- **ECR Repository:** Almacena las imagenes de contenedor de la funcion Lambda. Se crean multiples tags para comparar: imagen sin optimizar, imagen con multi-stage build, imagen con diferentes bases.
- **Lambda Function (Container Image):** En lugar de un ZIP, la funcion usa una imagen de contenedor desde ECR. La imagen debe incluir el Lambda Runtime Interface Client (RIC) que implementa el Runtime API.
- **Dockerfile Multi-Stage:**
  - Stage 1 (builder): Instala dependencias de compilacion, descarga librerias, compila el codigo
  - Stage 2 (runtime): Imagen base minima (ej: `public.ecr.aws/lambda/python:3.12` o custom), copia solo los artefactos necesarios del stage 1
- **Lambda Runtime Interface Emulator (RIE):** Permite probar la imagen localmente antes de desplegarla, emulando el entorno de ejecucion de Lambda.
- **Benchmarking:** Se despliegan dos versiones de la misma funcion (ZIP y container) y se comparan metricas de cold start, duracion y memoria en CloudWatch.

## Pasos sugeridos

1. Crear una funcion Lambda con dependencias pesadas (ej: numpy, pandas, o un modelo ML) que justifiquen el uso de container image
2. Escribir un Dockerfile basico (sin optimizar) usando la imagen base oficial de Lambda
3. Implementar el Runtime Interface Client en el codigo de la funcion
4. Construir la imagen y probarla localmente usando el Runtime Interface Emulator (RIE)
5. Optimizar el Dockerfile con multi-stage build: compilar dependencias en un stage y copiar solo lo necesario al stage final
6. Comparar tamanos de imagen: sin optimizar vs multi-stage vs con imagen base distroless/minimalista
7. Crear el repositorio ECR y hacer push de las imagenes (cada variante con un tag diferente)
8. Desplegar la funcion Lambda usando la imagen de contenedor optimizada
9. Desplegar la misma funcion como ZIP package (si el tamano lo permite) para comparacion
10. Ejecutar benchmarks: invocar ambas versiones multiples veces, medir cold starts, duracion, y consumo de memoria en CloudWatch y X-Ray

## Validacion

- [ ] La imagen de contenedor se construye correctamente con multi-stage build
- [ ] La funcion Lambda funciona correctamente con la imagen de contenedor desde ECR
- [ ] El Runtime Interface Client esta correctamente implementado
- [ ] La imagen optimizada es significativamente mas pequena que la version sin optimizar
- [ ] Las pruebas locales con RIE funcionan antes del despliegue
- [ ] Las metricas de CloudWatch muestran las diferencias de cold start entre versiones
- [ ] El benchmarking compara ZIP vs container image con resultados documentados
- [ ] La funcion maneja correctamente las invocaciones concurrentes
