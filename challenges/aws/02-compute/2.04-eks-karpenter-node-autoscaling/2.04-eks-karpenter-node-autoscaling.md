# 2.04 EKS con Karpenter Node Autoscaling

## Metadata
| Campo | Valor |
|-------|-------|
| **Complejidad** | Experto |
| **Modalidad** | Terraform (EKS Blueprints) |
| **Servicios AWS** | EKS, Karpenter, EC2 (Spot/On-Demand), IRSA, Fargate |

## Descripcion

Desplegar un cluster EKS con Karpenter (en lugar de Cluster Autoscaler). Definir NodePools con constraints de instancias, configurar provisioning de Spot y On-Demand, implementar pod disruption budgets, y observar el comportamiento de consolidacion de Karpenter bajo carga variable.

## Objetivos de aprendizaje

- Dominar el autoscaling de nodos en Kubernetes con Karpenter
- Configurar NodePools y EC2NodeClasses para control granular de instancias
- Aplicar mejores practicas de Spot en Kubernetes
- Comprender bin-packing y consolidacion automatica de nodos

## Requisitos previos

- Conocimientos: Kubernetes intermedio (deployments, pods, scheduling), EC2 instance types, IAM roles, Helm
- Herramientas: AWS CLI, Terraform, kubectl, Helm, eksctl (opcional)
- Servicios AWS: Cuenta AWS con permisos para EKS, EC2, IAM, SQS (para interrupciones Spot)

## Arquitectura

Cluster EKS con Karpenter gestionando el autoscaling de nodos:

- **EKS Cluster:** Cluster Kubernetes gestionado. El control plane corre en AWS. Karpenter se despliega como un pod en un node group gestionado o Fargate.
- **Karpenter Controller:** Corre en el cluster y observa pods pending. Cuando detecta pods que no pueden ser schedulados, provisiona nodos automaticamente segun los constraints definidos.
- **NodePool (antes Provisioner):** Define las restricciones para los nodos: tipos de instancia permitidos, AZs, arquitectura (amd64/arm64), capacity type (on-demand/spot), limites de recursos, y politica de consolidation.
- **EC2NodeClass (antes AWSNodeTemplate):** Define parametros especificos de AWS: AMI family, subnets, security groups, instance profile, block device mappings, user data.
- **Pod Disruption Budgets (PDB):** Protegen las aplicaciones durante la consolidacion de nodos, asegurando que siempre haya un minimo de replicas disponibles.
- **Consolidacion:** Karpenter continuamente evalua si los nodos existentes estan subutilizados y los reemplaza con nodos mas eficientes (bin-packing).
- **SQS Queue:** Recibe notificaciones de interrupciones Spot e instancias scheduled para mantenimiento. Karpenter las procesa para hacer drain de nodos de forma anticipada.

## Pasos sugeridos

1. Crear la infraestructura base con Terraform: VPC, subnets, EKS cluster usando el modulo terraform-aws-eks o EKS Blueprints
2. Crear un node group pequeno (o Fargate profile) para alojar los componentes del sistema (Karpenter, CoreDNS, kube-proxy)
3. Configurar IRSA (IAM Roles for Service Accounts) para Karpenter con los permisos necesarios (EC2, IAM, SSM, SQS, EKS)
4. Instalar Karpenter via Helm chart en el cluster
5. Crear un EC2NodeClass definiendo AMI family (AL2023), subnets, security groups y tags
6. Crear un NodePool con constraints: multiples tipos de instancia, ambas arquitecturas (amd64), capacity types (spot preferido, on-demand como fallback), y limites de CPU/memoria
7. Desplegar una aplicacion de prueba con requests de recursos definidos y multiples replicas
8. Observar como Karpenter provisiona nodos optimizados para los pods pending
9. Crear Pod Disruption Budgets para la aplicacion y habilitar consolidacion en el NodePool
10. Escalar la aplicacion arriba y abajo, observando el bin-packing y la consolidacion automatica

## Validacion

- [ ] El cluster EKS esta corriendo con Karpenter instalado y funcional
- [ ] Karpenter provisiona nodos automaticamente cuando hay pods pending
- [ ] Los nodos provisionados cumplen con los constraints del NodePool (tipos, AZs, capacity type)
- [ ] Se utilizan instancias Spot cuando estan disponibles
- [ ] Pod Disruption Budgets son respetados durante la consolidacion
- [ ] Karpenter consolida nodos subutilizados automaticamente
- [ ] Al reducir la carga, Karpenter elimina nodos innecesarios en el tiempo configurado
- [ ] Las interrupciones Spot son manejadas correctamente via la SQS queue
- [ ] Los logs de Karpenter muestran las decisiones de scheduling claramente
