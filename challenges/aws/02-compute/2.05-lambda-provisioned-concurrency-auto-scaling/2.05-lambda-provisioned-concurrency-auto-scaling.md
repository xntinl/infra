After completing this exercise, you will be able to:

## Why Provisioned Concurrency Matters

Every Lambda invocation runs inside a micro-VM called an execution environment. When a request arrives and no warm environment is available, Lambda must create one from scratch -- downloading your code, initializing the runtime, and running your handler's initialization code. This "cold start" adds latency: typically 200-500ms for Python/Node.js, 1-3 seconds for Java/.NET, and up to 10 seconds for functions in a VPC (though VPC cold starts have improved dramatically with Hyperplane ENI caching).

For most workloads, occasional cold starts are acceptable. But for latency-sensitive APIs, payment processing, or real-time data pipelines, a 2-second spike in P99 latency is a real problem. Provisioned concurrency solves this by keeping a pool of pre-initialized execution environments ready. When a request arrives, it is served by a warm environment with no initialization delay -- the same fast response time as a warmed-up function, every single time.

The trade-off is cost: you pay for provisioned concurrency whether or not it is used ($0.0000041667 per GB-second). This is why auto scaling is essential. Instead of provisioning a fixed number of environments 24/7, you attach an Application Auto Scaling policy that scales provisioned concurrency up during peak hours and down during off-peak, matching capacity to actual demand.

> **Best Practice:** Auto-scale provisioned concurrency on a schedule or based on utilization metrics, rather than leaving it always-on at peak capacity. A function that needs 100 concurrent executions during business hours but only 5 at night will waste 95% of its provisioned concurrency cost without scaling.

### `lambda_code.tf`

```hcl
# Create a simple Python handler inline. In production you would
# build and package your code separately.
resource "local_file" "lambda_handler" {
  filename = "${path.module}/lambda/handler.py"
  content  = <<-PYTHON
    import json
    import time
    import os

    # Simulate initialization work (this runs during cold start)
    print("Initializing function -- this is the cold start cost")
    INIT_TIME = time.time()

    def handler(event, context):
        current_time = time.time()
        uptime = round(current_time - INIT_TIME, 2)

        return {
            "statusCode": 200,
            "body": json.dumps({
                "message": "Hello from Lambda!",
                "uptime_seconds": uptime,
                "function_version": context.function_version,
                "cold_start": uptime < 1,
                "memory_mb": context.memory_limit_in_mb,
                "region": os.environ.get("AWS_REGION", "unknown")
            })
        }
  PYTHON
}

data "archive_file" "lambda" {
  type        = "zip"
  source_dir  = "${path.module}/lambda"
  output_path = "${path.module}/lambda.zip"

  depends_on = [local_file.lambda_handler]
}
```

### `iam.tf`

```hcl
# IAM role for the Lambda function. Uses data "aws_iam_policy_document"
# for the trust policy as required by best practices.
data "aws_iam_policy_document" "lambda_assume" {
  statement {
    actions = ["sts:AssumeRole"]

    principals {
      type        = "Service"
      identifiers = ["lambda.amazonaws.com"]
    }
  }
}

resource "aws_iam_role" "this" {
  name               = "${var.project_name}-role"
  assume_role_policy = data.aws_iam_policy_document.lambda_assume.json

  tags = { Name = "${var.project_name}-role" }
}

# Attach the basic execution role for CloudWatch Logs access.
resource "aws_iam_role_policy_attachment" "basic_execution" {
  role       = aws_iam_role.this.name
  policy_arn = "arn:aws:iam::aws:policy/service-role/AWSLambdaBasicExecutionRole"
}
```

### `lambda.tf`

```hcl
# CloudWatch Log Group for Lambda logs (explicit creation with retention).
resource "aws_cloudwatch_log_group" "this" {
  name              = "/aws/lambda/${var.project_name}"
  retention_in_days = 14

  tags = { Name = "${var.project_name}-logs" }
}

# The Lambda function itself. Setting publish = true creates a new
# version on each update, which is required for alias-based deployments
# and provisioned concurrency.
resource "aws_lambda_function" "this" {
  function_name    = var.project_name
  role             = aws_iam_role.this.arn
  handler          = "handler.handler"
  runtime          = "python3.12"
  filename         = data.archive_file.lambda.output_path
  source_code_hash = data.archive_file.lambda.output_base64sha256
  memory_size      = 128
  timeout          = 10
  publish          = true

  tags = { Name = "${var.project_name}-function" }

  depends_on = [aws_cloudwatch_log_group.this]
}

# Lambda alias -- provisioned concurrency attaches to an alias, not $LATEST.
# The alias points to a specific published version for stable deployments.
resource "aws_lambda_alias" "live" {
  name             = "live"
  function_name    = aws_lambda_function.this.function_name
  function_version = aws_lambda_function.this.version
  description      = "Live alias with provisioned concurrency"
}

# Provisioned concurrency on the alias -- keeps N execution environments
# warm and initialized, eliminating cold starts for the first N concurrent
# requests. Additional requests beyond N still work but may cold-start.
resource "aws_lambda_provisioned_concurrency_config" "this" {
  function_name                  = aws_lambda_function.this.function_name
  qualifier                      = aws_lambda_alias.live.name
  provisioned_concurrent_executions = var.provisioned_min
}
```

### `autoscaling.tf`

```hcl
# Application Auto Scaling target -- registers the Lambda alias as a
# scalable resource. This tells Auto Scaling what to scale and its bounds.
resource "aws_appautoscaling_target" "this" {
  service_namespace  = "lambda"
  resource_id        = "function:${aws_lambda_function.this.function_name}:${aws_lambda_alias.live.name}"
  scalable_dimension = "lambda:function:ProvisionedConcurrency"
  min_capacity       = var.provisioned_min
  max_capacity       = var.provisioned_max
}

# Target tracking policy -- scales provisioned concurrency to maintain
# the target utilization. When utilization exceeds the target, Auto Scaling
# increases provisioned concurrency. When it drops, it scales down.
resource "aws_appautoscaling_policy" "this" {
  name               = "${var.project_name}-utilization"
  service_namespace  = aws_appautoscaling_target.this.service_namespace
  resource_id        = aws_appautoscaling_target.this.resource_id
  scalable_dimension = aws_appautoscaling_target.this.scalable_dimension
  policy_type        = "TargetTrackingScaling"

  target_tracking_scaling_policy_configuration {
    predefined_metric_specification {
      predefined_metric_type = "LambdaProvisionedConcurrencyUtilization"
    }
    target_value = var.target_utilization
  }
}

# Scheduled action -- scale up during business hours, scale down at night.
# This preemptively adds capacity before traffic arrives.
resource "aws_appautoscaling_scheduled_action" "scale_up" {
  name               = "${var.project_name}-scale-up"
  service_namespace  = aws_appautoscaling_target.this.service_namespace
  resource_id        = aws_appautoscaling_target.this.resource_id
  scalable_dimension = aws_appautoscaling_target.this.scalable_dimension
  schedule           = "cron(0 8 ? * MON-FRI *)"

  scalable_target_action {
    min_capacity = 5
    max_capacity = var.provisioned_max
  }
}

resource "aws_appautoscaling_scheduled_action" "scale_down" {
  name               = "${var.project_name}-scale-down"
  service_namespace  = aws_appautoscaling_target.this.service_namespace
  resource_id        = aws_appautoscaling_target.this.resource_id
  scalable_dimension = aws_appautoscaling_target.this.scalable_dimension
  schedule           = "cron(0 20 ? * MON-FRI *)"

  scalable_target_action {
    min_capacity = var.provisioned_min
    max_capacity = 5
  }
}
```

## Step 3 -- Invoke the Function and Observe Cold Starts

Invoke the alias directly:

```bash
aws lambda invoke \
  --function-name lambda-pc-lab:live \
  --payload '{}' \
  --cli-binary-format raw-in-base64-out \
  response.json && cat response.json | python3 -m json.tool
```

Expected: the response shows `"cold_start": false` because provisioned concurrency has already initialized the environment.

Now invoke the unaliased `$LATEST` version (which has no provisioned concurrency):

```bash
aws lambda invoke \
  --function-name lambda-pc-lab \
  --payload '{}' \
  --cli-binary-format raw-in-base64-out \
  response.json && cat response.json | python3 -m json.tool
```

Expected: the first invocation shows `"cold_start": true` and a higher latency.

## Best Practices

1. **Attach provisioned concurrency to an alias, not `$LATEST`.** The `$LATEST` version cannot have provisioned concurrency. Use a published version via an alias for stable, versioned deployments.

2. **Set `publish = true` on the Lambda function.** Without publishing, Terraform only updates `$LATEST`, and your alias has nothing to point to.

3. **Combine scheduled scaling with target tracking.** Scheduled actions proactively increase the minimum before predictable traffic spikes (e.g., morning business hours). Target tracking handles unexpected bursts within the scheduled bounds.

4. **Set target utilization to 0.7, not 1.0.** A target of 1.0 means Auto Scaling only reacts after all provisioned environments are in use. By then, overflow invocations are already cold-starting. A target of 0.7 triggers scale-out before you run out of warm capacity.

## Common Mistakes

### 1. Provisioned concurrency on `$LATEST`

Attempting to set provisioned concurrency on the `$LATEST` version fails with an error:

```
InvalidParameterValueException: Provisioned Concurrency is not supported on $LATEST
```

**Wrong:**

```hcl
resource "aws_lambda_provisioned_concurrency_config" "this" {
  function_name                  = aws_lambda_function.this.function_name
  qualifier                      = "$LATEST"
  provisioned_concurrent_executions = 5
}
```

**Fix -- use an alias or a published version number:**

```hcl
resource "aws_lambda_provisioned_concurrency_config" "this" {
  function_name                  = aws_lambda_function.this.function_name
  qualifier                      = aws_lambda_alias.live.name
  provisioned_concurrent_executions = 5
}
```

### 2. Forgetting `publish = true` on the Lambda function

Without `publish = true`, Terraform only updates the `$LATEST` version. Your alias still points to the old published version, so code changes never reach the alias.

**Wrong:**

```hcl
resource "aws_lambda_function" "this" {
  function_name = "my-function"
  # ... no publish attribute
}
```

**What happens:** You update the code, run `terraform apply`, and invocations through the alias still return the old response.

**Fix:**

```hcl
resource "aws_lambda_function" "this" {
  function_name = "my-function"
  publish       = true
  # ...
}
```

### 3. Auto Scaling target_value set to 1.0

Setting the utilization target to 100% means Auto Scaling only adds capacity after all provisioned environments are fully consumed. By then, overflow requests are already experiencing cold starts.

**Wrong:**

```hcl
target_tracking_scaling_policy_configuration {
  target_value = 1.0
}
```

**Fix -- leave headroom for bursts:**

```hcl
target_tracking_scaling_policy_configuration {
  target_value = 0.7
}
```

## Summary

- **Cold starts** happen when Lambda creates a new execution environment (downloading code, initializing runtime, running init code)
- **Provisioned concurrency** pre-initializes a pool of environments on an alias or published version, eliminating cold start latency
- **Application Auto Scaling** dynamically adjusts provisioned concurrency based on utilization metrics, preventing over-provisioning
- **Scheduled actions** proactively scale capacity before predictable traffic patterns (e.g., business hours)
- **Target utilization of 0.7** leaves headroom for bursts; 1.0 triggers scaling too late
- **`publish = true`** is required for versioning; provisioned concurrency cannot be set on `$LATEST`
