# 2.10 EKS con IRSA (IAM Roles for Service Accounts)

## Metadata
| Campo | Valor |
|-------|-------|
| **Complejidad** | Intermedio |
| **Modalidad** | Terraform |
| **Servicios AWS** | EKS, IAM, OIDC, S3, DynamoDB |

## Descripcion

Desplegar un cluster EKS y configurar IRSA para dar a pods especificos permisos AWS granulares. Verificar que pods sin la anotacion no pueden acceder a recursos AWS. Implementar politicas de least-privilege.

## Objetivos de aprendizaje

- Comprender IAM a nivel de pod con IRSA
- Configurar OIDC federation entre EKS e IAM
- Implementar principio de least-privilege en Kubernetes
- Dominar las anotaciones de service accounts para IRSA

## Requisitos previos

- Conocimientos: Kubernetes basico (pods, deployments, service accounts), IAM policies, conceptos de OIDC
- Herramientas: AWS CLI, Terraform, kubectl, eksctl (opcional)
- Servicios AWS: Cuenta AWS con permisos para EKS, IAM, S3, DynamoDB

## Arquitectura

Cluster EKS con IRSA habilitado para acceso granular a servicios AWS:

- **EKS Cluster:** Cluster Kubernetes con el OIDC provider habilitado. El OIDC issuer URL del cluster se registra como un Identity Provider en IAM.
- **OIDC Identity Provider:** Configurado en IAM, permite que los service accounts de Kubernetes asuman IAM roles. La relacion de confianza usa el OIDC issuer del cluster.
- **IAM Role "S3Reader":** Tiene una policy que permite solo `s3:GetObject` y `s3:ListBucket` en un bucket especifico. La trust policy permite que solo el service account `app-s3-reader` en el namespace `app` lo asuma.
- **IAM Role "DynamoDBWriter":** Tiene una policy que permite `dynamodb:PutItem`, `dynamodb:GetItem` en una tabla especifica. Solo el service account `app-dynamo-writer` puede asumirlo.
- **Pod "S3 Reader":** Usa el service account `app-s3-reader` con la anotacion `eks.amazonaws.com/role-arn`. AWS SDK automaticamente usa las credenciales temporales inyectadas.
- **Pod "DynamoDB Writer":** Usa el service account `app-dynamo-writer` con la anotacion correspondiente.
- **Pod "No Access":** Usa el service account default sin anotacion. No puede acceder a ningun recurso AWS.

## Pasos sugeridos

1. Crear el cluster EKS con Terraform, asegurandose de que el OIDC provider esta habilitado
2. Obtener el OIDC issuer URL del cluster y registrarlo como Identity Provider en IAM
3. Crear un bucket S3 y una tabla DynamoDB como recursos de prueba
4. Crear el IAM Role "S3Reader" con una policy de least-privilege para el bucket S3 y una trust policy que referencie el OIDC provider y el service account especifico
5. Crear el IAM Role "DynamoDBWriter" con policy para la tabla DynamoDB y trust policy equivalente
6. Crear los service accounts en Kubernetes con las anotaciones `eks.amazonaws.com/role-arn` apuntando a sus respectivos IAM roles
7. Desplegar el pod "S3 Reader" que intente leer del bucket S3 (debe funcionar) y escribir en DynamoDB (debe fallar)
8. Desplegar el pod "DynamoDB Writer" que intente escribir en DynamoDB (debe funcionar) y leer S3 (debe fallar)
9. Desplegar el pod "No Access" sin anotacion IRSA e intentar acceder a S3 y DynamoDB (ambos deben fallar)
10. Verificar que las credenciales temporales se inyectan correctamente examinando las variables de entorno del pod (AWS_ROLE_ARN, AWS_WEB_IDENTITY_TOKEN_FILE)

## Validacion

- [ ] El cluster EKS esta corriendo con el OIDC provider habilitado
- [ ] El Identity Provider OIDC esta registrado en IAM
- [ ] El pod "S3 Reader" puede leer del bucket S3 exitosamente
- [ ] El pod "S3 Reader" NO puede escribir en DynamoDB
- [ ] El pod "DynamoDB Writer" puede escribir en DynamoDB exitosamente
- [ ] El pod "DynamoDB Writer" NO puede leer del bucket S3
- [ ] El pod "No Access" NO puede acceder a ningun recurso AWS
- [ ] Las variables de entorno AWS_ROLE_ARN y AWS_WEB_IDENTITY_TOKEN_FILE estan presentes en los pods con IRSA
- [ ] Las IAM policies siguen el principio de least-privilege (recursos y acciones especificos)
