# 16.02 Pipeline de Promocion entre Entornos

## Metadata
| Campo | Valor |
|-------|-------|
| **Complejidad** | Avanzado |
| **Modalidad** | Terraform + GitHub Actions |
| **Features TF** | Workspaces, `-var-file`, remote state |

## Descripcion

Construye un pipeline donde los cambios de infraestructura fluyan: dev -> staging -> prod. Cada entorno usa la misma version de modulo pero distintos tfvars. Implementa plan-on-PR, apply-on-merge con aprobacion manual para prod.

## Objetivos de aprendizaje

- Disenar un pipeline de promocion de infraestructura entre entornos
- Implementar gates de aprobacion para entornos criticos
- Gestionar tfvars por entorno en un flujo de CI/CD
- Comprender el diseno de pipelines de infraestructura seguros

## Requisitos previos

- Conocimientos: Terraform, GitHub Actions, workflows YAML, Git branching
- Herramientas: Terraform >= 1.5, GitHub (repo con Actions habilitadas), AWS CLI
- Features/Servicios: S3 (backend), IAM (OIDC para GitHub Actions), los servicios del proyecto

## Arquitectura

```
.github/workflows/
  +-- terraform-plan.yml       (trigger: PR abierto/actualizado)
  +-- terraform-apply.yml      (trigger: merge a main)

terraform/
  +-- main.tf
  +-- variables.tf
  +-- envs/
        +-- dev.tfvars
        +-- staging.tfvars
        +-- prod.tfvars

Pipeline Flow:
  PR abierto
    -> plan dev    (automatico, comentario en PR)
    -> plan staging (automatico, comentario en PR)
    -> plan prod   (automatico, comentario en PR)

  Merge a main
    -> apply dev     (automatico)
    -> apply staging (automatico, tras exito de dev)
    -> apply prod    (manual approval requerido)
```

## Pasos sugeridos

1. Crear la estructura de Terraform con variables diferenciadas por entorno en archivos `.tfvars`
2. Configurar el backend S3 con workspace-aware state paths
3. Crear el workflow de GitHub Actions para `terraform plan` que se ejecute en cada PR
4. Implementar la generacion de plan para cada entorno y publicacion del resultado como comentario en el PR
5. Crear el workflow de apply que se ejecute al hacer merge a `main`
6. Configurar la ejecucion secuencial: dev primero, staging despues, prod al final
7. Implementar GitHub Environments con `environment` y `required_reviewers` para la aprobacion manual de prod
8. Configurar OIDC entre GitHub Actions y AWS para evitar credenciales estaticas
9. Agregar protecciones: verificar que el plan no ha cambiado entre el PR y el apply
10. Probar el flujo completo: crear PR, revisar planes, aprobar merge, verificar applies secuenciales

## Validacion

- [ ] El plan se ejecuta automaticamente en cada PR para todos los entornos
- [ ] Los resultados del plan se publican como comentarios en el PR
- [ ] El apply de dev se ejecuta automaticamente al hacer merge
- [ ] El apply de staging se ejecuta solo tras el exito de dev
- [ ] El apply de prod requiere aprobacion manual antes de ejecutarse
- [ ] OIDC se usa para la autenticacion con AWS (sin secrets estaticos)
- [ ] Cada entorno usa su archivo `.tfvars` correspondiente
- [ ] El pipeline falla de forma segura si algun paso falla
