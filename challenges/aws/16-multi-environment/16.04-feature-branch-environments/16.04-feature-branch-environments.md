# 16.04 Entornos Efimeros por Feature Branch

## Metadata
| Campo | Valor |
|-------|-------|
| **Complejidad** | Avanzado |
| **Modalidad** | Terraform + GitHub Actions |
| **Features TF** | Workspaces, dynamic naming, `terraform destroy` |

## Descripcion

Implementa entornos efimeros por feature branch: al abrir un PR, crea un entorno completo con nombrado unico. Al cerrar el PR, destruye todo. Usa Terraform workspaces con nombres derivados del branch y limpieza automatizada.

## Objetivos de aprendizaje

- Implementar entornos efimeros con ciclo de vida automatizado
- Dominar la creacion y destruccion automatica de infraestructura
- Comprender el control de costos con entornos temporales
- Mejorar la experiencia del desarrollador con entornos de preview

## Requisitos previos

- Conocimientos: Terraform, GitHub Actions, Workspaces, Git branching
- Herramientas: Terraform >= 1.5, GitHub (repo con Actions), AWS CLI
- Features/Servicios: S3 (backend), Lambda, API Gateway, DynamoDB (o servicios del proyecto)

## Arquitectura

```
.github/workflows/
  +-- preview-env-create.yml   (trigger: PR opened/synchronized)
  +-- preview-env-destroy.yml  (trigger: PR closed)

terraform/
  +-- main.tf                  (usa terraform.workspace para naming)
  +-- variables.tf
  +-- naming.tf                (local.prefix = "preview-${terraform.workspace}")

Flujo:
  PR abierto (branch: feature/user-auth)
    -> workspace: "preview-feature-user-auth"
    -> Recursos: preview-feature-user-auth-lambda, etc.
    -> Comentario en PR con URLs del entorno

  PR cerrado/mergeado
    -> terraform destroy
    -> terraform workspace delete
    -> Recursos eliminados completamente
```

## Pasos sugeridos

1. Disenar el naming scheme: derivar un nombre de workspace valido desde el nombre del branch
2. Crear la configuracion de Terraform que use `terraform.workspace` para prefijar todos los recursos
3. Asegurar que los nombres de recursos sean unicos y no colisionen con otros entornos
4. Crear el workflow de GitHub Actions que se dispare al abrir un PR
5. Implementar la creacion del workspace y `terraform apply` automatico
6. Publicar un comentario en el PR con las URLs y endpoints del entorno creado
7. Crear el workflow de destruccion que se dispare al cerrar el PR
8. Implementar `terraform destroy` seguido de `terraform workspace delete`
9. Agregar un job programado (cron) para limpiar entornos huerfanos (PRs abandonados)
10. Implementar limites: maximo de entornos simultaneos, tiempo de vida maximo

## Validacion

- [ ] Al abrir un PR se crea automaticamente un entorno completo
- [ ] Los nombres de recursos son unicos por branch y no colisionan
- [ ] El PR recibe un comentario con las URLs del entorno efimero
- [ ] Al cerrar el PR se destruyen todos los recursos del entorno
- [ ] El workspace se elimina tras la destruccion
- [ ] Actualizar el PR (push) actualiza el entorno existente
- [ ] Existe un mecanismo de limpieza para entornos huerfanos
- [ ] Los costos se minimizan con la destruccion automatica
