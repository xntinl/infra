# 16.08 Gestion de Secrets entre Entornos

## Metadata
| Campo | Valor |
|-------|-------|
| **Complejidad** | Avanzado |
| **Modalidad** | Terraform |
| **Features TF** | `sensitive`, Secrets Manager, KMS, SOPS integration |

## Descripcion

Implementa el manejo de secrets para multiples entornos: usa Terraform para crear secrets en Secrets Manager por entorno, integra con SOPS o AWS KMS para encriptar tfvars, y asegura que los secrets nunca aparezcan en el state o plan output usando `sensitive = true`.

## Objetivos de aprendizaje

- Gestionar secrets de forma segura en infraestructura como codigo
- Comprender como proteger valores sensibles en el state y plan output
- Implementar encriptacion at rest para archivos de configuracion
- Disenar patrones de rotacion y acceso a secrets por entorno

## Requisitos previos

- Conocimientos: IAM, KMS, Secrets Manager, encriptacion simetrica
- Herramientas: Terraform >= 1.5, AWS CLI, SOPS (mozilla/sops), cuenta AWS
- Features/Servicios: Secrets Manager, KMS, IAM, S3 (para state encriptado)

## Arquitectura

```
config/
  +-- secrets.dev.enc.yaml       (encriptado con SOPS + KMS)
  +-- secrets.staging.enc.yaml
  +-- secrets.prod.enc.yaml
  +-- .sops.yaml                 (reglas de encriptacion por entorno)

terraform/
  +-- main.tf
  +-- secrets.tf
  |     +-- aws_secretsmanager_secret "this"       (for_each)
  |     +-- aws_secretsmanager_secret_version "this"
  |     +-- variable "db_password" { sensitive = true }
  +-- kms.tf
  |     +-- aws_kms_key "secrets"                  (por entorno)
  |     +-- aws_kms_alias "secrets"

Flujo:
  1. Desarrollador edita secrets con: sops config/secrets.dev.enc.yaml
  2. SOPS encripta con KMS antes de guardar en Git
  3. CI/CD desencripta con SOPS y pasa valores a Terraform
  4. Terraform crea/actualiza secrets en Secrets Manager
  5. Aplicaciones leen secrets desde Secrets Manager en runtime
```

## Pasos sugeridos

1. Crear una KMS key por entorno para encriptacion de secrets
2. Instalar SOPS y configurar `.sops.yaml` con reglas que mapeen archivos a KMS keys
3. Crear archivos de secrets por entorno y encriptarlos con SOPS
4. Implementar variables de Terraform con `sensitive = true` para valores secretos
5. Crear recursos de Secrets Manager con `for_each` para cada secret por entorno
6. Verificar que `terraform plan` no muestra los valores sensibles en el output
7. Implementar un script de CI/CD que desencripte con SOPS y pase valores a Terraform via `-var`
8. Configurar el backend S3 con encriptacion server-side para proteger el state
9. Implementar outputs sensibles con `sensitive = true` para valores que otros modulos necesiten
10. Probar el flujo completo: editar secret -> encriptar -> commit -> CI/CD -> Secrets Manager

## Validacion

- [ ] Los archivos de secrets estan encriptados en el repositorio con SOPS
- [ ] Cada entorno usa su propia KMS key para encriptacion
- [ ] `terraform plan` oculta valores sensibles en el output ("sensitive value")
- [ ] Los secrets se crean correctamente en Secrets Manager por entorno
- [ ] El state de Terraform esta encriptado en S3
- [ ] El flujo de CI/CD desencripta y aplica secrets sin exponerlos en logs
- [ ] Los outputs marcados como `sensitive` no se muestran en consola
- [ ] La rotacion de secrets es posible sin recrear recursos
