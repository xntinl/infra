# 16.01 Workspaces vs Estructura de Directorios: Analisis Comparativo

## Metadata
| Campo | Valor |
|-------|-------|
| **Complejidad** | Intermedio |
| **Modalidad** | Terraform |
| **Features TF** | Workspaces, variable files, backend config |

## Descripcion

Implementa la misma infraestructura usando ambos enfoques: basado en workspaces (una sola configuracion, multiples workspaces) y basado en directorios (directorios separados por entorno). Compara en: aislamiento de estado, duplicacion de codigo, complejidad de CI/CD y flujo de trabajo del equipo.

## Objetivos de aprendizaje

- Comprender los trade-offs de cada estrategia de gestion de entornos
- Identificar cuando usar workspaces vs estructura de directorios
- Evaluar criterios como aislamiento de estado, DRY y complejidad operacional
- Tomar decisiones informadas sobre la estrategia de entornos para un proyecto

## Requisitos previos

- Conocimientos: Terraform basico, backends, variables, workspaces
- Herramientas: Terraform >= 1.5, AWS CLI, cuenta AWS con bucket S3 para state
- Features/Servicios: S3 (backend), DynamoDB (lock), los servicios del proyecto de ejemplo

## Arquitectura

```
Enfoque A: Workspaces
workspace-based/
  +-- main.tf
  +-- variables.tf
  +-- outputs.tf
  +-- terraform.tfvars         (defaults)
  +-- envs/
        +-- dev.tfvars
        +-- staging.tfvars
        +-- prod.tfvars

Enfoque B: Directorios
directory-based/
  +-- modules/
  |     +-- app/               (modulo compartido)
  +-- environments/
        +-- dev/
        |     +-- main.tf      (llama al modulo)
        |     +-- terraform.tfvars
        +-- staging/
        |     +-- main.tf
        |     +-- terraform.tfvars
        +-- prod/
              +-- main.tf
              +-- terraform.tfvars
```

## Pasos sugeridos

1. Definir una infraestructura de ejemplo (VPC + EC2 + S3) que se desplegara en multiples entornos
2. Implementar el enfoque de workspaces: una configuracion con `terraform.workspace` para diferenciar entornos
3. Crear archivos `.tfvars` por entorno y usar `-var-file` en los comandos
4. Configurar el backend S3 con workspace-aware key path
5. Implementar el enfoque de directorios: un modulo compartido llamado desde cada directorio de entorno
6. Configurar backends independientes para cada directorio de entorno
7. Desplegar ambos enfoques en dev y staging
8. Comparar: tamano del blast radius si un `terraform apply` falla
9. Comparar: complejidad del pipeline CI/CD para cada enfoque
10. Documentar una tabla comparativa con pros y contras de cada enfoque

## Validacion

- [ ] Ambos enfoques despliegan la misma infraestructura en dev y staging
- [ ] Los workspaces comparten configuracion pero tienen state separado
- [ ] Los directorios tienen state completamente aislado
