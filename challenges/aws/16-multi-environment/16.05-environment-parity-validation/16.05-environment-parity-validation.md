# 16.05 Validacion de Paridad entre Entornos

## Metadata
| Campo | Valor |
|-------|-------|
| **Complejidad** | Avanzado |
| **Modalidad** | Terraform + Python/Go |
| **Features TF** | `terraform show -json`, state comparison |

## Descripcion

Construye una herramienta que compare el state de Terraform entre entornos para verificar paridad. Comprueba que dev, staging y prod tienen los mismos tipos de recursos y configuraciones (modulo las diferencias esperadas como tamanos de instancia). Alerta sobre drift entre entornos.

## Objetivos de aprendizaje

- Comprender la introspeccion de estado de Terraform
- Validar paridad de configuracion entre entornos
- Disenar herramientas de validacion de infraestructura
- Detectar drift entre entornos de forma automatizada

## Requisitos previos

- Conocimientos: Terraform state, JSON processing, Python o Go
- Herramientas: Terraform >= 1.5, Python 3.x o Go 1.21+, jq, AWS CLI
- Features/Servicios: S3 (backend de state), los servicios desplegados en cada entorno

## Arquitectura

```
tools/
  +-- env-parity-checker/
        +-- main.py (o main.go)
        +-- config.yaml
        |     +-- environments: [dev, staging, prod]
        |     +-- expected_differences:
        |           +-- aws_instance: [instance_type]
        |           +-- aws_db_instance: [instance_class, multi_az]
        +-- report.py          (generacion de reporte)

Flujo:
  1. terraform show -json (por cada entorno)
  2. Parsear el state JSON
  3. Extraer tipos de recursos y sus configuraciones
  4. Comparar entre entornos ignorando diferencias esperadas
  5. Generar reporte de paridad / alertas de drift
```

## Pasos sugeridos

1. Desplegar una infraestructura de ejemplo en al menos dos entornos (dev, staging)
2. Usar `terraform show -json` para exportar el state de cada entorno
3. Estudiar la estructura del JSON de state: resources, types, attributes
4. Crear un script que parsee el state JSON y extraiga los tipos de recursos
5. Comparar los tipos de recursos entre entornos: alertar si dev tiene algo que staging no
6. Para recursos del mismo tipo, comparar atributos excluyendo diferencias esperadas (instance_type, etc.)
7. Implementar un archivo de configuracion con las diferencias esperadas por tipo de recurso
8. Generar un reporte con: recursos identicos, diferencias esperadas, diferencias inesperadas (drift)
9. Agregar salida en formato que sirva para CI/CD (exit code != 0 si hay drift)
10. Integrar como paso en un pipeline de GitHub Actions que se ejecute periodicamente

## Validacion

- [ ] `terraform show -json` exporta el state de cada entorno correctamente
- [ ] La herramienta parsea el JSON de state sin errores
- [ ] Se detectan recursos presentes en un entorno pero ausentes en otro
- [ ] Las diferencias esperadas (instance_type, etc.) se ignoran correctamente
- [ ] Las diferencias inesperadas se reportan como drift
- [ ] El reporte es claro y accionable
- [ ] El exit code refleja si hay drift (0 = paridad, 1 = drift)
- [ ] La herramienta es configurable para nuevos tipos de recursos y atributos
