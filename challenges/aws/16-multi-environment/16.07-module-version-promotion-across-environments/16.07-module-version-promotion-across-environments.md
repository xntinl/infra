# 16.07 Promocion de Versiones de Modulos entre Entornos

## Metadata
| Campo | Valor |
|-------|-------|
| **Complejidad** | Avanzado |
| **Modalidad** | Terraform + CI/CD |
| **Features TF** | Module version constraints, registry |

## Descripcion

Usa diferentes versiones de modulos por entorno: dev usa `~> 2.0` (ultimo minor), staging usa una version especifica `2.1.0`, prod usa `2.0.5` (anterior estable). Construye un flujo de promocion que suba versiones a traves de los entornos con validacion en cada etapa.

## Objetivos de aprendizaje

- Comprender estrategias de version constraints para modulos
- Disenar flujos de promocion de versiones entre entornos
- Implementar staged rollouts para cambios de infraestructura
- Gestionar el riesgo de actualizaciones con versionado controlado

## Requisitos previos

- Conocimientos: Modulos de Terraform, versionado semantico, registries
- Herramientas: Terraform >= 1.5, GitHub (para registry privado o tags), CI/CD
- Features/Servicios: S3 (backend), los servicios que el modulo gestiona

## Arquitectura

```
modules/
  +-- networking/              (modulo versionado via Git tags)
        +-- v2.0.5 (tag)      -> prod (estable anterior)
        +-- v2.1.0 (tag)      -> staging (version especifica)
        +-- v2.x   (branch)   -> dev (ultimo minor)

environments/
  +-- dev/
  |     main.tf:
  |       module "networking" {
  |         source  = "git::https://...?ref=v2.1.0"  # o ~> 2.0 con registry
  |       }
  +-- staging/
  |     main.tf:
  |       module "networking" {
  |         source  = "git::https://...?ref=v2.1.0"
  |       }
  +-- prod/
        main.tf:
          module "networking" {
            source  = "git::https://...?ref=v2.0.5"
          }

Flujo de Promocion:
  Nueva version v2.2.0
    -> Actualizar dev (automatico)
    -> Validar en dev
    -> Actualizar staging (PR)
    -> Validar en staging
    -> Actualizar prod (PR + approval)
```

## Pasos sugeridos

1. Crear un modulo de Terraform (ej: networking) con su propio repositorio o directorio
2. Tagear multiples versiones del modulo (v2.0.5, v2.1.0, v2.2.0) con cambios incrementales
3. Configurar dev para usar la version mas reciente (`~> 2.0` o referencia a branch)
4. Configurar staging para usar una version especifica (v2.1.0)
5. Configurar prod para usar la version estable anterior (v2.0.5)
6. Verificar que cada entorno usa la version correcta con `terraform init` y `terraform plan`
7. Crear un script o workflow que automatice la actualizacion de la referencia de version en cada entorno
8. Implementar el flujo de promocion: actualizar dev, validar, promover a staging, validar, promover a prod
9. Agregar validaciones automaticas entre cada promocion (plan exitoso, tests de infraestructura)
10. Documentar el proceso de rollback: revertir a la version anterior en caso de fallo

## Validacion

- [ ] Cada entorno referencia una version diferente del modulo
- [ ] `terraform init` descarga la version correcta para cada entorno
- [ ] El cambio de version en dev no afecta a staging ni prod
- [ ] La promocion de staging a prod requiere aprobacion
- [ ] `terraform plan` muestra los cambios entre versiones antes de aplicar
- [ ] El rollback a una version anterior funciona sin problemas
- [ ] El flujo de promocion esta documentado y es reproducible
- [ ] Las validaciones automaticas detectan problemas antes de promover
