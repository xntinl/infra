# 12.09 Kubernetes HPA, VPA & Advanced Autoscaling

## Metadata
| Campo | Valor |
|-------|-------|
| **Complejidad** | Avanzado |
| **Modalidad** | Terraform + Kubernetes |
| **Servicios AWS** | EKS, CloudWatch, Prometheus, KEDA, Karpenter |

## Descripcion

Implementar autoscaling avanzado en EKS: HPA con metricas custom (desde CloudWatch o Prometheus via KEDA), VPA para recomendaciones de right-sizing, y Cluster Proportional Autoscaler. Probar el comportamiento de escalado bajo carga y observar el aprovisionamiento de nodos con Karpenter.

## Objetivos de aprendizaje

- Dominar los patrones de autoscaling en Kubernetes (HPA, VPA, cluster-level)
- Configurar HPA con metricas personalizadas mas alla de CPU y memoria
- Implementar escalado event-driven con KEDA
- Comprender las recomendaciones de VPA para right-sizing de pods
- Integrar Karpenter para aprovisionamiento reactivo de nodos

## Requisitos previos

- **Conocimientos**: Kubernetes autoscaling (HPA basics), metricas de Prometheus, EKS node management
- **Herramientas**: AWS CLI, Terraform, kubectl, Helm, herramienta de carga (k6, hey, wrk)
- **Servicios AWS**: Cluster EKS funcional, permisos para instalar add-ons y crear nodos

## Arquitectura

La arquitectura de autoscaling multinivel incluye:

- **HPA (Horizontal Pod Autoscaler)**: Escala el numero de pods basado en metricas de CPU, memoria y custom metrics
- **VPA (Vertical Pod Autoscaler)**: Analiza el uso de recursos y recomienda (o aplica) ajustes de requests/limits
- **KEDA**: Escalador event-driven que extiende HPA con triggers de SQS, CloudWatch, Prometheus y otros
- **Karpenter**: Aprovisionador de nodos que reacciona a pods pending creando nodos del tamano optimo
- **Prometheus**: Fuente de metricas custom para HPA y base para KEDA ScaledObjects
- **Metrics Server**: Proveedor de metricas de CPU y memoria para HPA basico
- **Cluster Proportional Autoscaler**: Escala componentes del sistema (CoreDNS) proporcionalmente al tamano del cluster

## Pasos sugeridos

1. Verificar que Metrics Server esta instalado en el cluster EKS (requerido para HPA basico)
2. Desplegar una aplicacion de ejemplo con Deployment y Service, configurando resource requests/limits
3. Crear un HPA basico con target de CPU al 50% y probar con carga artificial
4. Instalar Prometheus (via kube-prometheus-stack) para metricas custom
5. Configurar un HPA con metricas custom de Prometheus (ej: requests per second) usando Prometheus Adapter
6. Instalar KEDA y crear un ScaledObject que escale basado en la profundidad de una cola SQS
7. Instalar VPA y configurarlo en modo "recommend" (sin aplicar cambios automaticamente)
8. Analizar las recomendaciones de VPA comparandolas con los requests/limits actuales
9. Instalar Karpenter y configurar un NodePool con constraints de instancia
10. Instalar Cluster Proportional Autoscaler para escalar CoreDNS proporcionalmente
11. Ejecutar pruebas de carga progresivas y observar: HPA escalando pods -> Karpenter creando nodos -> KEDA respondiendo a eventos
12. Documentar los tiempos de reaccion de cada componente y las configuraciones optimas

## Validacion

- [ ] HPA basico (CPU) escala pods correctamente bajo carga
- [ ] HPA con metricas custom de Prometheus funciona (ej: RPS)
- [ ] KEDA escala pods basado en un trigger externo (ej: profundidad de cola SQS)
- [ ] VPA genera recomendaciones de right-sizing coherentes con el uso real
- [ ] Karpenter aprovisiona nodos cuando hay pods pending y consolida cuando hay capacidad ociosa
- [ ] Cluster Proportional Autoscaler ajusta replicas de CoreDNS segun tamano del cluster
- [ ] Se realizaron pruebas de carga con metricas documentadas
- [ ] Se documentaron tiempos de reaccion y configuraciones recomendadas
