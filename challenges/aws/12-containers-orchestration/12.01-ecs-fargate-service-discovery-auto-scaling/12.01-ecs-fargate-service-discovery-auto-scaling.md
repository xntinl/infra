## The Scenario

Your team has been tasked with building a production-ready container platform on ECS Fargate. You must deploy two microservices -- an API service fronted by an ALB and a worker service driven by SQS -- with service-to-service discovery, auto-scaling, secrets management, and full observability. No hand-holding. No starter code. Build it from scratch.

## Constraints

1. VPC with 3-tier architecture spanning 2 AZs: public, private-app, and private-data subnets (6 total).
2. ECR repository with a lifecycle policy retaining only the last 5 images.
3. Two ECS Fargate services: `api-service` (ALB-fronted, port 8080) and `worker-service` (SQS-driven).
4. ECS Service Connect for api-to-worker internal discovery (not legacy Cloud Map Service Discovery).
5. App Auto Scaling: api on `ALBRequestCountPerTarget` (threshold: 100), worker on SQS `ApproximateNumberOfMessagesVisible` (threshold: 10).
6. SQS queue with DLQ (`maxReceiveCount`: 3) and an SNS topic for processed results.
7. Secrets Manager secret injected via task definition `secrets` block -- never as plaintext `environment`.
8. Container Insights enabled on the ECS cluster and an X-Ray sidecar container in both task definitions.
9. CloudWatch Dashboard with exactly 6 widgets: API latency, API 5xx rate, worker queue depth, task count, CPU utilization, memory utilization.
10. ECS Exec enabled on both services with `ssmmessages:*` permissions in the task role.

## Success Criteria

- VPC has exactly 6 subnets across 2 AZs (2 public, 2 private-app, 2 private-data)
- ECR lifecycle policy returns `"countNumber": 5` for `countType: "imageCountMoreThan"`
- Both ECS services show `"status": "ACTIVE"` and `"desiredCount" >= 1`
- Service Connect configuration visible in ECS service description
- ALB health check returns HTTP 200 on the api-service target group
- SQS queue has a `RedrivePolicy` with `maxReceiveCount` of 3
- SNS topic exists with at least one subscription
- Secrets Manager secret referenced in task definition (not as plaintext env var)
- Container Insights enabled on cluster (`containerInsights: "enabled"`)
- X-Ray sidecar container named `xray-daemon` exists in both task definitions
- Auto-scaling policies exist for both services (describe-scaling-policies returns 2+)
- CloudWatch dashboard exists with exactly 6 widgets
- ECS Exec enabled (`executeCommandConfiguration` present on cluster)
- Task role has `ssmmessages:*` permissions
- All resources tagged with `Name` prefix matching project name

## Verify

```bash
aws ecs describe-clusters --clusters <cluster-name> \
  --query "clusters[0].{Status:status,Insights:settings[?name=='containerInsights'].value|[0],ExecConfig:configuration.executeCommandConfiguration}" --output json
```
```bash
aws ecs describe-services --cluster <cluster-name> --services api-service worker-service \
  --query "services[].{Name:serviceName,Status:status,Desired:desiredCount,ServiceConnect:deployments[0].serviceConnectConfiguration.enabled}" --output table
```
```bash
aws sqs get-queue-attributes --queue-url <queue-url> --attribute-names RedrivePolicy \
  --query "Attributes.RedrivePolicy" --output text
```
```bash
aws application-autoscaling describe-scaling-policies --service-namespace ecs \
  --query "ScalingPolicies[].{Service:ResourceId,Metric:TargetTrackingScalingPolicyConfiguration.PredefinedMetricSpecification.PredefinedMetricType}" --output table
```
