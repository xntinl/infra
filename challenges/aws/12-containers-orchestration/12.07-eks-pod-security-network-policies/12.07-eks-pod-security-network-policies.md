# 12.07 EKS Pod Security & Network Policies

## Metadata
| Campo | Valor |
|-------|-------|
| **Complejidad** | Avanzado |
| **Modalidad** | Terraform + Kubernetes manifests |
| **Servicios AWS** | EKS, IAM, Calico, OPA Gatekeeper |

## Descripcion

Implementar seguridad de Kubernetes en EKS: Pod Security Standards (Restricted), Network Policies con Calico, constraints de OPA Gatekeeper (prohibir contenedores privilegiados, requerir resource limits, registros aprobados), y Kubernetes RBAC integrado con IAM.

## Objetivos de aprendizaje

- Endurecer la seguridad de un cluster EKS aplicando Pod Security Standards
- Implementar segmentacion de red con Network Policies usando Calico
- Configurar admission control con OPA Gatekeeper para enforcer politicas organizacionales
- Disenar RBAC de Kubernetes integrado con IAM de AWS
- Aplicar defensa en profundidad en contenedores

## Requisitos previos

- **Conocimientos**: Kubernetes security (RBAC, PSS, Network Policies), EKS IAM integration, OPA/Rego basics
- **Herramientas**: AWS CLI, Terraform, kubectl, Helm
- **Servicios AWS**: Cluster EKS, roles IAM para integracion con Kubernetes RBAC

## Arquitectura

La arquitectura de seguridad del cluster incluye multiples capas:

- **Pod Security Standards (PSS)**: Admission controller nativo que enforce niveles Baseline/Restricted a nivel de namespace
- **Calico**: CNI plugin que implementa Network Policies para controlar trafico entre pods y namespaces
- **OPA Gatekeeper**: Admission controller que evalua constraints personalizados en cada request al API server
- **Kubernetes RBAC**: Roles y RoleBindings que definen permisos de usuarios y service accounts
- **IAM Integration**: aws-auth ConfigMap y/o EKS Access Entries que mapean roles IAM a grupos de Kubernetes
- **Security Groups for Pods**: Asignar security groups de VPC directamente a pods para control de red a nivel AWS

## Pasos sugeridos

1. Crear o usar un cluster EKS existente y desplegar una aplicacion multi-tier (frontend, backend, database) para probar
2. Configurar Pod Security Standards: aplicar el label `pod-security.kubernetes.io/enforce: restricted` en namespaces de produccion
3. Verificar que pods con configuraciones no seguras (privileged, hostNetwork) son rechazados
4. Instalar Calico en el cluster EKS via Helm o como add-on
5. Crear Network Policies: el frontend solo puede hablar con el backend, el backend solo con la base de datos, denegar todo lo demas
6. Verificar la segmentacion de red probando conectividad entre pods
7. Instalar OPA Gatekeeper via Helm y crear ConstraintTemplates
8. Crear constraints: prohibir contenedores privilegiados, requerir resource requests/limits, permitir solo registros ECR aprobados
9. Configurar RBAC: crear roles para developers (read-only en prod, edit en dev) y admins (cluster-admin)
10. Integrar RBAC con IAM: mapear roles IAM a grupos de Kubernetes via aws-auth o EKS Access Entries
11. Probar cada capa de seguridad intentando violar las politicas y verificando que son bloqueadas
12. Documentar la postura de seguridad del cluster y las politicas implementadas

## Validacion

- [ ] Pod Security Standards Restricted esta aplicado en al menos un namespace
- [ ] Los pods con configuraciones privilegiadas son rechazados automaticamente
- [ ] Calico esta instalado y las Network Policies estan activas
- [ ] La segmentacion de red funciona: solo las comunicaciones permitidas son posibles
- [ ] OPA Gatekeeper esta instalado con al menos 3 constraints activos
- [ ] Los constraints bloquean: contenedores privilegiados, pods sin resource limits, imagenes de registros no aprobados
- [ ] RBAC esta configurado con roles diferenciados por equipo/entorno
- [ ] Los roles IAM se mapean correctamente a permisos de Kubernetes
