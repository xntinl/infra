# 12.02 EKS con App Mesh (Service Mesh)

## Metadata
| Campo | Valor |
|-------|-------|
| **Complejidad** | Experto |
| **Modalidad** | Terraform + Kubernetes manifests |
| **Servicios AWS** | EKS, App Mesh, ALB, CloudWatch, X-Ray |

## Descripcion

Desplegar una aplicacion multi-servicio en EKS e integrar AWS App Mesh. Configurar virtual nodes, virtual services y virtual routers. Implementar traffic shifting (canary al 95/5), politicas de retry, configuraciones de timeout y observar metricas del Envoy proxy.

## Objetivos de aprendizaje

- Comprender la arquitectura de un service mesh y el rol de los sidecar proxies
- Configurar los componentes de App Mesh: virtual nodes, virtual services y virtual routers
- Implementar gestion de trafico avanzada con canary deployments y traffic shifting
- Configurar patrones de resiliencia como retries y timeouts a nivel de mesh
- Observar y analizar metricas del Envoy proxy para diagnostico

## Requisitos previos

- **Conocimientos**: Kubernetes (Deployments, Services, Ingress), EKS, conceptos de service mesh, Envoy proxy basics
- **Herramientas**: AWS CLI, Terraform, kubectl, Helm, eksctl (opcional)
- **Servicios AWS**: Cluster EKS funcional, permisos para crear App Mesh resources, X-Ray

## Arquitectura

La arquitectura del service mesh incluye:

- **EKS Cluster**: Cluster Kubernetes gestionado donde se ejecutan los microservicios
- **App Mesh**: Service mesh que gestiona la comunicacion entre servicios via Envoy sidecar proxies
- **Virtual Nodes**: Representan cada version de un microservicio en el mesh
- **Virtual Services**: Abstracciones que los servicios usan para comunicarse entre si
- **Virtual Routers**: Controlan el enrutamiento de trafico con reglas de peso para canary/blue-green
- **Envoy Sidecar**: Proxy inyectado en cada pod que intercepta y gestiona el trafico
- **X-Ray**: Trazado distribuido integrado con Envoy para visibilidad end-to-end
- **ALB Ingress**: Punto de entrada al mesh desde internet

## Pasos sugeridos

1. Crear un cluster EKS con Terraform incluyendo los add-ons necesarios
2. Instalar el App Mesh Controller via Helm para gestionar los CRDs de App Mesh en Kubernetes
3. Crear el App Mesh mesh resource y configurar el namespace con la label de inyeccion de sidecar
4. Desplegar al menos 3 microservicios (ej: frontend, catalog, orders) como Kubernetes Deployments
5. Crear los Virtual Nodes para cada microservicio, configurando listeners y service discovery
6. Crear Virtual Services y Virtual Routers para definir como se comunican los servicios
7. Implementar traffic shifting canary: desplegar v2 del servicio catalog y configurar el router con peso 95/5
8. Configurar retry policies en los virtual nodes (maxRetries, httpRetryEvents, retryOn)
9. Configurar timeouts (perRequest, idle) para prevenir cascading failures
10. Habilitar X-Ray tracing en los sidecars y verificar las trazas end-to-end en la consola
11. Crear un dashboard de CloudWatch con metricas de Envoy: request count, latency p50/p95/p99, error rate

## Validacion

- [ ] El cluster EKS esta funcionando con el App Mesh Controller instalado
- [ ] Al menos 3 microservicios estan desplegados con Envoy sidecar inyectado
- [ ] Los virtual nodes, virtual services y virtual routers estan configurados correctamente
- [ ] El traffic shifting canary funciona con la distribucion 95/5 verificable en las metricas
- [ ] Las retry policies estan configuradas y se activan ante errores transitorios
- [ ] Los timeouts estan configurados para prevenir requests colgados
- [ ] X-Ray muestra trazas distribuidas de las llamadas entre servicios
- [ ] El dashboard de metricas de Envoy muestra datos reales
