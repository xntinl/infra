# 12.08 Container CI/CD con CodeBuild y ECR

## Metadata
| Campo | Valor |
|-------|-------|
| **Complejidad** | Intermedio |
| **Modalidad** | Ambos |
| **Servicios AWS** | CodeBuild, CodePipeline, ECR, ECS, ALB |

## Descripcion

Construir un pipeline completo de CI/CD para contenedores: multi-stage Docker build en CodeBuild, tests unitarios en la etapa de build, escaneo de imagenes en ECR, push a ECR de staging, deploy a ECS staging, ejecutar tests de integracion, promover a ECR de produccion, deploy a ECS produccion.

## Objetivos de aprendizaje

- Implementar mejores practicas de CI/CD para contenedores
- Construir Dockerfiles multi-stage eficientes
- Implementar promocion de imagenes entre entornos
- Disenar pipelines con progresion entre entornos (staging -> produccion)

## Requisitos previos

- **Conocimientos**: Docker multi-stage builds, CodeBuild buildspec, CodePipeline stages, ECS deployments
- **Herramientas**: AWS CLI, Terraform, Docker, Git
- **Servicios AWS**: Repositorio en CodeCommit o GitHub, permisos para crear CodeBuild projects, CodePipeline, ECR y ECS services

## Arquitectura

El pipeline de CI/CD para contenedores incluye:

- **Source Stage**: CodeCommit o GitHub como fuente del codigo
- **Build Stage (CodeBuild)**: Multi-stage Docker build con tests unitarios integrados
- **ECR Staging**: Repositorio donde se publica la imagen tras el build exitoso
- **ECR Scan**: Escaneo automatico de vulnerabilidades al hacer push
- **Deploy Staging (ECS)**: Despliegue automatico al cluster ECS de staging
- **Test Stage (CodeBuild)**: Tests de integracion ejecutados contra el entorno de staging
- **Approval**: Gate de aprobacion manual antes de produccion
- **ECR Production**: Repositorio de produccion donde se promueve la imagen validada
- **Deploy Production (ECS)**: Despliegue al cluster ECS de produccion con rolling update

## Pasos sugeridos

1. Crear una aplicacion de ejemplo con Dockerfile multi-stage (build stage con tests, runtime stage minimo)
2. Crear repositorios ECR para staging y produccion via Terraform
3. Configurar el proyecto de CodeBuild con buildspec que construya la imagen y ejecute tests unitarios
4. Configurar ECR scanning on push en ambos repositorios
5. Crear clusters ECS con servicios para staging y produccion (pueden compartir cluster con namespaces diferentes)
6. Crear el CodePipeline con las siguientes stages: Source -> Build -> Deploy Staging -> Test -> Approval -> Deploy Production
7. En el stage de Build, configurar el push de la imagen a ECR staging con tag del commit SHA
8. En el stage de Deploy Staging, actualizar el servicio ECS de staging con la nueva imagen
9. En el stage de Test, ejecutar tests de integracion contra el endpoint de staging
10. Configurar un manual approval gate antes del deploy a produccion
11. En el stage de Deploy Production, retaggear la imagen y pushearla a ECR produccion, luego actualizar ECS
12. Probar el pipeline completo con un commit y verificar la progresion end-to-end

## Validacion

- [ ] El Dockerfile multi-stage construye correctamente con tests integrados
- [ ] CodeBuild ejecuta el build y los tests unitarios exitosamente
- [ ] La imagen se publica en ECR staging con tag del commit SHA
- [ ] ECR escanea la imagen automaticamente y reporta vulnerabilidades
- [ ] El servicio ECS de staging se actualiza con la nueva imagen
- [ ] Los tests de integracion se ejecutan contra staging
- [ ] El approval gate bloquea el deploy a produccion hasta la aprobacion manual
- [ ] La imagen se promueve a ECR produccion y se despliega en ECS produccion
- [ ] El pipeline completo funciona end-to-end con un solo commit
