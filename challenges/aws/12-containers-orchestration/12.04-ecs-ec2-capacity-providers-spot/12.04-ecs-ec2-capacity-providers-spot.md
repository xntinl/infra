# 12.04 ECS EC2 con Capacity Providers y Spot

## Metadata
| Campo | Valor |
|-------|-------|
| **Complejidad** | Avanzado |
| **Modalidad** | Ambos |
| **Servicios AWS** | ECS, EC2, Auto Scaling, Spot, ALB |

## Descripcion

Desplegar ECS en EC2 con multiples Capacity Providers: uno para instancias On-Demand (servicios criticos) y otro para instancias Spot (workloads batch). Configurar managed scaling, managed termination protection y estrategias de capacity provider con distribucion ponderada.

## Objetivos de aprendizaje

- Comprender y configurar ECS Capacity Providers con diferentes tipos de instancia
- Implementar instancias Spot en ECS para optimizar costos en workloads tolerantes a interrupciones
- Configurar managed scaling para escalar automaticamente la capacidad del cluster
- Disenar estrategias de capacity provider con distribucion ponderada por criticidad

## Requisitos previos

- **Conocimientos**: ECS en modo EC2, Auto Scaling Groups, instancias Spot, ALB target groups
- **Herramientas**: AWS CLI, Terraform
- **Servicios AWS**: VPC con subnets, permisos para crear ECS clusters, ASGs, instancias EC2

## Arquitectura

La arquitectura de ECS con Capacity Providers incluye:

- **ECS Cluster**: Cluster en modo EC2 con multiples capacity providers registrados
- **On-Demand Capacity Provider**: Asociado a un ASG de instancias On-Demand para servicios criticos (API, auth)
- **Spot Capacity Provider**: Asociado a un ASG de instancias Spot diversificadas para workloads batch
- **Auto Scaling Groups**: Dos ASGs con managed scaling habilitado, uno On-Demand y otro Spot con mixed instances
- **ECS Services**: Servicios criticos con estrategia 100% On-Demand, servicios batch con estrategia 70% Spot / 30% On-Demand
- **ALB**: Balanceador de carga frente a los servicios que requieren acceso externo

## Pasos sugeridos

1. Crear la VPC con subnets publicas y privadas en al menos 2 AZs
2. Crear un Auto Scaling Group para instancias On-Demand con un solo tipo de instancia
3. Crear un Auto Scaling Group para instancias Spot con mixed instances policy (10+ tipos de instancia)
4. Crear el cluster ECS y registrar los dos Capacity Providers (On-Demand y Spot) asociados a sus ASGs
5. Habilitar managed scaling con target capacity al 100% en ambos capacity providers
6. Habilitar managed termination protection para evitar que el ASG termine instancias con tasks activas
7. Desplegar un servicio critico (ej: API) con capacity provider strategy 100% On-Demand
8. Desplegar un servicio batch (ej: worker) con capacity provider strategy base=1 On-Demand, weight 70% Spot
9. Configurar el ALB con target groups apuntando a los servicios correspondientes
10. Probar el comportamiento: escalar tasks y verificar que el cluster agrega instancias automaticamente del tipo correcto
11. Simular una interrupcion Spot y verificar que los tasks se reubican en instancias disponibles

## Validacion

- [ ] El cluster ECS tiene 2 capacity providers registrados (On-Demand y Spot)
- [ ] El ASG de Spot usa mixed instances policy con al menos 10 tipos de instancia
- [ ] Managed scaling esta habilitado y escala las instancias automaticamente
- [ ] Managed termination protection evita terminar instancias con tasks activas
- [ ] El servicio critico se ejecuta exclusivamente en instancias On-Demand
- [ ] El servicio batch distribuye tasks entre Spot y On-Demand segun la estrategia configurada
- [ ] El cluster escala correctamente cuando se incrementa el numero de tasks
- [ ] La interrupcion de instancias Spot no causa perdida de servicio
