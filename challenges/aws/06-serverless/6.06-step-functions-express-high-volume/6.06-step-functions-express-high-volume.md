# 6.06 Step Functions Express High Volume

## Metadata
| Campo | Valor |
|-------|-------|
| **Complejidad** | Intermedio |
| **Modalidad** | Ambos |
| **Servicios AWS** | Step Functions (Express), Lambda, EventBridge Pipes, CloudWatch Logs |

## Descripcion

Comparar workflows Standard vs Express de Step Functions. Construir un workflow Express para procesamiento de eventos de alto volumen (ej. validacion de datos IoT). Implementar el estado Map para procesamiento paralelo, integrar con EventBridge Pipes, y analizar logs de ejecucion.

## Objetivos de aprendizaje

- Comprender las diferencias entre workflows Express y Standard
- Implementar patrones de alto rendimiento con Express workflows
- Utilizar el estado Map para paralelismo masivo
- Optimizar costos con el modelo de ejecucion Express

## Requisitos previos

- Conocimientos: Step Functions basico, Lambda, conceptos de procesamiento de eventos
- Herramientas: AWS CLI, Terraform o CloudFormation, lenguaje para Lambda
- Servicios AWS: cuenta con permisos para Step Functions, Lambda, EventBridge Pipes, CloudWatch Logs

## Arquitectura

La arquitectura procesa eventos de alto volumen con Step Functions Express:

1. **EventBridge Pipes**: captura eventos desde una fuente (ej. SQS, Kinesis) y los envia al workflow Express
2. **Express Workflow**: state machine optimizada para alto volumen y baja latencia
3. **Estado Map**: procesa items en paralelo dentro de cada ejecucion
4. **Lambda - Validar**: valida el formato y schema de cada evento
5. **Lambda - Transformar**: transforma y enriquece los datos
6. **Lambda - Almacenar**: persiste los datos procesados en DynamoDB/S3
7. **CloudWatch Logs**: log group dedicado para logs de ejecucion del workflow Express
8. **Standard Workflow (comparacion)**: version Standard del mismo workflow para comparar costos y comportamiento

## Pasos sugeridos

1. Crear un workflow Standard de Step Functions con pasos de validacion, transformacion y almacenamiento
2. Crear el mismo workflow en version Express y comparar la configuracion
3. Implementar las Lambdas de validacion, transformacion y almacenamiento
4. Configurar el estado Map en el workflow Express para procesamiento paralelo de lotes
5. Configurar logging del workflow Express en CloudWatch Logs (nivel ALL)
6. Crear un EventBridge Pipe desde una cola SQS hacia el workflow Express
7. Generar carga de prueba (cientos/miles de eventos) y enviarla a la cola SQS
8. Analizar los logs de ejecucion en CloudWatch Logs Insights
9. Comparar costos entre Standard y Express para el mismo volumen de ejecuciones
10. Optimizar el workflow: ajustar concurrencia del Map state, timeout, retry policies
