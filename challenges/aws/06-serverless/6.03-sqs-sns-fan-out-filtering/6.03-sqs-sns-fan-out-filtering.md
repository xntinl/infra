## Learning Objectives

After completing this exercise, you will be able to:

- **Explain** the SNS fan-out pattern and how one message reaches multiple subscribers
- **Configure** SQS queues as SNS subscribers with filter policies based on message attributes
- **Verify** that filtered messages arrive only in the intended queues
- **Set up** dead-letter queues to capture messages that fail processing

## Why Fan-Out with Filtering

Many applications need to send the same event to multiple consumers. An order system might need to notify the warehouse, the billing department, and the analytics pipeline whenever an order is placed. Without fan-out, the order service would need to know about every consumer and call each one individually -- adding a new consumer means changing the order service.

SNS solves this with the pub/sub pattern: the order service publishes once to a topic, and every subscriber receives a copy. But not every subscriber wants every message. The warehouse only cares about physical goods, billing only cares about paid orders, and analytics wants everything. SNS filter policies let each subscriber declare what it wants. Messages that do not match are never delivered, saving SQS costs and Lambda invocations.

SQS acts as a buffer between SNS and your processors. Even if a Lambda function is throttled or temporarily failing, messages sit safely in the queue. A dead-letter queue catches messages that fail processing repeatedly, so you can investigate without losing data.

> **Best Practice:** Set the SQS visibility timeout to at least 6 times the Lambda function timeout. If Lambda takes 30 seconds to process a message but the visibility timeout is only 30 seconds, SQS will re-deliver the message to another consumer while the first is still processing, causing duplicate processing.

```hcl
# Warehouse queue: receives only messages with order_type = "physical"

  name                       = "${var.project_name}-warehouse"
  visibility_timeout_seconds = 180
  message_retention_seconds  = 86400

    Name = "${var.project_name}-warehouse"
  }

# Billing queue: receives only messages with status = "paid"
resource "aws_sqs_queue" "billing" {
  name                       = "${var.project_name}-billing"

  tags = {
    Name = "${var.project_name}-billing"
  }
}

# Analytics queue: receives ALL messages (no filter)
resource "aws_sqs_queue" "analytics" {
  name                       = "${var.project_name}-analytics"
  visibility_timeout_seconds = 180
  message_retention_seconds  = 86400

  tags = {
    Name = "${var.project_name}-analytics"
  }
}

# ---------------------------------------------------------------
# Dead-letter queues -- one per processing queue
# ---------------------------------------------------------------
resource "aws_sqs_queue" "warehouse_dlq" {
  name                      = "${var.project_name}-warehouse-dlq"
  message_retention_seconds = 1209600

  tags = {
    Name = "${var.project_name}-warehouse-dlq"
  }
}

resource "aws_sqs_queue" "billing_dlq" {
  name                      = "${var.project_name}-billing-dlq"
  message_retention_seconds = 1209600

  tags = {
    Name = "${var.project_name}-billing-dlq"
  }
}

resource "aws_sqs_queue" "analytics_dlq" {
  name                      = "${var.project_name}-analytics-dlq"
  message_retention_seconds = 1209600

  tags = {
    Name = "${var.project_name}-analytics-dlq"
  }
}

# ---------------------------------------------------------------
# Redrive policies -- send failed messages to DLQ after 3 attempts
# ---------------------------------------------------------------
resource "aws_sqs_queue_redrive_policy" "warehouse" {
  queue_url = aws_sqs_queue.warehouse.id
  redrive_policy = jsonencode({
    deadLetterTargetArn = aws_sqs_queue.warehouse_dlq.arn
    maxReceiveCount     = 3
  })
}

resource "aws_sqs_queue_redrive_policy" "billing" {
  queue_url = aws_sqs_queue.billing.id
  redrive_policy = jsonencode({
    deadLetterTargetArn = aws_sqs_queue.billing_dlq.arn
    maxReceiveCount     = 3
  })
}

resource "aws_sqs_queue_redrive_policy" "analytics" {
  queue_url = aws_sqs_queue.analytics.id
  redrive_policy = jsonencode({
    deadLetterTargetArn = aws_sqs_queue.analytics_dlq.arn
    maxReceiveCount     = 3
  })
}

# ---------------------------------------------------------------
# SQS policies -- allow SNS to send messages to each queue
# ---------------------------------------------------------------
data "aws_iam_policy_document" "sns_to_sqs" {
  for_each = {
    warehouse = aws_sqs_queue.warehouse.arn
    billing   = aws_sqs_queue.billing.arn
    analytics = aws_sqs_queue.analytics.arn
  }

  statement {
    actions   = ["sqs:SendMessage"]
    resources = [each.value]

    principals {
      type        = "Service"
      identifiers = ["sns.amazonaws.com"]
    }

    condition {
      test     = "ArnEquals"
      variable = "aws:SourceArn"
      values   = [aws_sns_topic.orders.arn]
    }
  }
}

resource "aws_sqs_queue_policy" "warehouse" {
  queue_url = aws_sqs_queue.warehouse.id
  policy    = data.aws_iam_policy_document.sns_to_sqs["warehouse"].json
}

resource "aws_sqs_queue_policy" "billing" {
  queue_url = aws_sqs_queue.billing.id
  policy    = data.aws_iam_policy_document.sns_to_sqs["billing"].json
}

resource "aws_sqs_queue_policy" "analytics" {
  queue_url = aws_sqs_queue.analytics.id
  policy    = data.aws_iam_policy_document.sns_to_sqs["analytics"].json
}
```

### `subscriptions.tf`

```hcl
# ---------------------------------------------------------------
# SNS subscriptions with filter policies
# ---------------------------------------------------------------

# Warehouse: only physical orders
resource "aws_sns_topic_subscription" "warehouse" {
  topic_arn = aws_sns_topic.orders.arn
  protocol  = "sqs"
  endpoint  = aws_sqs_queue.warehouse.arn

  # Filter: only deliver messages where order_type = "physical"
  filter_policy = jsonencode({
    order_type = ["physical"]
  })

  raw_message_delivery = true
}

# Billing: only paid orders
resource "aws_sns_topic_subscription" "billing" {
  topic_arn = aws_sns_topic.orders.arn
  protocol  = "sqs"
  endpoint  = aws_sqs_queue.billing.arn

  # Filter: only deliver messages where status = "paid"
  filter_policy = jsonencode({
    status = ["paid"]
  })

  raw_message_delivery = true
}

# Analytics: ALL orders (no filter policy = receive everything)
resource "aws_sns_topic_subscription" "analytics" {
  topic_arn = aws_sns_topic.orders.arn
  protocol  = "sqs"
  endpoint  = aws_sqs_queue.analytics.arn

  raw_message_delivery = true
}
```

## Best Practices

| Practice | Why |
|----------|-----|
| Visibility timeout >= 6x Lambda timeout | Prevents SQS from re-delivering a message while Lambda is still processing it |
| Filter policies on subscriptions | Reduces cost and processing load by filtering at SNS, not in consumer code |
| Raw message delivery | Removes the SNS envelope, so consumers see the original payload directly |
| DLQ per processing queue | Isolates failures per consumer; makes it clear which processor is failing |
| maxReceiveCount = 3 | Allows transient failures to retry while still catching poison messages |
| Message retention on DLQ = 14 days | Gives operators time to investigate and replay failed messages |

## Common Mistakes

### 1. Missing SQS queue policy for SNS

If you forget to allow SNS to send messages to an SQS queue, the subscription appears healthy but no messages arrive.

**Wrong -- no queue policy:**

```hcl
resource "aws_sns_topic_subscription" "warehouse" {
  topic_arn = aws_sns_topic.orders.arn
  protocol  = "sqs"
  endpoint  = aws_sqs_queue.warehouse.arn
}

# No aws_sqs_queue_policy allowing sns.amazonaws.com to SendMessage
```

**What happens:** `terraform apply` succeeds. You publish a message to SNS. No errors appear, but the SQS queue remains empty because SNS cannot deliver to it.

**Fix -- add a queue policy:**

```hcl
data "aws_iam_policy_document" "sns_to_sqs" {
  statement {
    actions   = ["sqs:SendMessage"]
    resources = [aws_sqs_queue.warehouse.arn]

    principals {
      type        = "Service"
      identifiers = ["sns.amazonaws.com"]
    }

    condition {
      test     = "ArnEquals"
      variable = "aws:SourceArn"
      values   = [aws_sns_topic.orders.arn]
    }
  }
}

resource "aws_sqs_queue_policy" "warehouse" {
  queue_url = aws_sqs_queue.warehouse.id
  policy    = data.aws_iam_policy_document.sns_to_sqs.json
}
```

### 2. Filter policy on wrong attribute scope

SNS filter policies can match on **message attributes** (default) or **message body**. If your publisher sets attributes but your filter looks at the body (or vice versa), nothing matches.

**Wrong -- filter_policy_scope mismatch:**

```hcl
resource "aws_sns_topic_subscription" "warehouse" {
  topic_arn = aws_sns_topic.orders.arn
  protocol  = "sqs"
  endpoint  = aws_sqs_queue.warehouse.arn

  # This filters on message BODY, but publisher sends attributes
  filter_policy_scope = "MessageBody"
  filter_policy = jsonencode({
    order_type = ["physical"]
  })
}
```

**What happens:** The publisher sets `order_type` as a message attribute, but the filter is inspecting the message body JSON. No messages match because the body does not contain a top-level `order_type` key.

**Fix -- use the default scope (MessageAttributes) or explicitly set it:**

```hcl
resource "aws_sns_topic_subscription" "warehouse" {
  topic_arn = aws_sns_topic.orders.arn
  protocol  = "sqs"
  endpoint  = aws_sqs_queue.warehouse.arn

  # Default scope is "MessageAttributes" -- matches our publisher
  filter_policy = jsonencode({
    order_type = ["physical"]
  })
}
```

### 3. Visibility timeout shorter than Lambda timeout

**Wrong -- visibility timeout too low:**

```hcl
resource "aws_sqs_queue" "warehouse" {
  name                       = "warehouse"
  visibility_timeout_seconds = 30  # Same as Lambda timeout!
}
```

**What happens:** Lambda starts processing a message that takes 25 seconds. At 30 seconds, the visibility timeout expires and SQS makes the message visible again. Another Lambda invocation picks it up, causing duplicate processing.

**Fix -- set visibility timeout to 6x Lambda timeout:**

```hcl
resource "aws_sqs_queue" "warehouse" {
  name                       = "warehouse"
  visibility_timeout_seconds = 180  # 6 x 30s Lambda timeout
}
```

### Test 3 -- Publish a digital, paid order (should NOT reach warehouse)

```bash
aws sns publish \
  --topic-arn $(terraform output -raw sns_topic_arn) \
  --message '{"order_id": "002", "item": "ebook", "amount": 15}' \
```

```bash

  --queue-url $(terraform output -raw warehouse_queue_url) \
  --max-number-of-messages 10 \
  --output json
```

Expected: no messages with `order_id = 002`. The billing and analytics queues should have it.

## Summary

- **SNS fan-out** delivers a single message to multiple subscribers simultaneously
- **Filter policies** on subscriptions let SNS drop non-matching messages before delivery, reducing cost and unnecessary processing
- **SQS queues** buffer messages between SNS and consumers, providing durability and retry capability
- **Dead-letter queues** with `maxReceiveCount` catch messages that fail processing repeatedly
- **Visibility timeout** must be at least 6x the Lambda function timeout to prevent duplicate processing
- **Raw message delivery** removes the SNS envelope so consumers parse the original payload directly
