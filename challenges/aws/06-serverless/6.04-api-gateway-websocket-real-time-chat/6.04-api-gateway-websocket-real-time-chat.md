# 6.04 API Gateway WebSocket Real-Time Chat

## Metadata
| Campo | Valor |
|-------|-------|
| **Complejidad** | Intermedio |
| **Modalidad** | Ambos |
| **Servicios AWS** | API Gateway (WebSocket), Lambda, DynamoDB, IAM |

## Descripcion

Construir una aplicacion de chat en tiempo real serverless con WebSocket API Gateway, funciones Lambda para las rutas $connect/$disconnect/$default, DynamoDB para gestion de conexiones, y broadcast de mensajes a todos los clientes conectados.

## Objetivos de aprendizaje

- Comprender el protocolo WebSocket sobre API Gateway
- Gestionar el ciclo de vida de conexiones (connect, disconnect, default)
- Implementar patrones stateful en arquitecturas serverless
- Configurar comunicacion en tiempo real bidireccional

## Requisitos previos

- Conocimientos: WebSockets basico, API Gateway, Lambda, DynamoDB
- Herramientas: AWS CLI, Terraform o CloudFormation, wscat o cliente WebSocket, lenguaje para Lambda
- Servicios AWS: cuenta con permisos para API Gateway, Lambda, DynamoDB, IAM

## Arquitectura

La arquitectura implementa un chat en tiempo real usando WebSockets serverless:

1. **WebSocket API Gateway**: gestiona las conexiones WebSocket con tres rutas
2. **Lambda $connect**: se ejecuta al conectar un cliente, almacena el connectionId en DynamoDB
3. **Lambda $disconnect**: se ejecuta al desconectar, elimina el connectionId de DynamoDB
4. **Lambda $default**: procesa mensajes entrantes y los reenvía a todos los clientes conectados
5. **DynamoDB Connections Table**: almacena connectionIds activos con TTL para limpieza automatica
6. **API Gateway Management API**: usada por Lambda para enviar mensajes de vuelta a los clientes

## Pasos sugeridos

1. Crear la tabla DynamoDB para gestionar conexiones (partition key: connectionId, TTL habilitado)
2. Implementar la Lambda de $connect que almacene el connectionId en DynamoDB
3. Implementar la Lambda de $disconnect que elimine el connectionId de DynamoDB
4. Implementar la Lambda de $default que lea todos los connectionIds y haga broadcast del mensaje
5. Crear el WebSocket API en API Gateway con las tres rutas
6. Configurar los permisos IAM para que Lambda pueda usar la Management API de API Gateway
7. Desplegar la API y obtener la URL de WebSocket (wss://...)
8. Probar con wscat conectando multiples clientes y enviando mensajes
9. Implementar manejo de conexiones obsoletas (stale connections) con GoneException
10. (Opcional) Agregar salas de chat con filtrado de broadcast por sala

## Validacion

- [ ] El WebSocket API se crea correctamente en API Gateway
- [ ] La Lambda $connect almacena el connectionId en DynamoDB al conectar
- [ ] La Lambda $disconnect elimina el connectionId al desconectar
- [ ] Los mensajes enviados por un cliente se reciben en todos los demas clientes conectados
- [ ] Las conexiones obsoletas se manejan correctamente (GoneException)
- [ ] El TTL en DynamoDB limpia conexiones viejas automaticamente
- [ ] Multiples clientes pueden conectarse y chatear simultaneamente
- [ ] La aplicacion funciona correctamente con wscat u otro cliente WebSocket
