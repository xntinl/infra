## Why This Matters

Building a single Lambda function that responds to HTTP requests is straightforward. Building a production-grade API is not. Production means your API needs to handle malicious traffic (WAF), give you visibility when something breaks (X-Ray tracing, CloudWatch dashboards), scale without capacity planning (DynamoDB on-demand), and follow the principle of least privilege (scoped IAM policies per function). Skipping any of these layers means your first real incident will be harder to diagnose and more expensive to fix.

API Gateway HTTP API (v2) is the newer, faster, and cheaper alternative to REST API (v1). It supports Lambda proxy integration natively, has lower latency, and costs up to 70% less. For most new APIs that do not need REST API-specific features like request validation models or usage plans, HTTP API is the right choice.

> **Best Practice:** Enable X-Ray tracing end-to-end -- on API Gateway, on every Lambda function, and on the DynamoDB client in your application code. Without tracing, debugging latency spikes in a distributed system requires correlating timestamps across separate CloudWatch log groups, which is slow and error-prone.

## The Challenge

Build a production-grade serverless REST API with the following components:

| Component | Service | Purpose |
|-----------|---------|---------|
| HTTP API | API Gateway v2 | Request routing, CORS, throttling |

| Tracing | X-Ray | End-to-end distributed tracing |

### Requirements

1. **DynamoDB table** with partition key `item_id` (String), on-demand billing, and point-in-time recovery enabled
2. **4 Lambda functions** -- `create-item`, `get-item`, `update-item`, `delete-item` -- each with X-Ray tracing active, 128 MB memory, 30s timeout
3. **API Gateway HTTP API** with routes: `POST /items`, `GET /items/{item_id}`, `PUT /items/{item_id}`, `DELETE /items/{item_id}`
4. **WAF WebACL** with AWS managed rule groups: `AWSManagedRulesCommonRuleSet` and `AWSManagedRulesAmazonIpReputationList`, plus a rate-based rule (1000 requests per 5 minutes)
5. **CloudWatch dashboard** showing API Gateway 4xx/5xx errors, Lambda duration, and DynamoDB consumed read/write capacity
6. **CloudWatch alarm** on API Gateway 5xx error rate exceeding 1% over 5 minutes

## Best Practices

| Practice | Why |
|----------|-----|
| DynamoDB on-demand billing | Unpredictable API traffic makes provisioned capacity risky; on-demand scales automatically with zero capacity planning |
| X-Ray tracing on all layers | Distributed tracing across API Gateway, Lambda, and DynamoDB lets you pinpoint latency bottlenecks in a single trace view |
| WAF on every public API | Without WAF, your API is exposed to automated attacks, scraping bots, and SQL injection attempts from day one |
| Lambda Powertools for structured logging | JSON-formatted logs with correlation IDs make CloudWatch Logs Insights queries possible and incident response faster |
| Separate Lambda per route | Keeps cold starts low, IAM permissions scoped, and deployment blast radius small |
| Point-in-time recovery on DynamoDB | Enables continuous backups; recovering from accidental deletes without PITR requires manual table restoration from on-demand backups |

<summary>Hint 1: DynamoDB table design</summary>

Start with the data layer. Create a DynamoDB table with on-demand billing and point-in-time recovery:

```hcl
resource "aws_dynamodb_table" "this" {
  name         = "${var.project_name}-items"
  billing_mode = "PAY_PER_REQUEST"
  hash_key     = "item_id"

  attribute {
    name = "item_id"
    type = "S"
  }

  point_in_time_recovery {
    enabled = true
  }

  tags = { Name = "${var.project_name}-items" }
}
```

The `PAY_PER_REQUEST` billing mode means you pay per read/write operation with no capacity planning. For an API with unpredictable traffic patterns, this avoids both over-provisioning (wasting money) and under-provisioning (throttling requests).

## Spot the Bug

A colleague wrote this Lambda handler for the API Gateway HTTP API proxy integration. Requests return `502 Bad Gateway` every time, but the function executes successfully in CloudWatch Logs. **What is wrong?**

```python
import json
import boto3
import os

dynamodb = boto3.resource('dynamodb')
table = dynamodb.Table(os.environ['TABLE_NAME'])

def handler(event, context):
    item_id = event['pathParameters']['item_id']
    response = table.get_item(Key={'item_id': item_id})
    item = response.get('Item', {})

    return {
        'body': json.dumps(item)
    }
```

<details>
<summary>Explain the bug</summary>

**The problem:** The Lambda handler is not returning the proper API Gateway proxy response format. API Gateway HTTP API (v2) with `AWS_PROXY` integration expects the response to include a `statusCode` field. Without `statusCode`, API Gateway cannot construct a valid HTTP response and returns `502 Bad Gateway` to the client.

The `body` field alone is not enough. API Gateway treats any response missing `statusCode` as a Lambda execution error.

**The fix:** Always include `statusCode` (and optionally `headers`) in the return object:

```python
def handler(event, context):
    item_id = event['pathParameters']['item_id']
    response = table.get_item(Key={'item_id': item_id})
    item = response.get('Item', {})

    return {
        'statusCode': 200,
        'headers': {'Content-Type': 'application/json'},
        'body': json.dumps(item)
    }
```

For error cases, return the appropriate status code:

```python
if 'Item' not in response:
    return {
        'statusCode': 404,
        'headers': {'Content-Type': 'application/json'},
        'body': json.dumps({'message': 'Item not found'})
    }
```

</details>

  policy = data.aws_iam_policy_document.lambda_dynamodb.json
}

# ---------------------------------------------------------------
# Lambda function code
# ---------------------------------------------------------------
data "archive_file" "lambda" {
  for_each = local.functions

  type        = "zip"
  output_path = "${path.module}/builds/${each.key}.zip"

  source {
    content = <<-PYTHON
      import json
      import os
      import uuid
      import boto3
      from aws_xray_sdk.core import patch_all

      patch_all()

      dynamodb = boto3.resource('dynamodb')
      table = dynamodb.Table(os.environ['TABLE_NAME'])

      def handler(event, context):
          method = '${local.functions[each.key].method}'
          path_params = event.get('pathParameters') or {}
          item_id = path_params.get('item_id', '')
          body = json.loads(event.get('body', '{}') or '{}')

          try:
              if method == 'POST':
                  item_id = str(uuid.uuid4())
                  item = {**body, 'item_id': item_id}
                  table.put_item(Item=item)
                  return _response(201, item)
              elif method == 'GET':
                  result = table.get_item(Key={'item_id': item_id})
                  if 'Item' not in result:
                      return _response(404, {'message': 'Item not found'})
                  return _response(200, result['Item'])
              elif method == 'PUT':
                  item = {**body, 'item_id': item_id}
                  table.put_item(Item=item)
                  return _response(200, item)
              elif method == 'DELETE':
                  table.delete_item(Key={'item_id': item_id})
                  return _response(204, None)
          except Exception as e:
              return _response(500, {'message': str(e)})

      def _response(status, body):
          return {
              'statusCode': status,
              'headers': {'Content-Type': 'application/json'},
              'body': json.dumps(body, default=str) if body else ''
          }
    PYTHON
    filename = "index.py"
  }
}

resource "aws_wafv2_web_acl" "this" {
  name        = "${var.project_name}-waf"

    allow {}

      none {}

      sampled_requests_enabled   = true

    name     = "aws-ip-reputation"

    name     = "rate-limit"

## Summary

- **API Gateway HTTP API (v2)** is faster and cheaper than REST API (v1) for Lambda proxy integrations
- **DynamoDB on-demand** eliminates capacity planning for unpredictable API traffic patterns
- **X-Ray tracing** across API Gateway, Lambda, and DynamoDB provides end-to-end visibility into request latency
- **WAF WebACL** with managed rule groups protects public APIs from common attacks with minimal configuration
- **CloudWatch dashboards and alarms** provide operational awareness without requiring third-party monitoring tools
- Every Lambda response to API Gateway **must include `statusCode`** -- omitting it causes 502 errors
