## Why This Matters

Lambda functions generate logs, but the default `print()` output is unstructured text that is painful to search and impossible to aggregate. When an error occurs in production, you need to trace a single request across multiple Lambda invocations, correlate logs from different services, and measure business metrics -- all without adding hundreds of lines of boilerplate to every function.

AWS Lambda Powertools solves this with three utilities. The **Logger** produces structured JSON logs with automatic correlation IDs that link related log entries. The **Tracer** wraps X-Ray with decorators that automatically capture function input/output and annotate traces with metadata. The **Metrics** utility emits CloudWatch metrics using the Embedded Metric Format (EMF), which means your metrics appear in CloudWatch without any custom metric API calls -- they are extracted from specially-formatted log lines. Together, these three utilities give you production-grade observability with minimal code changes.

> **Best Practice:** Use Powertools Logger for all logging instead of `print()`. The structured JSON output includes timestamp, level, correlation ID, and custom fields that make CloudWatch Insights queries fast and precise.

### `handler.py` (for reference -- use in `data "archive_file"`)

```python
# This is the handler code to embed in your archive_file source.
# It demonstrates Logger, Tracer, and Metrics from Powertools.

import json
from aws_lambda_powertools import Logger, Tracer, Metrics
from aws_lambda_powertools.metrics import MetricUnit
from aws_lambda_powertools.utilities.typing import LambdaContext

# Initialize Powertools utilities
# Logger: structured JSON logging with correlation ID
logger = Logger()
# Tracer: X-Ray tracing with automatic capture
tracer = Tracer()
# Metrics: CloudWatch metrics via Embedded Metric Format
metrics = Metrics()


@tracer.capture_lambda_handler
@logger.inject_lambda_context(log_event=True)
@metrics.log_metrics(capture_cold_start_metric=True)
def handler(event: dict, context: LambdaContext) -> dict:
    # Structured logging -- these fields appear in every log line
    logger.append_keys(order_id=event.get("order_id", "unknown"))

    logger.info("Processing order", extra={
        "amount": event.get("amount", 0),
        "customer": event.get("customer", "unknown")
    })

    # Custom metric -- appears in CloudWatch Metrics
    amount = event.get("amount", 0)
    metrics.add_metric(name="OrderAmount", unit=MetricUnit.Count, value=amount)
    metrics.add_metric(name="OrderProcessed", unit=MetricUnit.Count, value=1)

    # Add custom dimension for filtering in CloudWatch
    metrics.add_dimension(name="Environment", value="lab")

    # Business logic (simulated)
    result = process_order(event)

    logger.info("Order processed successfully", extra={"result": result})
    return {"statusCode": 200, "body": json.dumps(result)}


@tracer.capture_method
def process_order(event: dict) -> dict:
    """Simulated order processing with automatic X-Ray subsegment."""
    order_id = event.get("order_id", "unknown")
    amount = event.get("amount", 0)

    # Tracer automatically creates a subsegment for this method
    tracer.put_annotation(key="OrderId", value=order_id)
    tracer.put_metadata(key="order_details", value=event)

    return {
        "order_id": order_id,
        "status": "processed",
        "amount": amount
    }
```

## Best Practices

| Practice | Why |
|----------|-----|
| Powertools Logger over print() | Structured JSON with automatic correlation IDs, timestamps, and log levels |
| @metrics.log_metrics decorator | Ensures EMF metrics are flushed even if the function throws an exception |
| capture_cold_start_metric=True | Automatically tracks cold starts as a CloudWatch metric for performance monitoring |
| X-Ray Active tracing mode | Traces every invocation (not sampled); critical for debugging in production |
| Powertools Layer (managed) | AWS-managed layer is updated regularly; avoids bundling Powertools in your zip |
| POWERTOOLS_SERVICE_NAME env var | Sets the service name in all logs and traces; used for filtering in CloudWatch Insights |

## Spot the Bug

A colleague deployed this Lambda function with Powertools, but no custom metrics appear in CloudWatch. The function runs without errors. **What is wrong?**

```python
from aws_lambda_powertools import Logger, Metrics
from aws_lambda_powertools.metrics import MetricUnit

logger = Logger()
metrics = Metrics()

def handler(event, context):
    logger.info("Processing event")

    metrics.add_metric(name="EventProcessed", unit=MetricUnit.Count, value=1)

    return {"statusCode": 200}
```

<details>
<summary>Explain the bug</summary>

**The problem:** The `@metrics.log_metrics` decorator is missing from the handler function. Without this decorator, the Metrics utility never flushes the EMF log line to stdout at the end of the invocation. The `add_metric()` call stores the metric in memory, but it is never serialized and printed to CloudWatch Logs.

Additionally, without `@logger.inject_lambda_context`, the logger is missing the Lambda context fields (function name, memory size, cold start status) that make structured logs useful.

**The fix:** Add the decorators:

```python
from aws_lambda_powertools import Logger, Metrics
from aws_lambda_powertools.metrics import MetricUnit

logger = Logger()
metrics = Metrics()

@logger.inject_lambda_context
@metrics.log_metrics(capture_cold_start_metric=True)
def handler(event, context):
    logger.info("Processing event")

    metrics.add_metric(name="EventProcessed", unit=MetricUnit.Count, value=1)

    return {"statusCode": 200}
```

The `@metrics.log_metrics` decorator calls `metrics.serialize_metric_set()` after the handler returns (or throws), which prints the EMF-formatted JSON to stdout. CloudWatch Logs then extracts the metric values automatically.

## Summary

- **Powertools Logger** replaces `print()` with structured JSON logging that includes correlation IDs, timestamps, and custom fields
- **Powertools Tracer** wraps AWS X-Ray with decorators that automatically capture Lambda handler and method calls as subsegments
- **Powertools Metrics** emits CloudWatch metrics via Embedded Metric Format (EMF) -- metrics are extracted from specially-formatted log lines without calling the PutMetricData API
- The `@metrics.log_metrics` decorator is **required** to flush metrics at the end of each invocation
- The AWS-managed Powertools Layer (`017000801446`) provides the package without bundling it in your deployment zip
- `capture_cold_start_metric=True` automatically tracks Lambda cold starts as a CloudWatch metric
