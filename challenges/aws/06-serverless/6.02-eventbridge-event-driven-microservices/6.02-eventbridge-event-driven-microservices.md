## Why This Matters

Tightly coupled services create cascading failures. When service A calls service B synchronously, a slow response from B blocks A, and an outage in B crashes A. Event-driven architectures break this coupling: service A publishes an event to a bus, and any number of consumers react independently. If a consumer is down, events queue up and get processed when it recovers.

AWS EventBridge takes this further with content-based filtering. Instead of every consumer receiving every event and discarding what it does not care about, EventBridge evaluates rules against event attributes and routes only matching events to each target. A custom event bus isolates your application events from AWS service events on the default bus, giving you a clean namespace and separate access control. Combined with dead-letter queues for undeliverable events, this architecture is both resilient and cost-efficient.

> **Best Practice:** Always use a custom event bus for application events. The default bus receives AWS service events (EC2 state changes, GuardDuty findings, etc.) and mixing application events there makes filtering and access control harder.

## Best Practices

| Practice | Why |
|----------|-----|
| Custom event bus | Isolates application events from AWS service events; enables separate IAM policies |
| Content-based filtering | Reduces Lambda invocations and cost by filtering at the bus level, not in handler code |
| Dead-letter queues on every target | Prevents silent event loss when a target is throttled or unavailable |
| Structured event schema | Use consistent `source` and `detail-type` naming conventions across all producers |
| Least-privilege IAM | Each Lambda needs only `logs:*` permissions; EventBridge invokes them via resource-based policy |
| 14-day DLQ retention | Gives operators time to investigate and replay failed events |

## Spot the Bug

A colleague wrote this EventBridge rule to match high-value orders, but it matches nothing at all. **What is wrong?**

```hcl
resource "aws_cloudwatch_event_rule" "high_value" {
  name           = "high-value-order"
  event_bus_name = aws_cloudwatch_event_bus.this.name

  event_pattern = jsonencode({
    source      = ["com.myapp.orders"]
    detail-type = ["OrderCreated"]
    detail = {
      amount = ["> 1000"]
    }
  })
}
```

<details>
<summary>Explain the bug</summary>

**The problem:** The numeric comparison `["> 1000"]` is written as a plain string, not as a proper EventBridge numeric filter. EventBridge interprets this as an exact string match -- it looks for the literal string `"> 1000"` in the `amount` field. Since the actual `amount` field contains a number (like `5000`), it never matches the string `"> 1000"`, so the rule silently drops all events.

**The fix:** Use the proper numeric matching syntax with a nested object:

```hcl
event_pattern = jsonencode({
  source      = ["com.myapp.orders"]
  detail-type = ["OrderCreated"]
  detail = {
    amount = [{ numeric = [">", 1000] }]
  }
})
```

The `numeric` key tells EventBridge to perform a numeric comparison. The value `[">", 1000]` means "greater than 1000" where `1000` is an actual number, not a string.

</details>

```hcl
data "aws_iam_policy_document" "lambda_assume" {
  statement {
    actions = ["sts:AssumeRole"]

      type        = "Service"
      identifiers = ["lambda.amazonaws.com"]

resource "aws_iam_role" "lambda" {
  name               = "${var.project_name}-lambda-role"
  assume_role_policy = data.aws_iam_policy_document.lambda_assume.json

  tags = {
    Name = "${var.project_name}-lambda-role"
  }
}

resource "aws_iam_role_policy_attachment" "lambda_basic" {
  role       = aws_iam_role.lambda.name
  policy_arn = "arn:aws:iam::aws:policy/service-role/AWSLambdaBasicExecutionRole"
}

locals {
  consumers = {
    order_processor = {
      description = "Processes new order events"
    }
    payment_processor = {
      description = "Processes payment completion events"
    }
    vip_notifier = {
      description = "Notifies VIP team of high-value orders"
    }
  }
}

data "archive_file" "lambda" {
  for_each = local.consumers

  type        = "zip"
  output_path = "${path.module}/builds/${each.key}.zip"

  source {
    content = <<-PYTHON
      import json

      def handler(event, context):
          print(f"Consumer: ${each.key}")
          print(f"Event: {json.dumps(event)}")

          source = event.get('source', 'unknown')
          detail_type = event.get('detail-type', 'unknown')
          detail = event.get('detail', {})

          print(f"Source: {source}, DetailType: {detail_type}")
          print(f"Detail: {json.dumps(detail)}")

          return {
              'statusCode': 200,
              'body': json.dumps({
                  'consumer': '${each.key}',
                  'processed': True
              })
          }
    PYTHON
    filename = "index.py"
  }
}

resource "aws_lambda_function" "this" {
  for_each = local.consumers

  function_name = "${var.project_name}-${replace(each.key, "_", "-")}"
  description   = each.value.description
  role          = aws_iam_role.lambda.arn
  handler       = "index.handler"
  runtime       = "python3.12"
  timeout       = 30
  memory_size   = 128
  publish       = true

  filename         = data.archive_file.lambda[each.key].output_path
  source_code_hash = data.archive_file.lambda[each.key].output_base64sha256

  tags = {
    Name = "${var.project_name}-${replace(each.key, "_", "-")}"
  }
}

resource "aws_cloudwatch_log_group" "lambda" {
  for_each = local.consumers

  name              = "/aws/lambda/${var.project_name}-${replace(each.key, "_", "-")}"
  retention_in_days = 14

  tags = {
    Name = "${var.project_name}-${replace(each.key, "_", "-")}-logs"
```

## Summary

- A **custom EventBridge bus** isolates application events from AWS service events on the default bus
- **Event rules** with content-based filtering route events to specific targets based on source, detail-type, and detail field values
- **Numeric matching** (`[{ "numeric": [">", 1000] }]`) enables filtering on numeric fields without invoking consumers
- **Dead-letter queues** on every EventBridge target prevent silent event loss when targets are throttled or unavailable
- **Lambda resource-based policies** (`aws_lambda_permission`) allow EventBridge to invoke functions without adding IAM policies to the bus
- EventBridge supports up to **5 targets per rule** and up to **300 rules per bus** (soft limits)
