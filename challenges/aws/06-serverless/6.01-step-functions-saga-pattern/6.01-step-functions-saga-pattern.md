## Why This Matters

In a monolithic application, a database transaction guarantees that either all operations succeed or all roll back. Microservices break this guarantee because each service owns its own data store. An order that reserves inventory, charges a payment, and arranges shipping involves three independent services -- if payment fails after inventory is reserved, you need a way to release that inventory.

The saga pattern solves this by pairing every forward action with a compensating action. AWS Step Functions orchestrates the saga: it calls each Lambda in sequence and, if any step fails, executes the compensation steps in reverse order. DynamoDB tracks the saga state so you can audit what happened and retry if needed. Without this pattern, distributed systems accumulate inconsistent state that is expensive to detect and painful to fix.

> **Best Practice:** Every forward step in a saga must have a corresponding compensating action. Design compensations to be idempotent -- they may be called more than once if the state machine retries.

## Best Practices

| Practice | Why |
|----------|-----|
| Idempotent compensations | Compensating actions may run more than once if retries occur; use conditional writes in DynamoDB |
| DynamoDB for saga state | On-demand billing and single-digit-ms latency make it ideal for tracking distributed transaction state |
| Step Functions Standard (not Express) | Standard workflows support executions up to one year and provide exactly-once semantics per state |
| Separate compensation Lambdas | Keeps forward and backward logic isolated; easier to test and reason about |
| Catch + Retry in ASL | Use `Retry` for transient errors and `Catch` for permanent failures that need compensation |
| CloudWatch Log Groups with retention | Prevents unbounded log growth; 14 days is sufficient for debugging |

## Spot the Bug

A colleague wrote this Step Functions definition. The saga is supposed to roll back inventory when payment fails, but the compensation never runs when shipping fails. **What is wrong?**

```json
{
  "StartAt": "ReserveInventory",
  "States": {
    "ReserveInventory": {
      "Type": "Task",
      "Resource": "arn:aws:lambda:us-east-1:123456789012:function:reserve-inventory",
      "Next": "ChargePayment"
    },
    "ChargePayment": {
      "Type": "Task",
      "Resource": "arn:aws:lambda:us-east-1:123456789012:function:charge-payment",
      "Next": "ArrangeShipping",
      "Catch": [{
        "ErrorEquals": ["States.ALL"],
        "Next": "CancelInventory"
      }]
    },
    "ArrangeShipping": {
      "Type": "Task",
      "Resource": "arn:aws:lambda:us-east-1:123456789012:function:arrange-shipping",
      "End": true
    },
    "CancelInventory": {
      "Type": "Task",
      "Resource": "arn:aws:lambda:us-east-1:123456789012:function:cancel-inventory",
      "End": true
    }
  }
}
```

<details>
<summary>Explain the bug</summary>

**The problem:** The `ArrangeShipping` state has no `Catch` block. If the `ChargePayment` step fails, the `CancelInventory` compensation runs correctly. But if `ArrangeShipping` fails, the entire execution fails without running any compensation -- both the payment charge and the inventory reservation are left in an inconsistent state.

Additionally, when `ArrangeShipping` fails, both `refund_payment` and `cancel_inventory` should run (since both payment and inventory need to be reversed), but only `CancelInventory` exists as a compensation target.

**The fix:** Add `Catch` blocks to every Task state that has preceding forward steps. When `ArrangeShipping` fails, it should chain through `RefundPayment` then `CancelInventory`:

```json
"ArrangeShipping": {
  "Type": "Task",
  "Resource": "arn:aws:lambda:us-east-1:123456789012:function:arrange-shipping",
  "End": true,
  "Catch": [{
    "ErrorEquals": ["States.ALL"],
    "Next": "RefundPayment"
  }]
},
"RefundPayment": {
  "Type": "Task",
  "Resource": "arn:aws:lambda:us-east-1:123456789012:function:refund-payment",
  "Next": "CancelInventory"
},
"CancelInventory": {
  "Type": "Task",
  "Resource": "arn:aws:lambda:us-east-1:123456789012:function:cancel-inventory",
  "End": true
}
```

</details>

<summary>TODO 2 -- Lambda functions for saga steps (lambda.tf)</summary>

```hcl
locals {
  saga_functions = {
    reserve_inventory = { description = "Reserve inventory for order" }
    charge_payment    = { description = "Charge payment for order" }
    arrange_shipping  = { description = "Arrange shipping for order" }
    cancel_inventory  = { description = "Compensate: cancel inventory reservation" }
    refund_payment    = { description = "Compensate: refund payment" }
  }
}

data "archive_file" "lambda" {
  for_each = local.saga_functions

  type        = "zip"
  output_path = "${path.module}/builds/${each.key}.zip"

  source {
    content = <<-PYTHON
      import json
      import os
      import boto3
      from datetime import datetime

      dynamodb = boto3.resource('dynamodb')
      table = dynamodb.Table(os.environ['TABLE_NAME'])

      def handler(event, context):
          order_id = event.get('order_id', 'unknown')
          step_name = '${each.key}'

          # Record saga step in DynamoDB
          table.update_item(
              Key={'order_id': order_id},
              UpdateExpression='SET #step = :status, updated_at = :ts',
              ExpressionAttributeNames={'#step': step_name},
              ExpressionAttributeValues={
                  ':status': 'completed',
                  ':ts': datetime.utcnow().isoformat()
              }
          )

          print(f"Step {step_name} completed for order {order_id}")
          return {
              'order_id': order_id,
              'step': step_name,
              'status': 'completed'
          }
    PYTHON
    filename = "index.py"
  }
}

resource "aws_lambda_function" "this" {
  for_each = local.saga_functions

  function_name = "${var.project_name}-${replace(each.key, "_", "-")}"
  description   = each.value.description
  role          = aws_iam_role.lambda.arn
  handler       = "index.handler"
  runtime       = "python3.12"
  timeout       = 30
  memory_size   = 128
  publish       = true

  filename         = data.archive_file.lambda[each.key].output_path
  source_code_hash = data.archive_file.lambda[each.key].output_base64sha256

  environment {
    variables = {
      TABLE_NAME = aws_dynamodb_table.saga_state.name
    }
  }

  tags = {
    Name = "${var.project_name}-${replace(each.key, "_", "-")}"
  }
}
```

<summary>TODO 5 -- Step Functions state machine (step_function.tf)</summary>

```hcl
resource "aws_sfn_state_machine" "this" {
  name     = "${var.project_name}-order-saga"
  role_arn = aws_iam_role.sfn.arn
  type     = "STANDARD"

  definition = jsonencode({
    Comment = "Order processing saga with compensation"
    StartAt = "ReserveInventory"
    States = {
      ReserveInventory = {
        Type     = "Task"
        Resource = aws_lambda_function.this["reserve_inventory"].arn
        Next     = "ChargePayment"
        Catch = [{
          ErrorEquals = ["States.ALL"]
          Next        = "SagaFailed"
          ResultPath  = "$.error"
        }]
      }
      ChargePayment = {
        Type     = "Task"
        Resource = aws_lambda_function.this["charge_payment"].arn
        Next     = "ArrangeShipping"
        Catch = [{
          ErrorEquals = ["States.ALL"]
          Next        = "CancelInventory"
          ResultPath  = "$.error"
        }]
      }
      ArrangeShipping = {
        Type     = "Task"
        Resource = aws_lambda_function.this["arrange_shipping"].arn
        Next     = "SagaSucceeded"
        Catch = [{
          ErrorEquals = ["States.ALL"]
          Next        = "RefundPayment"
          ResultPath  = "$.error"
        }]
      }
      RefundPayment = {
        Type     = "Task"
        Resource = aws_lambda_function.this["refund_payment"].arn
        Next     = "CancelInventory"
      }
      CancelInventory = {
        Type     = "Task"
        Resource = aws_lambda_function.this["cancel_inventory"].arn
        Next     = "SagaFailed"
      }
      SagaSucceeded = {
        Type = "Succeed"
      }
      SagaFailed = {
        Type  = "Fail"
        Error = "SagaCompensated"
        Cause = "One or more saga steps failed; compensations executed"
      }
    }
  })

  tags = {
    Name = "${var.project_name}-order-saga"
  }
}
```

## Summary

- The **saga pattern** replaces distributed transactions with a sequence of local transactions paired with compensating actions
- **Step Functions** orchestrates the saga, calling Lambda functions in sequence and branching to compensations on failure
- **Catch blocks** in ASL redirect execution to compensation states when a Task throws an error
- **DynamoDB** provides low-latency, serverless state tracking for auditing each saga step
- Every compensation must be **idempotent** because retries may invoke it more than once
- **Standard** workflows (not Express) provide the durability and exactly-once semantics needed for financial transactions
