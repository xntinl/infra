# 15.09 Meta-Arguments en Profundidad

## Metadata
| Campo | Valor |
|-------|-------|
| **Complejidad** | Intermedio |
| **Modalidad** | Terraform |
| **Features TF** | All meta-arguments, lifecycle rules |

## Descripcion

Practica todos los meta-arguments: `depends_on` para dependencias ocultas, `lifecycle` (create_before_destroy, prevent_destroy, ignore_changes, replace_triggered_by), `provider` para multi-region. Comprende cuando cada uno es necesario y sus anti-patrones.

## Objetivos de aprendizaje

- Dominar el ciclo de vida de recursos y como controlarlo con `lifecycle`
- Comprender la gestion de dependencias explicitas con `depends_on`
- Usar replacement triggers para forzar recreacion de recursos
- Identificar patrones de proteccion contra destruccion accidental

## Requisitos previos

- Conocimientos: Recursos de Terraform, grafo de dependencias, plan/apply
- Herramientas: Terraform >= 1.5, AWS CLI, cuenta AWS
- Features/Servicios: EC2, RDS, S3, Security Groups, IAM

## Arquitectura

```
main.tf
  +-- aws_db_instance "this"
  |     lifecycle {
  |       prevent_destroy = true           # Proteccion contra borrado
  |       ignore_changes  = [password]     # Ignorar cambios externos
  |     }
  |
  +-- aws_instance "this"
  |     lifecycle {
  |       create_before_destroy = true     # Zero-downtime replacement
  |       replace_triggered_by  = [        # Trigger de reemplazo
  |         aws_security_group.this.id
  |       ]
  |     }
  |
  +-- aws_s3_bucket "this"
  |     depends_on = [aws_iam_role.this]   # Dependencia explicita
  |
  +-- Recursos con provider = aws.secondary  # Meta-arg provider
```

## Pasos sugeridos

1. Crear una instancia RDS con `prevent_destroy = true` y verificar que `terraform destroy` la protege
2. Implementar `ignore_changes` en un recurso cuyo password se rota externamente
3. Crear una instancia EC2 con `create_before_destroy = true` y observar el orden de operaciones al cambiar la AMI
4. Implementar `replace_triggered_by` para forzar recreacion de una instancia cuando cambie su security group
5. Usar `depends_on` para declarar una dependencia que Terraform no puede inferir automaticamente (ej: bucket S3 que depende de una IAM policy)
6. Demostrar por que `depends_on` es un anti-patron cuando la dependencia es inferible y cuando es necesario
7. Combinar `lifecycle` con `ignore_changes = all` para recursos gestionados externamente
8. Implementar el patron de "import and protect" para recursos existentes
9. Probar cada meta-argument individualmente y observar el comportamiento en `terraform plan`
10. Documentar los anti-patrones comunes y sus alternativas preferidas

## Validacion

- [ ] `prevent_destroy` bloquea la destruccion del recurso con un error claro
- [ ] `ignore_changes` permite que cambios externos no generen drift en el plan
- [ ] `create_before_destroy` crea el nuevo recurso antes de destruir el anterior
- [ ] `replace_triggered_by` fuerza la recreacion cuando el trigger cambia
- [ ] `depends_on` establece el orden correcto para dependencias ocultas
- [ ] Se identifican los casos validos vs anti-patrones para cada meta-argument
- [ ] `terraform plan` refleja el comportamiento esperado en cada escenario
- [ ] Se documentan las alternativas preferidas a cada anti-patron
