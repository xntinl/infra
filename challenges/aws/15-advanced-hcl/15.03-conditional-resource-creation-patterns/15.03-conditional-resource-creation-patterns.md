# 15.03 Patrones de Creacion Condicional de Recursos

## Metadata
| Campo | Valor |
|-------|-------|
| **Complejidad** | Intermedio |
| **Modalidad** | Terraform |
| **Features TF** | `count`, ternary, `one()`, `try()`, conditional modules |

## Descripcion

Implementa diversos patrones condicionales: `count = var.enabled ? 1 : 0`, modulos condicionales, expresiones ternarias para atributos, funcion `one()` para data sources opcionales, y `try()` para degradacion graceful.

## Objetivos de aprendizaje

- Dominar los patrones condicionales disponibles en Terraform
- Implementar feature flags en infraestructura como codigo
- Comprender la creacion opcional de recursos y modulos
- Usar `one()` y `try()` para manejar datos opcionales de forma segura

## Requisitos previos

- Conocimientos: HCL basico, `count`, expresiones condicionales
- Herramientas: Terraform >= 1.5, AWS CLI, cuenta AWS
- Features/Servicios: Lambda, CloudWatch, SNS, S3

## Arquitectura

```
variables.tf
  +-- enable_monitoring    = bool
  +-- enable_notifications = bool
  +-- enable_backup        = bool
  +-- environment          = string
  |
  v
main.tf
  +-- aws_lambda_function "this"                (siempre)
  +-- aws_cloudwatch_metric_alarm "this"        (count = var.enable_monitoring ? 1 : 0)
  +-- module "notifications"                    (count = var.enable_notifications ? 1 : 0)
  +-- aws_s3_bucket "backup"                    (count = var.enable_backup ? 1 : 0)
  +-- Atributos condicionales via ternary
  +-- data sources opcionales con one()
  +-- Valores con try() para fallback seguro
```

Se crea una infraestructura base con componentes opcionales controlados por variables booleanas, demostrando multiples patrones condicionales.

## Pasos sugeridos

1. Definir variables booleanas para cada feature flag (`enable_monitoring`, `enable_notifications`, etc.)
2. Crear un recurso base (Lambda) que siempre existe
3. Implementar `count = var.enabled ? 1 : 0` para recursos opcionales como alarmas de CloudWatch
4. Crear un modulo condicional para notificaciones SNS usando `count` en el bloque `module`
5. Usar expresiones ternarias para atributos condicionales (ej: `timeout = var.environment == "prod" ? 30 : 10`)
6. Implementar `one()` para acceder de forma segura a data sources que pueden no existir
7. Usar `try()` para proveer valores por defecto cuando una expresion puede fallar
8. Combinar `coalesce()` y `coalescelist()` para seleccionar el primer valor no nulo
9. Crear outputs condicionales que solo se emitan cuando el recurso existe
10. Probar activando y desactivando cada feature flag para verificar el comportamiento

## Validacion

- [ ] Los recursos opcionales se crean solo cuando su feature flag esta activo
- [ ] Los modulos condicionales funcionan correctamente con `count`
- [ ] Las expresiones ternarias asignan valores distintos segun el entorno
- [ ] `one()` maneja correctamente data sources opcionales sin errores
- [ ] `try()` provee valores de fallback cuando la expresion falla
- [ ] Los outputs condicionales se emiten solo cuando corresponde
- [ ] Cambiar un flag de `true` a `false` destruye solo el recurso asociado
- [ ] `terraform plan` es limpio sin warnings inesperados
