# 15.04 Provisioners: local-exec y remote-exec

## Metadata
| Campo | Valor |
|-------|-------|
| **Complejidad** | Intermedio |
| **Modalidad** | Terraform |
| **Features TF** | `local-exec`, `remote-exec`, `null_resource`, `terraform_data` |

## Descripcion

Implementa patrones de provisioners: `local-exec` para ejecutar scripts tras la creacion de recursos (configurar contexto de kubectl, poblar una base de datos), `remote-exec` para bootstrapping de EC2. Comprende por que los provisioners son un ultimo recurso y explora alternativas.

## Objetivos de aprendizaje

- Comprender los patrones de uso de provisioners y cuando son apropiados
- Implementar `local-exec` para tareas post-creacion en la maquina local
- Implementar `remote-exec` para configuracion inicial de instancias EC2
- Conocer `terraform_data` como reemplazo moderno de `null_resource`
- Entender las alternativas preferidas a provisioners (user_data, SSM, etc.)

## Requisitos previos

- Conocimientos: HCL basico, EC2, SSH, scripting bash
- Herramientas: Terraform >= 1.5, AWS CLI, SSH key pair, cuenta AWS
- Features/Servicios: EC2, VPC, Security Groups, SSM (para alternativas)

## Arquitectura

```
main.tf
  +-- aws_instance "web"
  |     +-- provisioner "remote-exec" { ... }
  |     +-- provisioner "local-exec" { ... }
  |     +-- connection { type = "ssh" ... }
  |
  +-- terraform_data "config_update"
  |     +-- triggers_replace = [timestamp()]
  |     +-- provisioner "local-exec" { ... }
  |
  +-- null_resource "legacy_example"  (para comparacion)
  |     +-- triggers = { ... }
  |     +-- provisioner "local-exec" { ... }
  |
  +-- Alternativas demostradas:
        +-- aws_instance con user_data
        +-- aws_ssm_document para configuracion
```

## Pasos sugeridos

1. Crear una VPC con subnet publica y security group que permita SSH
2. Generar o importar un key pair para conexion SSH
3. Crear una instancia EC2 con un provisioner `remote-exec` inline que instale nginx
4. Agregar un provisioner `local-exec` que registre la IP publica en un archivo local
5. Configurar el bloque `connection` con tipo SSH y la clave privada
6. Implementar `on_failure = continue` y `on_failure = fail` para manejar errores
7. Crear un `terraform_data` con `triggers_replace` que ejecute un script local cuando cambie un valor
8. Comparar con `null_resource` y sus `triggers` para entender la evolucion
9. Implementar la misma funcionalidad usando `user_data` como alternativa preferida
10. Documentar cuando usar cada enfoque y sus trade-offs

## Validacion

- [ ] `remote-exec` ejecuta comandos en la instancia EC2 tras su creacion
- [ ] `local-exec` ejecuta scripts en la maquina local tras crear recursos
- [ ] El bloque `connection` establece la sesion SSH correctamente
- [ ] `on_failure` maneja errores segun la configuracion
- [ ] `terraform_data` reemplaza correctamente a `null_resource`
- [ ] `triggers_replace` provoca la re-ejecucion cuando cambian los valores
- [ ] Se documenta por que `user_data` o SSM son preferibles a provisioners
- [ ] Los provisioners se ejecutan solo en la creacion (no en updates)
