# 20.10 Regression Testing para Module Upgrades

## Metadata
| Campo | Valor |
|-------|-------|
| **Complejidad** | Avanzado |
| **Modalidad** | Terraform + CI |
| **Features TF** | Module versioning, `terraform plan`, state comparison |

## Descripcion

Antes de actualizar la version de un modulo, ejecuta tests de regresion: despliega con la version anterior, captura los atributos de los recursos, actualiza el modulo, ejecuta plan para identificar cambios y valida que no haya modificaciones inesperadas. Automatiza la comparacion en un pipeline de CI.

## Objetivos de aprendizaje

- Implementar un flujo seguro de actualizacion de modulos
- Usar `terraform plan` para detectar cambios inesperados al actualizar versiones
- Automatizar la comparacion de state antes y despues del upgrade
- Disenar regression tests que validen la compatibilidad de versiones

## Requisitos previos

- Conocimientos: Terraform avanzado, modulos versionados, CI/CD
- Herramientas: Terraform >= 1.0, jq, diff, Git
- Features TF: Module version constraints, `terraform show -json`, plan output

## Arquitectura

Mantener un modulo versionado en un registry (Git tags o S3). En el pipeline CI, desplegar con la version actual, capturar el state. Luego actualizar la version del modulo, ejecutar `terraform plan`, y comparar los cambios propuestos contra una lista de cambios esperados.

## Pasos sugeridos

1. Configurar un modulo con version pinning (e.g., `version = "1.0.0"`)
2. Desplegar la infraestructura con la version actual y capturar `terraform show -json > baseline.json`
3. Crear un script que extraiga atributos clave del state JSON (resource types, names, key attributes)
4. Actualizar la version del modulo en la configuracion (e.g., `version = "2.0.0"`)
5. Ejecutar `terraform plan -out=upgrade.plan` y convertir a JSON con `terraform show -json upgrade.plan`
6. Parsear el plan JSON para identificar creates, updates, y deletes
7. Comparar contra una lista de cambios esperados (allowlist)
8. Fallar el pipeline si hay cambios no esperados (deletes o recreates no autorizados)
9. Automatizar todo el flujo en GitHub Actions o similar

## Validacion

- [ ] El baseline del state se captura correctamente en JSON
- [ ] El plan de upgrade se genera sin errores
- [ ] Los cambios esperados se identifican correctamente en el plan
- [ ] Cambios inesperados (deletes, recreates) se detectan y reportan
- [ ] El pipeline falla automaticamente ante cambios no autorizados
- [ ] El flujo completo se ejecuta en CI sin intervencion manual
