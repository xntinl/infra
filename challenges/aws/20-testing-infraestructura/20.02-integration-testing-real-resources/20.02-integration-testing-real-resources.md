# 20.02 Integration Testing con recursos reales

## Metadata
| Campo | Valor |
|-------|-------|
| **Complejidad** | Avanzado |
| **Modalidad** | Terraform |
| **Features TF** | `terraform test`, `command = apply`, lifecycle |

## Descripcion

Escribe integration tests que desplieguen infraestructura real, la validen (llamadas HTTP, verificaciones con SDK) y la destruyan. Usa `terraform test` con `command = apply`. Maneja la limpieza ante fallos de tests. Ejecuta los tests en una cuenta AWS aislada.

## Objetivos de aprendizaje

- Implementar patrones de integration testing que despliegan recursos reales
- Validar infraestructura desplegada con verificaciones funcionales
- Disenar estrategias de limpieza robustas ante fallos de tests

## Requisitos previos

- Conocimientos: Terraform avanzado, `terraform test` basico (ver challenge 20.01), conceptos de testing de integracion
- Herramientas: Terraform >= 1.6, AWS CLI, cuenta AWS aislada para testing
- Features: `terraform test` con `command = apply`, bloques `run` con lifecycle, data sources para validacion

## Arquitectura

Tests de integracion que despliegan y validan infraestructura real:

```
module/
  main.tf
  variables.tf
  outputs.tf
  tests/
    setup/
      main.tf          # Infraestructura auxiliar para tests
    integration.tftest.hcl
    cleanup.tftest.hcl
```

El flujo es: setup (crear dependencias) -> apply (desplegar modulo) -> validate (verificar recursos) -> destroy (limpiar todo). Los tests usan `command = apply` para crear recursos reales y data sources para validar su estado.

## Pasos sugeridos

1. Configurar una cuenta AWS aislada dedicada exclusivamente a testing
2. Crear un modulo de Terraform para testear (por ejemplo, ALB + Lambda + API Gateway)
3. Escribir un modulo de setup auxiliar que cree dependencias necesarias (VPC, subnets)
4. Crear el archivo `.tftest.hcl` con `command = apply` para desplegar recursos reales
5. Agregar bloques `run` que validen los recursos creados usando data sources y outputs
6. Implementar validaciones funcionales (verificar que un endpoint HTTP responde correctamente)
7. Configurar el manejo de cleanup para que los recursos se destruyan incluso si un test falla
8. Agregar tests que verifiquen propiedades de seguridad (cifrado habilitado, acceso publico bloqueado)
9. Ejecutar los tests completos y verificar que el ciclo create-validate-destroy funciona
10. Documentar los tiempos de ejecucion y estrategias para optimizar la velocidad

## Validacion

- [ ] Los tests usan `command = apply` y crean recursos reales en AWS
- [ ] El modulo de setup prepara correctamente las dependencias necesarias
- [ ] Las validaciones funcionales verifican que los recursos operan correctamente (no solo que existen)
- [ ] Los tests verifican propiedades de seguridad de los recursos desplegados
- [ ] La limpieza destruye todos los recursos incluso cuando un test intermedio falla
- [ ] Los tests se ejecutan en una cuenta aislada sin afectar produccion
- [ ] Los outputs del modulo son correctos y accesibles para validacion
- [ ] El ciclo completo (setup -> apply -> validate -> destroy) se ejecuta sin residuos
