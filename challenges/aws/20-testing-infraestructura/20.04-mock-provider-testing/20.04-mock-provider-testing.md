# 20.04 Mock Provider Testing

## Metadata
| Campo | Valor |
|-------|-------|
| **Complejidad** | Intermedio |
| **Modalidad** | Terraform |
| **Features TF** | `mock_provider`, `override_resource`, `override_data` |

## Descripcion

Usa mock providers de Terraform 1.7+ para testear logica de modulos sin desplegar recursos reales. Simula respuestas del provider AWS. Testea logica condicional, validacion de variables y calculos de outputs puramente en memoria.

## Objetivos de aprendizaje

- Utilizar mock providers para testing rapido sin interaccion con la nube
- Testear logica de modulos (condicionales, loops, locals) de forma aislada
- Validar calculos de outputs y transformaciones de datos sin costo ni latencia

## Requisitos previos

- Conocimientos: Terraform intermedio, `terraform test` basico (ver challenge 20.01), logica de modulos (count, for_each, conditionals)
- Herramientas: Terraform >= 1.7
- Features: `mock_provider`, `override_resource`, `override_data`, `terraform test`

## Arquitectura

Estructura de tests con mock providers:

```
module/
  main.tf           # Logica condicional, locals, recursos
  variables.tf      # Variables con validaciones
  outputs.tf        # Outputs calculados
  tests/
    mocks/
      mock_aws.tftest.hcl    # Mock provider definitions
    logic.tftest.hcl          # Tests de logica condicional
    validation.tftest.hcl     # Tests de validacion de variables
    outputs.tftest.hcl        # Tests de calculos de outputs
```

Los mock providers simulan el comportamiento del provider AWS, permitiendo ejecutar plan y apply sin conexion a AWS. Se pueden override recursos y data sources con valores especificos para cada escenario de test.

## Pasos sugeridos

1. Crear un modulo con logica compleja: condicionales, for_each, locals calculados
2. Configurar un `mock_provider` para AWS que simule respuestas basicas
3. Escribir tests que usen `override_resource` para definir valores de recursos simulados
4. Crear tests con `override_data` para simular data sources (AMIs, VPCs existentes)
5. Testear logica condicional: verificar que `count` y `for_each` generan los recursos correctos
6. Validar que las validation rules de variables rechazan inputs invalidos
7. Verificar que los outputs calculados (locals, funciones) producen los valores esperados
8. Comparar tiempos de ejecucion: mock tests vs. plan-only tests vs. integration tests

## Validacion

- [ ] El `mock_provider` esta correctamente configurado y Terraform lo reconoce
- [ ] `override_resource` simula valores de recursos sin crear nada en AWS
- [ ] `override_data` simula data sources con valores controlados
- [ ] Los tests de logica condicional verifican los caminos correctos (count 0 vs 1, for_each)
- [ ] Los tests de validacion rechazan inputs invalidos con los mensajes esperados
- [ ] Los calculos de outputs producen valores correctos con datos simulados
- [ ] Los tests se ejecutan en segundos (sin latencia de red ni API calls)
- [ ] Los tests cubren multiples escenarios de configuracion del modulo
