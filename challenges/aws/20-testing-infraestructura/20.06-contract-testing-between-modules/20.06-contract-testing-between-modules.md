# 20.06 Contract Testing entre modulos

## Metadata
| Campo | Valor |
|-------|-------|
| **Complejidad** | Avanzado |
| **Modalidad** | Terraform |
| **Features TF** | Output types, `terraform test`, validation blocks |

## Descripcion

Define "contratos" entre modulos usando tipos de output y validacion. Verifica que los modulos consumidores reciben los outputs esperados de los modulos proveedores. Testea que los cambios de interfaz se detectan antes del despliegue.

## Objetivos de aprendizaje

- Disenar contratos de interfaz entre modulos de Terraform
- Implementar testing de interfaces para detectar breaking changes
- Usar validation blocks y output types para reforzar contratos
- Prevenir roturas de integracion entre modulos antes del despliegue

## Requisitos previos

- Conocimientos: Terraform avanzado, diseno de modulos, composicion de modulos, tipos de datos en Terraform
- Herramientas: Terraform >= 1.6
- Features: Output types, `terraform test`, variable validation blocks, type constraints

## Arquitectura

Estructura de modulos con contract testing:

```
modules/
  network/            # Modulo proveedor
    outputs.tf        # Contrato: vpc_id, subnet_ids, security_group_ids
  compute/            # Modulo consumidor
    variables.tf      # Espera: vpc_id (string), subnet_ids (list), sg_ids (list)
  database/           # Modulo consumidor
    variables.tf      # Espera: vpc_id (string), db_subnet_ids (list)
tests/
  contracts/
    network_outputs.tftest.hcl     # Valida outputs del proveedor
    compute_inputs.tftest.hcl      # Valida que compute acepta outputs de network
    database_inputs.tftest.hcl     # Valida que database acepta outputs de network
    integration.tftest.hcl         # Valida composicion completa
```

Los contratos definen que outputs produce cada modulo proveedor y que inputs espera cada consumidor. Los tests verifican compatibilidad y detectan breaking changes.

## Pasos sugeridos

1. Disenar los modulos proveedor y consumidor con interfaces claras (tipos explicitos en outputs y variables)
2. Definir contratos formales: documentar los tipos, formatos y restricciones de cada output/input
3. Implementar validation blocks en las variables de los modulos consumidores
4. Crear tests que verifiquen que los outputs del modulo proveedor cumplen el contrato de tipo
5. Escribir tests que verifiquen que los modulos consumidores aceptan los outputs del proveedor
6. Crear tests de integracion que compongan proveedor + consumidor y validen el flujo completo
7. Simular un breaking change (cambiar tipo de output) y verificar que los tests lo detectan
8. Simular una adicion retrocompatible y verificar que los tests siguen pasando
9. Documentar los contratos como referencia para el equipo
10. Integrar los contract tests en CI para ejecutarse en cada cambio a modulos

## Validacion

- [ ] Los outputs de los modulos proveedores tienen tipos explicitos definidos
- [ ] Las variables de los modulos consumidores tienen validation blocks estrictos
- [ ] Los tests verifican que los outputs del proveedor cumplen el contrato
- [ ] Los tests verifican que el consumidor acepta correctamente los outputs del proveedor
- [ ] Un breaking change en el tipo de output es detectado por los tests antes del despliegue
- [ ] Una adicion retrocompatible no rompe los tests existentes
- [ ] Los tests de integracion validan la composicion completa de modulos
- [ ] Los contract tests estan integrados en CI y se ejecutan automaticamente
