# 3.10 EFS Cross-AZ y Pruebas de Rendimiento

## Metadata
| Campo | Valor |
|-------|-------|
| **Complejidad** | Intermedio |
| **Modalidad** | Ambos |
| **Servicios AWS** | EFS, EC2, VPC, CloudWatch |

## Descripcion

Desplegar EFS con diferentes modos de rendimiento (General Purpose vs Max I/O) y modos de throughput (Bursting vs Provisioned vs Elastic). Montar desde instancias EC2 en multiples AZs, ejecutar benchmarks de I/O, comparar las caracteristicas de rendimiento, e implementar access points.

## Objetivos de aprendizaje

- Comprender los patrones de almacenamiento compartido de archivos con EFS
- Seleccionar el modo de rendimiento adecuado segun el caso de uso
- Planificar el throughput necesario y entender los creditos de burst
- Implementar access points para control de acceso granular
- Configurar seguridad NFS con security groups y policies

## Requisitos previos

- **Conocimientos**: Fundamentos de sistemas de archivos, NFS basico, conceptos de VPC y EC2, I/O performance
- **Herramientas**: AWS CLI v2, Terraform >= 1.5, herramientas de benchmarking (fio, dd, iozone)
- **Servicios AWS**: VPC con subnets en multiples AZs, permisos para EFS, EC2 y CloudWatch

## Arquitectura

Dos file systems EFS: uno con General Purpose performance mode y otro con Max I/O. Cada uno tiene mount targets en 3 AZs de la VPC. Instancias EC2 en cada AZ montan ambos file systems usando NFS v4.1. Se configuran EFS access points para diferentes aplicaciones/equipos con directorios raiz, UIDs/GIDs y permisos distintos. Se ejecutan benchmarks de I/O (lectura secuencial, escritura secuencial, lectura aleatoria, escritura aleatoria) en ambos file systems desde multiples instancias simultaneamente. CloudWatch recopila metricas de throughput, IOPS, latencia y burst credits.

## Pasos sugeridos

1. Crear la VPC con subnets en al menos 3 AZs y configurar security groups para NFS (puerto 2049)
2. Crear el primer file system EFS con General Purpose performance mode y Bursting throughput
3. Crear el segundo file system EFS con Max I/O performance mode y Provisioned throughput
4. Crear mount targets en cada AZ para ambos file systems
5. Lanzar instancias EC2 en diferentes AZs y montar ambos file systems via NFS v4.1
6. Crear access points con diferentes configuraciones (root directory, POSIX user, creation info)
7. Montar usando access points y verificar el aislamiento de directorios y permisos
8. Ejecutar benchmarks con fio: lecturas/escrituras secuenciales y aleatorias en General Purpose
9. Ejecutar los mismos benchmarks en Max I/O y comparar resultados (latencia vs throughput)
10. Analizar metricas de CloudWatch (TotalIOBytes, PermittedThroughput, BurstCreditBalance) y documentar conclusiones

## Validacion

- [ ] Ambos file systems EFS estan creados con los modos de rendimiento correctos
- [ ] Mount targets existen en al menos 3 AZs para cada file system
- [ ] Las instancias EC2 en diferentes AZs pueden montar y acceder a los file systems
- [ ] Los access points restringen correctamente el acceso a directorios y aplican permisos POSIX
- [ ] Los benchmarks de I/O muestran las diferencias entre General Purpose y Max I/O
- [ ] General Purpose muestra menor latencia para workloads de baja concurrencia
- [ ] Max I/O muestra mayor throughput agregado con multiples clientes concurrentes
- [ ] Las metricas de CloudWatch reflejan el rendimiento observado en los benchmarks
