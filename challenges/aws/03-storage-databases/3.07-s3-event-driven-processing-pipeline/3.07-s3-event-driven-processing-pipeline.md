# 3.07 S3 Event-Driven Processing Pipeline

## Metadata
| Campo | Valor |
|-------|-------|
| **Complejidad** | Intermedio |
| **Modalidad** | Ambos |
| **Servicios AWS** | S3, Lambda, SQS, DynamoDB, CloudWatch |

## Descripcion

Construir un pipeline donde las subidas de objetos a S3 disparan una funcion Lambda que valida archivos, los transforma (por ejemplo, redimensionamiento de imagenes o conversion CSV-to-Parquet), almacena los resultados en un bucket destino, y registra metadata en DynamoDB. Manejar fallos con SQS dead-letter queues.

## Objetivos de aprendizaje

- Comprender la arquitectura event-driven y sus ventajas para procesamiento de archivos
- Dominar los patrones de S3 event notifications y sus configuraciones
- Implementar manejo robusto de errores con Dead Letter Queues (DLQ)
- Disenar procesamiento idempotente para manejar reintentos sin duplicados

## Requisitos previos

- **Conocimientos**: Fundamentos de S3 (buckets, event notifications), Lambda basico, conceptos de SQS, DynamoDB basico
- **Herramientas**: AWS CLI v2, Terraform >= 1.5, runtime de Lambda (Python o Node.js)
- **Servicios AWS**: Permisos para S3, Lambda, SQS, DynamoDB y CloudWatch

## Arquitectura

Dos buckets S3: uno de entrada (`raw/`) y otro de salida (`processed/`). S3 event notifications en el bucket de entrada disparan una funcion Lambda cuando se sube un objeto. La Lambda: (1) valida el archivo (tipo, tamano, formato), (2) transforma el contenido (por ejemplo, CSV a Parquet o resize de imagen), (3) almacena el resultado en el bucket de salida, y (4) registra metadata (nombre, tamano, timestamp, estado) en una tabla DynamoDB. Si la Lambda falla despues de los reintentos configurados, el evento va a una SQS Dead Letter Queue para analisis posterior. CloudWatch monitorea invocaciones, errores y duracion.

## Pasos sugeridos

1. Crear los buckets S3 de entrada y salida con configuraciones apropiadas
2. Crear la tabla DynamoDB para metadata de procesamiento (PK: objeto, SK: timestamp)
3. Crear la cola SQS que servira como Dead Letter Queue
4. Desarrollar la funcion Lambda con logica de validacion, transformacion y registro
5. Configurar el IAM role de Lambda con permisos para S3 (read/write), DynamoDB (write) y SQS
6. Configurar S3 event notification para disparar Lambda en eventos `s3:ObjectCreated:*`
7. Agregar filtros de prefijo y sufijo en la notificacion para procesar solo archivos especificos
8. Configurar la Lambda con reintentos (maxRetryAttempts) y DLQ destination
9. Probar con archivos validos e invalidos, verificar el flujo completo
10. Implementar idempotencia usando conditional writes en DynamoDB para evitar reprocesamiento

## Validacion

- [ ] La subida de un archivo al bucket de entrada dispara la Lambda automaticamente
- [ ] Los archivos validos se transforman y almacenan en el bucket de salida
- [ ] Los archivos invalidos son rechazados con un registro de error en DynamoDB
- [ ] La metadata de cada procesamiento se registra en DynamoDB
- [ ] Los fallos despues de reintentos terminan en la SQS Dead Letter Queue
- [ ] El procesamiento es idempotente (subir el mismo archivo dos veces no duplica resultados)
- [ ] CloudWatch muestra metricas de invocaciones exitosas y fallidas
