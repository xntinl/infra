# 3.03 DynamoDB Global Tables y Resolucion de Conflictos

## Metadata
| Campo | Valor |
|-------|-------|
| **Complejidad** | Avanzado |
| **Modalidad** | Ambos |
| **Servicios AWS** | DynamoDB Global Tables, Lambda, API Gateway, Route 53 |

## Descripcion

Configurar DynamoDB Global Tables en 3 regiones. Construir una aplicacion que escribe en multiples regiones simultaneamente, observar la resolucion de conflictos last-writer-wins, e implementar deteccion de conflictos a nivel de aplicacion usando atributos de version.

## Objetivos de aprendizaje

- Comprender el modelo de datos active-active multi-region con Global Tables
- Observar y medir la consistencia eventual entre regiones
- Implementar estrategias de resolucion de conflictos a nivel de aplicacion
- Medir y entender el replication lag entre regiones de Global Tables

## Requisitos previos

- **Conocimientos**: Fundamentos de DynamoDB (tablas, items, escrituras), conceptos de consistencia eventual, APIs REST basicas
- **Herramientas**: AWS CLI v2, Terraform >= 1.5, herramienta para pruebas HTTP (curl, Postman)
- **Servicios AWS**: Tres regiones habilitadas, permisos para DynamoDB, Lambda, API Gateway y Route 53

## Arquitectura

Una tabla DynamoDB Global Table replicada en 3 regiones (por ejemplo, `us-east-1`, `eu-west-1`, `ap-northeast-1`). Cada region tiene un API Gateway + Lambda que expone endpoints REST para leer y escribir items. Route 53 con latency-based routing dirige a los usuarios a la region mas cercana. La aplicacion incluye un atributo `version` en cada item para detectar conflictos a nivel de aplicacion. Un Lambda adicional en cada region monitorea DynamoDB Streams para registrar conflictos detectados.

## Pasos sugeridos

1. Crear la tabla DynamoDB en la region primaria con DynamoDB Streams habilitado
2. Agregar replicas en las otras dos regiones para formar la Global Table
3. Desplegar una funcion Lambda con API Gateway en cada region para operaciones CRUD
4. Configurar Route 53 con latency-based routing apuntando a los tres API Gateways
5. Escribir el mismo item desde dos regiones simultaneamente y observar last-writer-wins
6. Implementar logica de versionado en la aplicacion (atributo `version` con conditional writes)
7. Medir el replication lag escribiendo en una region y leyendo en otra con timestamps
8. Crear un Lambda que procese DynamoDB Streams y detecte/registre conflictos
9. Simular un escenario de particion de red (deshabilitando temporalmente una replica) y observar el comportamiento
10. Documentar los patrones de conflicto observados y las estrategias de mitigacion

## Validacion

- [ ] La Global Table replica datos en las 3 regiones correctamente
- [ ] Las escrituras desde cualquier region son visibles en las otras regiones (con eventual consistency)
- [ ] El mecanismo last-writer-wins se puede observar al escribir simultaneamente
- [ ] El versionado a nivel de aplicacion detecta escrituras conflictivas
- [ ] El replication lag esta medido y documentado (tipicamente < 1 segundo)
- [ ] Route 53 dirige correctamente al endpoint mas cercano por latencia
- [ ] Los conflictos detectados se registran para analisis posterior
