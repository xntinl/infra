## Why Encryption at Rest Matters

Encryption at rest protects your data if someone gains unauthorized access to the physical storage infrastructure. While AWS secures its data centers, encryption adds a second layer of defense: even if storage media is compromised, the data is unreadable without the encryption key. For regulated industries (healthcare, finance, government), encryption at rest is not optional -- it is a compliance requirement.

S3 offers three server-side encryption (SSE) methods. **SSE-S3** uses keys managed entirely by AWS -- it is the simplest option and has been the default for all new S3 objects since January 2023. **SSE-KMS** uses AWS Key Management Service, giving you control over key rotation, key policies, and audit trails in CloudTrail. You can use the AWS-managed `aws/s3` key or create your own customer-managed key (CMK). **SSE-C** lets you provide your own encryption key with every request -- AWS uses it to encrypt/decrypt but never stores it.

For most workloads, SSE-KMS with a customer-managed key is the best choice because it provides an audit trail (every use of the key is logged in CloudTrail), fine-grained access control (you control who can use the key via key policy), and automatic annual rotation. The main downside is cost: each KMS API call costs $0.03 per 10,000 requests. This is where S3 Bucket Keys come in -- they generate a bucket-level data key that handles encryption locally, reducing KMS API calls by up to 99%.

### `kms.tf`

```hcl
# Look up the current AWS account ID and region for the key policy.
data "aws_caller_identity" "current" {}

# Customer-managed KMS key for S3 encryption.
# Unlike the AWS-managed aws/s3 key, a CMK gives you:
# 1. Full control over the key policy (who can use the key)
# 2. Automatic annual key rotation
# 3. Ability to revoke access and render data unreadable
# 4. CloudTrail logging of every encrypt/decrypt operation
resource "aws_kms_key" "s3" {
  description             = "CMK for S3 bucket encryption"
  deletion_window_in_days = 7
  enable_key_rotation     = true

  policy = data.aws_iam_policy_document.kms_key.json

  tags = {
    Name = "${var.project_name}-kms-key"
  }
}

# Human-readable alias for the KMS key.
resource "aws_kms_alias" "s3" {
  name          = "alias/${var.project_name}-key"
  target_key_id = aws_kms_key.s3.key_id
}

# Key policy: the root account has full control, and the
# current IAM principal can use the key for S3 operations.
data "aws_iam_policy_document" "kms_key" {
  # Allow the root account full management of the key
  statement {
    sid    = "EnableRootAccount"
    effect = "Allow"
    principals {
      type        = "AWS"
      identifiers = ["arn:aws:iam::${data.aws_caller_identity.current.account_id}:root"]
    }
    actions   = ["kms:*"]
    resources = ["*"]
  }

  # Allow the S3 service to use the key for encryption
  statement {
    sid    = "AllowS3ServiceUsage"
    effect = "Allow"
    principals {
      type        = "Service"
      identifiers = ["s3.amazonaws.com"]
    }
    actions = [
      "kms:GenerateDataKey",
      "kms:Decrypt",
    ]
    resources = ["*"]
  }
}
```

### `s3.tf`

```hcl
resource "random_id" "suffix" {
  byte_length = 4
}

# ----------------------------------------------------------
# Bucket 1: SSE-S3 encryption (AWS-managed keys)
# ----------------------------------------------------------
# This is the simplest encryption method. AWS manages the
# keys entirely. No KMS costs, no key management overhead.
resource "aws_s3_bucket" "sse_s3" {
  bucket        = "${var.project_name}-sse-s3-${random_id.suffix.hex}"
  force_destroy = true

  tags = {
    Name       = "${var.project_name}-sse-s3"
    Encryption = "SSE-S3"
  }
}

resource "aws_s3_bucket_public_access_block" "sse_s3" {
  bucket = aws_s3_bucket.sse_s3.id

  block_public_acls       = true
  block_public_policy     = true
  ignore_public_acls      = true
  restrict_public_buckets = true
}

resource "aws_s3_bucket_server_side_encryption_configuration" "sse_s3" {
  bucket = aws_s3_bucket.sse_s3.id

  rule {
    apply_server_side_encryption_by_default {
      sse_algorithm = "aws:kms"
    }
    # Note: even with sse_algorithm = "aws:kms" and no key specified,
    # AWS uses the aws/s3 managed key. We set it explicitly to SSE-S3
    # to demonstrate the difference.
  }
}

# Override the above to use pure SSE-S3 (AES-256)
resource "aws_s3_bucket_server_side_encryption_configuration" "sse_s3_actual" {
  bucket = aws_s3_bucket.sse_s3.id

  rule {
    apply_server_side_encryption_by_default {
      sse_algorithm = "AES256"
    }
  }
}

# Upload a test object -- it will be encrypted with SSE-S3 automatically
resource "aws_s3_object" "sse_s3_test" {
  bucket       = aws_s3_bucket.sse_s3.id
  key          = "test-sse-s3.txt"
  content      = "This object is encrypted with SSE-S3 (AES-256)."
  content_type = "text/plain"

  tags = {
    Name = "${var.project_name}-sse-s3-test"
  }
}

# ----------------------------------------------------------
# Bucket 2: SSE-KMS encryption with customer-managed key
# ----------------------------------------------------------
resource "aws_s3_bucket" "sse_kms" {
  bucket        = "${var.project_name}-sse-kms-${random_id.suffix.hex}"
  force_destroy = true

  tags = {
    Name       = "${var.project_name}-sse-kms"
    Encryption = "SSE-KMS"
  }
}

resource "aws_s3_bucket_public_access_block" "sse_kms" {
  bucket = aws_s3_bucket.sse_kms.id

  block_public_acls       = true
  block_public_policy     = true
  ignore_public_acls      = true
  restrict_public_buckets = true
}

# SSE-KMS with a customer-managed key and S3 Bucket Keys enabled.
# bucket_key_enabled = true generates a bucket-level data key,
# reducing KMS API calls from one-per-object to one-per-bucket-key-rotation.
resource "aws_s3_bucket_server_side_encryption_configuration" "sse_kms" {
  bucket = aws_s3_bucket.sse_kms.id

  rule {
    apply_server_side_encryption_by_default {
      sse_algorithm     = "aws:kms"
      kms_master_key_id = aws_kms_key.s3.arn
    }
    bucket_key_enabled = true
  }
}

# Upload a test object -- it will be encrypted with the CMK automatically
resource "aws_s3_object" "sse_kms_test" {
  bucket       = aws_s3_bucket.sse_kms.id
  key          = "test-sse-kms.txt"
  content      = "This object is encrypted with SSE-KMS using a customer-managed key."
  content_type = "text/plain"

  tags = {
    Name = "${var.project_name}-sse-kms-test"
  }
}

# Upload a second object explicitly specifying the KMS key
# (demonstrates per-object encryption override)
resource "aws_s3_object" "sse_kms_explicit" {
  bucket               = aws_s3_bucket.sse_kms.id
  key                  = "test-explicit-kms.txt"
  content              = "This object explicitly specifies the KMS key at upload time."
  content_type         = "text/plain"
  server_side_encryption = "aws:kms"
  kms_key_id           = aws_kms_key.s3.arn

  tags = {
    Name = "${var.project_name}-sse-kms-explicit"
  }
}
```

> **Best Practice:** Use S3 Bucket Keys with SSE-KMS to reduce KMS API calls by up to 99%. Without bucket keys, every `PUT` and `GET` operation calls KMS individually. With bucket keys, S3 generates a short-lived bucket-level data key and handles encryption locally. At high request volumes, this can save thousands of dollars per month in KMS costs.

```bash
aws s3api head-object \
  --bucket "$(terraform output -raw sse_s3_bucket_name)" \
  --key "test-sse-s3.txt" \
  --query "{ServerSideEncryption:ServerSideEncryption,SSEKMSKeyId:SSEKMSKeyId}" \
  --output table
```

Expected: `ServerSideEncryption: AES256`, `SSEKMSKeyId: None`.

```bash

  --bucket "$(terraform output -raw sse_kms_bucket_name)" \
  --key "test-sse-kms.txt" \
  --query "{ServerSideEncryption:ServerSideEncryption,SSEKMSKeyId:SSEKMSKeyId,BucketKeyEnabled:BucketKeyEnabled}" \
```

Expected: `ServerSideEncryption: aws:kms`, `SSEKMSKeyId` showing your CMK ARN, `BucketKeyEnabled: True`.

## Step 5 -- Test Encryption Enforcement

Try to upload an unencrypted object to the KMS bucket (the default encryption will apply automatically):

```bash
echo "Unencrypted upload test" | aws s3 cp - \
  "s3://$(terraform output -raw sse_kms_bucket_name)/test-default-encrypt.txt"
```

Verify it was encrypted:

```bash
aws s3api head-object \
  --bucket "$(terraform output -raw sse_kms_bucket_name)" \
  --key "test-default-encrypt.txt" \
  --query "{ServerSideEncryption:ServerSideEncryption,SSEKMSKeyId:SSEKMSKeyId}"
```

Expected: the object is encrypted with `aws:kms` using your CMK, even though the upload did not specify encryption -- the bucket default applied automatically.

> **Best Practice:** Combine default bucket encryption with a bucket policy that **denies** `s3:PutObject` requests missing the `s3:x-amz-server-side-encryption` header. This prevents any client from bypassing encryption, even if they explicitly set `--sse none` on their upload. Default encryption is a fallback, not an enforcement mechanism.

## Common Mistakes

### 1. Not specifying a KMS key and getting the AWS-managed key

When you set `sse_algorithm = "aws:kms"` without providing `kms_master_key_id`, AWS uses the default `aws/s3` managed key. This key cannot be rotated on a custom schedule, cannot be shared cross-account, and does not appear in your KMS key list.

**Wrong -- SSE-KMS without specifying a key:**

```hcl
resource "aws_s3_bucket_server_side_encryption_configuration" "this" {
  bucket = aws_s3_bucket.this.id
  rule {
    apply_server_side_encryption_by_default {
      sse_algorithm = "aws:kms"
      # No kms_master_key_id -- defaults to aws/s3 managed key
    }
  }
}
```

**Fix -- specify your customer-managed key:**

```hcl
resource "aws_s3_bucket_server_side_encryption_configuration" "this" {
  bucket = aws_s3_bucket.this.id
  rule {
    apply_server_side_encryption_by_default {
      sse_algorithm     = "aws:kms"
      kms_master_key_id = aws_kms_key.s3.arn
    }
    bucket_key_enabled = true
  }
}
```

### 2. Forgetting to enable bucket keys and getting high KMS costs

Without `bucket_key_enabled = true`, every S3 `PUT` and `GET` operation on SSE-KMS objects makes an individual KMS API call. At $0.03 per 10,000 requests, a bucket handling 100 million requests/month would cost $300/month in KMS fees alone.

**Wrong -- missing bucket_key_enabled:**

```hcl
rule {
  apply_server_side_encryption_by_default {
    sse_algorithm     = "aws:kms"
    kms_master_key_id = aws_kms_key.s3.arn
  }
  # No bucket_key_enabled -- every operation hits KMS
}
```

**Fix -- enable bucket keys:**

```hcl
rule {
  apply_server_side_encryption_by_default {
    sse_algorithm     = "aws:kms"
    kms_master_key_id = aws_kms_key.s3.arn
  }
  bucket_key_enabled = true
}
```

### 3. Deleting a KMS key before emptying the bucket

If you delete a KMS key (or let it expire past its deletion window), all objects encrypted with that key become permanently unreadable. There is no recovery.

**What happens:** `terraform destroy` deletes the KMS key first, then fails to delete the bucket because S3 cannot decrypt objects to process the deletion.

**Fix:** Always use `force_destroy = true` on lab buckets, and in production, ensure your KMS key `deletion_window_in_days` is at least 7 (the minimum) to give you time to recover.

## Summary

- **SSE-S3** (AES-256) is the simplest encryption method -- AWS manages the keys entirely
- **SSE-KMS** gives you control over key policies, rotation, and CloudTrail audit logging
- **Customer-managed keys** (CMK) provide fine-grained access control and the ability to revoke access
- **S3 Bucket Keys** reduce KMS API calls by up to 99% when using SSE-KMS
- **Default bucket encryption** applies automatically to objects uploaded without explicit encryption headers
- Always use `data "aws_iam_policy_document"` for KMS key policies (not `jsonencode()`)
- Deleting a KMS key renders all encrypted data permanently unreadable
