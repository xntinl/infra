# 3.06 ElastiCache Redis Cluster con Pub/Sub

## Metadata
| Campo | Valor |
|-------|-------|
| **Complejidad** | Avanzado |
| **Modalidad** | Ambos |
| **Servicios AWS** | ElastiCache Redis, VPC, CloudWatch, EC2/Lambda |

## Descripcion

Desplegar un cluster ElastiCache Redis con cluster mode habilitado (multiples shards). Implementar un leaderboard en tiempo real usando Sorted Sets, almacenamiento de sesiones con TTL, y un canal de notificaciones pub/sub. Configurar failover automatico y probarlo.

## Objetivos de aprendizaje

- Dominar las estructuras de datos de Redis (Sorted Sets, Hashes, Strings) y sus casos de uso
- Comprender el sharding en cluster mode y como se distribuyen las keys
- Implementar patrones de pub/sub para comunicacion en tiempo real
- Configurar y probar el comportamiento de failover automatico
- Planificar el dimensionamiento de cache (memoria, conexiones, throughput)

## Requisitos previos

- **Conocimientos**: Fundamentos de Redis (comandos basicos, tipos de datos), conceptos de VPC (subnets, security groups), patrones de caching
- **Herramientas**: AWS CLI v2, Terraform >= 1.5, redis-cli, SDK con cliente Redis (ioredis, redis-py)
- **Servicios AWS**: VPC con subnets privadas, permisos para ElastiCache, EC2

## Arquitectura

Un cluster ElastiCache Redis con cluster mode enabled, compuesto por 3 shards con 1 replica cada uno (6 nodos total), desplegado en subnets privadas de la VPC. Una aplicacion en EC2 implementa tres funcionalidades: (1) leaderboard en tiempo real usando ZADD/ZRANGE sobre Sorted Sets, (2) almacenamiento de sesiones de usuario usando Hashes con TTL, y (3) canal pub/sub para notificaciones en tiempo real entre multiples instancias de la aplicacion. CloudWatch monitorea metricas de memoria, CPU, conexiones y cache hits.

## Pasos sugeridos

1. Crear la VPC con subnets privadas en al menos 3 AZs (o usar una existente)
2. Crear el subnet group y el parameter group de ElastiCache para Redis cluster mode
3. Crear el replication group con cluster mode enabled (3 shards, 1 replica por shard)
4. Configurar security groups para permitir trafico en el puerto 6379 desde las instancias de aplicacion
5. Desplegar instancias EC2 y conectar al cluster Redis usando redis-cli para verificar
6. Implementar el leaderboard con Sorted Sets (ZADD para agregar scores, ZREVRANGE para top N)
7. Implementar almacenamiento de sesiones con Hashes y TTL (HSET, EXPIRE)
8. Implementar pub/sub con SUBSCRIBE/PUBLISH para notificaciones entre instancias
9. Simular un failover (reboot del primary de un shard) y observar la promocion automatica de la replica
10. Configurar CloudWatch alarms para metricas criticas (EngineCPUUtilization, DatabaseMemoryUsagePercentage, CurrConnections)

## Validacion

- [ ] El cluster Redis esta desplegado con 3 shards y cluster mode enabled
- [ ] El leaderboard funciona correctamente con Sorted Sets (agregar, consultar top N, rank de un usuario)
- [ ] Las sesiones se almacenan con TTL y expiran automaticamente
- [ ] Pub/sub funciona entre multiples suscriptores conectados a diferentes nodos
- [ ] El failover automatico promueve una replica a primary sin intervencion manual
- [ ] La aplicacion reconecta automaticamente despues del failover
- [ ] Las metricas de CloudWatch muestran el estado del cluster
