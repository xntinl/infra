## Why Lifecycle Policies Matter

Data access patterns change over time. A log file uploaded today is read frequently for the first week, occasionally for the next month, and rarely after that. But without lifecycle policies, that log file stays in S3 Standard forever -- costing $0.023/GB/month even if nobody ever reads it again.

Lifecycle policies automate the movement of objects through storage tiers based on age. A typical policy transitions objects from Standard to Standard-IA after 30 days (saving ~45% on storage), then to Glacier Flexible Retrieval after 90 days (saving ~68%), and finally deletes them after 365 days. For a 1TB data lake growing at 100GB/month, well-designed lifecycle policies can reduce annual storage costs from ~$28,000 to under $5,000.

Lifecycle rules also manage versioned buckets. Without expiration rules for noncurrent versions, old versions accumulate indefinitely, consuming storage that serves no purpose once your retention window has passed. Combining lifecycle transitions with noncurrent version expiration gives you both data protection (during the retention window) and cost control (after it expires).

```hcl
resource "random_id" "suffix" {
  byte_length = 4
}

# ----------------------------------------------------------
# Bucket with comprehensive lifecycle policies
# ----------------------------------------------------------
resource "aws_s3_bucket" "this" {
  bucket        = "${var.project_name}-${random_id.suffix.hex}"
  force_destroy = true

  tags = {
    Name        = "${var.project_name}-bucket"
    Environment = "lab"
  }
}

resource "aws_s3_bucket_public_access_block" "this" {
  bucket = aws_s3_bucket.this.id

  block_public_acls       = true
  block_public_policy     = true
  ignore_public_acls      = true
  restrict_public_buckets = true
}

# Enable versioning so we can demonstrate noncurrent version
# lifecycle rules. Versioning must be enabled before lifecycle
# rules that target noncurrent versions.
resource "aws_s3_bucket_versioning" "this" {
  bucket = aws_s3_bucket.this.id

  versioning_configuration {
    status = "Enabled"
  }
}

# Lifecycle configuration with multiple rules targeting
# different prefixes and object states.
resource "aws_s3_bucket_lifecycle_configuration" "this" {
  bucket = aws_s3_bucket.this.id

  depends_on = [aws_s3_bucket_versioning.this]

  # Rule 1: Standard lifecycle transitions for all objects
  # Standard -> Standard-IA (30 days) -> Glacier (90 days)
  # -> Glacier Deep Archive (180 days) -> Delete (365 days)
  rule {
    id     = "standard-tiering"
    status = "Enabled"

    filter {
      prefix = ""
    }

    transition {
      days          = 30
      storage_class = "STANDARD_IA"
    }

    transition {
      days          = 90
      storage_class = "GLACIER"
    }

    transition {
      days          = 180
      storage_class = "DEEP_ARCHIVE"
    }

    expiration {
      days = 365
    }

    # Expire noncurrent versions after 30 days.
    # This prevents old versions from accumulating indefinitely.
    noncurrent_version_expiration {
      noncurrent_days = 30
    }

    # Transition noncurrent versions to cheaper storage first.
    noncurrent_version_transition {
      noncurrent_days = 7
      storage_class   = "GLACIER"
    }
  }

  # Rule 2: Logs prefix -- shorter retention, faster transitions
  rule {
    id     = "logs-cleanup"
    status = "Enabled"

    filter {
      prefix = "logs/"
    }

    transition {
      days          = 7
      storage_class = "STANDARD_IA"
    }

    transition {
      days          = 30
      storage_class = "GLACIER"
    }

    expiration {
      days = 90
    }

    noncurrent_version_expiration {
      noncurrent_days = 7
    }
  }

  # Rule 3: Temp prefix -- expire quickly
  rule {
    id     = "temp-cleanup"
    status = "Enabled"

    filter {
      prefix = "tmp/"
    }

    expiration {
      days = 1
    }

    # Also clean up incomplete multipart uploads after 1 day.
    # Without this, abandoned multipart uploads consume storage
    # indefinitely.
    abort_incomplete_multipart_upload {
      days_after_initiation = 1
    }
  }

  # Rule 4: Delete expired delete markers.
  # When all versions of an object are deleted (via noncurrent
  # expiration), the delete marker remains. This rule removes
  # orphaned delete markers to keep the bucket clean.
  rule {
    id     = "cleanup-delete-markers"
    status = "Enabled"

    filter {}

    expiration {
      expired_object_delete_marker = true
    }
  }

  # Rule 5: Clean up incomplete multipart uploads globally.
  # Multipart uploads that are never completed consume storage
  # and cost money. This rule aborts them after 7 days.
  rule {
    id     = "abort-incomplete-uploads"
    status = "Enabled"

    filter {}

    abort_incomplete_multipart_upload {
      days_after_initiation = 7
    }
  }
}
```

> **Best Practice:** Lifecycle rules should always include a `filter {}` block, even if it is empty. Terraform requires this for lifecycle rules that apply to all objects. An empty `filter {}` means "match all objects." Omitting it causes validation errors.

## Step 4 -- Understand the Storage Class Transition Path

S3 enforces a waterfall model for transitions. Objects can only move "down" the hierarchy:

```
Standard --> Standard-IA --> Intelligent-Tiering --> One Zone-IA
    |              |                |                    |
    v              v                v                    v
Glacier Instant Retrieval --> Glacier Flexible Retrieval --> Glacier Deep Archive
```

Key constraints:
- Minimum 30 days in Standard before transitioning to Standard-IA
- Minimum 30 days in Standard-IA before transitioning to Glacier
- Objects smaller than 128 KB are not transitioned to Standard-IA (they would cost more)
- You cannot transition directly from Standard-IA to Deep Archive -- you must go through Glacier first

## Step 5 -- Verify Object Storage Classes

Check the current storage class of each object:

```bash
aws s3api list-objects-v2 \
  --bucket "$(terraform output -raw bucket_name)" \
  --query "Contents[].{Key:Key,StorageClass:StorageClass,Size:Size}" \
  --output table
```

Expected: all objects show `STANDARD` because lifecycle transitions happen asynchronously (typically within 24-48 hours of meeting the age criteria, not instantly).

> **Best Practice:** Design lifecycle policies with a cost-first mindset. The most impactful rule is often the simplest: expire incomplete multipart uploads and noncurrent versions. These are "invisible" storage costs that accumulate silently. Add an `abort_incomplete_multipart_upload` rule and `noncurrent_version_expiration` to every bucket, even if you do not need storage class transitions.

## Common Mistakes

### 1. Missing the minimum transition days between tiers

S3 requires a minimum of 30 days between certain transitions. Trying to go from Standard to Standard-IA in 7 days causes a validation error.

**Wrong -- transition too early:**

```hcl
transition {
  days          = 7     # Error: minimum 30 days for Standard-IA
  storage_class = "STANDARD_IA"
}
```

**What happens:**

```
Error: creating S3 Lifecycle Configuration: InvalidArgument:
  'Days' in the Transition action for StorageClass 'STANDARD_IA'
  must be at least 30
```

**Fix -- respect the minimum:**

```hcl
transition {
  days          = 30
  storage_class = "STANDARD_IA"
}
```

### 2. Forgetting to handle incomplete multipart uploads

Multipart uploads that are started but never completed (due to application crashes, network failures, or bugs) leave orphaned parts in S3. These parts are invisible to `ListObjects` but consume storage and incur charges.

**What happens:** Your S3 bill grows unexpectedly. You cannot see the orphaned parts using `aws s3 ls`.

**Fix -- always include this rule:**

```hcl
rule {
  id     = "abort-incomplete-uploads"
  status = "Enabled"
  filter {}
  abort_incomplete_multipart_upload {
    days_after_initiation = 7
  }
}
```

### 3. Overlapping lifecycle rules without understanding precedence

When multiple rules match the same object, S3 applies the most cost-effective action. For transitions, S3 picks the cheaper storage class. For expiration, S3 picks the shorter retention.

**Gotcha:** A rule with `prefix = ""` (all objects) and a rule with `prefix = "logs/"` both match objects in `logs/`. If the global rule transitions at 30 days and the logs rule transitions at 7 days, objects in `logs/` will transition at 7 days (the shorter period wins).

## Summary

- **Lifecycle rules** automate storage class transitions and object expiration based on age
- **Transitions** follow a waterfall model: objects can only move to cheaper (lower-access) tiers
- **Noncurrent version expiration** prevents old versions from accumulating indefinitely in versioned buckets
- **Incomplete multipart upload cleanup** removes orphaned upload parts that are invisible to normal listing
- **Delete marker cleanup** removes orphaned delete markers after all versions expire
- Storage class transitions are asynchronous and may take 24-48 hours to execute
- Every lifecycle rule must include a `filter {}` block, even if empty
