# 3.02 DynamoDB Single-Table Design con GSI y LSI

## Metadata
| Campo | Valor |
|-------|-------|
| **Complejidad** | Avanzado |
| **Modalidad** | Ambos (diseno manual de tabla + Terraform para infraestructura) |
| **Servicios AWS** | DynamoDB, Lambda, DynamoDB Streams, CloudWatch |

## Descripcion

Modelar un dominio complejo (por ejemplo, e-commerce con usuarios, ordenes, productos y resenas) usando el patron single-table design de DynamoDB. Implementar GSI para patrones de acceso como "todas las ordenes por estado" y LSI para "ordenes de un usuario ordenadas por fecha". Usar DynamoDB Streams para disparar Lambda y realizar denormalizacion.

## Objetivos de aprendizaje

- Dominar los patrones de single-table design y cuando aplicarlos
- Comprender los trade-offs entre GSI y LSI (costo, flexibilidad, limites)
- Implementar overloaded keys para soportar multiples entidades en una tabla
- Usar DynamoDB Streams como mecanismo de Change Data Capture (CDC)
- Planificar read/write capacity y entender los modos on-demand vs provisioned

## Requisitos previos

- **Conocimientos**: Fundamentos de DynamoDB (particiones, sort keys, tipos de datos), conceptos de modelado de datos NoSQL, Lambda basico
- **Herramientas**: AWS CLI v2, Terraform >= 1.5, NoSQL Workbench (recomendado para visualizar el modelo)
- **Servicios AWS**: Cuenta con permisos para DynamoDB, Lambda y CloudWatch

## Arquitectura

Una unica tabla DynamoDB modela todas las entidades del dominio e-commerce. La partition key (`PK`) y sort key (`SK`) usan prefijos para diferenciar entidades: `USER#123`, `ORDER#456`, `PRODUCT#789`. Un GSI con `GSI1PK` = `STATUS#pending` y `GSI1SK` = timestamp permite consultar ordenes por estado. Un LSI usa la misma PK del usuario pero con sort key de fecha para ordenar ordenes cronologicamente. DynamoDB Streams captura cambios y dispara una funcion Lambda que mantiene contadores denormalizados (total de ordenes por usuario, rating promedio por producto).

## Pasos sugeridos

1. Analizar el dominio e-commerce e identificar todas las entidades y patrones de acceso
2. Disenar el esquema single-table con PK/SK overloaded (usar NoSQL Workbench para visualizar)
3. Crear la tabla DynamoDB con el LSI definido al momento de creacion (no se puede agregar despues)
4. Crear el GSI para el patron de acceso "ordenes por estado"
5. Insertar datos de ejemplo para todas las entidades (usuarios, productos, ordenes, resenas)
6. Escribir queries para cada patron de acceso y verificar que funcionan correctamente
7. Habilitar DynamoDB Streams en la tabla con vista `NEW_AND_OLD_IMAGES`
8. Crear la funcion Lambda que procesa los eventos del stream para denormalizacion
9. Probar la denormalizacion creando/actualizando ordenes y verificando contadores
10. Configurar CloudWatch alarms para throttling, latencia y errores del stream

## Validacion

- [ ] La tabla soporta todas las entidades (usuarios, ordenes, productos, resenas) en una sola tabla
- [ ] Se pueden consultar ordenes por estado usando el GSI
- [ ] Se pueden consultar ordenes de un usuario ordenadas por fecha usando el LSI
- [ ] DynamoDB Streams captura todos los cambios en la tabla
- [ ] La funcion Lambda denormaliza correctamente (contadores, promedios)
- [ ] No hay hot partitions (distribucion uniforme de accesos)
- [ ] Los queries usan key conditions en lugar de filter expressions para eficiencia
