# 3.04 DAX Cluster y Patrones de Cache

## Metadata
| Campo | Valor |
|-------|-------|
| **Complejidad** | Intermedio |
| **Modalidad** | Ambos |
| **Servicios AWS** | DAX, DynamoDB, VPC, CloudWatch, EC2/Lambda |

## Descripcion

Desplegar un cluster DAX dentro de una VPC, configurar la aplicacion para usar DAX en workloads de lectura intensiva. Implementar y comparar estrategias de caching write-through vs. write-around. Medir las mejoras de latencia y los ratios de cache hit.

## Objetivos de aprendizaje

- Comprender el caching in-memory para DynamoDB con DAX y sus casos de uso
- Gestionar un cluster DAX (nodos, parameter groups, subnet groups)
- Implementar patrones de invalidacion de cache y entender sus trade-offs
- Alcanzar y medir latencia de lectura en microsegundos

## Requisitos previos

- **Conocimientos**: Fundamentos de DynamoDB (reads, writes, capacity), conceptos de VPC (subnets, security groups), patrones de caching
- **Herramientas**: AWS CLI v2, Terraform >= 1.5, SDK de AWS (Python/Node.js con cliente DAX)
- **Servicios AWS**: VPC con subnets privadas, permisos para DAX, DynamoDB, EC2 o Lambda

## Arquitectura

Un cluster DAX de 3 nodos desplegado en subnets privadas de la VPC. Una tabla DynamoDB con datos de prueba (por ejemplo, catalogo de productos). Una aplicacion en EC2 o Lambda dentro de la misma VPC se conecta al cluster DAX usando el DAX SDK en lugar del cliente DynamoDB estandar. La aplicacion implementa dos modos: write-through (escrituras pasan por DAX) y write-around (escrituras van directo a DynamoDB, DAX se invalida en la siguiente lectura). CloudWatch recopila metricas de cache hit/miss, latencia y errores.

## Pasos sugeridos

1. Crear una VPC con subnets privadas en al menos 2 AZs (o usar una existente)
2. Crear la tabla DynamoDB y cargar datos de prueba (miles de items)
3. Crear el DAX subnet group y el parameter group
4. Crear el IAM role para DAX con permisos sobre la tabla DynamoDB
5. Desplegar el cluster DAX de 3 nodos en las subnets privadas
6. Desplegar la aplicacion (EC2 o Lambda en VPC) usando el DAX SDK
7. Ejecutar pruebas de lectura sin DAX (directo a DynamoDB) y registrar latencias
8. Ejecutar las mismas pruebas de lectura via DAX y comparar latencias
9. Implementar y probar write-through vs write-around, medir consistencia y cache hit ratio
10. Configurar CloudWatch dashboards con metricas de DAX (ItemCacheHits, ItemCacheMisses, QueryCacheHits)

## Validacion

- [ ] El cluster DAX esta desplegado con 3 nodos en subnets privadas
- [ ] La aplicacion se conecta a DAX usando el DAX SDK
- [ ] La latencia de lectura via DAX es significativamente menor que directo a DynamoDB (microsegundos vs milisegundos)
- [ ] El cache hit ratio es visible en CloudWatch y aumenta con lecturas repetidas
- [ ] Write-through actualiza el cache inmediatamente despues de escribir
- [ ] Write-around muestra un cache miss en la primera lectura post-escritura
- [ ] El cluster sobrevive la caida de un nodo sin interrupcion del servicio
