## Why S3 Matters

Amazon S3 is the foundation of storage on AWS. Nearly every AWS architecture uses S3 in some way -- as a data lake, a static website host, an artifact store, a backup destination, or a log archive. S3 stores objects (files) in buckets (containers), and every object is identified by a unique key (its full path within the bucket). Unlike a traditional filesystem, S3 is a flat key-value store: the key `reports/2024/january/sales.csv` is not actually stored inside nested directories. The forward slashes are just characters in the key name. The AWS Console renders them as folders for convenience, but understanding this flat namespace is critical for designing efficient access patterns.

Every object in S3 has metadata -- a set of key-value pairs that travel with the object. System metadata includes `Content-Type`, `Content-Length`, and `Last-Modified`. You can also attach custom metadata (prefixed with `x-amz-meta-`) to store application-specific information like processing status, original filename, or data classification. This metadata is returned in HTTP headers when you retrieve the object, making it useful for downstream processing without needing to read the object body.

S3 offers multiple storage classes optimized for different access patterns and cost profiles. Standard is the default (frequent access), Standard-IA is for infrequent access at lower storage cost but with retrieval fees, and Glacier tiers are for archival data. Choosing the right storage class from day one -- or automating transitions with lifecycle policies -- can cut storage costs by 50-90%.

```hcl
}
```

### `s3.tf`

```hcl
# Random suffix to ensure globally unique bucket names.
# S3 bucket names must be unique across ALL AWS accounts worldwide.
resource "random_id" "suffix" {
  byte_length = 4
}

# Primary S3 bucket for learning fundamentals.
# force_destroy = true allows Terraform to delete the bucket even
# when it contains objects -- essential for lab cleanup.
resource "aws_s3_bucket" "this" {
  bucket        = "${var.project_name}-${random_id.suffix.hex}"
  force_destroy = true

  tags = {
    Name        = "${var.project_name}-bucket"
    Environment = "lab"
  }
}

# Block ALL public access. This is the single most important
# security setting on any S3 bucket. It overrides any bucket
# policy or ACL that would otherwise grant public access.
resource "aws_s3_bucket_public_access_block" "this" {
  bucket = aws_s3_bucket.this.id

  block_public_acls       = true
  block_public_policy     = true
  ignore_public_acls      = true
  restrict_public_buckets = true
}

# Upload a test object to the root of the bucket.
# The key is the object's full path (no leading slash).
resource "aws_s3_object" "hello" {
  bucket       = aws_s3_bucket.this.id
  key          = "hello.txt"
  content      = "Hello from S3! This is my first object."
  content_type = "text/plain"

  metadata = {
    author      = "terraform"
    environment = "lab"
  }

  tags = {
    Name = "${var.project_name}-hello"
  }
}

# Upload an object with a prefix (simulated folder structure).
# The key "reports/2024/summary.csv" creates the appearance
# of nested folders in the AWS Console, but S3 stores this
# as a single flat key.
resource "aws_s3_object" "report" {
  bucket       = aws_s3_bucket.this.id
  key          = "reports/2024/summary.csv"
  content      = "date,revenue,cost\n2024-01-01,50000,30000\n2024-02-01,55000,31000"
  content_type = "text/csv"

  metadata = {
    department = "finance"
    year       = "2024"
  }

  tags = {
    Name = "${var.project_name}-report"
  }
}

# Upload a JSON configuration object to demonstrate
# different content types and deeper prefix hierarchies.
resource "aws_s3_object" "config" {
  bucket       = aws_s3_bucket.this.id
  key          = "config/app/settings.json"
  content      = jsonencode({
    app_name    = "my-app"
    version     = "1.0.0"
    debug       = false
    max_retries = 3
  })
  content_type = "application/json"

  metadata = {
    managed_by = "terraform"
  }

  tags = {
    Name = "${var.project_name}-config"
  }
}

# Upload an object with a specific storage class.
# STANDARD_IA (Infrequent Access) costs less per GB stored
# but charges a per-GB retrieval fee.
resource "aws_s3_object" "archive" {
  bucket        = aws_s3_bucket.this.id
  key           = "archive/old-data.txt"
  content       = "This data is rarely accessed but must be retained."
  content_type  = "text/plain"
  storage_class = "STANDARD_IA"

  metadata = {
    retention = "7-years"
    data_class = "cold"
  }

  tags = {
    Name = "${var.project_name}-archive"
  }
}
```

> **Best Practice:** Always enable `aws_s3_bucket_public_access_block` with all four flags set to `true` on every bucket. This is your last line of defense against accidental public exposure, even if a bucket policy is misconfigured. AWS now enables this by default on new buckets, but Terraform should declare it explicitly for clarity and drift detection.

## Step 4 -- Inspect Object Metadata

Retrieve the metadata for the hello.txt object using the `head-object` API. This returns metadata without downloading the object body:

```bash
aws s3api head-object \
  --bucket "$(terraform output -raw bucket_name)" \
  --key "hello.txt"
```

Expected output includes `ContentType`, `ContentLength`, `Metadata`, and `LastModified`. Notice the custom metadata keys are lowercased:

```json
{
    "ContentType": "text/plain",
    "ContentLength": 39,
    "Metadata": {
        "author": "terraform",
        "environment": "lab"
    },
    "StorageClass": "STANDARD"
}
```

Check the storage class of the archive object:

```bash
aws s3api head-object \
  --bucket "$(terraform output -raw bucket_name)" \
  --key "archive/old-data.txt" \
  --query "{StorageClass:StorageClass,ContentType:ContentType,Metadata:Metadata}" \
  --output table
```

Expected: `StorageClass` shows `STANDARD_IA`, confirming the object was placed in the Infrequent Access tier.

> **Best Practice:** Use `head-object` (not `get-object`) when you only need metadata. `head-object` does not transfer the object body, so it costs less and is faster for metadata-only lookups. This is especially important for large objects.

## Common Mistakes

### 1. Forgetting the random suffix on bucket names

S3 bucket names are globally unique across all AWS accounts. If you hardcode a name like `my-bucket`, someone else likely already owns it.

**Wrong -- hardcoded bucket name:**

```hcl
resource "aws_s3_bucket" "this" {
  bucket = "my-data-bucket"
}
```

**What happens:** Terraform fails with `BucketAlreadyExists`:

```
Error: creating Amazon S3 Bucket (my-data-bucket): BucketAlreadyExists
```

**Fix -- use a random suffix:**

```hcl
resource "random_id" "suffix" {
  byte_length = 4
}

resource "aws_s3_bucket" "this" {
  bucket = "${var.project_name}-${random_id.suffix.hex}"
}
```

### 2. Using deprecated inline bucket configuration

Older Terraform examples put versioning, lifecycle, and encryption directly inside the `aws_s3_bucket` resource block. The AWS provider v4+ deprecated this pattern in favor of separate resources.

**Wrong -- inline configuration:**

```hcl
resource "aws_s3_bucket" "this" {
  bucket = "my-bucket"

  versioning {
    enabled = true
  }

  server_side_encryption_configuration {
    rule {
      apply_server_side_encryption_by_default {
        sse_algorithm = "aws:kms"
      }
    }
  }
}
```

**What happens:** Terraform may show warnings or unexpected drift, and the configuration is harder to manage as complexity grows.

**Fix -- use separate resources:**

```hcl
resource "aws_s3_bucket" "this" {
  bucket = "my-bucket"
}

resource "aws_s3_bucket_versioning" "this" {
  bucket = aws_s3_bucket.this.id
  versioning_configuration {
    status = "Enabled"
  }
}

resource "aws_s3_bucket_server_side_encryption_configuration" "this" {
  bucket = aws_s3_bucket.this.id
  rule {
    apply_server_side_encryption_by_default {
      sse_algorithm = "aws:kms"
    }
  }
}
```

### 3. Omitting force_destroy on lab buckets

If you skip `force_destroy = true`, Terraform cannot delete a bucket that contains objects. You must empty the bucket manually first.

**Wrong -- no force_destroy:**

```hcl
resource "aws_s3_bucket" "this" {
  bucket = "my-lab-bucket-abc123"
}
```

**What happens:** `terraform destroy` fails:

```
Error: deleting Amazon S3 Bucket: BucketNotEmpty: The bucket you tried to delete is not empty
```

**Fix -- add force_destroy for lab environments:**

```hcl
resource "aws_s3_bucket" "this" {
  bucket        = "my-lab-bucket-abc123"
  force_destroy = true
}
```

## Summary

- An **S3 bucket** is a container for objects with a globally unique name
- An **object** consists of a key (path), body (data), and metadata (headers)
- **Prefixes** (forward slashes in keys) create the illusion of folders, but S3 is a flat key-value store
- **Storage classes** (Standard, Standard-IA, Glacier) optimize cost based on access patterns
- **Custom metadata** attaches application-specific key-value pairs to objects
- **Block Public Access** should always be enabled with all four flags set to `true`
- Use **separate Terraform resources** (`aws_s3_bucket_versioning`, etc.) instead of deprecated inline blocks
