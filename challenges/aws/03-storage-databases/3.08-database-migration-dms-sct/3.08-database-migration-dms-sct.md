# 3.08 Migracion de Base de Datos con DMS y SCT

## Metadata
| Campo | Valor |
|-------|-------|
| **Complejidad** | Avanzado |
| **Modalidad** | Manual (consola DMS) + Terraform para infraestructura destino |
| **Servicios AWS** | DMS, SCT, Aurora PostgreSQL, EC2, VPC |

## Descripcion

Migrar una base de datos PostgreSQL ejecutandose en EC2 a Aurora PostgreSQL usando AWS Database Migration Service. Usar Schema Conversion Tool para cualquier transformacion de esquema. Configurar replicacion continua para un cutover con near-zero downtime.

## Objetivos de aprendizaje

- Comprender las estrategias de migracion de bases de datos (lift-and-shift, replatform)
- Implementar replicacion CDC (Change Data Capture) para migracion continua
- Usar Schema Conversion Tool para analizar y convertir esquemas
- Planificar y ejecutar un cutover con minimo downtime
- Validar la integridad de datos post-migracion

## Requisitos previos

- **Conocimientos**: Administracion basica de PostgreSQL (DDL, DML, replicacion logica), conceptos de VPC y networking, fundamentos de Aurora
- **Herramientas**: AWS CLI v2, Terraform >= 1.5, psql CLI, AWS SCT (aplicacion de escritorio)
- **Servicios AWS**: VPC con subnets, permisos para DMS, RDS/Aurora, EC2

## Arquitectura

Una instancia EC2 ejecuta PostgreSQL (source) con una base de datos de ejemplo con multiples tablas, relaciones y datos. Un replication instance de DMS en la misma VPC se conecta tanto al source (EC2 PostgreSQL) como al target (Aurora PostgreSQL cluster). SCT analiza el esquema origen y genera el DDL para Aurora. DMS ejecuta una full load inicial seguida de CDC para capturar cambios continuos. Durante el cutover, la aplicacion se detiene brevemente, DMS termina de replicar los cambios pendientes, y la aplicacion se redirige a Aurora.

## Pasos sugeridos

1. Crear la VPC con subnets y security groups que permitan comunicacion entre EC2, DMS y Aurora
2. Desplegar una instancia EC2 con PostgreSQL instalado y configurar la base de datos origen con datos de ejemplo
3. Configurar PostgreSQL para replicacion logica (`wal_level = logical`, `max_replication_slots`)
4. Crear el cluster Aurora PostgreSQL como destino de la migracion
5. Instalar y ejecutar AWS SCT para analizar el esquema origen y generar el reporte de compatibilidad
6. Aplicar las transformaciones de esquema recomendadas por SCT en Aurora
7. Crear el DMS replication instance, source endpoint (EC2 PostgreSQL) y target endpoint (Aurora)
8. Crear y ejecutar la DMS migration task con full load + CDC
9. Monitorear la task: verificar que la full load completo y que CDC esta replicando cambios
10. Ejecutar el cutover: detener escrituras en el origen, esperar que DMS sincronice, verificar datos, redirigir aplicacion a Aurora

## Validacion

- [ ] PostgreSQL en EC2 tiene la base de datos origen con datos de ejemplo
- [ ] SCT genera un reporte de compatibilidad sin errores criticos
- [ ] El esquema se crea correctamente en Aurora PostgreSQL
- [ ] La full load de DMS completa sin errores y los row counts coinciden
- [ ] CDC replica cambios en tiempo real del origen al destino
- [ ] El cutover se ejecuta con downtime minimo (< 1 minuto)
- [ ] La validacion post-migracion confirma integridad de datos (row counts, checksums)
- [ ] Las constraints (foreign keys, indexes) estan presentes y funcionales en Aurora
