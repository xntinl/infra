# 3.01 S3 Cross-Region Replication & Lifecycle

## Metadata
| Campo | Valor |
|-------|-------|
| **Complejidad** | Intermedio |
| **Modalidad** | Ambos |
| **Servicios AWS** | S3, S3 Replication, CloudWatch, SNS |

## Descripcion

Configurar S3 Cross-Region Replication (CRR) entre dos regiones con versionado habilitado. Implementar politicas de lifecycle para transicionar objetos a traves de las storage classes (Standard -> IA -> Glacier -> Deep Archive). Agregar reglas de replicacion con filtros para prefijos y tags especificos.

## Objetivos de aprendizaje

- Comprender la configuracion de replicacion entre regiones (CRR) y sus requisitos
- Dominar la economia de las storage classes y cuando usar cada una
- Implementar lifecycle transitions y entender los tiempos minimos entre transiciones
- Comprender las implicaciones del versionado en replicacion y lifecycle
- Configurar Replication Time Control (RTC) para SLAs predecibles

## Requisitos previos

- **Conocimientos**: Fundamentos de S3 (buckets, objetos, versionado), conceptos basicos de IAM roles
- **Herramientas**: AWS CLI v2, Terraform >= 1.5 (opcional)
- **Servicios AWS**: Dos regiones habilitadas, permisos para crear buckets S3 y roles IAM

## Arquitectura

La arquitectura consiste en dos buckets S3 en regiones distintas (por ejemplo, `us-east-1` y `eu-west-1`), ambos con versionado habilitado. Un IAM role permite a S3 replicar objetos del bucket origen al destino. Las reglas de replicacion filtran por prefijo (`data/`) y tags (`environment=production`). Las lifecycle policies en ambos buckets transicionan objetos: a Standard-IA tras 30 dias, a Glacier tras 90 dias, y a Deep Archive tras 180 dias. CloudWatch monitorea metricas de replicacion y SNS notifica fallos.

## Pasos sugeridos

1. Crear dos buckets S3 en regiones diferentes con versionado habilitado
2. Crear el IAM role y la policy que permite a S3 realizar la replicacion
3. Configurar la regla de replicacion basica (todos los objetos) y verificar que funciona
4. Agregar filtros de replicacion por prefijo (`data/`) y por tags
5. Habilitar Replication Time Control (RTC) y metricas de replicacion
6. Configurar lifecycle policies en el bucket origen (Standard -> IA -> Glacier -> Deep Archive)
7. Configurar lifecycle policies equivalentes en el bucket destino
8. Subir objetos de prueba con diferentes prefijos y tags, verificar replicacion selectiva
9. Configurar CloudWatch alarms para replicacion fallida y latencia
10. Configurar SNS notifications para alertas de replicacion

## Validacion

- [ ] Ambos buckets tienen versionado habilitado
- [ ] Los objetos con prefijo `data/` y tag correcto se replican al bucket destino
- [ ] Los objetos sin el filtro correcto NO se replican
- [ ] Las lifecycle policies transicionan objetos segun los periodos configurados
- [ ] Las metricas de replicacion son visibles en CloudWatch
- [ ] RTC muestra el SLA de 15 minutos en el dashboard
- [ ] Las alertas SNS se disparan cuando hay fallos de replicacion
