## Why This Matters

Cross-Region Replication (CRR) is how you protect S3 data against region-level failures and meet regulatory requirements for geographic redundancy. If your primary region experiences an outage, a replicated bucket in another region contains a near-real-time copy of your data. CRR is also used for compliance (some regulations require data copies in specific jurisdictions), latency reduction (serving data from a closer region), and aggregating logs from multiple regions into a central bucket.

CRR replicates new objects asynchronously -- typically within 15 minutes for most objects, though S3 provides no SLA without Replication Time Control (RTC). Existing objects are not replicated retroactively unless you configure S3 Batch Replication. Both the source and destination buckets must have versioning enabled because CRR relies on version IDs to track what has and has not been replicated. The replication IAM role needs permission to read from the source and write to the destination, and the destination bucket policy must allow the source account to replicate into it (for cross-account scenarios).

> **Best Practice:** CRR requires versioning on both source and destination buckets. Enable versioning before configuring replication -- if you enable replication first, Terraform may fail because the replication configuration requires versioning to already be in place.

## Spot the Bug

A colleague wrote this replication configuration, but no objects are being replicated. The replication status on objects stays `PENDING` indefinitely. **What is wrong?**

```hcl
data "aws_iam_policy_document" "replication_policy" {
  statement {
    effect = "Allow"
    actions = [
      "s3:GetReplicationConfiguration",
      "s3:ListBucket",
    ]
    resources = [aws_s3_bucket.source.arn]
  }

  statement {
    effect = "Allow"
    actions = [
      "s3:GetObjectVersion",           # <-- BUG
      "s3:GetObjectVersionAcl",
    ]
    resources = ["${aws_s3_bucket.source.arn}/*"]
  }

  statement {
    effect = "Allow"
    actions = [
      "s3:ReplicateObject",
      "s3:ReplicateDelete",
    ]
    resources = ["${aws_s3_bucket.destination.arn}/*"]
  }
}
```

<details>
<summary>Explain the bug</summary>

The IAM policy uses `s3:GetObjectVersion` instead of `s3:GetObjectVersionForReplication`. These are different IAM actions:

- `s3:GetObjectVersion` allows downloading a specific version of an object
- `s3:GetObjectVersionForReplication` allows S3 to read the object specifically for replication purposes

Without `s3:GetObjectVersionForReplication`, the replication service cannot read objects from the source bucket. Objects will show `PENDING` replication status because S3 keeps retrying but the IAM policy denies the read.

The fix:

```hcl
actions = [
  "s3:GetObjectVersionForReplication",
  "s3:GetObjectVersionAcl",
  "s3:GetObjectVersionTagging",
]
```

Also note the missing `s3:ReplicateTags` action in the destination statement, which would cause tags to not be replicated.

</details>

<summary>TODO 3 -- IAM policy for replication permissions (s3-source.tf)</summary>

```hcl
data "aws_iam_policy_document" "replication" {
  # Read replication config and list source bucket
  statement {
    effect = "Allow"

    actions = [
      "s3:GetReplicationConfiguration",
      "s3:ListBucket",
    ]

    resources = [aws_s3_bucket.source.arn]
  }

  # Read object versions from source bucket
  statement {
    effect = "Allow"

    actions = [
      "s3:GetObjectVersionForReplication",
      "s3:GetObjectVersionAcl",
      "s3:GetObjectVersionTagging",
    ]

    resources = ["${aws_s3_bucket.source.arn}/*"]
  }

  # Write replicated objects to destination bucket
  statement {
    effect = "Allow"

    actions = [
      "s3:ReplicateObject",
      "s3:ReplicateDelete",
      "s3:ReplicateTags",
    ]

    resources = ["${aws_s3_bucket.destination.arn}/*"]
  }
}

resource "aws_iam_role_policy" "replication" {
  name   = "${var.project_name}-replication-policy"
  role   = aws_iam_role.replication.id
  policy = data.aws_iam_policy_document.replication.json
}
```

## Verify What You Learned

### Step 1 -- Verify replication configuration exists

```bash
aws s3api get-bucket-replication \
  --bucket "$(terraform output -raw source_bucket_name)" \
  --query "ReplicationConfiguration.Rules[].{ID:ID,Status:Status,Dest:Destination.Bucket}" \
  --output table
```

Expected: one rule with `Status: Enabled` and the destination bucket ARN.

### Step 2 -- Upload a test object and check replication

```bash
echo "cross-region replication test $(date)" | \
  aws s3 cp - "s3://$(terraform output -raw source_bucket_name)/replication-verify.txt"
```

Wait 30-60 seconds, then check the destination:

```bash
aws s3 ls "s3://$(terraform output -raw destination_bucket_name)/replication-verify.txt" \
  --region "$(terraform output -raw destination_region)"
```

Expected: the object appears in the destination bucket.

### Step 3 -- Check replication status on the source object

```bash
aws s3api head-object \
  --bucket "$(terraform output -raw source_bucket_name)" \
  --key "replication-verify.txt" \
  --query "ReplicationStatus" \
  --output text
```

Expected: `COMPLETED` (or `PENDING` if checked within seconds of upload).

### Step 4 -- Verify versioning on both buckets

```bash
aws s3api get-bucket-versioning \
```


Expected: both show `Status: Enabled`.

## Summary

- **Cross-Region Replication (CRR)** asynchronously copies objects to a bucket in another region
- **Versioning must be enabled** on both source and destination buckets
- The **replication IAM role** needs `GetObjectVersionForReplication` on the source and `ReplicateObject` on the destination
- **Existing objects** are not replicated retroactively -- use S3 Batch Replication for that
- **Replication status** is tracked per object: `PENDING`, `COMPLETED`, `FAILED`, or `REPLICA`
- Use **multi-provider Terraform** (`provider = aws.destination`) to manage resources in different regions
