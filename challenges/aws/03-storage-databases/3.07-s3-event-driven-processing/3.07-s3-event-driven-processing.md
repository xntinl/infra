## Why This Matters

Event-driven architectures decouple producers (who upload files) from consumers (who process them). When a CSV file lands in S3, the processing pipeline should start automatically -- no polling, no cron jobs, no manual triggers. S3 event notifications make this possible by emitting events for every `PutObject`, `DeleteObject`, and other operations.

The simplest pattern is S3 directly invoking Lambda. But this has a critical weakness: if Lambda is throttled, has a cold start timeout, or returns an error, S3 retries the notification -- but only twice, and with no dead-letter queue. If all retries fail, the event is lost. You will never know the file was uploaded.

Placing SQS between S3 and Lambda solves this. S3 sends the event to an SQS queue (which is virtually impossible to throttle), SQS retains the message for up to 14 days, and Lambda polls the queue at its own pace. If Lambda fails, SQS retries automatically with configurable visibility timeout. After the maximum number of retries, the message moves to a dead-letter queue (DLQ) for investigation. No event is ever lost.

> **Best Practice:** Use SQS between S3 and Lambda for retry and DLQ support. Direct S3-to-Lambda invocation has limited retry capability and no built-in dead-letter queue. The SQS pattern adds ~100ms of latency but provides much stronger reliability guarantees.

### `lambda.tf`

```hcl
# Lambda function source code (inline for lab simplicity)
data "archive_file" "processor" {
  type        = "zip"
  output_path = "${path.module}/processor.zip"

  source {
    content  = <<-PYTHON
import json
import boto3
import os
import urllib.parse

s3_client = boto3.client('s3')
OUTPUT_BUCKET = os.environ['OUTPUT_BUCKET']

def handler(event, context):
    """Process S3 events from SQS messages."""
    for record in event['Records']:
        # Parse the SQS message body (which contains the S3 event)
        body = json.loads(record['body'])

        # Handle S3 test events (sent when notification is first created)
        if 'Event' in body and body['Event'] == 's3:TestEvent':
            print("Received S3 test event, skipping")
            continue

        for s3_record in body.get('Records', []):
            bucket = s3_record['s3']['bucket']['name']
            key = urllib.parse.unquote_plus(
                s3_record['s3']['object']['key']
            )
            size = s3_record['s3']['object']['size']

            print(f"Processing: s3://{bucket}/{key} ({size} bytes)")

            # Read the source object
            response = s3_client.get_object(Bucket=bucket, Key=key)
            content = response['Body'].read().decode('utf-8')

            # Simple processing: count lines and add metadata
            line_count = len(content.strip().split('\n'))
            result = {
                'source_bucket': bucket,
                'source_key': key,
                'size_bytes': size,
                'line_count': line_count,
                'status': 'processed'
            }

            # Write result to output bucket
            output_key = key.replace('uploads/', 'processed/')
            output_key = output_key.replace('.csv', '-result.json')

            s3_client.put_object(
                Bucket=OUTPUT_BUCKET,
                Key=output_key,
                Body=json.dumps(result, indent=2),
                ContentType='application/json'
            )

            print(f"Result written to s3://{OUTPUT_BUCKET}/{output_key}")

    return {'statusCode': 200, 'processed': len(event['Records'])}
    PYTHON
    filename = "index.py"
  }
}
```

## Spot the Bug

A colleague configured S3 event notifications but the Lambda function never executes. The SQS queue shows 0 messages. CloudWatch shows no Lambda invocations. **What is wrong?**

```hcl
resource "aws_s3_bucket_notification" "this" {
  bucket = aws_s3_bucket.source.id

  queue {
    queue_arn     = aws_sqs_queue.processing.arn
    events        = ["s3:ObjectCreated:*"]
    filter_prefix = "uploads/"
    filter_suffix = ".csv"
  }
}

# Missing: SQS queue policy
# The SQS queue has no resource policy allowing S3 to send messages
```

<details>
<summary>Explain the bug</summary>

The SQS queue is missing a **resource-based policy** (queue policy) that allows S3 to send messages to it. By default, only the queue owner can send messages. When S3 tries to deliver an event notification, SQS rejects it because the S3 service principal has no `sqs:SendMessage` permission.

The fix is to add an SQS queue policy:

```hcl
data "aws_iam_policy_document" "sqs_policy" {
  statement {
    effect = "Allow"

    principals {
      type        = "*"
      identifiers = ["*"]
    }

    actions   = ["sqs:SendMessage"]
    resources = [aws_sqs_queue.processing.arn]

    condition {
      test     = "ArnEquals"
      variable = "aws:SourceArn"
      values   = [aws_s3_bucket.source.arn]
    }
  }
}

resource "aws_sqs_queue_policy" "this" {
  queue_url = aws_sqs_queue.processing.id
  policy    = data.aws_iam_policy_document.sqs_policy.json
}
```

The `aws:SourceArn` condition restricts which S3 buckets can send to this queue, following the principle of least privilege.

</details>

```hcl
resource "aws_sqs_queue" "dlq" {
  name                      = "${var.project_name}-dlq"

    Name = "${var.project_name}-dlq"

  name                       = "${var.project_name}-queue"
  visibility_timeout_seconds = 300     # 5 min (6x Lambda timeout)
  message_retention_seconds  = 86400   # 1 day

  redrive_policy = jsonencode({
    deadLetterTargetArn = aws_sqs_queue.dlq.arn
    maxReceiveCount     = 3

  tags = {
    Name = "${var.project_name}-queue"
  }
}
```

```hcl
data "aws_iam_policy_document" "lambda_permissions" {
  # Read from source bucket
  statement {
    effect    = "Allow"
    actions   = ["s3:GetObject"]
    resources = ["${aws_s3_bucket.source.arn}/*"]
  }

  # Write to output bucket
  statement {
    effect    = "Allow"
    actions   = ["s3:PutObject"]
    resources = ["${aws_s3_bucket.output.arn}/*"]
  }

  # Consume messages from SQS
  statement {
    effect = "Allow"
    actions = [
      "sqs:ReceiveMessage",
      "sqs:DeleteMessage",
      "sqs:GetQueueAttributes",
    ]
    resources = [aws_sqs_queue.processing.arn]
  }

  # Write logs to CloudWatch
  statement {
    effect = "Allow"
    actions = [
      "logs:CreateLogGroup",
      "logs:CreateLogStream",
      "logs:PutLogEvents",
    ]
    resources = ["arn:aws:logs:${data.aws_region.current.name}:${data.aws_caller_identity.current.account_id}:*"]
  }
}

resource "aws_iam_role_policy" "lambda" {
  name   = "${var.project_name}-lambda-policy"
  role   = aws_iam_role.lambda.id
  policy = data.aws_iam_policy_document.lambda_permissions.json
```

## Summary

- **S3 event notifications** trigger downstream processing when objects are created, deleted, or restored
- **SQS between S3 and Lambda** provides reliable delivery with retry and dead-letter queue support
- **Dead-letter queues** capture failed messages for investigation instead of losing them
- The **SQS queue policy** must allow S3 to send messages (resource-based policy with `aws:SourceArn` condition)
- **Visibility timeout** should be at least 6x the Lambda timeout to prevent duplicate processing
- Lambda **event source mapping** polls SQS and invokes the function with batches of messages
- Use `data "aws_iam_policy_document"` for all IAM policies (never `jsonencode()`)
