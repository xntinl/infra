After completing this exercise, you will be able to:

- **Demonstrate** delete markers and how S3 "deletes" without deleting

- **Describe** MFA Delete and when it provides additional protection

## Why Versioning Matters

Without versioning, overwriting an object in S3 permanently replaces it. Deleting an object permanently removes it. There is no undo. In production, this means a single mistyped `aws s3 rm` command or a buggy deployment script can destroy critical data with no recovery path.

Versioning changes this by preserving every version of every object. When you overwrite `config.json`, S3 keeps the old version and assigns the new one a unique version ID. When you delete an object, S3 does not actually remove it -- instead, it inserts a "delete marker" that hides the object from normal `GET` and `LIST` operations. The actual data remains intact and recoverable. This makes versioning essential for compliance, audit trails, and disaster recovery.

MFA Delete adds another layer of protection by requiring a multi-factor authentication code to permanently delete a version or disable versioning. This prevents even an IAM administrator with full S3 permissions from destroying data without physical access to an MFA device. MFA Delete can only be enabled by the root account using the AWS CLI -- Terraform cannot manage it. We will explore the concept and configuration in this exercise, but enabling MFA Delete requires root credentials that should never be used in lab environments.

> **Best Practice:** Enable versioning on day one for every bucket that stores data you care about. Versioning itself is free -- you only pay for the storage consumed by additional versions. Combine it with lifecycle policies (exercise 3.05) to automatically expire old versions and control costs.

### `s3.tf`

```hcl
resource "random_id" "suffix" {
  byte_length = 4
}

# The bucket itself -- keep it minimal. Configuration goes
# into separate resources (versioning, encryption, etc.).
resource "aws_s3_bucket" "this" {
  bucket        = "${var.project_name}-${random_id.suffix.hex}"
  force_destroy = true

  tags = {
    Name        = "${var.project_name}-bucket"
    Environment = "lab"
  }
}

resource "aws_s3_bucket_public_access_block" "this" {
  bucket = aws_s3_bucket.this.id

  block_public_acls       = true
  block_public_policy     = true
  ignore_public_acls      = true
  restrict_public_buckets = true
}

# Enable versioning as a separate resource.
# Once enabled, versioning can be "Suspended" but never
# fully disabled -- the bucket remembers it was versioned.
resource "aws_s3_bucket_versioning" "this" {
  bucket = aws_s3_bucket.this.id

  versioning_configuration {
    status = "Enabled"
  }
}

# Upload the initial version of a configuration file.
# After apply, we will overwrite this using the CLI to
# create multiple versions and demonstrate recovery.
resource "aws_s3_object" "config" {
  bucket       = aws_s3_bucket.this.id
  key          = "config.json"
  content      = jsonencode({
    version     = "1.0.0"
    environment = "production"
    debug       = false
    max_retries = 3
  })
  content_type = "application/json"

  tags = {
    Name = "${var.project_name}-config"
  }

  depends_on = [aws_s3_bucket_versioning.this]
}

# Upload a second object to demonstrate delete markers.
resource "aws_s3_object" "data" {
  bucket       = aws_s3_bucket.this.id
  key          = "data/important.txt"
  content      = "This data must not be lost."
  content_type = "text/plain"

  tags = {
    Name = "${var.project_name}-data"
  }

  depends_on = [aws_s3_bucket_versioning.this]
}
```

### `outputs.tf`

output "bucket_name" {

## Step 3 -- Create Multiple Versions

Overwrite the config.json object twice using the CLI to create three total versions:

```bash
BUCKET=$(terraform output -raw bucket_name)

echo '{"version":"2.0.0","environment":"production","debug":true,"max_retries":5}' | \
  aws s3 cp - "s3://${BUCKET}/config.json" --content-type "application/json"

echo '{"version":"3.0.0","environment":"production","debug":false,"max_retries":10}' | \
  aws s3 cp - "s3://${BUCKET}/config.json" --content-type "application/json"
```

Now list all versions of config.json:

```bash
aws s3api list-object-versions \
  --bucket "${BUCKET}" \
  --prefix "config.json" \
  --query "Versions[].{Key:Key,VersionId:VersionId,IsLatest:IsLatest,LastModified:LastModified,Size:Size}" \
  --output table
```

Expected: three versions listed. The most recent has `IsLatest: True`. Each has a unique `VersionId`.

## Step 4 -- Demonstrate Delete Markers

Delete the data object (without specifying a version ID):

```bash
aws s3 rm "s3://${BUCKET}/data/important.txt"
```

Try to read it:

```bash
aws s3 cp "s3://${BUCKET}/data/important.txt" -
```

Expected: `An error occurred (NoSuchKey)` -- the object appears deleted.

Now check what actually happened under the hood:

```bash
aws s3api list-object-versions \
  --bucket "${BUCKET}" \
  --prefix "data/important.txt" \
  --query "{Versions:Versions[].{Key:Key,VersionId:VersionId,IsLatest:IsLatest},DeleteMarkers:DeleteMarkers[].{Key:Key,VersionId:VersionId,IsLatest:IsLatest}}" \
  --output table
```

Expected: one version (the original data) AND one delete marker. The delete marker is marked `IsLatest: True`. The original version still exists.

## Step 5 -- Recover Deleted and Overwritten Objects

Recover the "deleted" data object by removing the delete marker:

```bash
# Get the delete marker version ID
DELETE_MARKER_ID=$(aws s3api list-object-versions \
  --bucket "${BUCKET}" \
  --prefix "data/important.txt" \
  --query "DeleteMarkers[0].VersionId" \
  --output text)

# Remove the delete marker (this "undeletes" the object)
aws s3api delete-object \
  --bucket "${BUCKET}" \
  --key "data/important.txt" \
  --version-id "${DELETE_MARKER_ID}"
```

Verify the object is back:

```bash
aws s3 cp "s3://${BUCKET}/data/important.txt" -
```

Expected: `This data must not be lost.`

Now recover version 1.0.0 of config.json by downloading a specific version:

```bash
# Get the oldest (first) version ID
FIRST_VERSION=$(aws s3api list-object-versions \
  --bucket "${BUCKET}" \
  --prefix "config.json" \
  --query "Versions[-1].VersionId" \
  --output text)

# Download that specific version
aws s3api get-object \
  --bucket "${BUCKET}" \
  --key "config.json" \
  --version-id "${FIRST_VERSION}" \
  /dev/stdout
```

Expected: the original `{"version":"1.0.0",...}` content.

> **Best Practice:** To "restore" an old version as the current version, copy it back on top of itself. This creates a new version with the old content, preserving the full version history:
>
> ```bash
> aws s3api copy-object \
>   --bucket "${BUCKET}" \
>   --key "config.json" \
>   --copy-source "${BUCKET}/config.json?versionId=${FIRST_VERSION}"
> ```

## Step 6 -- Understand MFA Delete (Conceptual)

MFA Delete adds two protections:

1. **Permanently deleting** an object version requires an MFA code
2. **Disabling versioning** on the bucket requires an MFA code

MFA Delete can only be enabled by the **root account** using the CLI:

```bash
# DO NOT RUN THIS IN A LAB -- this requires root credentials
# and a configured MFA device. Shown for reference only.
aws s3api put-bucket-versioning \
  --bucket "my-bucket" \
  --versioning-configuration Status=Enabled,MFADelete=Enabled \
  --mfa "arn:aws:iam:::<account-id>:mfa/<device-name> <mfa-code>"
```

Terraform cannot enable MFA Delete because it requires real-time MFA token input. In production, enable MFA Delete on buckets storing critical data (backups, compliance archives, financial records) using a manual process documented in your runbook.

## Common Mistakes

### 1. Trying to delete a versioned object without understanding delete markers

When you run `aws s3 rm` on a versioned bucket, S3 does not delete the object -- it creates a delete marker. The object appears gone but all versions remain.

**Wrong assumption:** "I deleted the object, so the data is gone and I have freed up storage."

**What actually happens:** The object is hidden by a delete marker. All versions still consume storage. To permanently delete all versions:

```bash
# Delete ALL versions of an object (truly permanent)
aws s3api list-object-versions \
  --bucket "${BUCKET}" \
  --prefix "config.json" \
  --query "Versions[].{Key:Key,VersionId:VersionId}" \
  --output text | while read key vid; do
    aws s3api delete-object --bucket "${BUCKET}" --key "${key}" --version-id "${vid}"
  done
```

### 2. Forgetting that versioning cannot be fully disabled

Once you enable versioning, you can only **suspend** it -- not disable it. Suspended versioning means new overwrites get a version ID of `null`, but existing versions remain. This catches people off guard when they expect `Disabled` to remove version history.

**Wrong -- trying to disable versioning:**

```hcl
resource "aws_s3_bucket_versioning" "this" {
  bucket = aws_s3_bucket.this.id
  versioning_configuration {
    status = "Disabled"  # Invalid! Only "Enabled" or "Suspended"
  }
}
```

**Fix -- suspend if you want to stop new versions:**

```hcl
resource "aws_s3_bucket_versioning" "this" {
  bucket = aws_s3_bucket.this.id
  versioning_configuration {
    status = "Suspended"
  }
}
```

## Summary

- **Versioning** preserves every version of every object in the bucket
- A **version ID** uniquely identifies each version of an object
- **Deleting** an object in a versioned bucket creates a **delete marker** -- the data is hidden, not removed
- You can **recover** any previous version by its version ID
- Versioning can be **Enabled** or **Suspended**, but never fully disabled
- **MFA Delete** requires a multi-factor authentication code to permanently delete versions or disable versioning
- Terraform manages versioning via `aws_s3_bucket_versioning` (not inline on the bucket resource)
