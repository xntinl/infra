## Why Multipart and Transfer Acceleration Matter

S3 handles objects up to 5 TB, but uploading a 5 GB file as a single PUT is fragile -- if the connection drops at 4.9 GB, the entire upload starts over. Multipart upload splits the file into parts (each 5 MB to 5 GB), uploads them in parallel, and assembles the object server-side. If one part fails, you retry only that part. AWS recommends multipart for any file over 100 MB.

The hidden cost trap is incomplete multipart uploads. When a multipart upload starts but never completes (network failure, client crash, abandoned process), the uploaded parts remain in S3 and you pay for the storage indefinitely. Without a lifecycle rule to abort these stale uploads, they accumulate silently. Teams have discovered hundreds of gigabytes of orphaned parts years after the fact.

Transfer Acceleration routes uploads through CloudFront edge locations using optimized network paths. For cross-region uploads (e.g., a user in Tokyo uploading to us-east-1), acceleration can improve throughput by 50-500% depending on distance and network conditions. It costs $0.04-$0.08/GB extra but can be worth it when upload speed matters.

> **Best Practice:** Always add a lifecycle rule to abort incomplete multipart uploads after 7 days. This single rule prevents a common and invisible storage cost leak. Add it to every bucket from day one -- the cost of forgetting is unbounded.

## Spot the Bug

A colleague created an S3 bucket for a data pipeline that processes thousands of multipart uploads daily. After six months, storage costs tripled even though the number of completed objects stayed flat. **What is wrong?**

```hcl
resource "aws_s3_bucket" "pipeline" {
  bucket        = "data-pipeline-uploads-abc123"
  force_destroy = true

  tags = { Name = "data-pipeline" }
}

resource "aws_s3_bucket_lifecycle_configuration" "pipeline" {
  bucket = aws_s3_bucket.pipeline.id

  rule {
    id     = "transition-old-objects"
    status = "Enabled"

    transition {
      days          = 30
      storage_class = "STANDARD_IA"
    }

    transition {
      days          = 90
      storage_class = "GLACIER"
    }
  }

  # No abort_incomplete_multipart_upload rule!
}
```

<details>
<summary>Explain the bug</summary>

**The problem:** The lifecycle configuration transitions completed objects to cheaper storage classes but has **no rule to abort incomplete multipart uploads**. When a multipart upload is initiated but never completed (network failure, client crash, timeout), the uploaded parts remain in the bucket as hidden storage.

These orphaned parts:
1. **Do not appear** in `aws s3 ls` -- they are invisible without `aws s3api list-multipart-uploads`
2. **Are not transitioned** by the existing lifecycle rules (lifecycle transitions only apply to completed objects)
3. **Accumulate indefinitely** at STANDARD storage rates ($0.023/GB/month)

With thousands of daily uploads, even a 1% failure rate means dozens of incomplete uploads per day. Over six months, this can easily reach hundreds of gigabytes of orphaned parts.

**The fix:** Add an `abort_incomplete_multipart_upload` rule:

```hcl
resource "aws_s3_bucket_lifecycle_configuration" "pipeline" {
  bucket = aws_s3_bucket.pipeline.id

  rule {
    id     = "abort-incomplete-multipart"
    status = "Enabled"

    abort_incomplete_multipart_upload {
      days_after_initiation = 7
    }
  }

  rule {
    id     = "transition-old-objects"
    status = "Enabled"

    transition {
      days          = 30
      storage_class = "STANDARD_IA"
    }

    transition {
      days          = 90
      storage_class = "GLACIER"
    }
  }
}
```

To find existing orphaned parts before the rule takes effect:

```bash
aws s3api list-multipart-uploads \
  --bucket data-pipeline-uploads-abc123 \
  --query "Uploads[].{Key:Key,Initiated:Initiated,UploadId:UploadId}" \
  --output table
```

</details>

## Summary

- **Multipart upload** splits large files into parts for parallel, resumable uploads -- use for files over 100 MB
- **Transfer Acceleration** routes uploads through CloudFront edge locations for faster cross-region transfers
- **Incomplete multipart uploads** are invisible storage cost leaks -- always add a lifecycle rule to abort them
- **Byte-range fetches** enable parallel downloads of specific portions of an object
- **Presigned URLs** grant time-limited upload/download access without sharing AWS credentials
- **Versioning** protects against accidental overwrites during concurrent multipart uploads
- Bucket names **cannot contain dots** when using Transfer Acceleration (dots break the SSL certificate)
