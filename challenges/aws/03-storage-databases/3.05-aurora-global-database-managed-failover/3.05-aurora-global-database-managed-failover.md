# 3.05 Aurora Global Database y Failover Gestionado

## Metadata
| Campo | Valor |
|-------|-------|
| **Complejidad** | Avanzado |
| **Modalidad** | Ambos |
| **Servicios AWS** | Aurora, Aurora Global Database, RDS, Route 53, CloudWatch |

## Descripcion

Desplegar una Aurora Global Database que abarque dos regiones. Configurar el cluster primario con read replicas y un secundario cross-region. Realizar un failover planificado (managed), luego simular un failover no planificado. Medir RPO/RTO para cada escenario.

## Objetivos de aprendizaje

- Comprender la arquitectura de Global Database y su replicacion fisica a nivel de storage
- Medir y entender el replication lag cross-region
- Ejecutar failovers planificados vs no planificados y entender las diferencias
- Medir RPO (Recovery Point Objective) y RTO (Recovery Time Objective) en ambos escenarios

## Requisitos previos

- **Conocimientos**: Fundamentos de bases de datos relacionales, conceptos de Aurora (clusters, instances, endpoints), alta disponibilidad y disaster recovery
- **Herramientas**: AWS CLI v2, Terraform >= 1.5, cliente de base de datos (psql o mysql CLI)
- **Servicios AWS**: Dos regiones habilitadas, permisos para RDS/Aurora, VPC con subnets en ambas regiones

## Arquitectura

Un Aurora Global Database con un cluster primario en `us-east-1` (1 writer + 2 reader instances) y un cluster secundario en `eu-west-1` (1 reader instance que se promueve a writer durante failover). La replicacion es a nivel de storage con lag tipico < 1 segundo. Route 53 health checks monitorean los endpoints del cluster. CloudWatch recopila metricas de replication lag, conexiones y performance. Una aplicacion de prueba genera carga constante para medir RPO/RTO durante los failovers.

## Pasos sugeridos

1. Crear VPCs con subnet groups para Aurora en ambas regiones
2. Crear el cluster Aurora primario en la region principal con una writer instance y dos reader instances
3. Agregar la region secundaria al Global Database, creando el cluster secundario con una reader instance
4. Verificar la replicacion midiendo el `AuroraGlobalDBReplicationLag` en CloudWatch
5. Configurar Route 53 health checks apuntando a los endpoints de los clusters
6. Generar carga de escritura constante en el cluster primario con un script de prueba
7. Ejecutar un managed planned failover (switchover) al cluster secundario y medir RTO
8. Verificar que el antiguo primario se convirtio en secundario automaticamente
9. Simular un failover no planificado (detach y promote el secundario) y medir RPO/RTO
10. Documentar y comparar las metricas de RPO/RTO para ambos escenarios

## Validacion

- [ ] El Global Database esta configurado con clusters en dos regiones
- [ ] El cluster primario tiene 1 writer + 2 readers
- [ ] El replication lag cross-region es menor a 1 segundo bajo carga normal
- [ ] El managed planned failover completa sin perdida de datos (RPO = 0)
- [ ] El RTO del planned failover esta medido (tipicamente < 1-2 minutos)
- [ ] El unplanned failover se ejecuta correctamente promoviendo el secundario
- [ ] El RPO del unplanned failover esta medido y documentado
- [ ] Las aplicaciones pueden reconectarse al nuevo endpoint primario
