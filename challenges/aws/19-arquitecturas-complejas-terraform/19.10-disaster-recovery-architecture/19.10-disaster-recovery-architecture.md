# 19.10 Disaster Recovery Architecture

## Metadata
| Campo | Valor |
|-------|-------|
| **Complejidad** | Experto |
| **Modalidad** | Terraform |
| **Servicios AWS** | EC2, RDS, S3, Route 53, Step Functions, Lambda |

## Descripcion

Implementa DR con Terraform: region primaria (stack completo), region DR (pilot light con recursos escalados hacia abajo). Construye una configuracion Terraform de "failover" que escala los recursos DR. Incluye runbook automatizado (Step Functions) para orquestacion del failover.

## Objetivos de aprendizaje

- Implementar infraestructura de disaster recovery como codigo con Terraform
- Aplicar el patron pilot light para DR cost-effective
- Automatizar el proceso de failover con Step Functions
- Planificar y documentar RPO/RTO para la arquitectura

## Requisitos previos

- Conocimientos: Terraform experto, estrategias de DR (pilot light, warm standby, active-active), RDS replicacion, networking multi-region
- Herramientas: Terraform >= 1.6, AWS CLI configurado para multiples regiones
- Servicios: EC2, RDS (cross-region read replica), S3, Route 53, Step Functions, Lambda, CloudWatch

## Arquitectura

Disaster Recovery con patron pilot light y failover automatizado:

1. **Region primaria**: Stack completo en produccion (EC2 Auto Scaling, RDS Multi-AZ, S3, ALB)
2. **Region DR (pilot light)**: Recursos minimos mantenidos activos: RDS cross-region read replica, S3 replicado, AMIs sincronizadas, infraestructura de red preparada
3. **Failover config**: Configuracion Terraform separada que escala los recursos DR a capacidad de produccion (promueve RDS replica, escala EC2, activa ALB)
4. **Orquestacion**: Step Functions state machine que ejecuta el runbook de failover paso a paso
5. **DNS**: Route 53 health checks que detectan fallos y redirigen trafico
6. **Monitoreo**: CloudWatch alarmas que detectan condiciones de DR y pueden activar el failover

## Pasos sugeridos

1. Desplegar el stack completo de produccion en la region primaria (VPC, EC2 ASG, RDS, ALB, S3)
2. Configurar la VPC y subnets en la region DR con la misma estructura de red
3. Crear la RDS cross-region read replica en la region DR
4. Configurar S3 Cross-Region Replication para datos criticos
5. Implementar un pipeline de sincronizacion de AMIs entre regiones
6. Crear la configuracion Terraform de "failover" que escala los recursos DR
7. Implementar la Step Functions state machine con el runbook de failover (promover RDS, escalar EC2, actualizar DNS)
8. Crear funciones Lambda para cada paso del failover (promover replica, verificar health, actualizar Route 53)
9. Configurar Route 53 health checks y failover routing policies
10. Ejecutar un simulacro de DR completo: activar failover, validar servicios, ejecutar failback

## Validacion

- [ ] La region primaria tiene el stack completo funcional
- [ ] La region DR tiene pilot light activo (RDS replica sincronizada, S3 replicado)
- [ ] La configuracion de failover escala correctamente los recursos DR a capacidad de produccion
- [ ] RDS read replica se promueve a master exitosamente durante failover
- [ ] Step Functions ejecuta el runbook de failover completo sin errores
- [ ] Route 53 redirige trafico a la region DR tras el failover
- [ ] La aplicacion es funcional en la region DR despues del failover
- [ ] RPO medido: perdida de datos dentro del objetivo definido
- [ ] RTO medido: tiempo de recuperacion dentro del objetivo definido
- [ ] El proceso de failback devuelve la operacion a la region primaria
