# 19.04 Multi-Region Active-Active Architecture

## Metadata
| Campo | Valor |
|-------|-------|
| **Complejidad** | Experto |
| **Modalidad** | Terraform |
| **Servicios AWS** | ECS, DynamoDB Global Tables, S3, Route 53, EventBridge, CloudWatch |

## Descripcion

Despliega stacks identicos en 2 regiones: ECS + DynamoDB Global Tables + S3 CRR. Route 53 con ruteo basado en latencia. EventBridge global endpoints para replicacion de eventos. Monitoreo que cubre ambas regiones. Todo gestionado desde un unico codebase de Terraform usando provider aliases.

## Objetivos de aprendizaje

- Gestionar infraestructura multi-region con Terraform usando provider aliases
- Implementar patrones active-active con replicacion de datos
- Configurar ruteo global basado en latencia con Route 53
- Manejar recursos globales y regionales desde un solo codebase

## Requisitos previos

- Conocimientos: Terraform avanzado (provider aliases, multi-region), DynamoDB Global Tables, replicacion S3, DNS avanzado
- Herramientas: Terraform >= 1.6, AWS CLI configurado para multiples regiones
- Servicios: ECS Fargate, DynamoDB Global Tables, S3 Cross-Region Replication, Route 53, EventBridge, CloudWatch

## Arquitectura

Arquitectura active-active distribuida en dos regiones:

1. **Region primaria y secundaria**: Stacks identicos con ECS Fargate, ALB y networking completo
2. **Datos**: DynamoDB Global Tables con replicacion bidireccional automatica; S3 con Cross-Region Replication (CRR)
3. **Ruteo global**: Route 53 con latency-based routing para dirigir usuarios a la region mas cercana
4. **Eventos**: EventBridge global endpoints para replicar eventos entre regiones
5. **Observabilidad**: CloudWatch dashboards cross-region con vista unificada

Terraform usa provider aliases (`aws.primary` y `aws.secondary`) para gestionar ambas regiones desde un solo state.

## Pasos sugeridos

1. Configurar providers con aliases para ambas regiones en Terraform
2. Crear modulo de VPC reutilizable y desplegarlo en ambas regiones
3. Desplegar ECS Fargate con ALB en cada region usando el mismo modulo
4. Configurar DynamoDB Global Tables con replicacion entre ambas regiones
5. Crear buckets S3 con Cross-Region Replication (CRR) bidireccional
6. Configurar Route 53 con health checks y latency-based routing
7. Implementar EventBridge global endpoints para replicacion de eventos
8. Crear CloudWatch dashboards que muestren metricas de ambas regiones
9. Configurar alarmas cross-region con SNS para notificaciones unificadas
10. Probar failover: deshabilitar una region y verificar que el trafico se redirige

## Validacion

- [ ] Ambas regiones tienen stacks identicos funcionales
- [ ] DynamoDB Global Tables replica datos bidireccional en menos de 1 segundo
- [ ] S3 CRR replica objetos entre regiones correctamente
- [ ] Route 53 dirige trafico a la region de menor latencia
- [ ] Route 53 health checks detectan fallos y redirigen trafico
- [ ] EventBridge replica eventos entre ambas regiones
- [ ] CloudWatch dashboard muestra metricas unificadas de ambas regiones
- [ ] El failover de una region es transparente para los usuarios
- [ ] Todo se gestiona desde un unico Terraform state con provider aliases
