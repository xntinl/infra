# 19.07 Event-Driven Architecture

## Metadata
| Campo | Valor |
|-------|-------|
| **Complejidad** | Avanzado |
| **Modalidad** | Terraform |
| **Servicios AWS** | EventBridge, Step Functions, SQS, SNS, Lambda, DynamoDB |

## Descripcion

Despliega un sistema event-driven completo: EventBridge como bus central, Step Functions para orquestacion, SQS para buffering, SNS para notificaciones, Lambda para procesamiento y DynamoDB para estado. Incluye event schema registry, DLQs y capacidades de replay.

## Objetivos de aprendizaje

- Disenar e implementar arquitectura event-driven como infraestructura como codigo
- Configurar orquestacion de workflows con Step Functions
- Implementar patrones de manejo de errores con DLQs y retry policies
- Gestionar esquemas de eventos y capacidades de replay

## Requisitos previos

- Conocimientos: Terraform intermedio-avanzado, patrones event-driven, workflows asincrono, Step Functions ASL
- Herramientas: Terraform >= 1.6, AWS CLI, lenguaje para Lambda (Python/Node.js)
- Servicios: EventBridge, Step Functions, SQS, SNS, Lambda, DynamoDB

## Arquitectura

Sistema event-driven con orquestacion y manejo de errores:

1. **Bus de eventos**: EventBridge custom event bus como punto central de comunicacion
2. **Schema Registry**: Registro de esquemas para documentar y versionar los formatos de eventos
3. **Procesamiento**: Funciones Lambda activadas por reglas de EventBridge segun patrones de evento
4. **Orquestacion**: Step Functions para workflows complejos que requieren multiples pasos coordinados
5. **Buffering**: Colas SQS como buffer entre productores y consumidores de alta carga
6. **Notificaciones**: SNS para fan-out de eventos a multiples suscriptores
7. **Estado**: DynamoDB para persistir el estado de las entidades del dominio
8. **Resiliencia**: DLQs en cada cola, retry policies en Step Functions y archive/replay en EventBridge

## Pasos sugeridos

1. Crear el EventBridge custom event bus y el schema registry
2. Definir los esquemas de eventos del dominio y registrarlos en el registry
3. Implementar funciones Lambda productoras y consumidoras de eventos
4. Configurar reglas de EventBridge con patrones de filtrado para cada consumidor
5. Desplegar Step Functions state machines para workflows de multiples pasos
6. Configurar colas SQS con DLQs, visibility timeout y retry policies
7. Crear SNS topics con suscripciones para notificaciones fan-out
8. Configurar DynamoDB tables para persistencia de estado con streams habilitados
9. Implementar EventBridge archive y replay para reprocesamiento de eventos
10. Configurar monitoreo con CloudWatch: metricas de cada componente y alarmas

## Validacion

- [ ] EventBridge custom bus recibe y enruta eventos segun las reglas configuradas
- [ ] Schema Registry contiene los esquemas de todos los tipos de eventos
- [ ] Lambda functions procesan eventos correctamente activadas por EventBridge
- [ ] Step Functions ejecuta workflows completos con manejo de errores y reintentos
- [ ] SQS procesa mensajes con buffering y DLQ captura mensajes fallidos
- [ ] SNS distribuye notificaciones a multiples suscriptores simultaneamente
- [ ] DynamoDB persiste el estado y sus streams propagan cambios como eventos
- [ ] EventBridge archive almacena eventos y replay los reprocesa correctamente
- [ ] CloudWatch muestra metricas y alarmas de todos los componentes del sistema
