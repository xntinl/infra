# 8.07 Terraform Custom Provider Development

## Metadata
| Campo | Valor |
|-------|-------|
| **Complejidad** | Experto |
| **Modalidad** | Terraform + Go |
| **Servicios AWS** | Cualquier API externa como recurso gestionado |

## Descripcion

Escribir un provider personalizado de Terraform en Go usando el Terraform Plugin Framework. El provider debe gestionar un recurso personalizado. Implementar operaciones CRUD, data sources, acceptance tests y documentacion.

## Objetivos de aprendizaje

- Comprender la arquitectura interna de un provider de Terraform
- Utilizar el Plugin Framework para implementar providers modernos
- Disenar schemas de recursos con tipos, validaciones y atributos computed
- Escribir acceptance tests que validen el ciclo de vida completo de los recursos
- Documentar y distribuir un provider personalizado

## Requisitos previos

- Conocimientos: Go (intermedio), Terraform (avanzado), APIs REST, testing en Go
- Herramientas: Go >= 1.21, Terraform >= 1.5, Git, make, editor con soporte Go (VS Code + gopls)
- Servicios AWS: Opcional (el provider puede gestionar cualquier API externa)

## Arquitectura

El provider se estructura siguiendo el patron del Terraform Plugin Framework. El directorio principal contiene la definicion del provider (configuracion, autenticacion). Los subdirectorios definen recursos y data sources, cada uno con su schema, operaciones CRUD (Create, Read, Update, Delete) y modelos de datos. Los acceptance tests crean recursos reales contra la API target, verifican su estado y los destruyen. La documentacion se genera automaticamente con tfplugindocs. El provider compilado se puede distribuir como binario local o publicar en el Terraform Registry.

## Pasos sugeridos

1. Configurar el proyecto Go con la estructura de directorios estandar para un provider
2. Implementar la funcion Provider con la configuracion (endpoint, API key, etc.)
3. Disenar el schema del recurso principal con atributos required, optional y computed
4. Implementar la operacion Create: llamar a la API y guardar el estado en Terraform
5. Implementar la operacion Read: consultar la API y sincronizar el estado
6. Implementar la operacion Update: detectar cambios y actualizar via API
7. Implementar la operacion Delete: eliminar el recurso via API
8. Implementar un data source para consultar recursos existentes sin gestionarlos
9. Escribir acceptance tests que ejecuten el ciclo completo (create -> read -> update -> delete)
10. Generar documentacion con tfplugindocs y probar el provider con un ejemplo de configuracion

## Validacion

- [ ] El provider compila correctamente y se puede usar con `terraform init` (dev overrides)
- [ ] La configuracion del provider acepta credenciales y parametros de conexion
- [ ] `terraform apply` crea el recurso correctamente en la API target
- [ ] `terraform plan` detecta drift cuando el recurso se modifica externamente
- [ ] `terraform apply` actualiza el recurso cuando se cambia la configuracion
- [ ] `terraform destroy` elimina el recurso correctamente
- [ ] El data source consulta recursos existentes y expone sus atributos
- [ ] Los acceptance tests pasan exitosamente con `go test -v ./...`
- [ ] La documentacion se genera correctamente con tfplugindocs
