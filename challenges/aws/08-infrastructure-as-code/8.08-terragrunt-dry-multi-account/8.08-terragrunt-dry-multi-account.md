# 8.08 Terragrunt DRY Multi-Account

## Metadata
| Campo | Valor |
|-------|-------|
| **Complejidad** | Avanzado |
| **Modalidad** | Terraform + Terragrunt |
| **Servicios AWS** | Cualquier servicio AWS |

## Descripcion

Reestructurar un proyecto Terraform usando Terragrunt. Implementar una jerarquia (account/region/environment/stack) con configuraciones compartidas usando bloques `include`, gestion de dependencias entre stacks y referencia a remote state via bloques `dependency`.

## Objetivos de aprendizaje

- Aplicar patrones de Terragrunt para eliminar repeticion en codigo de infraestructura (DRY)
- Gestionar dependencias entre stacks de forma explicita y segura
- Estructurar proyectos de infraestructura multi-cuenta y multi-region de forma escalable
- Comprender las ventajas y limitaciones de Terragrunt vs Terraform nativo

## Requisitos previos

- Conocimientos: Terraform intermedio-avanzado, gestion multi-cuenta, concepto de stacks de infraestructura
- Herramientas: Terraform >= 1.5, Terragrunt >= 0.55, AWS CLI, Git
- Servicios AWS: Cuenta AWS (preferiblemente multiples cuentas) con permisos para los servicios a desplegar

## Arquitectura

El proyecto se organiza en una jerarquia de directorios que refleja la estructura organizacional: `account/region/environment/stack`. En el nivel raiz, un `terragrunt.hcl` define la configuracion del backend remoto y providers. Cada nivel de la jerarquia tiene un `terragrunt.hcl` que hereda configuracion del nivel superior via `include` blocks. Los stacks individuales (vpc, ecs, rds) referencian modulos Terraform compartidos y declaran dependencias entre si con bloques `dependency` (ejemplo: ecs depende de vpc). Terragrunt gestiona el orden de ejecucion automaticamente.

## Pasos sugeridos

1. Disenar la jerarquia de directorios: `{account}/{region}/{env}/{stack}/terragrunt.hcl`
2. Crear el `terragrunt.hcl` raiz con la configuracion del backend S3 (con generate block)
3. Crear modulos Terraform reutilizables en un directorio separado (`modules/vpc`, `modules/ecs`, etc.)
4. Configurar `include` blocks para heredar configuracion del nivel superior
5. Usar `locals` y `read_terragrunt_config` para extraer valores de la jerarquia (account_id, region, env)
6. Implementar el stack de VPC como primer componente sin dependencias
7. Implementar el stack de ECS con un bloque `dependency` que referencia los outputs de VPC
8. Implementar el stack de RDS con dependencias a VPC (subnets) y ECS (security groups)
9. Ejecutar `terragrunt run-all plan` para verificar el orden de ejecucion y las dependencias
10. Replicar la estructura para una segunda cuenta/region y verificar la reutilizacion de configuracion

## Validacion

- [ ] La jerarquia de directorios refleja la estructura account/region/environment/stack
- [ ] El `include` block hereda configuracion correctamente en todos los niveles
- [ ] La configuracion del backend se genera automaticamente con paths unicos por stack
- [ ] Los bloques `dependency` referencian outputs de otros stacks correctamente
- [ ] `terragrunt run-all plan` resuelve el orden de dependencias automaticamente
- [ ] `terragrunt run-all apply` despliega todos los stacks en el orden correcto
- [ ] La misma estructura se puede replicar para otra cuenta/region sin duplicar codigo
- [ ] Los cambios en configuracion compartida se propagan a todos los stacks hijos
