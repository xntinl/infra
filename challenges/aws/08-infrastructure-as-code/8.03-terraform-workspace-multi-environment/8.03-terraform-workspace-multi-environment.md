# 8.03 Terraform Workspace Multi-Environment

## Metadata
| Campo | Valor |
|-------|-------|
| **Complejidad** | Intermedio |
| **Modalidad** | Terraform |
| **Servicios AWS** | Cualquier servicio AWS, S3 (backend), DynamoDB (locking) |

## Descripcion

Estructurar un proyecto Terraform usando workspaces (dev, staging, prod) con archivos de variables especificos por entorno. Implementar creacion condicional de recursos basada en el workspace, usar data sources para descubrimiento de entorno y construir un Makefile para la gestion de workspaces.

## Objetivos de aprendizaje

- Implementar una estrategia de workspaces para gestionar multiples entornos con el mismo codigo
- Mantener paridad entre entornos mientras se permite configuracion especifica
- Aplicar logica condicional en Terraform para crear recursos segun el entorno
- Gestionar variables por entorno de forma organizada y escalable
- Comprender las limitaciones de workspaces y cuando usar alternativas

## Requisitos previos

- Conocimientos: Terraform basico (resources, variables, outputs), concepto de entornos (dev/staging/prod)
- Herramientas: Terraform >= 1.5, AWS CLI, Make
- Servicios AWS: Cuenta AWS con permisos para los servicios a desplegar, S3, DynamoDB

## Arquitectura

Un unico directorio de Terraform gestiona multiples entornos usando workspaces. Cada workspace (dev, staging, prod) tiene su propio state file en S3 (separado por prefix). Archivos `.tfvars` especificos por entorno definen los valores de variables (tamano de instancias, cantidad de replicas, etc.). La logica condicional con `terraform.workspace` controla la creacion de recursos opcionales (ejemplo: NAT gateway solo en prod, replicas de lectura de RDS solo en staging/prod). Un Makefile simplifica las operaciones de workspace.

## Pasos sugeridos

1. Crear la estructura del proyecto con archivos `.tfvars` por entorno (dev.tfvars, staging.tfvars, prod.tfvars)
2. Configurar el backend S3 con workspace_key_prefix para separar state files
3. Crear los workspaces: `terraform workspace new dev`, `staging`, `prod`
4. Definir variables con valores por defecto para dev y overrides para staging/prod
5. Implementar logica condicional con `count` o `for_each` basada en `terraform.workspace`
6. Crear recursos que solo existan en ciertos entornos (ejemplo: WAF solo en prod)
7. Usar data sources para descubrir recursos del entorno (VPC existente, AMI mas reciente)
8. Construir un Makefile con targets: `make plan ENV=dev`, `make apply ENV=prod`, etc.
9. Desplegar en los tres entornos y verificar las diferencias
10. Documentar las limitaciones de workspaces encontradas y cuando usar directorios separados

## Validacion

- [ ] Los tres workspaces (dev, staging, prod) estan creados correctamente
- [ ] Cada workspace tiene su state file separado en S3
- [ ] Los archivos `.tfvars` definen valores apropiados por entorno
- [ ] La logica condicional crea/omite recursos segun el workspace
- [ ] Los recursos de dev son mas pequenos/menos que los de prod
- [ ] El Makefile simplifica las operaciones de plan/apply por entorno
- [ ] `terraform plan` en cada workspace muestra los recursos esperados
- [ ] Se documentan las limitaciones de workspaces vs directorios separados
