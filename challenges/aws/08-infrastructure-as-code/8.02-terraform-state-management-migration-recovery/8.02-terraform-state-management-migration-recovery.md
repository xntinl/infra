# 8.02 Terraform State Management, Migration y Recovery

## Metadata
| Campo | Valor |
|-------|-------|
| **Complejidad** | Avanzado |
| **Modalidad** | Terraform |
| **Servicios AWS** | S3, DynamoDB, IAM, KMS |

## Descripcion

Comenzar con state local, migrar a backend S3 con locking en DynamoDB. Practicar operaciones de state: `terraform import` para recursos existentes, `terraform state mv` para refactorizacion, `terraform state rm` para adopcion de recursos. Simular y recuperarse de corrupcion de state.

## Objetivos de aprendizaje

- Configurar backends remotos con S3 y DynamoDB locking para trabajo en equipo
- Dominar el locking de state para evitar conflictos en entornos colaborativos
- Ejecutar workflows de import para incorporar recursos existentes a Terraform
- Manipular el state de forma segura para refactorizaciones y reorganizaciones
- Implementar estrategias de recuperacion ante desastres del state file

## Requisitos previos

- Conocimientos: Terraform basico (resources, state concept), S3, DynamoDB, cifrado con KMS
- Herramientas: Terraform >= 1.5, AWS CLI, jq (para inspeccion de state)
- Servicios AWS: Cuenta AWS con permisos para S3, DynamoDB, KMS, IAM, EC2 o cualquier servicio para importar

## Arquitectura

El ejercicio progresa a traves de varias fases de gestion de state. Se inicia con state local, se migra a un backend S3 con cifrado KMS y locking DynamoDB. Se crean recursos manualmente en AWS para practicar import. Se refactoriza el codigo moviendo recursos entre modulos con `state mv`. Se simula corrupcion del state y se practica la recuperacion usando backups automaticos de S3 versioning.

## Pasos sugeridos

1. Crear un proyecto Terraform simple con recursos basicos (VPC, security group, EC2) usando state local
2. Crear el bucket S3 con versionado y cifrado KMS para el backend remoto
3. Crear la tabla DynamoDB para state locking
4. Migrar el state local al backend S3 con `terraform init -migrate-state`
5. Verificar el locking: abrir dos terminales e intentar `terraform plan` simultaneamente
6. Crear recursos manualmente en la consola de AWS (un security group, una subnet)
7. Importar los recursos manuales con `terraform import` y escribir la configuracion correspondiente
8. Refactorizar: mover recursos a un modulo con `terraform state mv`
9. Practicar `terraform state rm` para dejar de gestionar un recurso sin destruirlo
10. Simular corrupcion del state (modificar el JSON manualmente) y recuperar usando S3 versioning

## Validacion

- [ ] El state se migro exitosamente de local a S3
- [ ] El backend S3 tiene versionado y cifrado KMS habilitados
- [ ] El locking con DynamoDB previene operaciones simultaneas
- [ ] Los recursos existentes se importaron correctamente con `terraform import`
- [ ] `terraform plan` muestra "no changes" despues de importar y escribir la configuracion
- [ ] Los recursos se movieron entre modulos con `terraform state mv` sin recreacion
- [ ] `terraform state rm` desvincula un recurso sin destruirlo en AWS
- [ ] La recuperacion de state corrupto funciona usando S3 versioning
