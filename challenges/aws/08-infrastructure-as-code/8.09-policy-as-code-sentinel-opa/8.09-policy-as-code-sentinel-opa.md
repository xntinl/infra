# 8.09 Policy as Code con Sentinel y OPA

## Metadata
| Campo | Valor |
|-------|-------|
| **Complejidad** | Avanzado |
| **Modalidad** | Terraform + OPA/Sentinel |
| **Servicios AWS** | Cualquier servicio AWS objetivo de las politicas |

## Descripcion

Implementar policy-as-code usando Open Policy Agent (OPA) o HashiCorp Sentinel. Escribir politicas para: todos los buckets S3 deben estar cifrados, las instancias EC2 deben usar AMIs aprobadas, no se pueden crear recursos en regiones no autorizadas, todos los recursos deben tener tags requeridos.

## Objetivos de aprendizaje

- Comprender los patrones de policy-as-code y su importancia en governance de infraestructura
- Aprender el lenguaje Rego de OPA para escribir politicas declarativas
- Implementar validacion pre-plan y post-plan de configuraciones Terraform
- Automatizar el cumplimiento de compliance en pipelines de infraestructura

## Requisitos previos

- Conocimientos: Terraform intermedio, conceptos de compliance y governance, logica booleana, JSON
- Herramientas: Terraform >= 1.5, OPA (conftest), Git, jq, Python (opcional)
- Servicios AWS: Cuenta AWS con permisos para S3, EC2, IAM (para probar las politicas)

## Arquitectura

El flujo de validacion se integra en el pipeline de CI/CD de Terraform. Antes de aplicar cambios, se ejecutan dos capas de validacion. La primera capa (pre-plan) usa `conftest` con politicas Rego para validar los archivos `.tf` directamente (estructura, naming conventions). La segunda capa (post-plan) ejecuta `terraform plan -out=plan.json`, convierte el plan a JSON y lo evalua contra politicas Rego que verifican el compliance de los recursos a crear/modificar. Solo si ambas capas pasan, se permite ejecutar `terraform apply`.

## Pasos sugeridos

1. Instalar OPA y conftest en el entorno de desarrollo
2. Crear una configuracion Terraform de ejemplo con recursos S3, EC2 e IAM (algunos compliant, otros no)
3. Escribir una politica Rego que verifique que todos los buckets S3 tienen cifrado habilitado
4. Escribir una politica que valide que las instancias EC2 usan AMIs de una lista aprobada
5. Escribir una politica que bloquee la creacion de recursos en regiones no autorizadas
6. Escribir una politica que requiera tags obligatorios (Environment, Team, Project) en todos los recursos
7. Configurar conftest para evaluar las politicas contra los archivos `.tf` (validacion pre-plan)
8. Generar un `terraform plan` en formato JSON y evaluar las politicas post-plan con conftest
9. Crear un script o Makefile que integre ambas validaciones en un workflow unificado
10. Probar con configuraciones que violen las politicas y verificar que se bloquean correctamente

## Validacion

- [ ] Las politicas Rego estan escritas y se parsean correctamente con OPA
- [ ] La politica de cifrado S3 bloquea buckets sin cifrado y permite buckets cifrados
- [ ] La politica de AMIs aprobadas bloquea instancias con AMIs no autorizadas
- [ ] La politica de regiones bloquea recursos en regiones no permitidas
- [ ] La politica de tags requeridos bloquea recursos sin los tags obligatorios
- [ ] La validacion pre-plan funciona contra archivos `.tf` directamente
- [ ] La validacion post-plan funciona contra el output JSON de `terraform plan`
- [ ] Las configuraciones compliant pasan todas las validaciones
- [ ] Las configuraciones no compliant son bloqueadas con mensajes de error claros
