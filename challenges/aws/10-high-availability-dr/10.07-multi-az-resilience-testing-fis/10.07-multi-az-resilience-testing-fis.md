# 10.07 Multi-AZ Resilience Testing con FIS

## Metadata
| Campo | Valor |
|-------|-------|
| **Complejidad** | Avanzado |
| **Modalidad** | Ambos |
| **Servicios AWS** | FIS, EC2, ECS, RDS Multi-AZ, ALB, CloudWatch |

## Descripcion

Desplegar una aplicacion de 3 capas distribuida en 3 AZs. Usar AWS Fault Injection Simulator para simular: fallo de una AZ completa, interrupcion de red, failover de RDS y terminacion de tareas ECS. Verificar que la aplicacion se mantiene disponible durante cada fallo.

## Objetivos de aprendizaje

- Comprender y practicar pruebas de resiliencia con inyeccion de fallos
- Utilizar AWS Fault Injection Simulator para escenarios de fallo controlados
- Validar la independencia de AZ en la arquitectura
- Verificar la recuperacion automatica ante diferentes tipos de fallo
- Establecer una cultura de chaos engineering

## Requisitos previos

- Conocimientos: Arquitectura multi-AZ, ECS, RDS Multi-AZ, ALB, conceptos de resiliencia
- Herramientas: AWS CLI, Terraform
- Servicios AWS: Cuenta con permisos para FIS, EC2, ECS, RDS, ALB, CloudWatch

## Arquitectura

La aplicacion de 3 capas se distribuye en 3 Availability Zones. La capa de presentacion es un ALB que distribuye trafico entre las AZs. La capa de aplicacion ejecuta tareas ECS Fargate distribuidas en las 3 AZs (o instancias EC2 en un ASG multi-AZ). La capa de datos usa RDS Multi-AZ con standby automatico. AWS FIS ejecuta experimentos controlados que simulan fallos: terminacion de instancias/tareas, interrupcion de red (via Network ACLs o security groups), fallo de AZ completa (combinacion de las anteriores), y failover forzado de RDS. CloudWatch monitorea la disponibilidad y latencia durante cada experimento para validar que la aplicacion sigue sirviendo trafico.

## Pasos sugeridos

1. Desplegar la aplicacion de 3 capas en 3 AZs: ALB, ECS Fargate (o EC2+ASG), RDS Multi-AZ
2. Configurar monitoreo detallado en CloudWatch (disponibilidad, latencia, errores)
3. Crear un IAM role para FIS con permisos necesarios para inyectar fallos
4. Crear un experimento FIS que termine tareas ECS y verificar recuperacion automatica
5. Crear un experimento FIS que simule interrupcion de red en una AZ
6. Crear un experimento FIS que fuerce el failover de RDS Multi-AZ
7. Crear un experimento FIS que simule el fallo completo de una AZ (combinando acciones)
8. Ejecutar cada experimento y monitorear la disponibilidad en tiempo real
9. Analizar los resultados: tiempo de deteccion, tiempo de recuperacion, impacto en usuarios
10. Documentar hallazgos y crear runbooks para cada escenario de fallo

## Validacion

- [ ] La aplicacion funciona correctamente distribuida en 3 AZs
- [ ] La terminacion de tareas ECS resulta en reemplazo automatico sin impacto visible
- [ ] La interrupcion de red en una AZ no afecta la disponibilidad general
- [ ] El failover de RDS completa sin errores permanentes en la aplicacion
- [ ] El fallo simulado de una AZ completa mantiene la aplicacion disponible
- [ ] CloudWatch muestra metricas claras del impacto y recuperacion de cada experimento
- [ ] Los tiempos de recuperacion estan documentados para cada tipo de fallo
