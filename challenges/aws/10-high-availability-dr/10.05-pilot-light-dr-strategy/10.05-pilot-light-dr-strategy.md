# 10.05 Estrategia DR Pilot Light

## Metadata
| Campo | Valor |
|-------|-------|
| **Complejidad** | Avanzado |
| **Modalidad** | Terraform + automatizacion personalizada |
| **Servicios AWS** | EC2 AMI, RDS, S3, Lambda, Step Functions, Route 53, Auto Scaling |

## Descripcion

Implementar una estrategia DR de tipo pilot light: mantener infraestructura minima en la region DR (AMIs pre-configuradas, replica de lectura RDS, S3 CRR). Construir automatizacion (Lambda + Step Functions) que escale el entorno DR cuando se active. Probar el failover y medir el tiempo hasta alcanzar capacidad completa.

## Objetivos de aprendizaje

- Comprender el espectro de estrategias DR (backup-restore, pilot light, warm standby, multi-site)
- Evaluar trade-offs entre costo y RTO en cada estrategia
- Construir automatizacion confiable para escalar infraestructura DR
- Ejecutar y validar pruebas de failover end-to-end
- Medir tiempos reales de recuperacion y compararlos con los objetivos

## Requisitos previos

- Conocimientos: Estrategias de DR, EC2 AMI, RDS replication, Step Functions, Auto Scaling
- Herramientas: AWS CLI, Terraform, Python/Node.js para Lambda
- Servicios AWS: Cuenta con permisos en dos regiones para EC2, RDS, S3, Lambda, Step Functions, Route 53

## Arquitectura

La region primaria ejecuta la aplicacion completa: instancias EC2 en un Auto Scaling Group detras de un ALB, conectadas a RDS. La region DR mantiene solo lo minimo encendido (pilot light): AMIs actualizadas copiadas periodicamente, una replica de lectura RDS cross-region, y S3 CRR para datos estaticos. Cuando se detecta un fallo (o se decide activar el DR manualmente), una Step Functions state machine orquesta la activacion: promueve la replica RDS a escritora, lanza instancias EC2 desde las AMIs en un ASG, configura el ALB, y actualiza Route 53. El proceso de scale-up tiene un tiempo medible que constituye el RTO de esta estrategia.

## Pasos sugeridos

1. Desplegar la aplicacion completa en la region primaria: VPC, ALB, ASG, EC2, RDS
2. Crear AMIs de las instancias EC2 y copiarlas a la region DR periodicamente (Lambda scheduled)
3. Configurar una replica de lectura RDS cross-region en la region DR
4. Configurar S3 CRR para datos estaticos y de configuracion
5. Crear la infraestructura minima de red en la region DR (VPC, subnets, security groups)
6. Desarrollar funciones Lambda para cada paso del scale-up (promote RDS, launch ASG, configure ALB)
7. Orquestar las funciones con una Step Functions state machine
8. Configurar Route 53 health checks y failover routing
9. Ejecutar el failover activando la state machine y medir el tiempo hasta capacidad completa
10. Documentar RTO medido, comparar con objetivo, e identificar cuellos de botella

## Validacion

- [ ] Las AMIs se copian periodicamente a la region DR y estan actualizadas
- [ ] La replica de lectura RDS se mantiene sincronizada con el primario
- [ ] S3 CRR replica los datos criticos a la region DR
- [ ] La Step Functions state machine ejecuta todos los pasos de scale-up en orden
- [ ] La replica RDS se promueve a escritora exitosamente
- [ ] Las instancias EC2 se lanzan desde las AMIs y pasan los health checks del ALB
- [ ] Route 53 redirige el trafico a la region DR
- [ ] El RTO medido esta documentado y es aceptable para los requisitos del negocio
