# 10.01 Multi-Region Active-Passive con Aurora Global Database

## Metadata
| Campo | Valor |
|-------|-------|
| **Complejidad** | Experto |
| **Modalidad** | Terraform recomendado |
| **Servicios AWS** | Aurora Global Database, ECS, ALB, Route 53, CloudWatch, S3 (CRR) |

## Descripcion

Construir una arquitectura multi-region active-passive: la region primaria sirve todo el trafico via ALB + ECS + Aurora. La region secundaria tiene una replica de lectura Aurora (via Global Database). Route 53 con failover routing dirige el trafico. Probar failover planificado y no planificado. Medir RPO/RTO.

## Objetivos de aprendizaje

- Disenar arquitecturas multi-region y comprender sus trade-offs
- Medir y validar RPO (Recovery Point Objective) y RTO (Recovery Time Objective)
- Implementar failover de base de datos con Aurora Global Database
- Configurar DNS failover con Route 53 health checks
- Garantizar consistencia de datos entre regiones

## Requisitos previos

- Conocimientos: Arquitectura multi-region, Aurora, ECS, Route 53, conceptos de DR
- Herramientas: AWS CLI, Terraform, Docker
- Servicios AWS: Cuenta con permisos en dos regiones para Aurora, ECS, ALB, Route 53, S3

## Arquitectura

La region primaria ejecuta la carga completa: un ALB distribuye trafico a un servicio ECS Fargate que se conecta a un cluster Aurora como escritor. La region secundaria mantiene una replica de Aurora Global Database (replicacion asincrona con lag tipico < 1 segundo) y una infraestructura de computo minima o pre-desplegada. Route 53 utiliza failover routing con health checks activos contra el ALB primario. En caso de fallo de la region primaria, Route 53 redirige el trafico a la region secundaria. El cluster Aurora secundario se promueve a escritor (failover planificado via switchover o no planificado via detach-and-promote). S3 Cross-Region Replication mantiene sincronizados los assets estaticos.

## Pasos sugeridos

1. Desplegar la infraestructura en la region primaria: VPC, subnets, ALB, ECS Fargate, Aurora cluster
2. Crear el Aurora Global Database y agregar un cluster secundario en la segunda region
3. Desplegar la infraestructura de la region secundaria: VPC, subnets, ALB, ECS Fargate (conectado al cluster Aurora secundario)
4. Configurar S3 CRR para replicar assets estaticos entre regiones
5. Crear registros Route 53 con failover routing policy y health checks sobre los ALBs
6. Verificar que el trafico fluye normalmente a la region primaria
7. Ejecutar un failover planificado (switchover) de Aurora Global Database y medir el tiempo
8. Validar que la region secundaria sirve trafico correctamente tras el failover
9. Simular un failover no planificado (detach-and-promote) y medir RPO/RTO
10. Documentar los tiempos de RPO/RTO medidos y comparar con los objetivos

## Validacion

- [ ] La region primaria sirve todo el trafico normalmente via Route 53
- [ ] Aurora Global Database replica datos a la region secundaria con lag < 1 segundo
- [ ] El failover planificado completa sin perdida de datos (RPO = 0)
- [ ] Route 53 redirige el trafico a la region secundaria tras detectar fallo
- [ ] La region secundaria puede servir trafico de lectura y escritura tras la promocion
- [ ] El RTO medido esta dentro de los objetivos definidos (tipicamente < 5 minutos para DNS + DB promotion)
- [ ] S3 CRR mantiene los assets sincronizados entre regiones
