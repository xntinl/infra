# 10.02 Multi-Region Active-Active con DynamoDB Global Tables

## Metadata
| Campo | Valor |
|-------|-------|
| **Complejidad** | Experto |
| **Modalidad** | Ambos |
| **Servicios AWS** | DynamoDB Global Tables, API Gateway, Lambda, Route 53, CloudWatch |

## Descripcion

Desplegar una API en dos regiones, ambas leyendo y escribiendo en DynamoDB Global Tables. Route 53 con latency-based routing dirige a los usuarios a la region mas cercana. Implementar manejo de conflictos en la capa de aplicacion. Probar escrituras simultaneas en ambas regiones.

## Objetivos de aprendizaje

- Comprender patrones active-active y sus implicaciones
- Manejar consistencia eventual y resolucion de conflictos
- Configurar DynamoDB Global Tables y entender su modelo de replicacion
- Implementar latency-based routing con Route 53
- Disenar aplicaciones tolerantes a escrituras concurrentes multi-region

## Requisitos previos

- Conocimientos: DynamoDB avanzado, API Gateway, Lambda, conceptos de consistencia eventual
- Herramientas: AWS CLI, Terraform, lenguaje para Lambda (Python/Node.js)
- Servicios AWS: Cuenta con permisos en dos regiones para DynamoDB, API Gateway, Lambda, Route 53

## Arquitectura

Ambas regiones ejecutan stacks identicos: API Gateway (regional endpoint) + Lambda + DynamoDB. DynamoDB Global Tables replica automaticamente los datos entre las tablas de ambas regiones con consistencia eventual (replicacion tipica < 1 segundo). Route 53 utiliza latency-based routing para dirigir a cada usuario a la region con menor latencia. En caso de fallo de una region, Route 53 health checks redirigen todo el trafico a la region sana. La aplicacion implementa estrategias de resolucion de conflictos para manejar escrituras simultaneas al mismo item desde ambas regiones (last-writer-wins es el default de Global Tables, pero la aplicacion puede agregar logica adicional).

## Pasos sugeridos

1. Desplegar la infraestructura en la region primaria: API Gateway, Lambda, DynamoDB
2. Desplegar la misma infraestructura en la region secundaria
3. Convertir las tablas DynamoDB a Global Tables (agregar replica en la segunda region)
4. Configurar Route 53 con latency-based routing y health checks para ambas regiones
5. Verificar que los usuarios son dirigidos a la region mas cercana
6. Escribir datos desde la region A y verificar la replicacion a la region B
7. Implementar logica de resolucion de conflictos en la aplicacion (ej: versionado, timestamps)
8. Ejecutar escrituras simultaneas al mismo item desde ambas regiones
9. Verificar que los conflictos se resuelven segun la estrategia implementada
10. Simular la caida de una region y verificar que el trafico migra automaticamente

## Validacion

- [ ] Ambas regiones sirven trafico de lectura y escritura simultaneamente
- [ ] Los datos escritos en una region se replican a la otra en menos de 1 segundo
- [ ] Route 53 dirige a los usuarios a la region con menor latencia
- [ ] Las escrituras simultaneas al mismo item se resuelven sin errores
- [ ] La estrategia de resolucion de conflictos funciona segun lo disenado
- [ ] Al simular la caida de una region, el trafico migra automaticamente a la otra
- [ ] No hay perdida de datos durante el failover
