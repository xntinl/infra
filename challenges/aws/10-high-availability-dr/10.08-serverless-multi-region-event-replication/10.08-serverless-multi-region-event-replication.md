# 10.08 Serverless Multi-Region con Event Replication

## Metadata
| Campo | Valor |
|-------|-------|
| **Complejidad** | Experto |
| **Modalidad** | Ambos |
| **Servicios AWS** | API Gateway, Lambda, DynamoDB Global Tables, EventBridge, Route 53 |

## Descripcion

Construir una aplicacion serverless que se replique entre regiones usando: API Gateway (regional endpoints) + Lambda + DynamoDB Global Tables. Implementar EventBridge global endpoints para replicacion de eventos. Usar Route 53 para gestion de trafico.

## Objetivos de aprendizaje

- Implementar DR en arquitecturas serverless sin servidores que gestionar
- Configurar replicacion de eventos con EventBridge global endpoints
- Utilizar DynamoDB Global Tables en un contexto serverless real
- Disenar failover de compute stateless (Lambda)
- Gestionar trafico multi-region con Route 53

## Requisitos previos

- Conocimientos: API Gateway, Lambda, DynamoDB, EventBridge avanzado, Route 53
- Herramientas: AWS CLI, Terraform o SAM, lenguaje para Lambda (Python/Node.js)
- Servicios AWS: Cuenta con permisos en dos regiones para API Gateway, Lambda, DynamoDB, EventBridge, Route 53

## Arquitectura

La aplicacion serverless se despliega identicamente en dos regiones. Cada region tiene: un API Gateway regional endpoint, funciones Lambda para la logica de negocio, y una tabla DynamoDB (parte de Global Tables para replicacion bidireccional). EventBridge global endpoints replican eventos entre regiones: cuando un evento se publica en una region, el global endpoint lo envia automaticamente a la otra region, asegurando que los flujos event-driven se ejecuten en ambas. Route 53 con health checks y failover/latency-based routing dirige a los usuarios. En condiciones normales, el trafico se distribuye por latencia. Si una region falla, todo el trafico se redirige automaticamente a la region sana. El compute (Lambda) es inherentemente stateless, lo que simplifica el failover.

## Pasos sugeridos

1. Desplegar el stack serverless en la region primaria: API Gateway, Lambda functions, DynamoDB
2. Desplegar el mismo stack en la region secundaria
3. Convertir las tablas DynamoDB a Global Tables (replicacion bidireccional)
4. Configurar EventBridge global endpoints para replicar eventos entre regiones
5. Crear las reglas EventBridge en ambas regiones para procesar los eventos replicados
6. Configurar Route 53 con latency-based routing y health checks para ambos API Gateways
7. Verificar que las solicitudes API y eventos se procesan correctamente en ambas regiones
8. Probar la replicacion de datos: escribir en una region y leer en la otra
9. Simular la caida de una region y verificar el failover automatico
10. Medir tiempos de failover y validar que no hay perdida de eventos durante la transicion

## Validacion

- [ ] Ambas regiones sirven trafico API correctamente
- [ ] DynamoDB Global Tables replica datos bidireccional entre regiones
- [ ] EventBridge global endpoints replican eventos entre regiones
- [ ] Los eventos replicados disparan los flujos correspondientes en la region destino
- [ ] Route 53 dirige trafico basado en latencia en condiciones normales
- [ ] Al simular la caida de una region, el trafico migra automaticamente a la otra
- [ ] No se pierden eventos durante el failover
- [ ] El failover es transparente para el usuario final
