# 10.04 Route 53 Application Recovery Controller

## Descripcion

Implementar Route 53 ARC con readiness checks y routing controls. Definir cells (por AZ o por region) y recovery groups. Crear routing control health checks asociados a registros DNS de Route 53. Practicar failover controlado usando el cluster endpoint.

## Objetivos de aprendizaje

- Comprender el failover controlado con Route 53 ARC
- Implementar readiness assessment para validar la preparacion de los recursos
- Configurar routing controls para failover granular
- Disenar recovery groups y definir cells con blast radius controlado
- Gestionar el blast radius en arquitecturas multi-AZ y multi-region

## Requisitos previos

- Conocimientos: Route 53 avanzado, alta disponibilidad, conceptos de DR, health checks
- Herramientas: AWS CLI, Terraform
- Servicios AWS: Cuenta con permisos para Route 53 ARC, Route 53, ALB, ECS/EC2, CloudWatch

## Arquitectura

Route 53 ARC proporciona control operativo sobre el failover de aplicaciones distribuidas. Se definen cells que representan unidades independientes de fallo (una AZ o una region), agrupadas en recovery groups. Los readiness checks validan continuamente que cada cell tiene los recursos necesarios (ASG capacidad, ALB listeners, certificados, etc.) para recibir trafico. Los routing controls son interruptores on/off que controlan el flujo de trafico hacia cada cell. Estos routing controls se asocian como health checks a los registros DNS de Route 53. El cluster endpoint de ARC (cinco endpoints regionales para alta disponibilidad) permite activar/desactivar routing controls de forma segura, incluso durante una interrupcion regional. Safety rules previenen acciones peligrosas como desactivar todas las cells simultaneamente.

## Pasos sugeridos

1. Desplegar aplicacion con ALB + ECS/EC2 en dos regiones (o multiples AZs)
2. Crear un cluster de Route 53 ARC (genera cinco cluster endpoints regionales)
3. Definir cells para cada unidad de fallo (por region o por AZ)
4. Crear un recovery group que agrupe las cells
5. Configurar readiness checks para validar recursos en cada cell (ASG, ALB, etc.)
6. Crear routing controls (uno por cell) dentro de un control panel
7. Asociar los routing controls como health checks a los registros Route 53
8. Configurar safety rules (ej: al menos una cell debe estar activa)
9. Practicar failover controlado activando/desactivando routing controls via cluster endpoint
10. Verificar que el trafico se redirige correctamente y que las safety rules previenen errores

## Validacion

- [ ] El cluster de ARC esta creado con cinco endpoints regionales accesibles
- [ ] Los readiness checks validan correctamente el estado de recursos en cada cell
- [ ] Los routing controls estan asociados como health checks a los registros Route 53
- [ ] Al desactivar un routing control, el trafico deja de fluir a esa cell
- [ ] Al activar un routing control, el trafico se restaura a esa cell
- [ ] Las safety rules previenen la desactivacion de todas las cells simultaneamente
- [ ] El failover controlado completa sin interrupciones perceptibles para el usuario
