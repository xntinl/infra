# 13.09 Disaster Recovery del Terraform State

## Metadata
| Campo | Valor |
|-------|-------|
| **Complejidad** | Avanzado |
| **Modalidad** | Terraform + AWS services |
| **Features TF** | S3 backend, versioning, lifecycle rules |

## Descripcion

Implementar un plan completo de DR para el Terraform state: S3 Cross-Region Replication para el bucket del state, backup automatizado con Lambda, alerting sobre cambios en el state, y un runbook para recovery del state. Probar la restauracion completa del state desde backup en un escenario de DR.

## Objetivos de aprendizaje

- Garantizar la durabilidad del state file de Terraform
- Disenar estrategias de DR para Infrastructure as Code
- Automatizar backups del state con servicios AWS
- Crear procedimientos de recovery y probarlos

## Requisitos previos

- Conocimientos: Terraform backends, S3 replication, Lambda, CloudWatch/EventBridge
- Herramientas: Terraform >= 1.0, AWS CLI, Python (para Lambda)
- Servicios/Features: S3 CRR, Lambda, EventBridge, SNS, CloudWatch

## Arquitectura

La arquitectura de DR protege el state file en multiples capas:

- **S3 Primary Bucket**: Bucket principal del state con versioning habilitado en la region primaria
- **S3 Replica Bucket**: Bucket de replica en una region secundaria via Cross-Region Replication (CRR)
- **Lambda Backup**: Funcion Lambda disparada por EventBridge que crea copias adicionales del state con timestamp
- **Alerting**: EventBridge rules que detectan cambios en el state file y envian notificaciones via SNS
- **Runbook**: Procedimiento documentado para restaurar el state desde cualquier fuente de backup
- **DR Test**: Simulacion periodica de perdida del state y ejecucion del procedimiento de recovery

## Pasos sugeridos

1. Configurar el bucket S3 primario del state con versioning y lifecycle rules
2. Crear un bucket replica en una region secundaria con CRR habilitado
3. Verificar que los state files se replican automaticamente a la region secundaria
4. Crear una funcion Lambda que copie el state a un bucket de backup con timestamp en cada cambio
5. Configurar EventBridge para disparar la Lambda cuando se modifica el state file
6. Configurar alertas SNS para notificar al equipo cuando el state cambia
7. Escribir el runbook de DR: pasos para restaurar el state desde cada fuente de backup
8. Simular un escenario de DR: eliminar el state file del bucket primario
9. Ejecutar el procedimiento de recovery restaurando desde la replica o el backup
10. Verificar con `terraform plan` que el state restaurado es consistente (cero cambios)

## Validacion

- [ ] El bucket S3 del state tiene versioning y CRR habilitados
- [ ] Los state files se replican automaticamente a la region secundaria
- [ ] La Lambda de backup se ejecuta en cada cambio del state
- [ ] Las alertas SNS notifican cuando el state cambia
- [ ] El runbook de DR esta documentado con pasos claros
- [ ] La restauracion del state desde backup fue probada exitosamente
- [ ] `terraform plan` confirma cero cambios despues de la restauracion
