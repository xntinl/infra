# 13.01 S3 Backend con Native State Locking

## Metadata
| Campo | Valor |
|-------|-------|
| **Complejidad** | Intermedio |
| **Modalidad** | Terraform |
| **Features TF** | S3 backend, `use_lockfile`, versioning, encryption |

## Descripcion

Configurar un S3 backend con native S3 state locking (Terraform 1.10+), reemplazando el locking basado en DynamoDB. Habilitar versioning y encryption. Probar applies concurrentes para verificar que el locking funciona correctamente. Implementar una estrategia de backup del state.

## Objetivos de aprendizaje

- Comprender los fundamentos del remote state en Terraform
- Configurar S3-native locking con `use_lockfile`
- Implementar encryption del state en reposo
- Dominar la configuracion de backends

## Requisitos previos

- Conocimientos: Terraform basico, conceptos de state, AWS S3
- Herramientas: Terraform >= 1.10, AWS CLI, cuenta AWS con permisos de S3
- Features: S3 backend, `use_lockfile`, server-side encryption

## Arquitectura

La arquitectura consiste en un bucket S3 dedicado al almacenamiento del state file, con las siguientes caracteristicas:

- **S3 Bucket**: Almacena el state file con versioning habilitado para historial y rollback
- **S3 Native Locking**: Utiliza `use_lockfile = true` para locking nativo sin DynamoDB
- **Encryption**: Server-side encryption (SSE-S3 o SSE-KMS) para proteger datos sensibles en el state
- **Backup Strategy**: Lifecycle rules y/o replicacion para respaldo del state

El flujo de operacion es: `terraform init` configura el backend -> cada `terraform apply` adquiere un lock via S3 -> escribe el state encriptado -> libera el lock.

## Pasos sugeridos

1. Crear un bucket S3 con versioning habilitado y server-side encryption por defecto
2. Configurar el bloque `backend "s3"` en Terraform con `use_lockfile = true`
3. Ejecutar `terraform init` para inicializar el backend remoto
4. Desplegar un recurso simple (por ejemplo, un S3 bucket adicional) y verificar que el state se almacena en S3
5. Abrir dos terminales y ejecutar `terraform apply` simultaneamente para comprobar que el locking previene conflictos
6. Verificar en S3 que el archivo `.tflock` se crea durante el apply y se elimina al finalizar
7. Confirmar que el versioning del bucket guarda versiones anteriores del state
8. Implementar una estrategia de backup (lifecycle rules o script de copia periodica)
9. Probar la restauracion del state desde una version anterior usando S3 versioning

## Validacion

- [ ] El backend S3 esta configurado con `use_lockfile = true`
- [ ] El bucket tiene versioning habilitado
- [ ] El state file esta encriptado en reposo
- [ ] Applies concurrentes son bloqueados correctamente por el lock nativo
- [ ] Se pueden recuperar versiones anteriores del state desde S3 versioning
- [ ] La estrategia de backup esta documentada e implementada
