# 13.10 Workspaces Efimeros para Feature Branches

## Metadata
| Campo | Valor |
|-------|-------|
| **Complejidad** | Avanzado |
| **Modalidad** | Terraform + GitHub Actions |
| **Features TF** | Workspaces, dynamic backend config, `-var-file` |

## Descripcion

Construir un sistema donde cada feature branch obtiene su propio workspace de Terraform e infraestructura. Al abrir un PR, crear el workspace y aplicar. Al hacer merge o cerrar el PR, destruir y eliminar el workspace. Integrar con GitHub Actions y usar tfvars especificos por workspace.

## Objetivos de aprendizaje

- Implementar entornos efimeros con workspaces de Terraform
- Gestionar el ciclo de vida completo de workspaces automaticamente
- Integrar Terraform con pipelines CI/CD avanzados
- Controlar costos de infraestructura efimera

## Requisitos previos

- Conocimientos: Terraform workspaces, GitHub Actions, branching strategies, YAML
- Herramientas: Terraform >= 1.0, GitHub Actions, AWS CLI
- Features: Terraform workspaces, `-var-file`, dynamic backend config, GitHub Actions workflows

## Arquitectura

El sistema automatiza el ciclo de vida completo de entornos efimeros:

- **PR Open Trigger**: GitHub Actions detecta la apertura de un PR y extrae el nombre de la rama
- **Workspace Creation**: Se crea un workspace de Terraform con el nombre derivado de la rama (sanitizado)
- **Dynamic tfvars**: Archivo de variables especifico por workspace con dimensionamiento reducido para efimeros
- **Infrastructure Deploy**: `terraform apply` despliega infraestructura aislada para la feature branch
- **PR Comment**: El workflow publica un comentario en el PR con la URL/endpoints del entorno efimero
- **Cleanup Trigger**: Al hacer merge o cerrar el PR, se ejecuta `terraform destroy` y se elimina el workspace
- **Cost Control**: Tags de recurso con TTL y alertas para workspaces que excedan el tiempo esperado

## Pasos sugeridos

1. Disenar la convencion de nombrado de workspaces a partir de nombres de rama (sanitizar caracteres especiales)
2. Crear un archivo tfvars para entornos efimeros con recursos de tamano minimo
3. Escribir el workflow de GitHub Actions para el evento `pull_request` (opened, synchronize)
4. Implementar el paso de creacion de workspace: `terraform workspace new` o `select`
5. Ejecutar `terraform apply` con el tfvars efimero y el workspace seleccionado
6. Agregar un paso que publique un comentario en el PR con los outputs del despliegue
7. Escribir el workflow para el evento `pull_request` (closed) que ejecute destroy
8. Implementar `terraform destroy` seguido de `terraform workspace delete`
9. Agregar tags de TTL a los recursos y un mecanismo de limpieza para workspaces huerfanos
10. Probar el flujo completo: crear rama -> abrir PR -> verificar infra -> merge -> verificar limpieza

## Validacion

- [ ] Al abrir un PR se crea automaticamente un workspace y se despliega infraestructura
- [ ] El nombre del workspace se deriva correctamente del nombre de la rama
- [ ] El PR recibe un comentario con los endpoints/outputs del entorno efimero
- [ ] Al cerrar/mergear el PR la infraestructura se destruye automaticamente
- [ ] El workspace se elimina despues del destroy
- [ ] Los recursos efimeros tienen tags apropiados (TTL, branch, PR number)
- [ ] Existe un mecanismo de limpieza para workspaces huerfanos
