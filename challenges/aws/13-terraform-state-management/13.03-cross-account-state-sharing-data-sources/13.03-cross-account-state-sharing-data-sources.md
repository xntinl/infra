# 13.03 State Sharing Cross-Account con Data Sources

## Metadata
| Campo | Valor |
|-------|-------|
| **Complejidad** | Avanzado |
| **Modalidad** | Terraform |
| **Features TF** | `terraform_remote_state`, data sources, S3 bucket policies |

## Descripcion

Configurar el Terraform state en una cuenta compartida. Usar el data source `terraform_remote_state` desde cuentas de workload para leer outputs (VPC IDs, subnet IDs). Implementar politicas IAM que restrinjan el acceso de lectura al state por entorno.

## Objetivos de aprendizaje

- Implementar patrones de comparticion de state entre cuentas
- Configurar acceso cross-account seguro para state files
- Consumir outputs de otros estados remotos
- Aplicar seguridad granular al state

## Requisitos previos

- Conocimientos: AWS IAM, S3 bucket policies, Terraform backends, multi-account AWS
- Herramientas: Terraform >= 1.0, AWS CLI con perfiles para multiples cuentas
- Servicios/Features: S3, IAM, `terraform_remote_state` data source

## Arquitectura

La arquitectura sigue un patron hub-and-spoke para gestion de state:

- **Cuenta Shared Services (Hub)**: Bucket S3 centralizado que almacena los state files de infraestructura base (VPC, subnets, security groups compartidos)
- **Cuentas Workload (Spokes)**: Cuentas de dev/staging/prod que consumen outputs del state compartido via `terraform_remote_state`
- **IAM Policies**: Roles cross-account con permisos minimos que permiten solo lectura del state correspondiente a cada entorno
- **S3 Bucket Policies**: Restricciones a nivel de bucket que limitan acceso por prefijo (path) del state segun la cuenta solicitante

## Pasos sugeridos

1. Crear un bucket S3 en la cuenta de shared services para almacenar states compartidos
2. Desplegar infraestructura base (VPC, subnets) en la cuenta compartida con outputs bien definidos
3. Configurar una S3 bucket policy que permita acceso cross-account de lectura
4. Crear IAM roles en las cuentas workload con permisos para asumir rol en la cuenta compartida
5. Implementar `terraform_remote_state` data source en la configuracion de una cuenta workload
6. Consumir outputs del state remoto (VPC ID, subnet IDs) para desplegar recursos en la cuenta workload
7. Restringir acceso por entorno: la cuenta de dev solo puede leer el state de dev
8. Verificar que una cuenta no autorizada no puede leer states de otros entornos
9. Documentar el patron y las politicas de acceso implementadas

## Validacion

- [ ] El state compartido esta almacenado en un bucket S3 centralizado
- [ ] `terraform_remote_state` consume correctamente outputs desde otra cuenta
- [ ] Las bucket policies restringen acceso por entorno/cuenta
- [ ] Los IAM roles cross-account siguen el principio de minimo privilegio
- [ ] Una cuenta de workload puede desplegar recursos usando datos del state compartido
- [ ] El acceso no autorizado a states de otros entornos es denegado
