# 13.02 Migracion de State entre Backends

## Metadata
| Campo | Valor |
|-------|-------|
| **Complejidad** | Avanzado |
| **Modalidad** | Terraform |
| **Features TF** | `terraform init -migrate-state`, `moved` blocks, backend config |

## Descripcion

Migrar el state desde un backend local a un backend S3, y luego a Terraform Cloud. Practicar migracion parcial de state usando `terraform state mv`. Manejar el bloque `moved` para refactorizar paths de modulos sin destruir recursos.

## Objetivos de aprendizaje

- Dominar la migracion de state entre diferentes backends
- Ejecutar cambios de backend sin tiempo de inactividad
- Utilizar `moved` blocks para refactorizacion segura de recursos
- Comprender las implicaciones de cada tipo de backend

## Requisitos previos

- Conocimientos: Terraform backends, S3, conceptos de state, Terraform Cloud basico
- Herramientas: Terraform >= 1.5, AWS CLI, cuenta de Terraform Cloud (free tier)
- Features: S3 backend, Terraform Cloud backend, `moved` blocks

## Arquitectura

El ejercicio simula un ciclo de vida real de evolucion del backend:

- **Fase 1 - Local**: State almacenado en `terraform.tfstate` local con recursos desplegados
- **Fase 2 - S3**: Migracion a S3 backend con `terraform init -migrate-state`
- **Fase 3 - Terraform Cloud**: Segunda migracion hacia Terraform Cloud como backend
- **Refactorizacion**: Uso de `moved` blocks para reorganizar recursos entre modulos sin recrearlos

En cada fase, los recursos existentes en AWS permanecen intactos mientras el state se transfiere entre backends.

## Pasos sugeridos

1. Crear una configuracion con backend local y desplegar varios recursos (VPC, subnets, security groups)
2. Configurar un bucket S3 para usarlo como backend remoto
3. Modificar el bloque `backend` a S3 y ejecutar `terraform init -migrate-state`
4. Verificar que el state se migro correctamente con `terraform plan` (sin cambios)
5. Crear una cuenta en Terraform Cloud y configurar un workspace
6. Modificar el backend a `cloud` y ejecutar `terraform init -migrate-state` nuevamente
7. Practicar `terraform state mv` para mover un recurso a una direccion diferente
8. Implementar un bloque `moved` para renombrar un modulo y verificar con `terraform plan`
9. Documentar las diferencias y trade-offs de cada backend utilizado

## Validacion

- [ ] El state se migro exitosamente de local a S3 sin perdida de recursos
- [ ] El state se migro exitosamente de S3 a Terraform Cloud
- [ ] `terraform plan` muestra cero cambios despues de cada migracion
- [ ] Se utilizo `terraform state mv` para mover al menos un recurso
- [ ] Un bloque `moved` fue implementado y validado con plan sin destruccion de recursos
- [ ] Los recursos en AWS permanecen intactos durante todo el proceso
