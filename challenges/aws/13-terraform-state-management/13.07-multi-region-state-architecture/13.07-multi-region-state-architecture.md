# 13.07 Arquitectura de State Multi-Region

## Metadata
| Campo | Valor |
|-------|-------|
| **Complejidad** | Avanzado |
| **Modalidad** | Terraform |
| **Features TF** | Multiple backends, partial config, `terraform_remote_state` |

## Descripcion

Disenar una arquitectura de state para un despliegue multi-region: state files separados por region, state global compartido para Route 53/CloudFront/IAM, referencias cross-state. Implementar con convenciones de nombrado de buckets e IAM adecuados.

## Objetivos de aprendizaje

- Disenar arquitecturas de state para despliegues multi-region
- Implementar descomposicion de state (global vs regional)
- Diferenciar recursos globales de regionales en la gestion de state
- Aplicar convenciones de nombrado y acceso para multiples state files

## Requisitos previos

- Conocimientos: Terraform avanzado, AWS multi-region, Route 53, CloudFront, IAM
- Herramientas: Terraform >= 1.0, AWS CLI con acceso a multiples regiones
- Servicios/Features: S3, Route 53, CloudFront, IAM, `terraform_remote_state`

## Arquitectura

La arquitectura separa el state en capas global y regional:

- **State Global**: Un unico state file que gestiona recursos globales de AWS (Route 53 hosted zones, CloudFront distributions, IAM roles/policies). Almacenado en un bucket S3 en la region primaria.
- **State Regional (us-east-1)**: State dedicado para recursos de la region primaria (VPC, EC2, RDS, ALB)
- **State Regional (eu-west-1)**: State dedicado para recursos de la region secundaria
- **Cross-State References**: Los states regionales consumen outputs del state global (zone IDs, distribution IDs) via `terraform_remote_state`
- **Naming Convention**: Buckets nombrados como `{project}-tfstate-{global|region}`, keys como `{layer}/terraform.tfstate`

## Pasos sugeridos

1. Definir la estrategia de separacion: que recursos son globales vs regionales
2. Crear buckets S3 para state en cada region con convenciones de nombrado consistentes
3. Implementar la configuracion global (Route 53, CloudFront, IAM) con su propio backend
4. Definir outputs claros en la configuracion global para consumo por configuraciones regionales
5. Implementar la configuracion regional para us-east-1 con su backend dedicado
6. Usar `terraform_remote_state` para referenciar outputs del state global
7. Replicar la configuracion regional para eu-west-1 usando partial backend config
8. Configurar IAM policies que restrinjan acceso a cada state por region/capa
9. Probar el flujo completo: desplegar global primero, luego regionales
10. Documentar el orden de despliegue y las dependencias entre states

## Validacion

- [ ] El state global gestiona exclusivamente recursos de alcance global (Route 53, IAM, CloudFront)
- [ ] Cada region tiene su propio state file independiente
- [ ] Los states regionales consumen outputs del state global via `terraform_remote_state`
- [ ] Las convenciones de nombrado de buckets son consistentes y claras
- [ ] Los IAM policies restringen acceso apropiadamente por capa y region
- [ ] El orden de despliegue esta documentado y se respeta en el pipeline
