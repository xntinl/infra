# 13.05 State Surgery y Recovery

## Metadata
| Campo | Valor |
|-------|-------|
| **Complejidad** | Avanzado |
| **Modalidad** | Terraform |
| **Features TF** | `terraform state` subcommands, JSON state format |

## Descripcion

Practicar operaciones avanzadas de state: `terraform state rm` para desasociar un recurso, `terraform state mv` para renombrar/reestructurar, `terraform state pull/push` para edicion manual. Simular corrupcion del state (modificar el JSON del state) y recuperar usando S3 versioning.

## Objetivos de aprendizaje

- Comprender la estructura interna del state file de Terraform
- Ejecutar operaciones de emergencia en el state
- Dominar el resource addressing para operaciones de state
- Entender la anatomia del archivo de state JSON

## Requisitos previos

- Conocimientos: Terraform state, JSON, S3 versioning, resource addressing
- Herramientas: Terraform >= 1.0, AWS CLI, jq para manipulacion de JSON
- Features: `terraform state` subcommands, S3 backend con versioning

## Arquitectura

El ejercicio trabaja directamente con la estructura interna del state:

- **State File**: Archivo JSON que contiene el mapeo entre recursos HCL y recursos reales en AWS
- **S3 Backend con Versioning**: Permite recuperar versiones anteriores del state en caso de corrupcion
- **Operaciones de State**: Comandos que manipulan el state sin afectar la infraestructura real
- **Escenario de Corrupcion**: Modificacion intencional del JSON del state para practicar recovery

Las operaciones de state son herramientas de ultimo recurso pero esenciales para situaciones de emergencia.

## Pasos sugeridos

1. Desplegar una configuracion con multiples recursos (VPC, subnets, instancias, security groups) usando S3 backend con versioning
2. Ejecutar `terraform state list` para inventariar todos los recursos en el state
3. Usar `terraform state show <resource>` para examinar un recurso especifico en detalle
4. Practicar `terraform state rm` para desasociar un recurso del state (el recurso sigue en AWS)
5. Usar `terraform state mv` para renombrar un recurso en el state (simular refactorizacion)
6. Ejecutar `terraform state pull` para obtener el state JSON y examinarlo con jq
7. Modificar manualmente el JSON del state (simular corrupcion) y hacer `terraform state push`
8. Verificar que `terraform plan` detecta inconsistencias tras la corrupcion
9. Recuperar el state desde una version anterior usando S3 versioning
10. Confirmar la recuperacion exitosa con `terraform plan` mostrando cero cambios

## Validacion

- [ ] Se uso `terraform state rm` para desasociar un recurso exitosamente
- [ ] Se uso `terraform state mv` para renombrar un recurso sin destruirlo
- [ ] Se examino el state JSON con `terraform state pull` y se comprende su estructura
- [ ] Se simulo corrupcion del state modificando el JSON manualmente
- [ ] Se recupero el state corrupto usando S3 versioning
- [ ] `terraform plan` confirma cero cambios despues de la recuperacion
- [ ] Se documento el procedimiento de recovery como runbook
