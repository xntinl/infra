# 13.06 Estrategia de Workspaces para Multiples Entornos

## Metadata
| Campo | Valor |
|-------|-------|
| **Complejidad** | Intermedio |
| **Modalidad** | Terraform |
| **Features TF** | Workspaces, `terraform.workspace`, conditional expressions |

## Descripcion

Disenar e implementar una estrategia de despliegue basada en workspaces para dev/staging/prod. Usar `terraform.workspace` en las configuraciones para dimensionamiento especifico por entorno. Construir un pipeline CI/CD que seleccione el workspace segun la rama. Documentar los trade-offs frente al enfoque basado en directorios.

## Objetivos de aprendizaje

- Comprender los patrones de uso de workspaces en Terraform
- Implementar aislamiento de entornos con workspaces
- Integrar workspaces en pipelines CI/CD
- Evaluar las limitaciones de workspaces frente a otras estrategias

## Requisitos previos

- Conocimientos: Terraform basico, CI/CD conceptos, branching strategies
- Herramientas: Terraform >= 1.0, AWS CLI, GitHub Actions o similar
- Features: Terraform workspaces, S3 backend, conditional expressions

## Arquitectura

La arquitectura utiliza workspaces para gestionar multiples entornos desde una unica base de codigo:

- **Workspace dev**: Recursos de menor tamano (t3.micro, single AZ) con costos reducidos
- **Workspace staging**: Configuracion intermedia que replica parcialmente produccion
- **Workspace prod**: Recursos de mayor tamano (t3.large, multi-AZ) con alta disponibilidad
- **CI/CD Pipeline**: GitHub Actions que mapea ramas a workspaces (develop->dev, release->staging, main->prod)
- **State Isolation**: Cada workspace mantiene su propio state file en el mismo backend S3

## Pasos sugeridos

1. Configurar un S3 backend que soporte multiples workspaces (key con prefijo de workspace)
2. Crear los workspaces: `terraform workspace new dev`, `staging`, `prod`
3. Definir un mapa de configuracion por entorno usando `terraform.workspace` como clave
4. Implementar dimensionamiento condicional (instance types, counts, storage) segun el workspace
5. Crear un pipeline de GitHub Actions que seleccione workspace segun la rama
6. Desplegar infraestructura en los tres entornos y verificar las diferencias de configuracion
7. Probar el flujo completo: push a rama -> seleccion de workspace -> plan/apply
8. Documentar los trade-offs: workspaces vs directorios vs Terragrunt
9. Identificar escenarios donde los workspaces no son la mejor opcion

## Validacion

- [ ] Los tres workspaces (dev, staging, prod) estan creados y funcionales
- [ ] `terraform.workspace` se usa para diferenciar configuraciones por entorno
- [ ] Los recursos en cada entorno tienen el dimensionamiento correcto
- [ ] El pipeline CI/CD selecciona el workspace correcto segun la rama
- [ ] Cada workspace tiene su state file aislado en S3
- [ ] Los trade-offs frente al enfoque de directorios estan documentados
