# 11.04 Automated Idle Resource Cleanup

## Metadata
| Campo | Valor |
|-------|-------|
| **Complejidad** | Intermedio |
| **Modalidad** | Ambos (Lambda code + Terraform for infra) |
| **Servicios AWS** | Lambda, EventBridge, SNS, EC2, RDS, S3 |

## Descripcion

Construir un sistema automatizado que identifique y limpie recursos ociosos: volumenes EBS no adjuntos, Elastic IPs no utilizadas, snapshots antiguos, buckets S3 vacios, instancias RDS ociosas y NAT Gateways no asociados. Usar Lambda + EventBridge schedules con notificaciones SNS antes de la eliminacion.

## Objetivos de aprendizaje

- Gestionar el ciclo de vida de recursos en AWS de forma automatizada
- Identificar las fuentes mas comunes de desperdicio de costos
- Implementar gobernanza automatizada de recursos
- Disenar mecanismos de seguridad como modo dry-run para evitar eliminaciones accidentales

## Requisitos previos

- **Conocimientos**: Python o Node.js para Lambda, boto3/AWS SDK, EventBridge schedules
- **Herramientas**: AWS CLI, Terraform, lenguaje de programacion para Lambda (Python recomendado)
- **Servicios AWS**: Permisos para listar y eliminar recursos EC2, RDS, S3; crear funciones Lambda y reglas EventBridge

## Arquitectura

El sistema de limpieza automatizada se compone de:

- **EventBridge Scheduler**: Dispara la ejecucion periodica (diaria o semanal) de las funciones Lambda
- **Lambda Functions**: Una o varias funciones que escanean la cuenta buscando recursos ociosos por tipo
- **SNS Topic**: Canal de notificacion para alertar antes de eliminar recursos (periodo de gracia)
- **DynamoDB** (opcional): Tabla para rastrear recursos marcados para eliminacion y su periodo de gracia
- **CloudWatch Logs**: Registro de todas las acciones realizadas y recursos identificados

El flujo es: EventBridge dispara Lambda -> Lambda escanea recursos -> identifica ociosos -> notifica via SNS -> espera periodo de gracia -> elimina si no se ha revertido la decision.

## Pasos sugeridos

1. Disenar la arquitectura definiendo que tipos de recursos se escanearan y los criterios de "ocioso" para cada uno
2. Crear la infraestructura base con Terraform: funciones Lambda, EventBridge rules, SNS topic, roles IAM
3. Implementar la funcion Lambda para detectar volumenes EBS no adjuntos (estado "available")
4. Agregar deteccion de Elastic IPs no asociadas y snapshots con mas de X dias de antiguedad
5. Implementar modo dry-run que solo reporte sin eliminar, controlado por variable de entorno
6. Agregar notificacion SNS con detalle de recursos encontrados y accion planificada
7. Implementar la logica de eliminacion con periodo de gracia (ej: marcar hoy, eliminar en 7 dias)
8. Extender el escaneo a instancias RDS detenidas por mas de 7 dias y NAT Gateways sin rutas
9. Agregar tags de exclusion (ej: `cleanup:exclude`) para proteger recursos que deben mantenerse
10. Probar el sistema completo en modo dry-run y verificar las notificaciones antes de activar eliminacion

## Validacion

- [ ] Las funciones Lambda se ejecutan segun el schedule de EventBridge
- [ ] Se detectan correctamente volumenes EBS no adjuntos
- [ ] Se detectan Elastic IPs no asociadas y snapshots antiguos
- [ ] El modo dry-run funciona correctamente sin eliminar recursos
- [ ] Se envian notificaciones SNS con el detalle de recursos ociosos encontrados
- [ ] El periodo de gracia funciona antes de la eliminacion
- [ ] Los tags de exclusion protegen recursos marcados
- [ ] Los logs en CloudWatch registran todas las acciones
