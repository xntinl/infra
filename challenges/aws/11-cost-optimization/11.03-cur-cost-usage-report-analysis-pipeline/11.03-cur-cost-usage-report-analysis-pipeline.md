# 11.03 CUR (Cost and Usage Report) Analysis Pipeline

## Metadata
| Campo | Valor |
|-------|-------|
| **Complejidad** | Avanzado |
| **Modalidad** | Terraform for setup, SQL for analysis |
| **Servicios AWS** | CUR, S3, Glue, Athena, QuickSight |

## Descripcion

Configurar el Cost and Usage Report (CUR) con entrega a S3 en formato Parquet. Crear un Glue Crawler y una tabla de Athena para consultar los datos del CUR. Construir queries para identificar: servicios con mayor gasto, recursos sin etiquetas, recursos ociosos, costos de data transfer y oportunidades de ahorro.

## Objetivos de aprendizaje

- Implementar visibilidad completa de costos mediante el CUR
- Comprender el esquema del Cost and Usage Report y sus columnas clave
- Disenar e implementar cost allocation tags para atribucion de costos
- Realizar analisis de FinOps con SQL sobre datos reales de facturacion
- Identificar costos de data transfer y su impacto en la factura

## Requisitos previos

- **Conocimientos**: SQL basico-intermedio, conceptos de data lake, modelo de costos de AWS
- **Herramientas**: AWS CLI, Terraform, cliente SQL (Athena console o herramienta local)
- **Servicios AWS**: Permisos para configurar CUR, crear buckets S3, Glue Crawlers y bases de datos Athena

## Arquitectura

El pipeline de analisis de costos consta de:

- **CUR**: Genera reportes detallados de uso y costo, entregados periodicamente a S3 en formato Parquet
- **S3**: Almacena los reportes CUR comprimidos y particionados por fecha
- **Glue Crawler**: Descubre automaticamente el esquema de los archivos Parquet y lo registra en el Glue Data Catalog
- **Athena**: Motor de consultas serverless que permite ejecutar SQL directamente sobre los datos en S3
- **QuickSight** (opcional): Visualizacion de dashboards interactivos conectados a Athena

## Pasos sugeridos

1. Crear un bucket S3 con politica de ciclo de vida para almacenar los reportes CUR
2. Configurar el Cost and Usage Report via Terraform o consola con formato Parquet, compresion, y granularidad horaria
3. Esperar la primera entrega del CUR (puede tardar hasta 24 horas)
4. Crear un Glue Crawler apuntando al prefijo S3 del CUR y ejecutarlo
5. Verificar que la tabla se creo correctamente en el Glue Data Catalog
6. Escribir queries en Athena para identificar los 10 servicios con mayor gasto
7. Crear queries para encontrar recursos sin cost allocation tags
8. Desarrollar queries para detectar recursos potencialmente ociosos (bajo uso de CPU, volumenes sin I/O)
9. Analizar los costos de data transfer desglosados por tipo (inter-AZ, inter-region, internet egress)
10. Documentar las oportunidades de ahorro identificadas con estimaciones de impacto

## Validacion

- [ ] El CUR esta configurado y entregando reportes en formato Parquet a S3
- [ ] El Glue Crawler ejecuto correctamente y creo la tabla en el Data Catalog
- [ ] Se pueden ejecutar queries en Athena sobre los datos del CUR
- [ ] Se identificaron los servicios con mayor gasto
- [ ] Se listaron recursos sin cost allocation tags
- [ ] Se detectaron recursos potencialmente ociosos
- [ ] Se analizaron y desglosaron los costos de data transfer
- [ ] Se documentaron al menos 3 oportunidades de ahorro concretas
