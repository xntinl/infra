# 11.02 Right-Sizing con Compute Optimizer

## Metadata
| Campo | Valor |
|-------|-------|
| **Complejidad** | Intermedio |
| **Modalidad** | Manual + Terraform for changes |
| **Servicios AWS** | Compute Optimizer, EC2, EBS, Lambda, CloudWatch |

## Descripcion

Habilitar AWS Compute Optimizer en todas las cuentas. Analizar las recomendaciones para EC2, EBS, Lambda y Auto Scaling Groups. Implementar cambios de right-sizing usando un enfoque por fases. Medir el impacto en costos antes y despues de los cambios.

## Objetivos de aprendizaje

- Implementar right-sizing de recursos de forma sistematica
- Evaluar trade-offs entre rendimiento y costo
- Interpretar las recomendaciones de AWS Compute Optimizer
- Aplicar un enfoque de optimizacion por fases para minimizar riesgo

## Requisitos previos

- **Conocimientos**: Tipos de instancias EC2, volumenes EBS, configuracion de Lambda, metricas de CloudWatch
- **Herramientas**: AWS CLI, Terraform, acceso a la consola de Compute Optimizer
- **Servicios AWS**: Cuenta con recursos EC2, EBS y Lambda activos durante al menos 14 dias (idealmente 30)

## Arquitectura

El flujo de trabajo se compone de:

- **Compute Optimizer**: Servicio centralizado que analiza metricas de CloudWatch para generar recomendaciones de right-sizing
- **CloudWatch**: Fuente de metricas historicas de CPU, memoria, red y disco
- **EC2, EBS, Lambda**: Recursos objetivo para la optimizacion
- **Terraform**: Herramienta para aplicar los cambios de tipo de instancia, tamano de volumen y configuracion de memoria de forma controlada y reproducible

## Pasos sugeridos

1. Habilitar AWS Compute Optimizer a nivel de cuenta u organizacion desde la consola o CLI
2. Esperar a que Compute Optimizer genere recomendaciones (requiere al menos 14 dias de metricas)
3. Exportar las recomendaciones para EC2, EBS y Lambda desde la consola o via API
4. Clasificar las recomendaciones por impacto estimado de ahorro (mayor a menor)
5. Seleccionar un grupo de recursos no criticos para la primera fase de right-sizing
6. Registrar las metricas de rendimiento actuales (CPU, memoria, latencia) como baseline
7. Aplicar los cambios via Terraform (modificar instance_type, volume_size, memory_size segun corresponda)
8. Monitorear las metricas de rendimiento durante 48-72 horas post-cambio
9. Comparar costos antes y despues usando Cost Explorer
10. Proceder con la segunda fase aplicando cambios a recursos de mayor criticidad

## Validacion

- [ ] Compute Optimizer esta habilitado y generando recomendaciones
- [ ] Se exportaron y clasificaron las recomendaciones por ahorro estimado
- [ ] Se registro un baseline de rendimiento antes de los cambios
- [ ] Se aplicaron cambios de right-sizing en al menos 3 recursos via Terraform
- [ ] Se monitorearon metricas post-cambio sin degradacion significativa
- [ ] Se documento el ahorro medido en Cost Explorer
- [ ] Se definio un plan para las fases siguientes de optimizacion
