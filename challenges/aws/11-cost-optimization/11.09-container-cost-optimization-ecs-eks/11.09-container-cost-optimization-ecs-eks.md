# 11.09 Container Cost Optimization (ECS & EKS)

## Metadata
| Campo | Valor |
|-------|-------|
| **Complejidad** | Avanzado |
| **Modalidad** | Ambos |
| **Servicios AWS** | ECS, EKS, Fargate, Container Insights, Karpenter, Cost Explorer |

## Descripcion

Implementar optimizacion de costos de contenedores: right-size de task definitions de ECS usando datos de Container Insights, implementar consolidacion con Karpenter en EKS, comparar costos de Fargate vs EC2 launch type, y usar Kubecost o Split Cost Allocation para atribucion de costos por servicio.

## Objetivos de aprendizaje

- Comprender los modelos de costos de contenedores en AWS (ECS vs EKS, Fargate vs EC2)
- Realizar right-sizing de contenedores basado en datos reales de utilizacion
- Implementar atribucion de costos (cost allocation) a nivel de servicio/equipo
- Comparar y seleccionar el launch type mas eficiente segun la carga de trabajo

## Requisitos previos

- **Conocimientos**: ECS task definitions, EKS/Kubernetes, Fargate, metricas de contenedores, Karpenter basics
- **Herramientas**: AWS CLI, Terraform, kubectl, Helm
- **Servicios AWS**: Cluster ECS y/o EKS con workloads activos, Container Insights habilitado

## Arquitectura

La optimizacion de costos de contenedores abarca multiples componentes:

- **ECS con Container Insights**: Metricas detalladas de CPU y memoria por task/service para identificar sobredimensionamiento
- **EKS con Karpenter**: Node provisioner que consolida pods en nodos del tamano optimo, eliminando capacidad ociosa
- **Fargate vs EC2**: Comparacion side-by-side del mismo workload en ambos launch types
- **Kubecost / Split Cost Allocation**: Herramientas para atribuir costos de infraestructura compartida a servicios individuales
- **Cost Explorer**: Vision agregada de costos de contenedores con filtros por servicio y tag

## Pasos sugeridos

1. Habilitar Container Insights en los clusters ECS y/o EKS existentes
2. Recolectar metricas de utilizacion de CPU y memoria durante al menos 1 semana
3. Identificar tasks/pods sobredimensionados comparando limites configurados vs uso real
4. Ajustar las task definitions de ECS con valores de CPU y memoria alineados al uso real (con margen del 20-30%)
5. En EKS, instalar y configurar Karpenter para reemplazar Cluster Autoscaler
6. Configurar Karpenter Provisioners/NodePools con consolidation habilitado para eliminar nodos infrautilizados
7. Desplegar el mismo workload en Fargate y EC2 launch type, midiendo costos durante 1 semana
8. Analizar el break-even point: Fargate es mas simple pero EC2 puede ser mas barato a escala
9. Instalar Kubecost en EKS o habilitar Split Cost Allocation en ECS para atribucion de costos
10. Documentar las optimizaciones aplicadas, el ahorro obtenido y las recomendaciones por tipo de carga

## Validacion

- [ ] Container Insights esta habilitado y recolectando metricas de utilizacion
- [ ] Se identificaron al menos 3 tasks/pods sobredimensionados con datos concretos
- [ ] Se ajustaron las task definitions con valores optimizados
- [ ] Karpenter esta instalado y consolidando nodos activamente (si aplica EKS)
- [ ] Se comparo el costo de Fargate vs EC2 para un workload equivalente
- [ ] Se implemento atribucion de costos por servicio con Kubecost o Split Cost Allocation
- [ ] Se documento el ahorro total y las recomendaciones
