# 11.08 Lambda Cost Optimization & Power Tuning

## Metadata
| Campo | Valor |
|-------|-------|
| **Complejidad** | Intermedio |
| **Modalidad** | Ambos |
| **Servicios AWS** | Lambda, Step Functions (Power Tuning), CloudWatch, Cost Explorer |

## Descripcion

Usar la herramienta AWS Lambda Power Tuning (basada en Step Functions) para encontrar la configuracion optima de memoria/costo para funciones Lambda. Comparar precios de ARM vs x86. Implementar un analisis de costos de provisioned concurrency. Construir un dashboard que muestre tendencias de costo por funcion.

## Objetivos de aprendizaje

- Comprender el modelo de precios de Lambda (duracion x memoria)
- Aplicar la metodologia de power tuning para optimizar funciones Lambda
- Evaluar las ventajas de costo de la arquitectura ARM (Graviton2)
- Analizar la economia de provisioned concurrency vs on-demand

## Requisitos previos

- **Conocimientos**: AWS Lambda, Step Functions, CloudWatch metrics, modelo de precios de Lambda
- **Herramientas**: AWS CLI, Terraform, SAR (Serverless Application Repository) o despliegue manual del power tuning tool
- **Servicios AWS**: Funciones Lambda existentes con trafico real o simulado, permisos para crear Step Functions

## Arquitectura

La solucion de optimizacion de Lambda se compone de:

- **Lambda Power Tuning Tool**: State machine de Step Functions que ejecuta una funcion Lambda con diferentes configuraciones de memoria y mide duracion y costo
- **Lambda Functions**: Las funciones objetivo a optimizar
- **CloudWatch**: Metricas de duracion, invocaciones, errores y concurrent executions
- **Cost Explorer**: Datos historicos de costo de Lambda por funcion
- **CloudWatch Dashboard**: Visualizacion consolidada de tendencias de costo y rendimiento

## Pasos sugeridos

1. Desplegar la herramienta AWS Lambda Power Tuning desde el Serverless Application Repository o via Terraform
2. Seleccionar 2-3 funciones Lambda representativas para optimizar (una CPU-bound, una I/O-bound)
3. Ejecutar el power tuning para cada funcion con un rango amplio de memoria (128 MB a 3008 MB)
4. Analizar los resultados: identificar el "sweet spot" donde el costo por invocacion es minimo
5. Comparar el rendimiento y costo entre arquitecturas x86 y ARM (Graviton2) para las mismas funciones
6. Para funciones con trafico constante, analizar si provisioned concurrency reduce costos vs cold starts
7. Calcular el costo de provisioned concurrency vs el costo de cold starts en funcion del patron de trafico
8. Aplicar las configuraciones optimas de memoria a las funciones en produccion via Terraform
9. Crear un CloudWatch Dashboard que muestre costo estimado, duracion y invocaciones por funcion
10. Documentar los ahorros obtenidos y las recomendaciones para futuras funciones

## Validacion

- [ ] La herramienta Lambda Power Tuning esta desplegada y funcional
- [ ] Se ejecuto el power tuning en al menos 2 funciones con diferentes perfiles de carga
- [ ] Se identifico la configuracion optima de memoria para cada funcion
- [ ] Se comparo el costo de ARM vs x86 para al menos 1 funcion
- [ ] Se analizo el costo de provisioned concurrency vs on-demand
- [ ] Se aplicaron las configuraciones optimas via Terraform
- [ ] Se creo un dashboard de CloudWatch con metricas de costo por funcion
- [ ] Se documento el ahorro total obtenido
