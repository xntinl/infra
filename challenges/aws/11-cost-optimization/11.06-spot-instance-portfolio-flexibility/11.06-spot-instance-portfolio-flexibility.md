# 11.06 Spot Instance Portfolio & Flexibility

## Metadata
| Campo | Valor |
|-------|-------|
| **Complejidad** | Avanzado |
| **Modalidad** | Ambos |
| **Servicios AWS** | EC2 Spot, Auto Scaling, SQS (job queue), CloudWatch, SNS |

## Descripcion

Disenar una estrategia de instancias Spot para procesamiento batch: crear una Spot fleet diversificada con mas de 10 tipos de instancia y 3 AZs, configurar asignacion capacity-optimized, implementar manejo graceful de interrupciones con el aviso de 2 minutos, y comparar costos con On-Demand.

## Objetivos de aprendizaje

- Comprender la dinamica del mercado Spot y su modelo de precios
- Disenar estrategias de diversificacion de instancias para maximizar disponibilidad
- Implementar manejo de interrupciones Spot de forma graceful
- Medir y cuantificar el ahorro real comparado con On-Demand

## Requisitos previos

- **Conocimientos**: EC2 instance types, Auto Scaling Groups, SQS, procesamiento batch
- **Herramientas**: AWS CLI, Terraform, lenguaje de scripting para manejo de interrupciones
- **Servicios AWS**: Permisos para crear Spot Fleet/EC2 Fleet, Auto Scaling Groups, SQS queues

## Arquitectura

La solucion de procesamiento batch con Spot se compone de:

- **SQS Queue**: Cola de trabajos pendientes que desacopla la ingesta del procesamiento
- **EC2 Spot Fleet / Auto Scaling Group**: Flota diversificada de instancias Spot con 10+ tipos en 3 AZs
- **Interruption Handler**: Script/daemon que escucha la notificacion de 2 minutos y realiza checkpoint del trabajo
- **SNS**: Notificaciones de interrupciones y estado de procesamiento
- **CloudWatch**: Metricas de procesamiento, interrupciones y costos
- **S3**: Almacenamiento de resultados y checkpoints

El flujo es: trabajos entran en SQS -> Spot Fleet escala segun profundidad de cola -> workers procesan mensajes -> ante interrupcion, checkpoint a S3 y devolver mensaje a SQS -> otro worker retoma.

## Pasos sugeridos

1. Investigar el Spot Instance Advisor para identificar los tipos de instancia con menor tasa de interrupcion
2. Seleccionar al menos 10 tipos de instancia compatibles con la carga de trabajo (ej: m5.large, m5a.large, m5d.large, m4.large, c5.large, etc.)
3. Crear una SQS queue para la cola de trabajos batch
4. Configurar un Auto Scaling Group con mixed instances policy usando Terraform, diversificando en 3 AZs
5. Configurar la estrategia de asignacion `capacity-optimized` para minimizar interrupciones
6. Implementar un script de user data que inicie el worker y registre el handler de interrupciones
7. Desarrollar la logica de manejo de interrupciones: detectar la notificacion de 2 minutos, guardar checkpoint, devolver mensaje a SQS
8. Configurar CloudWatch alarms para monitorear interrupciones y profundidad de cola
9. Ejecutar la carga de trabajo y recolectar metricas de costo, interrupciones y tiempo de procesamiento
10. Comparar el costo total con una ejecucion equivalente en On-Demand y documentar el ahorro

## Validacion

- [ ] La Spot Fleet incluye al menos 10 tipos de instancia en 3 AZs
- [ ] La estrategia de asignacion es capacity-optimized
- [ ] Los workers procesan trabajos de la cola SQS correctamente
- [ ] El handler de interrupciones detecta la notificacion de 2 minutos
- [ ] Se realiza checkpoint del trabajo antes de la terminacion
- [ ] Los trabajos interrumpidos se reintentaron automaticamente por otro worker
- [ ] Se recopilaron metricas de costo y se documento el ahorro vs On-Demand
- [ ] Las alarmas de CloudWatch se configuraron para interrupciones y estado de la cola
