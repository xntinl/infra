# 7.09 Application Signals y Service Level Objectives

## Metadata
| Campo | Valor |
|-------|-------|
| **Complejidad** | Avanzado |
| **Modalidad** | Ambos |
| **Servicios AWS** | CloudWatch Application Signals, X-Ray, CloudWatch, SLOs |

## Descripcion

Habilitar CloudWatch Application Signals en una aplicacion de microservicios. Definir SLOs (99.9% de disponibilidad, latencia p99 < 200ms) y configurar alertas de burn rate. Usar el service map auto-generado y los dashboards a nivel de operacion para identificar cuellos de botella de rendimiento.

## Objetivos de aprendizaje

- Definir Service Level Objectives (SLOs) basados en metricas reales de la aplicacion
- Configurar alertas de burn rate para detectar degradaciones antes de violar los SLOs
- Utilizar auto-instrumentacion para obtener observabilidad sin cambios en el codigo
- Monitorear aplicaciones a nivel de servicio y operacion para identificar problemas de rendimiento

## Requisitos previos

- Conocimientos: Conceptos de SLI/SLO/SLA, microservicios, CloudWatch, X-Ray, metricas de latencia y disponibilidad
- Herramientas: AWS CLI, Terraform (opcional), Docker, lenguaje de programacion (Java/Python/Node.js)
- Servicios AWS: Cuenta AWS con permisos para CloudWatch, X-Ray, ECS o EKS, IAM

## Arquitectura

Una aplicacion de microservicios esta instrumentada automaticamente con CloudWatch Application Signals. El sistema recopila metricas de latencia, disponibilidad y error rate por servicio y operacion sin necesidad de instrumentacion manual. Los SLOs se definen sobre estas metricas (ejemplo: 99.9% de llamadas exitosas, p99 latencia < 200ms). Las alertas de burn rate calculan la velocidad a la que se consume el error budget y notifican antes de que se agote. El service map auto-generado muestra las dependencias y el estado de cada servicio.

## Pasos sugeridos

1. Desplegar una aplicacion de microservicios (3-4 servicios) en ECS o EKS
2. Habilitar CloudWatch Application Signals en el entorno de ejecucion
3. Configurar la auto-instrumentacion (ADOT con Application Signals habilitado)
4. Verificar que las metricas de latencia, disponibilidad y error rate aparecen en Application Signals
5. Explorar el service map auto-generado y los dashboards por operacion
6. Definir un SLO de disponibilidad: 99.9% de llamadas exitosas en una ventana de 30 dias
7. Definir un SLO de latencia: p99 < 200ms en una ventana de 30 dias
8. Configurar alertas de burn rate para ambos SLOs
9. Generar carga con errores controlados para observar el consumo del error budget
10. Usar los dashboards de operacion para identificar el servicio o endpoint que mas contribuye a la degradacion

## Validacion

- [ ] Application Signals esta habilitado y recopilando metricas automaticamente
- [ ] El service map muestra todos los servicios y sus dependencias
- [ ] Los dashboards por operacion muestran latencia, error rate y throughput
- [ ] Los SLOs estan definidos correctamente con ventanas de tiempo apropiadas
- [ ] Las alertas de burn rate se disparan cuando el error budget se consume rapidamente
- [ ] Se puede identificar el servicio/operacion que causa degradacion desde el dashboard
- [ ] El error budget se calcula y visualiza correctamente
