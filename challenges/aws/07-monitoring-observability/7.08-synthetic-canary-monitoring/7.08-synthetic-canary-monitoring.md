## Why This Matters

Traditional monitoring is reactive -- you find out something is broken when a user reports it or when an error metric spikes. Synthetics canaries flip this model: they continuously simulate user interactions against your endpoints, detecting failures before real users are affected. A canary that runs every 5 minutes gives you a maximum detection window of 5 minutes, regardless of traffic volume. This is critical for APIs that receive low traffic outside business hours -- a broken endpoint at 3 AM might not trigger error-based alarms until the morning rush, but a canary catches it immediately.

Canaries store detailed artifacts in S3: HAR (HTTP Archive) files showing every request and response, screenshots for visual canaries, and execution logs. When a canary fails, you have a complete forensic trail without needing to reproduce the issue.

> **Best Practice:** Use API canaries for backend health checks and heartbeat canaries for simple uptime monitoring. API canaries can execute multi-step request sequences with assertions on response bodies, headers, and status codes. Heartbeat canaries simply check that a URL returns a 2xx response.

```hcl
#   - Create an aws_iam_role_policy attaching the permissions
#   - Tag the role with Name

# Hint: The canary needs s3:PutObject to store artifacts. Missing this

# =======================================================
# TODO 3 -- Synthetics canary
# =======================================================
# Requirements:
#   - Create an aws_synthetics_canary named
#     "${var.project_name}-api" (note: canary names must be
#     <= 21 characters, so use a shorter name like "api-health-check")
#   - artifact_s3_location = "s3://${bucket_name}/"
#   - execution_role_arn = the IAM role ARN from TODO 2
#   - handler = "index.handler"
#   - runtime_version = "syn-nodejs-puppeteer-9.1"
#   - zip_file = a path to a zip containing the canary script
#     (use data.archive_file below)
#   - start_canary = true
#   - schedule: expression = "rate(5 minutes)"
#   - run_config: timeout_in_seconds = 60
#   - Tag with Name
#
# Docs: https://registry.terraform.io/providers/hashicorp/aws/latest/docs/resources/synthetics_canary
# Hint: The canary name has a 21-character limit. Keep it short.

data "archive_file" "canary" {
  type        = "zip"
  output_path = "${path.module}/canary.zip"

  source {
    content = <<-NODEJS
    const { URL } = require('url');
    const synthetics = require('Synthetics');
    const log = require('SyntheticsLogger');

    const apiCanaryBlueprint = async function () {
      const url = 'https://jsonplaceholder.typicode.com/posts/1';
      const requestOptions = {
        hostname: new URL(url).hostname,
        method: 'GET',
        path: new URL(url).pathname,
        port: 443,
        protocol: 'https:',
        headers: { 'User-Agent': 'CloudWatch-Synthetics' }
      };

      log.info('Making request to: ' + url);

      const stepConfig = {
        includeRequestHeaders: true,
        includeResponseHeaders: true,
        includeRequestBody: true,
        includeResponseBody: true,
        continueOnHttpStepFailure: false
      };

      await synthetics.executeHttpStep(
        'verifyApiEndpoint',
        requestOptions,
        function (res) {
          return new Promise((resolve, reject) => {
            if (res.statusCode < 200 || res.statusCode > 299) {
              throw new Error(res.statusCode + ' ' + res.statusMessage);
            }

            let responseBody = '';
            res.on('data', (d) => { responseBody += d; });
            res.on('end', () => {
              const parsed = JSON.parse(responseBody);
              if (!parsed.id || !parsed.title) {
                throw new Error('Response missing required fields');
              }
              log.info('Response validated successfully');
              resolve();
            });
          });
        },
        stepConfig
      );
    };

    exports.handler = async () => {
      return await apiCanaryBlueprint();
    };
    NODEJS
    filename = "nodejs/node_modules/index.js"
  }
}
```

## Best Practices

1. **Run canaries on a schedule, not on demand** -- A canary running every 5 minutes gives you at most a 5-minute detection window for outages. Choose a frequency that balances cost ($0.0012/run) against acceptable detection latency.

2. **Store artifacts in a dedicated S3 bucket** -- Screenshots, HAR files, and execution logs are invaluable for debugging failures. Use lifecycle rules to expire old artifacts after 30-90 days to control storage costs.

3. **Use API canaries for backend services** -- API canaries can validate response status codes, headers, and body content. Heartbeat canaries only check reachability. For APIs that return structured data, always assert on the response body.

4. **Alarm on SuccessPercent, not individual failures** -- A single timeout does not necessarily mean your API is down. Alarm when the success rate drops below 100% over a 5-minute window to filter out transient network blips.

5. **Canary IAM roles need explicit S3 write access** -- The most common canary deployment failure is a missing `s3:PutObject` permission. The canary runs but cannot store artifacts, making failures impossible to debug.

## Spot the Bug

A colleague deployed this canary IAM role, but the canary runs successfully while no artifacts appear in S3. The canary status shows "Passed" in the console. **What is wrong?**

```hcl
data "aws_iam_policy_document" "canary_permissions" {
  statement {
    actions = [
      "s3:GetObject",
      "s3:ListBucket",
      "s3:GetBucketLocation",
      "logs:CreateLogGroup",
      "logs:CreateLogStream",
      "logs:PutLogEvents",
      "cloudwatch:PutMetricData"
    ]
    resources = ["*"]
  }
  # s3:PutObject is missing  # <-- BUG
}
```

<details>
<summary>Explain the bug</summary>

The IAM policy is missing `s3:PutObject`. The canary executes its HTTP checks and reports success metrics to CloudWatch, but it cannot write artifacts (HAR files, screenshots, logs) to the S3 bucket.

This bug is deceptive because the canary appears to work -- the CloudWatch Synthetics console shows "Passed" and the SuccessPercent metric is 100%. You only discover the problem when you try to debug a future failure and find the artifacts bucket is empty.

**Wrong -- missing PutObject:**

```hcl
actions = [
  "s3:GetObject",
  "s3:ListBucket",
  ...
]
```

**Fix -- add PutObject:**

```hcl
actions = [
  "s3:PutObject",
  "s3:GetObject",
  "s3:ListBucket",
  ...
]
```

</details>

<summary>TODO 2 -- IAM role for the canary</summary>

```hcl
data "aws_iam_policy_document" "canary_assume" {
  statement {
    actions = ["sts:AssumeRole"]
    principals {
      type        = "Service"
      identifiers = ["lambda.amazonaws.com"]
    }
  }
}

data "aws_iam_policy_document" "canary_permissions" {
  statement {
    actions = [
      "s3:PutObject",
      "s3:GetObject",
      "s3:ListBucket",
      "s3:GetBucketLocation"
    ]
    resources = [
      aws_s3_bucket.this.arn,
      "${aws_s3_bucket.this.arn}/*"
    ]
  }

  statement {
    actions = [
      "logs:CreateLogGroup",
      "logs:CreateLogStream",
      "logs:PutLogEvents"
    ]
    resources = ["arn:aws:logs:*:*:*"]
  }

  statement {
    actions   = ["cloudwatch:PutMetricData"]
    resources = ["*"]
    condition {
      test     = "StringEquals"
      variable = "cloudwatch:namespace"
      values   = ["CloudWatchSynthetics"]
    }
  }
}

resource "aws_iam_role" "canary" {
  name               = "${var.project_name}-canary-role"

  tags = { Name = "${var.project_name}-canary-role" }
}

  name   = "${var.project_name}-canary-policy"
```

## Summary

- **CloudWatch Synthetics canaries** simulate user interactions on a schedule, catching failures before real users are affected
- **API canaries** validate response codes, headers, and bodies; **heartbeat canaries** check simple uptime
- **Canary artifacts** (HAR files, screenshots, logs) stored in S3 provide a forensic trail for debugging failures
- **SuccessPercent metric** in the CloudWatchSynthetics namespace is the primary metric for alarming on canary health
- **IAM roles need explicit `s3:PutObject`** permission -- without it, canaries run but produce no debuggable artifacts
