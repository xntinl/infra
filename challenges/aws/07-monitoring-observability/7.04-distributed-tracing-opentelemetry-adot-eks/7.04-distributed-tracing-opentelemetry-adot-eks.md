# 7.04 Distributed Tracing con OpenTelemetry y ADOT en EKS

## Metadata
| Campo | Valor |
|-------|-------|
| **Complejidad** | Avanzado |
| **Modalidad** | Terraform + Kubernetes manifests |
| **Servicios AWS** | ADOT, X-Ray, CloudWatch, EKS, Prometheus |

## Descripcion

Desplegar el AWS Distro for OpenTelemetry (ADOT) collector en EKS como DaemonSet. Instrumentar una aplicacion multi-servicio con los SDKs de OpenTelemetry. Enviar trazas a X-Ray y metricas a CloudWatch. Configurar sampling rules y crear service maps.

## Objetivos de aprendizaje

- Instrumentar aplicaciones con los SDKs de OpenTelemetry para generar trazas distribuidas
- Configurar el ADOT Collector (pipelines, receivers, processors, exporters)
- Comprender la propagacion de contexto de trazas entre servicios
- Disenar estrategias de sampling para controlar volumen y costo
- Interpretar service maps para identificar cuellos de botella y dependencias

## Requisitos previos

- Conocimientos: Kubernetes (pods, DaemonSets, services), EKS, conceptos de tracing distribuido, YAML
- Herramientas: AWS CLI, kubectl, eksctl o Terraform, Helm, Docker
- Servicios AWS: Cuenta AWS con permisos para EKS, X-Ray, CloudWatch, IAM (IRSA)

## Arquitectura

Un cluster EKS ejecuta una aplicacion de microservicios (3-4 servicios). El ADOT Collector se despliega como DaemonSet en cada nodo, recibiendo trazas via OTLP de los servicios instrumentados. El collector procesa y exporta trazas a X-Ray y metricas a CloudWatch/Prometheus. Las sampling rules en X-Ray controlan el volumen de trazas recopiladas. El service map de X-Ray visualiza las dependencias y latencias entre servicios.

## Pasos sugeridos

1. Crear un cluster EKS con Terraform o eksctl (al menos 2 nodos)
2. Desplegar el ADOT Collector como DaemonSet usando el Helm chart oficial
3. Configurar el collector con pipelines para trazas (OTLP receiver -> X-Ray exporter) y metricas (OTLP receiver -> CloudWatch exporter)
4. Instrumentar los servicios con el SDK de OpenTelemetry (auto-instrumentacion o manual)
5. Configurar la propagacion de contexto (W3C Trace Context) entre servicios
6. Desplegar los servicios instrumentados en EKS con las variables de entorno de OTEL
7. Configurar sampling rules en X-Ray (reservoir + fixed rate) para controlar costos
8. Generar trafico de prueba y verificar que las trazas aparecen en X-Ray
9. Analizar el service map para identificar dependencias y latencias
10. Crear dashboards con metricas derivadas de las trazas

## Validacion

- [ ] El ADOT Collector esta desplegado como DaemonSet y corriendo en todos los nodos
- [ ] Los servicios generan trazas que se envian via OTLP al collector
- [ ] Las trazas aparecen correctamente en la consola de X-Ray
- [ ] La propagacion de contexto funciona entre todos los servicios (trace_id consistente)
- [ ] Las sampling rules se aplican correctamente
- [ ] El service map muestra las dependencias entre servicios
- [ ] Las metricas se exportan a CloudWatch correctamente
- [ ] Se pueden identificar cuellos de botella usando las trazas y el service map
