# 7.10 Cross-Account Centralized Observability

## Metadata
| Campo | Valor |
|-------|-------|
| **Complejidad** | Experto |
| **Modalidad** | Terraform recomendado |
| **Servicios AWS** | CloudWatch (cross-account), EventBridge, SNS, Grafana, Organizations |

## Descripcion

Configurar una cuenta de monitoreo centralizada que agregue metricas, logs y alarmas de CloudWatch de multiples cuentas de cargas de trabajo. Configurar acceso cross-account a CloudWatch, crear un dashboard central en Grafana e implementar un pipeline centralizado de enrutamiento de alarmas.

## Objetivos de aprendizaje

- Disenar una arquitectura de observabilidad multi-cuenta para entornos empresariales
- Configurar el acceso cross-account de CloudWatch para compartir metricas y logs
- Implementar alertas centralizadas que agreguen alarmas de multiples cuentas
- Obtener visibilidad operacional a escala en organizaciones con multiples cuentas AWS

## Requisitos previos

- Conocimientos: AWS Organizations, IAM roles cross-account, CloudWatch, EventBridge, arquitectura multi-cuenta
- Herramientas: AWS CLI, Terraform, multiples cuentas AWS (al menos 3: monitoring, workload-1, workload-2)
- Servicios AWS: Cuenta AWS con Organizations habilitado, permisos para CloudWatch, EventBridge, SNS, IAM, Grafana

## Arquitectura

La arquitectura consta de tres capas. Las cuentas de carga de trabajo generan metricas, logs y alarmas locales. Cada cuenta de workload comparte sus datos de CloudWatch con la cuenta de monitoreo central usando CloudWatch cross-account sharing. La cuenta de monitoreo central agrega todas las senales en dashboards unificados en Grafana, y un pipeline de EventBridge recibe eventos de alarma de todas las cuentas, los enriquece con contexto (nombre de cuenta, entorno) y los enruta a los destinos apropiados via SNS (email, Slack, PagerDuty).

## Pasos sugeridos

1. Configurar AWS Organizations con al menos 3 cuentas (monitoring, workload-1, workload-2)
2. Habilitar CloudWatch cross-account observability en la cuenta de monitoreo (monitoring account)
3. Configurar las cuentas de workload como source accounts para compartir metricas, logs y trazas
4. Verificar que las metricas y logs de las cuentas de workload son visibles desde la cuenta de monitoreo
5. Crear dashboards cross-account en CloudWatch que muestren metricas de todas las cuentas
6. Configurar EventBridge en las cuentas de workload para enviar eventos de alarma a la cuenta de monitoreo
7. Crear un pipeline de procesamiento de alarmas (EventBridge rule -> Lambda -> SNS) en la cuenta de monitoreo
8. Enriquecer las notificaciones con contexto (account_id, account_name, environment)
9. Desplegar Grafana en la cuenta de monitoreo con data sources cross-account
10. Crear un dashboard operacional unificado que muestre el estado de salud de todos los servicios en todas las cuentas

## Validacion

- [ ] CloudWatch cross-account observability esta configurado entre todas las cuentas
- [ ] Las metricas de las cuentas de workload son visibles desde la cuenta de monitoreo
- [ ] Los logs de las cuentas de workload se pueden consultar desde la cuenta central
- [ ] Los dashboards cross-account muestran datos de todas las cuentas simultaneamente
- [ ] Los eventos de alarma llegan a la cuenta de monitoreo via EventBridge
- [ ] Las notificaciones incluyen contexto enriquecido (cuenta, entorno, servicio)
- [ ] El dashboard operacional en Grafana muestra una vista unificada de toda la organizacion
- [ ] El pipeline de enrutamiento de alarmas funciona correctamente con multiples destinos
