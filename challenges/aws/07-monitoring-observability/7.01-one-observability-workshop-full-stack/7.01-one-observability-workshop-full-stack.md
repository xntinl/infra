## Why This Matters

Running a Lambda behind API Gateway without observability is like driving at night with the headlights off. You will not know when latency spikes, when errors start creeping in, or when someone modifies your infrastructure without authorization. A full-stack observability setup gives you four complementary signals:

- **Metrics** (CloudWatch) tell you what is happening right now -- request counts, error rates, latency percentiles
- **Logs** (CloudWatch Logs) tell you the details -- individual request payloads, stack traces, debug information
- **Traces** (X-Ray) tell you where time is spent -- which downstream service is slow, where the bottleneck lives
- **Audit events** (CloudTrail) tell you who changed what -- API calls to your AWS account, who deployed, who modified IAM

Without all four, you are blind to at least one failure mode. Metrics alone cannot tell you why a function is slow. Logs alone cannot show you the call chain across services. Traces alone cannot tell you that someone deleted an IAM policy at 3 AM. CloudTrail alone cannot help you debug a runtime error. The combination is what makes production workloads manageable.

> **Best Practice:** Always enable X-Ray active tracing on both API Gateway and Lambda. Enabling it on only one side gives you partial traces that end abruptly at the service boundary, making it impossible to correlate frontend latency with backend processing time.

## The Challenge

Build a complete observability stack for a Lambda function behind API Gateway. The stack must include:

| Component | Purpose |
|-----------|---------|
| Lambda function | Business logic -- returns a JSON response with simulated latency |
| API Gateway (REST) | HTTP frontend routing requests to Lambda |
| X-Ray tracing | End-to-end distributed tracing across API GW and Lambda |
| CloudWatch dashboard | Single-pane-of-glass with 6 widgets covering errors, latency, invocations |
| Metric alarms | Individual alarms for error rate, latency P99, and throttles |
| Composite alarm | Combines the three metric alarms into a single "service health" alarm |
| SNS topic | Notification target for the composite alarm |
| CloudTrail | Audit trail for all management events |
| CloudWatch Log Group | Lambda execution logs with 14-day retention |

### Requirements

1. **Lambda function** with X-Ray tracing enabled (`tracing_config { mode = "Active" }`), a CloudWatch Log Group with 14-day retention, and an IAM role granting `logs:*` and `xray:PutTraceSegments` / `xray:PutTelemetryRecords`
2. **API Gateway REST API** with a single `GET /health` route, X-Ray tracing enabled, and a stage named `v1` with access logging to CloudWatch
3. **CloudWatch dashboard** with 6 widgets arranged in a 24-column grid:
   - Row 1: Invocations (line), Errors (line), Throttles (line) -- each 8 columns wide
   - Row 2: Duration P50/P90/P99 (line), API Gateway 4xx/5xx (stacked bar), Concurrent Executions (line) -- each 8 columns wide
4. **Three metric alarms**: Lambda errors > 1 in 5 minutes, Lambda P99 duration > 3000ms, Lambda throttles > 0
5. **Composite alarm** combining all three with `OR` logic, sending to an SNS topic
6. **CloudTrail** trail logging management events to an S3 bucket with a lifecycle policy
7. All resources tagged with `Name = "${var.project_name}-..."`

## Best Practices

1. **Dashboard widget sizing** -- CloudWatch dashboards use a 24-column grid. Plan your layout on paper first. Widgets at `width: 8` give you three per row, which works well for Lambda's core metrics (invocations, errors, duration).

2. **Alarm thresholds** -- Use `datapoints_to_alarm` (M-of-N evaluation) to avoid false positives from transient spikes. A single data point above threshold is noise; 3 out of 5 is a trend.

3. **`treat_missing_data`** -- Set this to `"notBreaching"` for Lambda alarms. Lambda functions that are not invoked produce no metric data, which CloudWatch treats as `INSUFFICIENT_DATA` by default. This can trigger pages at 3 AM for an idle function.

4. **CloudTrail S3 bucket policy** -- CloudTrail requires a specific bucket policy before it can write logs. The principal is `cloudtrail.amazonaws.com` and it needs `s3:PutObject` with a condition on `s3:x-amz-acl = bucket-owner-full-control`.

5. **X-Ray sampling rules** -- In production, X-Ray's default sampling (1 req/sec + 5% thereafter) is usually sufficient. Do not set sampling to 100% unless you are debugging -- it increases cost significantly and can add latency.

<details>
<summary>Hint 4: Composite alarm with suppressor</summary>

Create three metric alarms first, then combine them with `aws_cloudwatch_composite_alarm`:

```hcl
resource "aws_cloudwatch_composite_alarm" "service_health" {
  alarm_name = "${var.project_name}-service-health"

  alarm_rule = join(" OR ", [
    "ALARM(${aws_cloudwatch_metric_alarm.lambda_errors.alarm_name})",
    "ALARM(${aws_cloudwatch_metric_alarm.lambda_p99.alarm_name})",
    "ALARM(${aws_cloudwatch_metric_alarm.lambda_throttles.alarm_name})"
  ])

  alarm_actions = [aws_sns_topic.alerts.arn]

  tags = { Name = "${var.project_name}-composite-alarm" }
}
```

The `alarm_rule` uses a simple expression language with `ALARM()`, `OK()`, `INSUFFICIENT_DATA()`, combined with `AND`, `OR`, `NOT`.

data "aws_caller_identity" "current" {}

data "aws_iam_policy_document" "cloudtrail_s3" {

    effect = "Allow"
    principals {
      type        = "Service"

    actions   = ["s3:GetBucketAcl"]
    resources = [aws_s3_bucket.cloudtrail.arn]

    sid    = "AWSCloudTrailWrite"

      type        = "Service"

    resources = ["${aws_s3_bucket.cloudtrail.arn}/AWSLogs/${data.aws_caller_identity.current.account_id}/*"]
    condition {
      test     = "StringEquals"
      variable = "s3:x-amz-acl"
      values   = ["bucket-owner-full-control"]
    }
  }
}

## Spot the Bug

A colleague wrote this CloudWatch dashboard but the Duration widget always shows a flat line at zero. The Invocations widget works fine. **What is wrong?**

```hcl
resource "aws_cloudwatch_dashboard" "this" {
  dashboard_name = "my-dashboard"

  dashboard_body = jsonencode({
    widgets = [
      {
        type   = "metric"
        x      = 0
        y      = 0
        width  = 12
        height = 6
        properties = {
          title   = "Invocations"
          region  = var.region
          metrics = [
            ["AWS/Lambda", "Invocations", "FunctionName", "my-function"]
          ]
          period = 300
          stat   = "Sum"
        }
      },
      {
        type   = "metric"
        x      = 12
        y      = 0
        width  = 12
        height = 6
        properties = {
          title   = "Duration P99"
          region  = var.region
          metrics = [
            ["AWS/Lambda", "Duration", "FunctionName", "my-function"]
          ]
          period = 300
          stat   = "Sum"                  # <-- BUG
        }
      }
    ]
  })
}
```

<details>
<summary>Explain the bug</summary>

The Duration widget uses `stat = "Sum"` instead of `stat = "p99"` (or `"Average"`, `"Maximum"`, etc.). The `Sum` statistic for Duration adds up all invocation durations in the period, which is meaningless for latency analysis and does not show "flat zero" -- it actually shows a large number that looks wrong.

More importantly, if the function has very few invocations, the Sum might be so small relative to the Y-axis auto-scale that it appears as a flat line.

The real issue is semantic: Duration should use a percentile or Average stat. For a P99 widget:

```hcl
stat = "p99"
```

For percentile stats in CloudWatch, use lowercase `p` followed by the percentile number: `p50`, `p90`, `p95`, `p99`.

</details>

# ---------------------------------------------------------------
# Lambda function code (inline Python)
# ---------------------------------------------------------------
data "archive_file" "lambda" {
  type        = "zip"
  output_path = "${path.module}/lambda.zip"

  source {
    content = <<-PYTHON
    import json
    import time
    import random
    import logging

    logger = logging.getLogger()
    logger.setLevel(logging.INFO)

    def handler(event, context):
        start = time.time()
        # Simulate variable latency
        delay = random.uniform(0.05, 0.5)
        time.sleep(delay)

        # Simulate occasional errors (10% rate)
        if random.random() < 0.1:
            logger.error("Simulated error occurred", extra={
                "request_id": context.aws_request_id,
                "delay_ms": int(delay * 1000)
            })
            return {
                "statusCode": 500,
                "body": json.dumps({"error": "Internal server error"})
            }

        duration = int((time.time() - start) * 1000)
        logger.info("Request processed successfully", extra={
            "request_id": context.aws_request_id,
            "duration_ms": duration,
            "delay_ms": int(delay * 1000)
        })
        return {
            "statusCode": 200,
            "body": json.dumps({
                "message": "healthy",
                "duration_ms": duration,
                "request_id": context.aws_request_id
            })
        }
    PYTHON
    filename = "index.py"
  }
}

# ---------------------------------------------------------------
# IAM role for Lambda
# ---------------------------------------------------------------
data "aws_iam_policy_document" "lambda_assume" {
  statement {
    actions = ["sts:AssumeRole"]
    principals {
      type        = "Service"
      identifiers = ["lambda.amazonaws.com"]
    }
  }
}

data "aws_iam_policy_document" "lambda_permissions" {
  statement {
    sid = "CloudWatchLogs"
    actions = [
      "logs:CreateLogGroup",
      "logs:CreateLogStream",
      "logs:PutLogEvents"
    ]
    resources = ["arn:aws:logs:*:*:*"]
  }

  statement {
    sid = "XRayWrite"
    actions = [
      "xray:PutTraceSegments",
      "xray:PutTelemetryRecords"
    ]
    resources = ["*"]
  }
}

resource "aws_iam_role" "lambda" {
  name               = "${var.project_name}-lambda-role"
  assume_role_policy = data.aws_iam_policy_document.lambda_assume.json

  tags = { Name = "${var.project_name}-lambda-role" }
}

  name   = "${var.project_name}-lambda-policy"

  policy = data.aws_iam_policy_document.lambda_permissions.json
}

# Lambda function

    mode = "Active"

## Summary

- **X-Ray active tracing** on both API Gateway and Lambda provides end-to-end request visibility with correlated trace IDs
- **CloudWatch dashboards** use a 24-column grid; plan widget layout with `x`, `y`, `width`, `height` before coding
- **Metric alarms** with `datapoints_to_alarm` (M-of-N) reduce false positives from transient spikes
- **`treat_missing_data = "notBreaching"`** prevents idle Lambda functions from triggering INSUFFICIENT_DATA alerts
- **Composite alarms** combine multiple signals into a single actionable alert, reducing notification fatigue
- **CloudTrail** provides an audit trail of every API call to your AWS account -- essential for security and compliance
- **All four pillars** (metrics, logs, traces, audit) are complementary; each answers different questions about system health
