## Why This Matters

In a production environment, a single Lambda function might have alarms for errors, latency, throttles, concurrent executions, and iterator age. If each alarm sends its own SNS notification, an on-call engineer receives five separate pages for what is really one incident. Composite alarms solve this by combining multiple metric alarms into a single parent alarm that fires only when a specific combination of child alarms triggers.

The suppressor pattern takes this further. During planned maintenance -- a deployment, a database migration, a load test -- you expect alarms to fire. A suppressor alarm is a special alarm that, when in ALARM state, silences the composite alarm's actions. You set the suppressor to ALARM before maintenance starts and back to OK when maintenance ends. The composite alarm still evaluates its rule (so you can see the state in the console), but it does not send notifications.

> **Best Practice:** Composite alarms should have a suppressor alarm for maintenance windows. Without it, every deployment that causes a brief error spike pages the on-call team, leading to alert fatigue and eventually ignored alerts.

## Best Practices

1. **Composite alarms reduce notification noise** -- Instead of receiving separate notifications for errors, latency, and throttles, combine them into a single "service health" alarm. The on-call engineer gets one page with a clear signal that something is wrong.

2. **Use M-of-N evaluation to avoid false positives** -- A single data point exceeding threshold is often noise. Setting `evaluation_periods = 5` and `datapoints_to_alarm = 3` means the alarm only fires when 3 out of 5 consecutive periods breach the threshold.

3. **`treat_missing_data = "notBreaching"`** -- Lambda functions that are idle produce no metric data. CloudWatch treats missing data as `INSUFFICIENT_DATA` by default, which can trigger unexpected state changes. Setting `notBreaching` means "no data = no problem."

4. **Suppressor alarm for maintenance** -- Use `actions_suppressor` on the composite alarm to silence notifications during planned maintenance. Push a custom metric value of `1` to activate it, and `0` to deactivate.

5. **Do not set `alarm_actions` on child alarms** -- When using composite alarms, only the composite alarm should have `alarm_actions`. If child alarms also send to SNS, you get duplicate notifications.

## Spot the Bug

A colleague wrote this composite alarm, but it fires during every deployment even though the suppressor alarm is configured. **What is wrong?**

```hcl
resource "aws_cloudwatch_composite_alarm" "service_health" {
  alarm_name = "my-service-health"

  alarm_rule = join(" OR ", [
    "ALARM(${aws_cloudwatch_metric_alarm.errors.alarm_name})",
    "ALARM(${aws_cloudwatch_metric_alarm.latency.alarm_name})"
  ])

  alarm_actions = [aws_sns_topic.alerts.arn]

  actions_suppressor                    = aws_cloudwatch_metric_alarm.maintenance.alarm_name  # <-- BUG
  actions_suppressor_wait_period        = 60
  actions_suppressor_extension_period   = 60

  tags = { Name = "my-service-health" }
}
```

<details>
<summary>Explain the bug</summary>

The `actions_suppressor` argument requires the **alarm ARN**, not the alarm name. Using the alarm name causes Terraform to create the resource, but CloudWatch cannot resolve the suppressor reference, so suppression never activates.

**Wrong:**

```hcl
actions_suppressor = aws_cloudwatch_metric_alarm.maintenance.alarm_name
```

**Fix -- use the ARN:**

```hcl
actions_suppressor = aws_cloudwatch_metric_alarm.maintenance.arn
```

This is a subtle bug because `terraform apply` succeeds without errors. The composite alarm works for evaluation, but the suppressor simply has no effect. You only discover the problem during the next deployment when the team gets paged despite the maintenance window being active.

</details>

## Summary

- **Composite alarms** combine multiple metric alarms into a single signal using `ALARM()`, `OK()`, `AND`, `OR`, `NOT` expressions
- **Suppressor alarms** silence composite alarm actions during planned maintenance without disabling the alarm evaluation
- **M-of-N evaluation** (`datapoints_to_alarm` < `evaluation_periods`) reduces false positives from transient spikes
- **`treat_missing_data = "notBreaching"`** is essential for Lambda alarms to prevent idle functions from triggering alerts
- **Child alarms should not have `alarm_actions`** when used with a composite alarm -- only the composite alarm sends notifications
