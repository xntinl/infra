## Prerequisites

- Terraform >= 1.7 installed
- AWS CLI configured with a sandbox account
- Basic understanding of Lambda functions

## Learning Objectives

After completing this exercise, you will be able to:

- **Explain** how CloudWatch Logs Insights queries are structured using `fields`, `filter`, `stats`, and `sort`
- **Identify** the built-in fields (`@timestamp`, `@message`, `@logStream`) available in every log group
- **Construct** queries that extract error rates, latency percentiles, and top-N patterns from Lambda logs

## Why Logs Insights

CloudWatch Logs captures every `console.log`, `print()`, and `logger.info()` from your Lambda functions. Without a query language, finding a specific error means scrolling through thousands of log lines in the console. CloudWatch Logs Insights gives you a SQL-like query language purpose-built for log data -- you can filter by time range, parse JSON fields, calculate percentiles, group by dimensions, and visualize results as time series or tables.

The key advantage over exporting logs to a separate analytics service is zero setup: Logs Insights works directly on your existing CloudWatch Log Groups with no pipelines, no ETL, and no additional infrastructure. The trade-off is cost: you pay per gigabyte of data scanned per query. This means time window filters are not just convenient -- they directly reduce your bill. A query scanning 7 days of logs costs 7x more than the same query scanning 1 day.

> **Best Practice:** Logs Insights is pay-per-query based on data scanned. Always use time window filters (`@timestamp` range picker in the console, or `--start-time` / `--end-time` in the CLI) to limit the scan range. For recurring analysis, consider CloudWatch Contributor Insights or export to S3 + Athena.

Structured logging (JSON) is the foundation of effective Logs Insights queries. When your Lambda emits `{"level": "ERROR", "duration_ms": 342, "user_id": "abc123"}`, Logs Insights automatically discovers `level`, `duration_ms`, and `user_id` as queryable fields. Plain text logs require `parse` commands with regex, which are slower and more fragile.

## Step 2 -- Initialize, Apply, and Generate Log Data

```bash
terraform init
terraform apply -auto-approve
```

Terraform will create 5 resources: the IAM role, IAM policy, log group, Lambda function, and the archive data source.

Now invoke the Lambda 50 times to generate a rich set of log data:

```bash
FUNC_NAME=$(terraform output -raw lambda_function_name)
for i in $(seq 1 50); do
  aws lambda invoke \
    --function-name "$FUNC_NAME" \
    --payload '{}' \
    --cli-binary-format raw-in-base64-out \
    /dev/null > /dev/null 2>&1
done
echo "Done -- 50 invocations sent"
```

Wait about 30 seconds for CloudWatch to ingest the logs, then proceed to the queries.

## Step 3 -- Run Logs Insights Queries

### Query 1: Recent log entries (basic fields)

This query returns the 20 most recent log entries with their timestamp and message:

```bash
LOG_GROUP=$(terraform output -raw log_group_name)
aws logs start-query \
  --log-group-name "$LOG_GROUP" \
  --start-time $(date -u -d '1 hour ago' +%s 2>/dev/null || date -u -v-1H +%s) \
  --end-time $(date -u +%s) \
  --query-string 'fields @timestamp, @message | sort @timestamp desc | limit 20' \
  --output json
```

This returns a `queryId`. Use it to fetch results:

```bash
aws logs get-query-results --query-id <QUERY_ID> --output table
```

The built-in `@timestamp` and `@message` fields are available in every log group. The `@` prefix denotes system-generated fields.

### Query 2: Filter errors only

```
fields @timestamp, level, endpoint, status_code, error_type, duration_ms
| filter level = "ERROR"
| sort @timestamp desc
| limit 20
```

The `filter` command works like SQL's `WHERE`. Because the Lambda emits JSON, Logs Insights automatically discovers `level`, `endpoint`, `status_code`, and every other JSON key as a queryable field.

### Query 3: Error rate by endpoint

```
filter ispresent(status_code)
| stats count(*) as total,
        sum(status_code >= 500) as errors,
        sum(status_code >= 500) * 100.0 / count(*) as error_rate
  by endpoint
| sort error_rate desc
```

The `stats` command aggregates data like SQL's `GROUP BY`. The `ispresent()` function filters out Lambda platform log lines (START, END, REPORT) that do not contain your JSON fields.

### Query 4: Latency percentiles by endpoint

```
filter ispresent(duration_ms)
| stats avg(duration_ms) as avg_ms,
        pct(duration_ms, 50) as p50,
        pct(duration_ms, 90) as p90,
        pct(duration_ms, 99) as p99,
        max(duration_ms) as max_ms
  by endpoint
| sort p99 desc
```

The `pct()` function calculates percentiles directly in the query engine -- no need to export data to a separate tool.

### Query 5: Top users by request count

```
filter ispresent(user_id)
| stats count(*) as requests,
        sum(status_code >= 400) as errors
  by user_id
| sort requests desc
| limit 10
```

### Query 6: Time-bucketed error rate (for graphing)

```
filter ispresent(status_code)
| stats count(*) as total,
        sum(status_code >= 500) as errors
  by bin(5m)
| sort @timestamp asc
```

The `bin()` function buckets results into time intervals. In the CloudWatch console, this query renders as a time-series chart. The CLI returns the raw data.

### Running queries from the CLI (complete example)

Here is a complete CLI workflow for Query 3:

```bash
LOG_GROUP=$(terraform output -raw log_group_name)

QUERY_ID=$(aws logs start-query \
  --log-group-name "$LOG_GROUP" \
  --start-time $(date -u -d '1 hour ago' +%s 2>/dev/null || date -u -v-1H +%s) \
  --end-time $(date -u +%s) \
  --query-string 'filter ispresent(status_code) | stats count(*) as total, sum(status_code >= 500) as errors, sum(status_code >= 500) * 100.0 / count(*) as error_rate by endpoint | sort error_rate desc' \
  --query "queryId" \
  --output text)

echo "Query started: $QUERY_ID"

# Poll for results (queries typically complete in 1-3 seconds)
sleep 3

aws logs get-query-results \
  --query-id "$QUERY_ID" \
  --output table
```

## Common Mistakes

### 1. Forgetting `ispresent()` when querying JSON fields

Lambda log groups contain platform messages (START, END, REPORT) that do not have your custom JSON fields. Without `ispresent()`, these lines show `null` for your fields and skew aggregations.

**Wrong -- includes platform log lines:**

```
stats avg(duration_ms) by endpoint
```

**What happens:** The query returns results where `endpoint` is empty and `avg(duration_ms)` is skewed by null values.

**Fix -- filter to lines that have your fields:**

```
filter ispresent(duration_ms) and ispresent(endpoint)
| stats avg(duration_ms) by endpoint
```

### 2. Not constraining the time window

Logs Insights charges per GB scanned. Querying with a wide time window scans all logs in that period.

**Wrong -- scanning 30 days of logs for a recent issue:**

```bash
aws logs start-query \
  --start-time $(date -u -d '30 days ago' +%s) \
  --end-time $(date -u +%s) \
  --query-string 'filter level = "ERROR" | limit 10'
```

**What happens:** You pay for scanning 30 days of data even though you only need the last hour.

**Fix -- narrow the time window:**

```bash
aws logs start-query \
  --start-time $(date -u -d '1 hour ago' +%s) \
  --end-time $(date -u +%s) \
  --query-string 'filter level = "ERROR" | limit 10'
```

### 3. Using `parse` when structured JSON would auto-discover

If your logs are plain text, you need `parse` with regex. But if they are JSON, Logs Insights auto-discovers every key.

**Unnecessary -- manually parsing JSON that is already auto-discovered:**

```
parse @message '{"level":"*","duration_ms":*,' as level, duration_ms
| filter level = "ERROR"
```

**Simpler -- just reference the JSON fields directly:**

```
filter level = "ERROR"
| fields @timestamp, duration_ms, endpoint
```

## Summary

- **CloudWatch Logs Insights** provides a SQL-like query language for log analysis directly on CloudWatch Log Groups
- **Structured JSON logging** enables automatic field discovery -- no `parse` commands needed
- **`filter ispresent(field)`** excludes Lambda platform log lines (START, END, REPORT) that lack your custom fields
- **`stats` with `pct()`** calculates latency percentiles without exporting data
- **`bin()` function** creates time-bucketed aggregations for time-series visualization
- **Cost control** comes from narrowing time windows -- Logs Insights charges per GB scanned, so always set the tightest time range possible
