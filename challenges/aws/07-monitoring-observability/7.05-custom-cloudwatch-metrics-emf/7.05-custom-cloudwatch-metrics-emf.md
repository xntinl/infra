## Why This Matters

The traditional way to publish custom CloudWatch metrics is the `PutMetricData` API. This works, but it adds latency to every Lambda invocation (an extra API call), costs money per API request, and separates your metric data from your log data. If you see a latency spike in a metric, you have to correlate timestamps manually to find the corresponding log entries.

Embedded Metric Format (EMF) solves all three problems. You write a specially structured JSON object to stdout (or CloudWatch Logs), and CloudWatch automatically extracts metrics from it -- no API call, no extra latency, no per-request cost beyond log ingestion. The same log line that produces the metric also contains your structured log fields, so you can query both in CloudWatch Logs Insights with a single search. EMF is the recommended approach for Lambda custom metrics in production.

> **Best Practice:** EMF timestamps must be in milliseconds since epoch (not seconds). If you pass seconds, CloudWatch silently accepts the data but places it at the wrong time -- typically January 1970. Always use `Date.now()` in Node.js or `int(time.time() * 1000)` in Python.

    import json
    import time
    import random

    def handler(event, context):
        start = time.time()

        # Simulate order processing
        order_id = f"ORD-{random.randint(1000, 9999)}"
        is_error = random.random() < 0.15
        processing_time = random.uniform(50, 500) if not is_error else random.uniform(800, 2000)

        # Simulate processing delay
        time.sleep(processing_time / 5000)

        if is_error:
            error_type = random.choice(["ValidationError", "PaymentError", "InventoryError"])
        else:
            error_type = "None"

        # Emit EMF-formatted log line
        emf_payload = {
            "_aws": {
                "Timestamp": int(time.time() * 1000),
                "CloudWatchMetrics": [
                    {
                        "Namespace": "Custom/OrderService",
                        "Dimensions": [["ServiceName"]],
                        "Metrics": [
                            {"Name": "OrderCount", "Unit": "Count"},
                            {"Name": "ProcessingTime", "Unit": "Milliseconds"},
                            {"Name": "ErrorCount", "Unit": "Count"}
                        ]
                    }
                ]
            },
            "ServiceName": "order-processor",
            "OrderCount": 1,
            "ProcessingTime": round(processing_time, 2),
            "ErrorCount": 1 if is_error else 0,
            "OrderId": order_id,
            "ErrorType": error_type
        }
        print(json.dumps(emf_payload))

        if is_error:
            return {"statusCode": 500, "body": json.dumps({"error": error_type})}

        return {"statusCode": 200, "body": json.dumps({"orderId": order_id})}
    PYTHON
    filename = "index.py"
  }
}

## Best Practices

1. **EMF produces metrics AND logs from one write** -- A single `print(json.dumps(emf_payload))` call in your Lambda creates both a CloudWatch metric data point and a searchable structured log entry. You never need to call `PutMetricData` separately.

2. **Timestamps must be milliseconds** -- The `_aws.Timestamp` field must be milliseconds since epoch. Python's `time.time()` returns seconds, so multiply by 1000 and cast to int. Getting this wrong is the single most common EMF bug.

3. **Use dimensions sparingly** -- Each unique dimension combination creates a separate metric stream. A dimension like `CustomerId` with 10,000 unique values creates 10,000 streams at $0.30/metric/month = $3,000/month. Use dimensions for low-cardinality fields like `ServiceName` or `Environment`.

4. **Use metric math for derived metrics** -- Instead of computing error rates in your application and emitting a separate metric, emit raw counts (errors, orders) and let CloudWatch compute the rate with a math expression. This keeps your Lambda logic simple and lets you change the formula without redeploying.

5. **Log Insights queries work on EMF fields** -- Since EMF writes structured JSON to CloudWatch Logs, you can query any field with Logs Insights:
   ```
   fields @timestamp, OrderId, ProcessingTime, ErrorType
   | filter ErrorType != "None"
   | sort ProcessingTime desc
   | limit 20
   ```

## Spot the Bug

A colleague wrote this EMF payload in their Lambda function, but custom metrics never appear in CloudWatch Metrics. The log entries are visible in CloudWatch Logs. **What is wrong?**

```python
emf_payload = {
    "_aws": {
        "Timestamp": int(time.time()),  # <-- BUG
        "CloudWatchMetrics": [
            {
                "Namespace": "Custom/OrderService",
                "Dimensions": [["ServiceName"]],
                "Metrics": [
                    {"Name": "OrderCount", "Unit": "Count"}
                ]
            }
        ]
    },
    "ServiceName": "order-processor",
    "OrderCount": 1
}
print(json.dumps(emf_payload))
```

<details>
<summary>Explain the bug</summary>

The `_aws.Timestamp` is set to `int(time.time())`, which returns **seconds** since epoch (e.g., `1709913600`). EMF requires **milliseconds** (e.g., `1709913600000`).

CloudWatch interprets the seconds value as milliseconds, placing the metric at `1709913600 ms` since epoch -- which is January 20, 1970. The metric is created but filed under a date over 50 years in the past, so it never shows up in the default dashboard time range.

**Wrong:**

```python
"Timestamp": int(time.time())
```

**Fix -- multiply by 1000:**

```python
"Timestamp": int(time.time() * 1000)
```

This is particularly insidious because `terraform apply` succeeds, the Lambda runs without errors, and the log entry appears in CloudWatch Logs with correct-looking `_aws` metadata. You only discover the problem when the metric does not appear on your dashboard within the expected time window.

</details>

resource "aws_cloudwatch_metric_alarm" "error_rate" {
  alarm_name          = "${var.project_name}-error-rate"
  alarm_description   = "Order processing error rate exceeds 25%"
  comparison_operator = "GreaterThanThreshold"
  evaluation_periods  = 2
  datapoints_to_alarm = 2
  threshold           = 25
  treat_missing_data  = "notBreaching"

  metric_query {
    id          = "error_rate"
    expression  = "IF(orders > 0, errors / orders * 100, 0)"
    label       = "Error Rate %"
    return_data = true
  }

  metric_query {
    id          = "errors"
    return_data = false

    metric {
      metric_name = "ErrorCount"
      namespace   = "Custom/OrderService"
      period      = 300
      stat        = "Sum"
      dimensions = {
        ServiceName = "order-processor"
      }
    }
  }

  metric_query {
    id          = "orders"
    return_data = false

    metric {
      metric_name = "OrderCount"
      namespace   = "Custom/OrderService"
      period      = 300
      stat        = "Sum"

      }

## Summary

- **Embedded Metric Format (EMF)** lets you emit custom CloudWatch metrics by writing structured JSON to stdout -- no PutMetricData API call needed
- **EMF timestamps must be milliseconds since epoch** -- using seconds silently places metrics at the wrong time (January 1970)
- **Each unique dimension combination creates a separate metric stream** -- use dimensions for low-cardinality fields only to control cost
- **Metric math expressions** in dashboards and alarms let CloudWatch compute derived values (like error rate) from raw counters without application-side computation
- **CloudWatch Logs Insights** can query any field in EMF payloads, giving you correlated metrics and structured logs from a single data source
