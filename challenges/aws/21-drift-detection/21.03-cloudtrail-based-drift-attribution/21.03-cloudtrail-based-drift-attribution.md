# 21.03 CloudTrail-Based Drift Attribution

## Metadata
| Campo | Valor |
|-------|-------|
| **Complejidad** | Avanzado |
| **Modalidad** | Terraform + Lambda/Python |
| **Servicios AWS** | CloudTrail, Lambda, EventBridge |

## Descripcion

Cuando se detecta drift, consultar CloudTrail para identificar QUIEN hizo el cambio manual y CUANDO. Construir una Lambda que correlacione el diff del plan con eventos de CloudTrail. Proveer atribucion en las notificaciones de drift.

## Objetivos de aprendizaje

- Implementar atribucion de drift mediante analisis forense de CloudTrail
- Correlacionar cambios detectados por Terraform con eventos de API de AWS
- Automatizar la investigacion de causas raiz de drift

## Requisitos previos

- **Conocimientos**: Terraform intermedio, Python, CloudTrail event structure, AWS SDK (boto3)
- **Herramientas**: Terraform CLI, Python 3.x, boto3, AWS CLI
- **Servicios**: CloudTrail (trail activo), Lambda, EventBridge

## Arquitectura

1. **Deteccion de drift**: Pipeline que ejecuta `terraform plan -detailed-exitcode` y genera plan JSON
2. **Lambda de atribucion**: Funcion Python que recibe el plan JSON, extrae los recursos con drift, y consulta CloudTrail
3. **Correlacion**: Mapeo de resource type/ID de Terraform a event names de CloudTrail (ej: `aws_security_group` -> `AuthorizeSecurityGroupIngress`)
4. **Notificacion enriquecida**: Mensaje que incluye recurso, tipo de cambio, usuario responsable y timestamp
5. **EventBridge**: Regla que conecta la deteccion con la Lambda de atribucion

## Pasos sugeridos

1. Desplegar infraestructura base con Terraform y asegurar que CloudTrail esta activo en la cuenta
2. Crear el mapeo de resource types de Terraform a event names de CloudTrail (tabla de referencia)
3. Implementar la Lambda en Python que recibe un plan JSON como input
4. Parsear el plan JSON para extraer recursos con drift: resource address, tipo de cambio, atributos modificados
5. Construir queries a CloudTrail usando `lookup_events` con filtros por resource type y time range
6. Correlacionar eventos de CloudTrail con los cambios del plan para obtener usuario (userIdentity), timestamp y source IP
7. Formatear la notificacion enriquecida con atribucion completa
8. Configurar EventBridge para orquestar el flujo: deteccion -> atribucion -> notificacion
9. Introducir drift manualmente con un usuario IAM especifico y validar que la atribucion es correcta
10. Desplegar toda la infraestructura de atribucion con Terraform

## Validacion

- [ ] CloudTrail esta activo y captura eventos de modificacion de recursos
- [ ] La Lambda parsea correctamente el plan JSON e identifica recursos con drift
- [ ] Las queries a CloudTrail retornan eventos correlacionados con el drift detectado
- [ ] La atribucion identifica correctamente al usuario que hizo el cambio manual
- [ ] Las notificaciones incluyen: recurso, cambio, usuario, timestamp
- [ ] El flujo completo funciona de forma automatizada via EventBridge
