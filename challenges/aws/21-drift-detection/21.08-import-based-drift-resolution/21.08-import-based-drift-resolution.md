# 21.08 Import-Based Drift Resolution

## Metadata
| Campo | Valor |
|-------|-------|
| **Complejidad** | Intermedio |
| **Modalidad** | Terraform |
| **Features TF** | import block, config generation, terraform plan |

## Descripcion

Practica la resolucion de drift mediante import: descubre recursos creados manualmente usando AWS CLI/SDK, genera la configuracion de Terraform correspondiente, importa usando bloques import, y verifica con `terraform plan` que no haya cambios pendientes.

## Objetivos de aprendizaje

- Adopcion de recursos existentes en Terraform
- Flujos de trabajo de import
- Generacion de configuracion que coincida con el recurso real

## Requisitos previos

- Conocimientos: Terraform basico, AWS CLI, comprension de recursos AWS
- Herramientas: Terraform CLI (1.5+), AWS CLI
- Servicios/Features: import block de Terraform, EC2, S3, IAM, security groups

## Arquitectura

El ejercicio simula un escenario comun en organizaciones: existen recursos creados manualmente que necesitan ser incorporados a Terraform. El flujo de trabajo tiene cuatro fases: descubrimiento (encontrar recursos no gestionados usando AWS CLI), generacion de configuracion (escribir el HCL que represente el recurso), import (usar bloques `import` para asociar el recurso real con la configuracion), y validacion (ejecutar `terraform plan` para confirmar que no hay diferencias).

## Pasos sugeridos

1. Crear varios recursos manualmente via AWS CLI: una instancia EC2, un bucket S3, un security group y un IAM role
2. Documentar los IDs y atributos de cada recurso creado
3. Usar AWS CLI para descubrir y listar los recursos creados (describe-instances, list-buckets, etc.)
4. Escribir la configuracion de Terraform para el primer recurso (EC2) basandote en los datos reales
5. Agregar el bloque `import` en la configuracion para asociar el recurso real
6. Ejecutar `terraform plan` y ajustar la configuracion hasta que no haya diferencias
7. Repetir el proceso para el bucket S3 y el security group
8. Importar el IAM role con sus policies adjuntas
9. Ejecutar `terraform plan` final confirmando cero cambios en todos los recursos
10. Ejecutar `terraform apply` para confirmar el estado y verificar que los recursos siguen intactos

## Validacion

- [ ] Se crearon recursos manualmente via AWS CLI sin Terraform
- [ ] Se descubrieron los recursos usando comandos de AWS CLI
- [ ] Se escribio la configuracion HCL que coincide con cada recurso real
- [ ] Los bloques `import` estan correctamente configurados con los IDs reales
- [ ] `terraform plan` muestra cero cambios despues del import de cada recurso
- [ ] Los recursos originales no fueron modificados ni recreados durante el proceso
