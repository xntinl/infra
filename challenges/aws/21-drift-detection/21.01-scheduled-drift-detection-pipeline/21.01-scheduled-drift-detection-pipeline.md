# 21.01 Scheduled Drift Detection Pipeline

## Metadata
| Campo | Valor |
|-------|-------|
| **Complejidad** | Intermedio |
| **Modalidad** | Terraform + GitHub Actions/EventBridge |
| **Features TF** | `-detailed-exitcode`, plan JSON output |

## Descripcion

Construir un pipeline que ejecute `terraform plan -detailed-exitcode` de forma programada (diariamente). Parsear el codigo de salida y la salida del plan. Enviar notificaciones a Slack con un resumen del drift detectado. Rastrear las tendencias de drift a lo largo del tiempo en un dashboard.

## Objetivos de aprendizaje

- Automatizar la deteccion de drift mediante pipelines programados
- Parsear la salida de `terraform plan` en formato JSON
- Implementar monitoreo operacional de infraestructura
- Construir notificaciones contextuales con resumen de cambios

## Requisitos previos

- **Conocimientos**: Terraform basico, CI/CD con GitHub Actions o EventBridge, formato JSON
- **Herramientas**: Terraform CLI, GitHub Actions o AWS EventBridge, Slack (webhook), jq
- **Features**: `terraform plan -detailed-exitcode`, `terraform show -json`

## Arquitectura

El pipeline se compone de:

1. **Trigger programado**: GitHub Actions cron o EventBridge rule que ejecuta diariamente
2. **Ejecucion del plan**: Contenedor o Lambda que ejecuta `terraform plan -detailed-exitcode` contra el state remoto
3. **Parser de resultados**: Script que interpreta el exit code (0=sin cambios, 1=error, 2=drift detectado) y parsea el JSON del plan
4. **Notificacion**: Integracion con Slack webhook que envia un resumen legible del drift
5. **Dashboard de tendencias**: Almacenamiento de resultados historicos (DynamoDB o S3) y visualizacion de tendencias

## Pasos sugeridos

1. Configurar un backend remoto de Terraform (S3 + DynamoDB) con infraestructura de prueba desplegada
2. Crear un workflow de GitHub Actions con schedule cron diario (o una EventBridge rule con Lambda)
3. Implementar el paso de `terraform plan -detailed-exitcode` capturando el exit code
4. Agregar `terraform show -json tfplan` para obtener el plan en formato JSON
5. Escribir un script que parsee el JSON del plan y extraiga los recursos con drift (added, changed, destroyed)
6. Configurar un Slack webhook e implementar el envio de notificaciones con resumen formateado
7. Almacenar los resultados de cada ejecucion en S3 o DynamoDB con timestamp
8. Introducir drift manualmente (cambiar un tag via consola) y verificar que el pipeline lo detecta
9. Construir un dashboard simple (CloudWatch, Grafana o HTML estatico en S3) que muestre tendencias historicas

## Validacion

- [ ] El pipeline se ejecuta automaticamente segun el schedule configurado
- [ ] `terraform plan -detailed-exitcode` retorna exit code 2 cuando hay drift
- [ ] La salida JSON del plan se parsea correctamente identificando recursos afectados
- [ ] Se envia una notificacion a Slack con resumen claro del drift
- [ ] Los resultados historicos se almacenan y son consultables
- [ ] El dashboard muestra tendencias de drift a lo largo del tiempo
