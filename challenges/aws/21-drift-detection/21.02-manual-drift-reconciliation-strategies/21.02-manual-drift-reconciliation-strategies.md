# 21.02 Manual Drift Reconciliation Strategies

## Metadata
| Campo | Valor |
|-------|-------|
| **Complejidad** | Intermedio |
| **Modalidad** | Terraform |
| **Features TF** | `import`, `state rm`, `refresh`, `apply -refresh-only` |

## Descripcion

Practicar diferentes enfoques de reconciliacion de drift: `terraform apply` para revertir cambios manuales, `terraform import` para adoptar adiciones manuales, `terraform state rm` + actualizar configuracion para aceptar cambios manuales, y `terraform refresh` para actualizar el state.

## Objetivos de aprendizaje

- Dominar las diferentes opciones de reconciliacion de drift
- Comprender los trade-offs de cada enfoque
- Saber cuando usar cada estrategia segun el escenario

## Requisitos previos

- **Conocimientos**: Terraform basico, gestion de state, ciclo de vida de recursos
- **Herramientas**: Terraform CLI, AWS CLI, consola de AWS
- **Features**: `terraform import`, `terraform state rm`, `terraform apply -refresh-only`

## Arquitectura

Se trabajara con un conjunto de recursos simples (EC2, Security Groups, S3 buckets) donde se introducira drift de diferentes formas:

1. **Escenario A (Revert)**: Cambio manual de tags via consola, revertido con `terraform apply`
2. **Escenario B (Adopt)**: Recurso creado manualmente, adoptado con `terraform import`
3. **Escenario C (Accept)**: Cambio manual aceptado con `terraform state rm` + nueva configuracion
4. **Escenario D (Refresh)**: Actualizacion de state con `terraform apply -refresh-only`

## Pasos sugeridos

1. Desplegar infraestructura base con Terraform: instancia EC2, Security Group, bucket S3
2. **Escenario Revert**: Modificar tags de la instancia EC2 via consola AWS, ejecutar `terraform plan` para ver el drift, aplicar `terraform apply` para revertir
3. **Escenario Adopt**: Crear un Security Group manualmente via AWS CLI, escribir la configuracion Terraform correspondiente, importar con `terraform import`
4. **Escenario Accept**: Modificar reglas de un Security Group via consola, usar `terraform state rm` para eliminar del state, reescribir la configuracion reflejando el estado actual, re-importar
5. **Escenario Refresh**: Modificar atributos de un bucket S3 via consola, ejecutar `terraform apply -refresh-only` para actualizar el state sin cambiar infraestructura
6. Documentar los trade-offs de cada enfoque: riesgo, complejidad, cuando es apropiado
7. Ejecutar `terraform plan` despues de cada reconciliacion para confirmar que no hay drift residual

## Validacion

- [ ] `terraform apply` revierte exitosamente cambios manuales de tags
- [ ] `terraform import` adopta un recurso creado manualmente sin errores
- [ ] `terraform state rm` + nueva configuracion acepta cambios manuales correctamente
- [ ] `terraform apply -refresh-only` actualiza el state sin modificar infraestructura
- [ ] Despues de cada reconciliacion, `terraform plan` muestra "No changes"
- [ ] Se documentan claramente los trade-offs de cada estrategia
