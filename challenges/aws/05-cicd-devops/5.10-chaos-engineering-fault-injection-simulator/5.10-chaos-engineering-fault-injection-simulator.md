# 5.10 Chaos Engineering Fault Injection Simulator

## Metadata
| Campo | Valor |
|-------|-------|
| **Complejidad** | Avanzado |
| **Modalidad** | Manual + Terraform para infraestructura |
| **Servicios AWS** | FIS, EC2, RDS, Lambda, CloudWatch, ECS |

## Descripcion

Ejecutar experimentos de caos usando AWS FIS (Fault Injection Simulator). Crear experiment templates que inyecten fallas (terminacion de instancias EC2, perdida de disponibilidad de AZ, throttling de Lambda, failover de RDS). Validar que la aplicacion se recupera dentro de los SLOs definidos. Integrar en CI/CD.

## Objetivos de aprendizaje

- Comprender los principios de chaos engineering
- Implementar inyeccion de fallas controlada con AWS FIS
- Validar la resiliencia de aplicaciones contra diversos tipos de fallas
- Verificar cumplimiento de SLOs bajo condiciones de falla
- Preparar game days para equipos de operaciones

## Requisitos previos

- Conocimientos: arquitectura de aplicaciones distribuidas, CloudWatch metricas y alarmas, conceptos de resiliencia
- Herramientas: AWS CLI, Terraform, jq (para analizar resultados)
- Servicios AWS: cuenta con permisos para FIS, EC2, RDS, Lambda, CloudWatch, ECS, IAM

## Arquitectura

La arquitectura despliega una aplicacion distribuida y ejecuta experimentos de caos contra ella:

1. **Aplicacion target**: una aplicacion multi-tier con EC2/ECS en frontend, Lambda para procesamiento, y RDS como base de datos
2. **FIS Experiment Templates**: plantillas que definen las fallas a inyectar (targets, actions, stop conditions)
3. **Stop Conditions**: alarmas de CloudWatch que detienen el experimento si el impacto excede los limites aceptables
4. **CloudWatch Dashboard**: panel para observar metricas durante los experimentos
5. **SLO definitions**: objetivos de nivel de servicio (latencia, disponibilidad) que deben mantenerse durante las fallas

## Pasos sugeridos

1. Desplegar una aplicacion multi-tier con Terraform (EC2/ECS, Lambda, RDS, ALB)
2. Configurar CloudWatch dashboards y alarmas para metricas clave (latencia, errores, disponibilidad)
3. Definir SLOs claros (ej. latencia p99 < 500ms, disponibilidad > 99.9%)
4. Crear un experiment template de FIS para terminacion de instancias EC2
5. Crear un experiment template para simulacion de perdida de una Availability Zone
6. Crear un experiment template para throttling de Lambda
7. Crear un experiment template para failover de RDS
8. Configurar stop conditions basadas en alarmas de CloudWatch
9. Ejecutar cada experimento y observar el comportamiento de la aplicacion
10. Documentar resultados, validar cumplimiento de SLOs e integrar experimentos en el pipeline CI/CD

## Validacion

- [ ] La aplicacion multi-tier se despliega correctamente con Terraform
- [ ] Los experiment templates de FIS se crean correctamente
- [ ] El experimento de terminacion de EC2 se ejecuta y la aplicacion se recupera
- [ ] El experimento de perdida de AZ se ejecuta y la aplicacion sigue disponible
- [ ] El experimento de throttling de Lambda se ejecuta y se observa degradacion controlada
- [ ] El failover de RDS se ejecuta y la aplicacion se reconecta automaticamente
- [ ] Las stop conditions detienen el experimento si se superan los limites
- [ ] Los SLOs se cumplen durante los experimentos (o se documentan las deficiencias)
- [ ] Los experimentos se pueden ejecutar desde el pipeline CI/CD
