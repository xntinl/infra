## Learning Objectives

By the end of this exercise you will be able to:

1. **Analyze** container image scan results to identify critical vulnerabilities and their remediation paths
2. **Implement** ECR scan-on-push with a CodeBuild step that gates the pipeline on scan results
3. **Configure** Amazon Inspector for continuous scanning of ECR repositories
4. **Design** a pipeline that fails automatically when CRITICAL or HIGH severity findings are detected
5. **Evaluate** the trade-offs between basic ECR scanning and Inspector enhanced scanning

## Why This Matters

Container images are the unit of deployment in modern architectures, and every image inherits vulnerabilities from its base layers and dependencies. A single unpatched CVE in your base image can expose your entire fleet. ECR scan-on-push catches known vulnerabilities at the moment an image enters your registry, but without a pipeline gate, those findings are informational -- developers see them only if they check the console.

The shift-left approach embeds scanning directly in the CI/CD pipeline. CodeBuild scans the image, parses the results, and fails the build if any CRITICAL or HIGH severity findings exist. This prevents vulnerable images from ever reaching production. Amazon Inspector adds continuous scanning -- it re-evaluates images when new CVEs are published, catching vulnerabilities that did not exist at build time.

> **Best Practice:** ECR image scan on push should be mandatory in all pipelines. Treat CRITICAL findings as build-breaking. HIGH findings should require explicit approval. MEDIUM and below should be tracked but not block deployment.

## Best Practices

- **Scan on push is non-negotiable**: Every image pushed to ECR should be scanned automatically. Do not rely on manual scans or periodic schedules
- **Gate on CRITICAL, alert on HIGH**: CRITICAL findings should fail the pipeline. HIGH findings should notify the team and require acknowledgment. MEDIUM and below are tracked in Security Hub
- **Use immutable tags**: Prevent tag reuse so you can always trace a running container back to its exact image and scan results
- **Pin base images**: Use specific digest references (`FROM nginx@sha256:abc...`) instead of floating tags (`FROM nginx:latest`) to avoid inheriting new vulnerabilities from upstream changes between builds
- **Lifecycle policies**: Configure ECR lifecycle policies to clean up old images. Keeping thousands of unscanned images wastes storage and creates a false sense of security

## Spot the Bug

A colleague set up the CodeBuild project for Docker builds, but the build always fails with "Cannot connect to the Docker daemon." **What is wrong?**

```hcl
resource "aws_codebuild_project" "this" {
  name         = "image-scanner"
  service_role = aws_iam_role.codebuild.arn

  artifacts {
    type = "CODEPIPELINE"
  }

  environment {
    compute_type                = "BUILD_GENERAL1_SMALL"
    image                       = "aws/codebuild/amazonlinux2-x86_64-standard:5.0"
    type                        = "LINUX_CONTAINER"
    privileged_mode             = false
    image_pull_credentials_type = "CODEBUILD"
  }

  source {
    type = "CODEPIPELINE"
  }

  tags = { Name = "image-scanner" }
}
```

<details>
<summary>Explain the bug</summary>

**The problem:** `privileged_mode = false` prevents the CodeBuild container from running the Docker daemon. Docker-in-Docker requires privileged access to mount the cgroup filesystem and start the `dockerd` process inside the build container.

Without privileged mode, the `docker build` command fails immediately because there is no Docker socket available:

```
Cannot connect to the Docker daemon at unix:///var/run/docker.sock.
Is the docker daemon running?
```

**The fix:** Set `privileged_mode = true`:

```hcl
resource "aws_codebuild_project" "this" {
  name         = "image-scanner"
  service_role = aws_iam_role.codebuild.arn

  artifacts {
    type = "CODEPIPELINE"
  }

  environment {
    compute_type                = "BUILD_GENERAL1_SMALL"
    image                       = "aws/codebuild/amazonlinux2-x86_64-standard:5.0"
    type                        = "LINUX_CONTAINER"
    privileged_mode             = true
    image_pull_credentials_type = "CODEBUILD"
  }

  source {
    type = "CODEPIPELINE"
  }

  tags = { Name = "image-scanner" }
}
```

Privileged mode is required for any CodeBuild project that builds Docker images. This is a security trade-off -- the build container has elevated capabilities -- so limit its IAM permissions to only what is needed.

</details>

```hcl
resource "aws_ecr_repository" "this" {
  name                 = "${var.project_name}-repo"

resource "aws_ecr_lifecycle_policy" "this" {
  repository = aws_ecr_repository.this.name

  policy = jsonencode({
    rules = [
      {
        rulePriority = 1
        description  = "Expire untagged images after 1 day"
        selection = {
          tagStatus   = "untagged"
          countType   = "sinceImagePushed"
          countUnit   = "days"
          countNumber = 1
        }
        action = {
          type = "expire"
        }
      },
      {
        rulePriority = 2
        description  = "Keep only 10 most recent tagged images"
        selection = {
          tagStatus   = "tagged"
          tagPrefixList = ["v"]
          countType   = "imageCountMoreThan"
          countNumber = 10
        }
        action = {
          type = "expire"
        }
      }
    ]
  })
}
```

## Summary

- **ECR scan-on-push** catches known vulnerabilities at image registration time -- it should be enabled on every repository
- **CodeBuild with privileged mode** is required for Docker-in-Docker builds; the trade-off is elevated container capabilities
- **Pipeline gating** on scan results prevents vulnerable images from reaching production -- fail on CRITICAL, alert on HIGH
- **Enhanced scanning** via Amazon Inspector provides continuous CVE monitoring, catching vulnerabilities discovered after the image was built
- **EventBridge integration** enables automated notification workflows when scan results arrive
- **Lifecycle policies** keep ECR repositories clean by expiring old and untagged images automatically
