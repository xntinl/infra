## Why This Matters

Blue/green deployments eliminate downtime and provide instant rollback capability. Instead of replacing containers in place (rolling update), CodeDeploy provisions an entirely new set of tasks (the "green" target group), shifts traffic from the old "blue" target group, and monitors CloudWatch alarms during a configurable wait period. If an alarm fires -- high error rate, elevated latency, failed health checks -- CodeDeploy automatically routes traffic back to the blue target group and terminates the green tasks. The entire rollback takes seconds, not minutes.

In production, this pattern is the standard for mission-critical services. A rolling update can leave you in a partially-deployed state where some tasks run the old version and others run the new version, making debugging difficult. Blue/green gives you an atomic switch: 100% old or 100% new, with a safety net that triggers automatically.

> **Best Practice:** CodeDeploy must have a CloudWatch alarm configured for auto-rollback. Without it, a broken deployment will complete successfully and route all traffic to failing containers. Always monitor at least error rate (5xx count) and p99 latency.

## The Challenge

Build a complete CI/CD pipeline that takes application code from an S3 source bucket, builds a Docker image with CodeBuild, pushes it to ECR, and deploys it to ECS Fargate using CodeDeploy blue/green deployment -- with automatic rollback on alarm.

```
                                                          │  Port 8080(test)│
```

## Best Practices

- **Immutable image tags**: Set ECR `image_tag_mutability = "IMMUTABLE"` to prevent overwriting deployed images. Use git SHA or build number as the tag, never `latest` in production
- **Alarm-gated rollback**: Always attach at least one CloudWatch alarm (5xx count, p99 latency) to the deployment group. Without alarms, CodeDeploy will complete even if the new version is broken
- **Test listener validation**: Use the test listener (port 8080) to run integration tests against the green target group before shifting production traffic
- **Terminate wait time**: Set `termination_wait_time_in_minutes` to at least 5 to give yourself time to manually roll back even after automatic traffic shift
- **Pipeline artifact encryption**: Enable S3 server-side encryption on the artifact bucket and use a KMS key for sensitive builds

<summary>Hint 1: VPC and networking foundation</summary>

You need a VPC with public and private subnets across at least two AZs. The ALB lives in public subnets; ECS tasks run in private subnets with a NAT Gateway for ECR image pulls.

```hcl
data "aws_availability_zones" "available" {
  state = "available"
}

locals {
  azs = slice(data.aws_availability_zones.available.names, 0, 2)
}

resource "aws_vpc" "this" {
  cidr_block           = "10.0.0.0/16"
  enable_dns_support   = true
  enable_dns_hostnames = true

  tags = { Name = "${var.project_name}-vpc" }
}

resource "aws_subnet" "public" {
  for_each = toset(local.azs)

  vpc_id                  = aws_vpc.this.id
  cidr_block              = cidrsubnet(aws_vpc.this.cidr_block, 8, index(local.azs, each.key))
  availability_zone       = each.key
  map_public_ip_on_launch = true

  tags = { Name = "${var.project_name}-public-${each.key}" }
}

resource "aws_subnet" "private" {
  for_each = toset(local.azs)

  vpc_id            = aws_vpc.this.id
  cidr_block        = cidrsubnet(aws_vpc.this.cidr_block, 8, index(local.azs, each.key) + 10)
  availability_zone = each.key

  tags = { Name = "${var.project_name}-private-${each.key}" }
}
```

Add an internet gateway, NAT gateway (in one public subnet), and route tables:
- Public route table: `0.0.0.0/0` -> IGW
- Private route table: `0.0.0.0/0` -> NAT GW

<summary>Hint 3: ALB with blue/green target groups</summary>

The ALB needs two target groups and two listeners. CodeDeploy will manage traffic shifting between them:

```hcl
resource "aws_lb" "this" {
  name               = "${var.project_name}-alb"
  internal           = false
  load_balancer_type = "application"
  security_groups    = [aws_security_group.alb.id]
  subnets            = [for s in aws_subnet.public : s.id]

  tags = { Name = "${var.project_name}-alb" }
}

resource "aws_lb_target_group" "blue" {
  name        = "${var.project_name}-blue"
  port        = 80
  protocol    = "HTTP"
  vpc_id      = aws_vpc.this.id
  target_type = "ip"

  health_check {
    path                = "/health"
    healthy_threshold   = 2
    unhealthy_threshold = 3
    timeout             = 5
    interval            = 10
    matcher             = "200"
  }

  tags = { Name = "${var.project_name}-blue-tg" }
}

resource "aws_lb_target_group" "green" {
  name        = "${var.project_name}-green"
  port        = 80
  protocol    = "HTTP"
  vpc_id      = aws_vpc.this.id
  target_type = "ip"

  health_check {
    path                = "/health"
    healthy_threshold   = 2
    unhealthy_threshold = 3
    timeout             = 5
    interval            = 10
    matcher             = "200"
  }

  tags = { Name = "${var.project_name}-green-tg" }
}

# Production listener -- serves live traffic
resource "aws_lb_listener" "prod" {
  load_balancer_arn = aws_lb.this.arn
  port              = 80
  protocol          = "HTTP"

  default_action {
    type             = "forward"
    target_group_arn = aws_lb_target_group.blue.arn
  }

  tags = { Name = "${var.project_name}-prod-listener" }

  lifecycle {
    ignore_changes = [default_action]
  }
}

# Test listener -- used during deployment to validate green
resource "aws_lb_listener" "test" {
  load_balancer_arn = aws_lb.this.arn
  port              = 8080
  protocol          = "HTTP"

  default_action {
    type             = "forward"
    target_group_arn = aws_lb_target_group.green.arn
  }

  tags = { Name = "${var.project_name}-test-listener" }

  lifecycle {
    ignore_changes = [default_action]
  }
}
```

The `lifecycle { ignore_changes = [default_action] }` is critical -- CodeDeploy will modify the listener's target group during deployment, and you do not want Terraform to revert it.

<summary>Hint 4: CodeDeploy with CloudWatch alarm rollback</summary>

Create the CodeDeploy application and deployment group with alarm-based rollback:

```hcl
resource "aws_codedeploy_app" "this" {
  compute_platform = "ECS"
  name             = "${var.project_name}-deploy"

  tags = { Name = "${var.project_name}-codedeploy-app" }
}

resource "aws_codedeploy_deployment_group" "this" {
  app_name               = aws_codedeploy_app.this.name
  deployment_group_name  = "${var.project_name}-dg"
  deployment_config_name = "CodeDeployDefault.ECSAllAtOnce"
  service_role_arn       = aws_iam_role.codedeploy.arn

  auto_rollback_configuration {
    enabled = true
    events  = ["DEPLOYMENT_FAILURE", "DEPLOYMENT_STOP_ON_ALARM"]
  }

  alarm_configuration {
    alarms  = [aws_cloudwatch_metric_alarm.target_5xx.alarm_name]
    enabled = true
  }

  blue_green_deployment_config {
    deployment_ready_option {
      action_on_timeout = "CONTINUE_DEPLOYMENT"
    }

    terminate_blue_instances_on_deployment_success {
      action                           = "TERMINATE"
      termination_wait_time_in_minutes = 5
    }
  }

  deployment_style {
    deployment_option = "WITH_TRAFFIC_CONTROL"
    deployment_type   = "BLUE_GREEN"
  }

  ecs_service {
    cluster_name = aws_ecs_cluster.this.name
    service_name = aws_ecs_service.this.name
  }

  load_balancer_info {
    target_group_pair_info {
      prod_traffic_route {
        listener_arns = [aws_lb_listener.prod.arn]
      }

      test_traffic_route {
        listener_arns = [aws_lb_listener.test.arn]
      }

      target_group {
        name = aws_lb_target_group.blue.name
      }

      target_group {
        name = aws_lb_target_group.green.name
      }
    }
  }

  tags = { Name = "${var.project_name}-deployment-group" }
}

resource "aws_cloudwatch_metric_alarm" "target_5xx" {
  alarm_name          = "${var.project_name}-target-5xx-high"
  comparison_operator = "GreaterThanThreshold"
  evaluation_periods  = 1
  metric_name         = "HTTPCode_Target_5XX_Count"
  namespace           = "AWS/ApplicationELB"
  period              = 60
  statistic           = "Sum"
  threshold           = 10
  alarm_description   = "Triggers rollback when target 5xx errors exceed threshold"
  treat_missing_data  = "notBreaching"

  dimensions = {
    LoadBalancer = aws_lb.this.arn_suffix
  }

  tags = { Name = "${var.project_name}-5xx-alarm" }
}
```

## Spot the Bug

A colleague configured the ECS service for blue/green deployment, but every deployment fails with "The ECS service cannot be updated." **What is wrong?**

```hcl
resource "aws_ecs_service" "this" {
  name            = "my-app-service"
  cluster         = aws_ecs_cluster.this.id
  task_definition = aws_ecs_task_definition.this.arn
  desired_count   = 2
  launch_type     = "FARGATE"

  deployment_controller {
    type = "ECS"
  }

  network_configuration {
    subnets          = [for s in aws_subnet.private : s.id]
    security_groups  = [aws_security_group.ecs.id]
    assign_public_ip = false
  }

  load_balancer {
    target_group_arn = aws_lb_target_group.blue.arn
    container_name   = "app"
    container_port   = 80
  }
}
```

<details>
<summary>Explain the bug</summary>

**The problem:** The `deployment_controller` type is set to `"ECS"` (rolling update). Blue/green deployments with CodeDeploy require `deployment_controller { type = "CODE_DEPLOY" }`.

When the deployment controller is `"ECS"`, CodeDeploy cannot manage the service -- it has no permission to create replacement task sets or shift traffic between target groups. The deployment fails immediately because CodeDeploy expects to own the deployment lifecycle.

**The fix:**

```hcl
resource "aws_ecs_service" "this" {
  name            = "my-app-service"
  cluster         = aws_ecs_cluster.this.id
  task_definition = aws_ecs_task_definition.this.arn
  desired_count   = 2
  launch_type     = "FARGATE"

  deployment_controller {
    type = "CODE_DEPLOY"
  }

  network_configuration {
    subnets          = [for s in aws_subnet.private : s.id]
    security_groups  = [aws_security_group.ecs.id]
    assign_public_ip = false
  }

  load_balancer {
    target_group_arn = aws_lb_target_group.blue.arn
    container_name   = "app"
    container_port   = 80
  }

  lifecycle {
    ignore_changes = [task_definition, load_balancer]
  }
}
```

Note the `lifecycle { ignore_changes }` block: once CodeDeploy manages deployments, it will update the task definition and load balancer target group. If Terraform tracks those fields, it will try to revert them on the next `apply`.

</details>

## Summary

- **CodePipeline** orchestrates multi-stage CI/CD pipelines using AWS-native services (S3 source, CodeBuild, CodeDeploy)
- **Blue/green deployment** provisions a complete replacement environment before shifting traffic, enabling instant rollback
- **CodeDeploy with ECS** requires `deployment_controller { type = "CODE_DEPLOY" }` on the ECS service -- using the default `ECS` controller will cause deployment failures
- **CloudWatch alarm rollback** is the safety net: without it, a broken deployment completes and routes all traffic to failing containers
- **Lifecycle ignore_changes** on `task_definition` and `load_balancer` prevents Terraform from reverting CodeDeploy-managed state
- **Immutable ECR tags** prevent accidental overwriting of deployed images and ensure reproducible deployments
