## Why This Matters

Schema migrations are the most dangerous part of any deployment. A missing column crashes your application. A dropped table loses data. A long-running migration locks your database and causes an outage. Despite this, many teams still run migrations manually -- SSH into a bastion, paste SQL commands, and hope nothing goes wrong.

Automating migrations in your CI/CD pipeline eliminates human error and enforces ordering. Every migration gets a version number. A tracking table in the database records which versions have been applied. The pipeline runs unapplied migrations in order, and only proceeds to deploy application code if all migrations succeed. If a migration fails, the pipeline stops -- your application continues running the old code against the old schema, which is a safe state.

> **Best Practice:** Always use forward-only migrations. Never drop a column in the same release as the code change that stops using it. Instead, deploy the code change first (which ignores the column), then deploy a second migration that drops it. This two-phase approach means either version of the code works with either version of the schema.

## Best Practices

| Practice | Why |
|----------|-----|
| Forward-only migrations | Backward migrations are unreliable for data changes; deploy a new forward migration to reverse |
| Never drop columns in the same release | Old code still running during deployment will crash querying a removed column |
| Migrations tracking table | A `schema_migrations` table prevents duplicate execution and provides an audit trail |
| Test on a snapshot first | Clone the database, run migrations on the clone, proceed only if successful |
| Lambda in VPC for private RDS | Lambda must be in the same VPC to reach RDS in private subnets |
| Secrets Manager for credentials | Never hardcode passwords; use `manage_master_user_password = true` |

## Spot the Bug

A colleague deployed this Lambda migration runner but it times out every time it tries to connect to RDS. The RDS instance is healthy and reachable from an EC2 bastion in the same VPC. **What is wrong?**

```hcl
resource "aws_lambda_function" "migration_runner" {
  function_name = "db-migration-runner"
  role          = aws_iam_role.lambda.arn
  handler       = "index.handler"
  runtime       = "python3.12"
  timeout       = 120
  memory_size   = 256

  filename         = data.archive_file.lambda.output_path
  source_code_hash = data.archive_file.lambda.output_base64sha256

  environment {
    variables = {
      DB_HOST       = aws_db_instance.this.address
      DB_PORT       = "5432"
      DB_NAME       = var.db_name
      DB_SECRET_ARN = aws_db_instance.this.master_user_secret[0].secret_arn
    }
  }

  tags = { Name = "db-migration-runner" }
}
```

<details>
<summary>Explain the bug</summary>

**The problem:** The Lambda function has no `vpc_config` block. Without VPC configuration, Lambda runs in AWS's shared network, which has no route to the private subnets where RDS lives. The connection hangs until the 120-second timeout expires.

The EC2 bastion works because it is inside the VPC. Lambda, by default, runs outside your VPC.

**The fix:** Add `vpc_config` and attach `AWSLambdaVPCAccessExecutionRole` to the IAM role:

```hcl
resource "aws_lambda_function" "migration_runner" {
  # ... same as above ...

  vpc_config {
    subnet_ids         = [for s in aws_subnet.private : s.id]
    security_group_ids = [aws_security_group.lambda.id]
  }
}
```

</details>

data "archive_file" "lambda" {
  type        = "zip"
  output_path = "${path.module}/builds/migration-runner.zip"

      import json, os, boto3

      def handler(event, context):
          db_host = os.environ['DB_HOST']
          db_name = os.environ['DB_NAME']
          secret_arn = os.environ['DB_SECRET_ARN']

          sm = boto3.client('secretsmanager')
          secret = json.loads(
              sm.get_secret_value(SecretId=secret_arn)['SecretString']
          )

          print(f"Connected to {db_host}/{db_name}")
          print("Would execute: CREATE TABLE IF NOT EXISTS schema_migrations ...")
          print("Would apply unapplied SQL files in version order")

          return {
              'statusCode': 200,
              'body': json.dumps({'message': 'Migrations executed', 'db_host': db_host})
          }

    filename = "index.py"
  }
}

  policy = data.aws_iam_policy_document.pipeline_policy.json

resource "aws_codepipeline" "this" {
  name     = "${var.project_name}-pipeline"
  role_arn = aws_iam_role.pipeline.arn

  artifact_store {
    location = aws_s3_bucket.migrations.id
    type     = "S3"
  }

  stage {
    name = "Source"
    action {
      name             = "MigrationScripts"
      category         = "Source"
      owner            = "AWS"
      provider         = "S3"
      version          = "1"
      output_artifacts = ["source_output"]
      configuration = {
        S3Bucket             = aws_s3_bucket.migrations.id
        S3ObjectKey          = "migrations.zip"
        PollForSourceChanges = "true"
      }
    }
  }

  stage {
    name = "Migrate"
    action {
      name            = "RunMigrations"
      category        = "Invoke"
      owner           = "AWS"
      provider        = "Lambda"
      version         = "1"
      input_artifacts = ["source_output"]
      configuration = {
        FunctionName = aws_lambda_function.this.function_name
      }
    }
  }

  tags = { Name = "${var.project_name}-pipeline" }
