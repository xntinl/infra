## Why This Matters

Deploying a new Lambda version to 100% of traffic immediately is a gamble. If the new code has a regression -- a memory leak, a timeout under load, an unhandled exception for a specific input -- every invocation fails until you manually roll back. Canary deployments shift a small percentage of traffic (typically 10%) to the new version first, then monitor CloudWatch alarms for a configurable period before shifting the remaining 90%.

If the alarm fires during the canary window -- error rate spikes, latency exceeds your SLA -- CodeDeploy automatically routes all traffic back to the previous version. The blast radius is limited to the 10% canary period, and the rollback happens in seconds without human intervention.

> **Best Practice:** Always set alarms on both error rate and p99 latency. A canary should run for at least 10 minutes before proceeding to full traffic shift. Use `CodeDeployDefault.LambdaCanary10Percent10Minutes` or a custom deployment config with similar properties.

## Best Practices

- **Canary duration matters**: Use `CodeDeployDefault.LambdaCanary10Percent10Minutes` or longer. A 1-minute canary does not give enough time for latency metrics to populate and alarms to evaluate
- **Alarm on both errors and latency**: Error rate catches crashes; p99 latency catches performance regressions like cold start increases or slow downstream calls
- **Use `treat_missing_data = "notBreaching"`**: During low-traffic periods, missing data points should not trigger rollback. If you use `"breaching"`, every quiet minute fires the alarm
- **Publish versions explicitly**: Set `publish = true` on the Lambda function. Without it, updates replace `$LATEST` in place and there is no version to roll back to
- **Alias lifecycle ignore**: Add `lifecycle { ignore_changes = [function_version, routing_config] }` to the alias resource so Terraform does not fight with CodeDeploy over version assignments

## Spot the Bug

A colleague set up the CloudWatch alarm for Lambda errors, but the alarm never fires during canary deployment even when the new version throws exceptions. **What is wrong?**

```hcl
resource "aws_cloudwatch_metric_alarm" "lambda_errors" {
  alarm_name          = "canary-lambda-errors"
  comparison_operator = "GreaterThanThreshold"
  evaluation_periods  = 1
  metric_name         = "Errors"
  namespace           = "AWS/Lambda"
  period              = 60
  statistic           = "Sum"
  threshold           = 5

  dimensions = {
    FunctionName = aws_lambda_function.this.function_name
  }

  tags = { Name = "canary-lambda-errors" }
}
```

<details>
<summary>Explain the bug</summary>

**The problem:** The alarm dimensions only include `FunctionName`, which aggregates errors across **all versions and aliases** of the function. During a canary deployment, only 10% of traffic goes to the new version, so the error count is diluted by the 90% of successful invocations on the old version. The alarm threshold may never be reached.

More critically, the alarm is not scoped to the alias that CodeDeploy is managing. CodeDeploy needs to see errors specifically from the `live` alias to make rollback decisions.

**The fix:** Add the `Resource` dimension set to the alias name. This scopes the alarm to only count errors from invocations routed through the `live` alias:

```hcl
resource "aws_cloudwatch_metric_alarm" "lambda_errors" {
  alarm_name          = "canary-lambda-errors"
  comparison_operator = "GreaterThanThreshold"
  evaluation_periods  = 1
  metric_name         = "Errors"
  namespace           = "AWS/Lambda"
  period              = 60
  statistic           = "Sum"
  threshold           = 5

  dimensions = {
    FunctionName = aws_lambda_function.this.function_name
    Resource     = "${aws_lambda_function.this.function_name}:live"
  }

  tags = { Name = "canary-lambda-errors" }
}
```

The `Resource` dimension format is `function-name:alias-name`. This ensures the alarm only monitors invocations routed through the `live` alias, giving CodeDeploy accurate error signal during the canary window.

<summary>TODO 5 -- Pre-traffic hook Lambda (codedeploy.tf)</summary>

First create `hook_code/hook.py`:

```python
import boto3
import json
import os

codedeploy = boto3.client("codedeploy")
lambda_client = boto3.client("lambda")

def handler(event, context):
    deployment_id = event["DeploymentId"]
    hook_id = event["LifecycleEventHookExecutionId"]
    status = "Succeeded"

    try:
        response = lambda_client.invoke(
            FunctionName=os.environ["TARGET_FUNCTION_NAME"],
            Qualifier="live",
            Payload=json.dumps({"test": True}),
        )
        payload = json.loads(response["Payload"].read())

        if response["StatusCode"] != 200:
            status = "Failed"
        if payload.get("statusCode") != 200:
            status = "Failed"

    except Exception as e:
        print(f"Pre-traffic validation failed: {e}")
        status = "Failed"

    codedeploy.put_lifecycle_event_hook_execution_status(
        deploymentId=deployment_id,
        lifecycleEventHookExecutionId=hook_id,
        status=status,
    )
    return {"status": status}
```

## Summary

- **Lambda aliases** (`live`) decouple the invocation endpoint from the version, allowing CodeDeploy to shift traffic between versions without changing callers
- **Canary deployment** shifts 10% of traffic first, waits for alarm evaluation, then shifts the remaining 90% -- limiting blast radius to the canary window
- **CloudWatch alarm dimensions** must include the `Resource` dimension (function:alias) to scope metrics to the alias being deployed, not the aggregate function
- **`publish = true`** is mandatory for versioned deployments -- without it, there are no immutable versions to roll back to
- **Pre-traffic hooks** validate the new version before any production traffic reaches it, catching configuration errors and missing permissions early
