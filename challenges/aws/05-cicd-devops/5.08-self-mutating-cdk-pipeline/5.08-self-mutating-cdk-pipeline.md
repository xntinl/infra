# 5.08 Self-Mutating CDK Pipeline

## Metadata
| Campo | Valor |
|-------|-------|
| **Complejidad** | Avanzado |
| **Modalidad** | CDK (TypeScript/Python) |
| **Servicios AWS** | CodePipeline, CodeBuild, CloudFormation, CDK |

## Descripcion

Construir un CDK Pipeline que pueda actualizarse a si mismo. Cuando se hacen cambios al codigo del pipeline, este primero actualiza su propia definicion y luego despliega las etapas de infraestructura de la aplicacion. Incluir etapas de despliegue cross-account y tests de integracion.

## Objetivos de aprendizaje

- Comprender el constructo CDK Pipelines y su mecanismo de auto-mutacion
- Implementar sintesis de etapas con CDK
- Configurar bootstrapping cross-account para CDK
- Disenar pipelines que evolucionen con el codigo de infraestructura

## Requisitos previos

- Conocimientos: AWS CDK (constructos, stacks, stages), TypeScript o Python, CloudFormation, conceptos de CI/CD
- Herramientas: AWS CDK CLI, Node.js/Python, AWS CLI, Git
- Servicios AWS: cuenta con permisos para CodePipeline, CodeBuild, CloudFormation, IAM, S3

## Arquitectura

La arquitectura se basa en el constructo `CodePipeline` de CDK:

1. **Self-mutate stage**: primera etapa del pipeline que sintetiza el CDK app y actualiza la definicion del propio pipeline si hay cambios
2. **Synth step**: ejecuta `cdk synth` para generar los templates de CloudFormation
3. **Application stages**: etapas que despliegan la infraestructura de la aplicacion (dev, staging, prod)
4. **Cross-account stages**: etapas que despliegan en cuentas AWS diferentes usando roles bootstrapped
5. **Integration tests**: steps de ShellStep que ejecutan tests de integracion tras cada despliegue

El flujo es: push al repo -> synth -> self-mutate (si cambio el pipeline) -> deploy dev -> tests -> deploy prod.

## Pasos sugeridos

1. Inicializar un proyecto CDK con TypeScript o Python
2. Ejecutar `cdk bootstrap` en todas las cuentas target con trust hacia la cuenta del pipeline
3. Crear un `CodePipeline` con la fuente apuntando al repositorio Git
4. Definir el `ShellStep` de sintesis que ejecuta `npm install && cdk synth`
5. Crear un `Stage` de aplicacion que contenga los stacks de infraestructura (ej. Lambda + DynamoDB)
6. Agregar el stage de dev al pipeline con `pipeline.addStage()`
7. Agregar un step de integracion post-despliegue que valide la infraestructura
8. Agregar un stage de produccion con pre-step de aprobacion manual
9. Hacer un cambio al pipeline (ej. agregar una etapa) y verificar que se auto-actualiza
10. Verificar que los cambios a la infraestructura de la aplicacion se despliegan correctamente

## Validacion

- [ ] El pipeline CDK se despliega correctamente con `cdk deploy`
- [ ] La etapa de self-mutate actualiza el pipeline cuando su codigo cambia
- [ ] La sintesis CDK genera los templates de CloudFormation correctamente
- [ ] La infraestructura de la aplicacion se despliega en la etapa de dev
- [ ] Los tests de integracion se ejecutan tras el despliegue en dev
- [ ] La aprobacion manual es requerida antes del despliegue a produccion
- [ ] Los cambios al pipeline se reflejan automaticamente tras self-mutation
- [ ] El bootstrapping cross-account funciona correctamente (si se configura)
