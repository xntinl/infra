## Why Feature Flags

Deploying new code and enabling new features are two separate concerns, but most teams conflate them. A feature flag lets you deploy code that contains a new feature in a dormant state, then enable it independently -- for a percentage of users, for a specific environment, or all at once. If the feature causes problems, you flip the flag off instead of rolling back the deployment.

AWS AppConfig provides managed feature flag infrastructure. You define flags as JSON configuration, deploy them through a strategy (instant, linear, or canary), and read them from your application at runtime. AppConfig caches configuration locally via a Lambda layer, so reads are fast and do not add latency to your request path. The deployment strategy controls how quickly changes propagate -- `AllAtOnce` pushes instantly (fine for dev), while `Linear50PercentEvery30Seconds` rolls out gradually (safer for production). If a validator detects a problem during rollout, AppConfig automatically rolls back.

> **Best Practice:** Use `AppConfig.AllAtOnce` only for development environments. For production, use `AppConfig.Linear50PercentEvery30Seconds` or a custom canary strategy that gives you time to detect problems before the change reaches all instances.

### `appconfig.tf`

```hcl
# AppConfig Application: the top-level container for environments
# and configuration profiles. Think of it as the "project."
resource "aws_appconfig_application" "this" {
  name        = "${var.project_name}-app"
  description = "Feature flags demo application"

  tags = {
    Name = "${var.project_name}-app"
  }
}

# Environment: represents a deployment target (dev, staging, prod).
# Each environment gets its own deployed configuration version.
resource "aws_appconfig_environment" "dev" {
  name           = "dev"
  description    = "Development environment"
  application_id = aws_appconfig_application.this.id

  tags = {
    Name = "${var.project_name}-dev-env"
  }
}

# Configuration profile: defines the type and location of
# configuration data. "AWS.AppConfig.FeatureFlags" is a built-in
# type optimized for boolean/variant flags.
resource "aws_appconfig_configuration_profile" "flags" {
  application_id = aws_appconfig_application.this.id
  name           = "feature-flags"
  description    = "Feature flags configuration profile"
  location_uri   = "hosted"
  type           = "AWS.AppConfig.FeatureFlags"

  tags = {
    Name = "${var.project_name}-flags-profile"
  }
}

# Hosted configuration version: the actual feature flag data.
# This is a JSON document following the AppConfig feature flags schema.
resource "aws_appconfig_hosted_configuration_version" "flags" {
  application_id           = aws_appconfig_application.this.id
  configuration_profile_id = aws_appconfig_configuration_profile.flags.configuration_profile_id
  content_type             = "application/json"

  content = jsonencode({
    version = "1"
    flags = {
      enable_new_checkout = {
        name = "enable_new_checkout"
      }
      max_items_per_page = {
        name = "max_items_per_page"
        attributes = {
          limit = {
            constraints = {
              type     = "number"
              required = true
            }
          }
        }
      }
      enable_dark_mode = {
        name = "enable_dark_mode"
      }
    }
    values = {
      enable_new_checkout = {
        enabled = true
      }
      max_items_per_page = {
        enabled = true
        limit   = 25
      }
      enable_dark_mode = {
        enabled = false
      }
    }
  })
}

# Deployment strategy: controls how fast configuration changes
# propagate. AllAtOnce is instant (suitable for dev only).
resource "aws_appconfig_deployment_strategy" "instant" {
  name                           = "${var.project_name}-all-at-once"
  description                    = "Deploy all at once (dev only)"
  deployment_duration_in_minutes = 0
  final_bake_time_in_minutes     = 0
  growth_factor                  = 100
  growth_type                    = "LINEAR"
  replicate_to                   = "NONE"

  tags = {
    Name = "${var.project_name}-instant-strategy"
  }
}

# A safer strategy for production-like environments:
# 50% every 30 seconds with a 1-minute bake time.
resource "aws_appconfig_deployment_strategy" "linear" {
  name                           = "${var.project_name}-linear-50-pct"
  description                    = "Linear 50% every 30 seconds (prod)"
  deployment_duration_in_minutes = 1
  final_bake_time_in_minutes     = 1
  growth_factor                  = 50
  growth_type                    = "LINEAR"
  replicate_to                   = "NONE"

  tags = {
    Name = "${var.project_name}-linear-strategy"
  }
}

# Deployment: pushes the hosted configuration version to the dev
# environment using the instant strategy.
resource "aws_appconfig_deployment" "dev" {
  application_id           = aws_appconfig_application.this.id
  configuration_profile_id = aws_appconfig_configuration_profile.flags.configuration_profile_id
  configuration_version    = aws_appconfig_hosted_configuration_version.flags.version_number
  deployment_strategy_id   = aws_appconfig_deployment_strategy.instant.id
  environment_id           = aws_appconfig_environment.dev.environment_id
  description              = "Initial feature flags deployment"

  tags = {
    Name = "${var.project_name}-dev-deployment"
  }
}
```

### `lambda.tf`

```hcl
# ---------------------------------------------------------------
# IAM Role for Lambda
# ---------------------------------------------------------------
data "aws_iam_policy_document" "lambda_assume" {
  statement {
    actions = ["sts:AssumeRole"]
    principals {
      type        = "Service"
      identifiers = ["lambda.amazonaws.com"]
    }
  }
}

resource "aws_iam_role" "lambda" {
  name               = "${var.project_name}-lambda-role"
  assume_role_policy = data.aws_iam_policy_document.lambda_assume.json

  tags = {
    Name = "${var.project_name}-lambda-role"
  }
}

resource "aws_iam_role_policy_attachment" "lambda_basic" {
  role       = aws_iam_role.lambda.name
  policy_arn = "arn:aws:iam::aws:policy/service-role/AWSLambdaBasicExecutionRole"
}

# AppConfig permissions: the Lambda extension needs these to
# retrieve configuration from AppConfig.
data "aws_iam_policy_document" "lambda_appconfig" {
  statement {
    actions = [
      "appconfig:StartConfigurationSession",
      "appconfig:GetLatestConfiguration"
    ]
    resources = ["*"]
  }
}

resource "aws_iam_role_policy" "lambda_appconfig" {
  name   = "${var.project_name}-lambda-appconfig"
  role   = aws_iam_role.lambda.id
  policy = data.aws_iam_policy_document.lambda_appconfig.json
}

# ---------------------------------------------------------------
# Lambda function code
# ---------------------------------------------------------------
data "archive_file" "lambda" {
  type        = "zip"
  output_path = "${path.module}/builds/feature-flags-reader.zip"

  source {
    content = <<-PYTHON
      import json
      import os
      import urllib.request

      # The AppConfig Lambda extension runs on localhost:2772
      # and caches configuration locally. This avoids calling
      # the AppConfig API on every Lambda invocation.
      APPCONFIG_PORT = os.environ.get('AWS_APPCONFIG_EXTENSION_HTTP_PORT', '2772')
      APP_NAME = os.environ['APPCONFIG_APP']
      ENV_NAME = os.environ['APPCONFIG_ENV']
      PROFILE_NAME = os.environ['APPCONFIG_PROFILE']

      def handler(event, context):
          # Fetch feature flags from the local AppConfig extension
          url = (
              f"http://localhost:{APPCONFIG_PORT}"
              f"/applications/{APP_NAME}"
              f"/environments/{ENV_NAME}"
              f"/configurations/{PROFILE_NAME}"
          )

          try:
              req = urllib.request.Request(url)
              with urllib.request.urlopen(req) as resp:
                  config = json.loads(resp.read())
          except Exception as e:
              return {
                  'statusCode': 500,
                  'body': json.dumps({'error': f'Failed to read config: {str(e)}'})
              }

          # Use feature flags to control behavior
          new_checkout = config.get('enable_new_checkout', {}).get('enabled', False)
          dark_mode = config.get('enable_dark_mode', {}).get('enabled', False)
          max_items = config.get('max_items_per_page', {}).get('limit', 10)

          return {
              'statusCode': 200,
              'headers': {'Content-Type': 'application/json'},
              'body': json.dumps({
                  'message': 'Feature flags loaded successfully',
                  'flags': {
                      'enable_new_checkout': new_checkout,
                      'enable_dark_mode': dark_mode,
                      'max_items_per_page': max_items
                  }
              })
          }
    PYTHON
    filename = "index.py"
  }
}

# Look up the AppConfig Lambda extension layer ARN for the region.
# The extension runs as a sidecar and caches configuration locally.
data "aws_region" "current" {}

resource "aws_lambda_function" "this" {
  function_name = "${var.project_name}-reader"
  description   = "Reads feature flags from AppConfig"
  role          = aws_iam_role.lambda.arn
  handler       = "index.handler"
  runtime       = "python3.12"
  timeout       = 30
  memory_size   = 128

  filename         = data.archive_file.lambda.output_path
  source_code_hash = data.archive_file.lambda.output_base64sha256

  # The AppConfig Lambda extension layer provides local caching.
  # Without this layer, you would need to call the AppConfig API
  # directly on every invocation, adding latency and cost.
  layers = [
    "arn:aws:lambda:${data.aws_region.current.name}:027255383542:layer:AWS-AppConfig-Extension:128"
  ]

  environment {
    variables = {
      APPCONFIG_APP     = aws_appconfig_application.this.id
      APPCONFIG_ENV     = aws_appconfig_environment.dev.environment_id
      APPCONFIG_PROFILE = aws_appconfig_configuration_profile.flags.configuration_profile_id
    }
  }

  tags = {
    Name = "${var.project_name}-reader"
  }
}

resource "aws_cloudwatch_log_group" "lambda" {
  name              = "/aws/lambda/${var.project_name}-reader"
  retention_in_days = 14

  tags = {
    Name = "${var.project_name}-reader-logs"
  }
}
```

## Best Practices

| Practice | Why |
|----------|-----|
| Always add the AppConfig Lambda layer | Without the extension layer, every Lambda invocation calls the AppConfig API directly, adding 50-100ms of latency and risking throttling |
| Use `AllAtOnce` only in dev | Instant deployment provides no safety net; a bad configuration reaches all instances immediately |
| `Linear50PercentEvery30Seconds` for prod | Gradual rollout gives you time to detect problems before the change reaches all instances |
| Feature flags schema validation | Define constraints in the configuration profile so AppConfig rejects invalid values before deployment |
| Keep flag names descriptive | Use `enable_new_checkout` not `flag_1`; flags accumulate and cryptic names cause confusion |
| Clean up stale flags | Feature flags are meant to be temporary; remove flags after the feature is fully rolled out |

## Common Mistakes

### 1. Not adding the AppConfig Lambda layer for caching

If you create the Lambda function without the AppConfig extension layer, the function has no local cache. You must call the AppConfig API directly using the SDK, which adds latency to every invocation and can trigger throttling under load.

**Wrong -- no layer, calling API directly:**

```python
import boto3

client = boto3.client('appconfig')
# This calls the API on every invocation -- slow and expensive
response = client.get_configuration(...)
```

**Fix -- add the extension layer and read from localhost:**

```python
import urllib.request

# The extension caches config locally -- fast and free
url = "http://localhost:2772/applications/..."
response = urllib.request.urlopen(url)
```

### 2. Using the wrong deployment strategy in production

Using `AllAtOnce` in production pushes configuration changes to 100% of instances instantly. If the new flag value breaks your application, every user is affected simultaneously.

**Wrong:**

```hcl
resource "aws_appconfig_deployment" "prod" {
  # ...
  deployment_strategy_id = aws_appconfig_deployment_strategy.instant.id  # BAD for prod
}
```

**Fix -- use a gradual strategy:**

```hcl
resource "aws_appconfig_deployment" "prod" {
  # ...
  deployment_strategy_id = aws_appconfig_deployment_strategy.linear.id
}
```

### 3. Hardcoding AppConfig identifiers in Lambda code

Passing the AppConfig application name, environment, and profile as hardcoded strings in the Lambda code means you must redeploy the function to change environments.

**Wrong:**

```python
url = "http://localhost:2772/applications/my-app/environments/dev/configurations/flags"
```

**Fix -- use environment variables:**

```python
url = f"http://localhost:2772/applications/{os.environ['APPCONFIG_APP']}/environments/{os.environ['APPCONFIG_ENV']}/configurations/{os.environ['APPCONFIG_PROFILE']}"
```

## Summary

- **AWS AppConfig** provides managed feature flag infrastructure with deployment strategies, validation, and automatic rollback
- The **AppConfig Lambda extension** (layer) caches configuration locally, avoiding API calls on every invocation
- **Deployment strategies** control rollout speed: `AllAtOnce` for dev, linear or canary for production
- Feature flags **decouple deployment from release** -- you can deploy code and enable features independently
- Configuration profiles with **type `AWS.AppConfig.FeatureFlags`** provide built-in schema support for boolean and variant flags
- AppConfig is **free for up to 1000 configuration retrievals per minute**, making it cost-effective for most workloads
