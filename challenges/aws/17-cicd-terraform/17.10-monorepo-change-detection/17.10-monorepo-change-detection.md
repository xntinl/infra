# 17.10 Deteccion de Cambios en Monorepo

## Metadata
| Campo | Valor |
|-------|-------|
| **Complejidad** | Avanzado |
| **Modalidad** | Terraform + GitHub Actions |
| **Features TF** | Project isolation, path-based triggers |

## Descripcion

En un monorepo con multiples proyectos de Terraform, detectar cuales proyectos cambiaron en un PR y solo hacer plan/apply de esos. Usar filtros por path, grafos de dependencia entre proyectos y caching inteligente para minimizar el tiempo de CI.

## Objetivos de aprendizaje

- Aplicar patrones de monorepo para proyectos de Terraform
- Implementar deteccion de cambios basada en paths
- Disenar CI consciente de dependencias entre proyectos
- Optimizar builds con caching y ejecucion selectiva

## Requisitos previos

- Conocimientos: Monorepos, GitHub Actions (path filters, matrices), dependencias entre modulos Terraform
- Herramientas: Terraform CLI, GitHub Actions, dorny/paths-filter o similar
- Features: Path-based triggers, aislamiento de proyectos, caching de providers

## Arquitectura

El monorepo contiene multiples proyectos Terraform independientes, cada uno en su propio directorio. Un grafo de dependencias define las relaciones entre proyectos (ej: `networking` -> `compute`). El pipeline detecta los archivos cambiados, identifica los proyectos afectados (incluyendo dependientes) y solo ejecuta plan/apply para esos.

```
Monorepo
  |
  +-- projects/
  |     +-- networking/    (VPC, subnets)
  |     +-- security/      (IAM, SGs)
  |     +-- compute/       (EC2, ECS) -- depende de networking, security
  |     +-- database/      (RDS)      -- depende de networking, security
  |     +-- monitoring/    (CloudWatch)
  |
  +-- modules/             (modulos compartidos)
  |
  +-- dependency-graph.yaml

PR cambia: projects/networking/ y modules/vpc/
  -> Detectar: networking (directo), compute y database (dependientes)
  -> Plan/Apply solo esos 3 proyectos
```

## Pasos sugeridos

1. Estructurar el monorepo con directorios separados por proyecto Terraform
2. Crear un archivo de grafo de dependencias (`dependency-graph.yaml` o similar)
3. Implementar deteccion de archivos cambiados usando `dorny/paths-filter` o `git diff`
4. Crear un script que, dado los archivos cambiados, determine los proyectos afectados
5. Resolver dependencias transitivas: si cambia `networking`, incluir `compute` y `database`
6. Considerar cambios en modulos compartidos: si cambia `modules/vpc/`, afecta a todos los que lo usan
7. Generar una matrix dinamica con los proyectos a planificar/aplicar
8. Implementar caching de providers y modulos de Terraform para acelerar los builds
9. Ordenar los applies respetando el grafo de dependencias (networking antes que compute)
10. Probar con diferentes escenarios: cambio en un proyecto, en un modulo compartido, en multiples proyectos

## Validacion

- [ ] Solo se planifican los proyectos con archivos cambiados
- [ ] Las dependencias transitivas se resuelven correctamente
- [ ] Cambios en modulos compartidos disparan plan en todos los proyectos que los usan
- [ ] Los applies se ejecutan en el orden correcto segun dependencias
- [ ] El caching reduce significativamente el tiempo de CI
- [ ] Proyectos sin cambios no se planifican ni aplican
- [ ] El pipeline escala con nuevos proyectos sin modificar la configuracion de CI
