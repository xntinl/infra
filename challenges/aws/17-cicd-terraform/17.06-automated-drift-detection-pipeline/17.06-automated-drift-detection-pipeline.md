# 17.06 Pipeline de Deteccion Automatica de Drift

## Metadata
| Campo | Valor |
|-------|-------|
| **Complejidad** | Avanzado |
| **Modalidad** | Terraform + GitHub Actions/EventBridge |
| **Features TF** | `terraform plan -detailed-exitcode`, plan JSON |

## Descripcion

Construir un pipeline programado (diario/semanal) que ejecute `terraform plan` en todos los workspaces y reporte drift. Parsear el JSON del plan para clasificar la severidad del drift. Enviar alertas por Slack/email con detalles. Rastrear metricas de drift a lo largo del tiempo.

## Objetivos de aprendizaje

- Implementar deteccion de drift automatizada para infraestructura gestionada con Terraform
- Parsear y analizar el output JSON de `terraform plan`
- Disenar monitoreo automatizado de infraestructura
- Aplicar buenas practicas de higiene operacional

## Requisitos previos

- Conocimientos: Terraform state y plan, GitHub Actions scheduled workflows (cron), JSON parsing
- Herramientas: Terraform CLI, jq, GitHub Actions, Slack/email para notificaciones
- Features: `terraform plan -detailed-exitcode`, `terraform show -json`

## Arquitectura

Un workflow programado (cron) ejecuta `terraform plan` periodicamente contra cada workspace. El exit code y el output JSON se analizan para determinar si hay drift y su severidad. Los resultados se reportan via notificaciones y se almacenan como metricas historicas.

```
Cron Schedule (diario/semanal)
  |
  v
Para cada workspace:
  terraform plan -detailed-exitcode
  |
  +-- Exit code 0: sin cambios
  +-- Exit code 2: drift detectado
        |
        v
  terraform show -json -> Clasificar severidad
        |
        v
  Alertas (Slack/Email) + Metricas
```

## Pasos sugeridos

1. Crear un workflow de GitHub Actions con trigger `schedule` (cron)
2. Definir la lista de workspaces/proyectos a escanear
3. Implementar un loop que ejecute `terraform plan -detailed-exitcode` por cada workspace
4. Capturar el exit code: 0 = sin cambios, 1 = error, 2 = drift detectado
5. Para los que tienen drift, generar el JSON: `terraform show -json tfplan`
6. Parsear el JSON con `jq` para extraer recursos afectados y tipo de cambio
7. Clasificar severidad: destrucciones (critico), modificaciones (alto), creaciones inesperadas (medio)
8. Enviar alerta a Slack con resumen: workspace, recursos afectados, severidad
9. Almacenar resultados (ej: como artefacto o en S3) para tracking historico
10. Crear un dashboard o reporte que muestre tendencias de drift

## Validacion

- [ ] El pipeline se ejecuta automaticamente segun el schedule configurado
- [ ] Detecta correctamente cuando hay drift (exit code 2)
- [ ] El parseo del JSON identifica los recursos especificos con drift
- [ ] La clasificacion de severidad es correcta (destrucciones vs modificaciones)
- [ ] Las alertas de Slack/email contienen informacion accionable
- [ ] Se mantiene un historial de detecciones de drift
- [ ] El pipeline no falla si un workspace tiene errores (manejo graceful)
