# 17.01 GitHub Actions Pipeline con OIDC Auth

## Metadata
| Campo | Valor |
|-------|-------|
| **Complejidad** | Intermedio |
| **Modalidad** | Terraform + GitHub Actions |
| **Features TF** | Plan output, OIDC provider |

## Descripcion

Construir un pipeline de GitHub Actions para Terraform que se autentique en AWS usando OIDC (sin credenciales de larga duracion). Implementar: plan en PR, comentar el output del plan en el PR, apply al hacer merge a main, jobs separados por entorno.

## Objetivos de aprendizaje

- Comprender la federacion OIDC entre GitHub y AWS
- Implementar autenticacion keyless (sin credenciales de larga duracion)
- Disenar workflows basados en PR con visualizacion del plan
- Configurar jobs separados por entorno en GitHub Actions

## Requisitos previos

- Conocimientos: GitHub Actions (workflows, jobs, steps), Terraform basico, IAM roles y trust policies
- Herramientas: GitHub (repositorio con Actions habilitado), Terraform CLI, AWS CLI
- Servicios: AWS IAM (OIDC Provider, Roles), GitHub Actions OIDC token

## Arquitectura

El pipeline se estructura en dos workflows principales:

1. **On Pull Request**: ejecuta `terraform plan`, captura el output y lo publica como comentario en el PR para revision del equipo.
2. **On Merge to Main**: ejecuta `terraform apply` automaticamente tras el merge, con jobs separados para cada entorno (dev, staging, prod) usando matrices o jobs secuenciales.

La autenticacion se realiza mediante un OIDC provider configurado en AWS que confia en los tokens JWT emitidos por GitHub Actions, eliminando la necesidad de access keys.

## Pasos sugeridos

1. Crear el recurso `aws_iam_openid_connect_provider` en Terraform apuntando a `token.actions.githubusercontent.com`
2. Crear un IAM role con trust policy que restrinja por repositorio y branch
3. Configurar el workflow de GitHub Actions para el evento `pull_request` con el step `aws-actions/configure-aws-credentials` usando OIDC
4. Agregar steps para `terraform init`, `terraform plan` con output capturado
5. Usar `actions/github-script` para comentar el plan como texto en el PR
6. Crear un segundo workflow (o job) para el evento `push` a `main` que ejecute `terraform apply`
7. Separar jobs por entorno (dev, staging, prod) con dependencias entre ellos
8. Probar el flujo completo: crear PR, verificar comentario del plan, hacer merge, verificar apply

## Validacion

- [ ] El workflow se dispara automaticamente al crear un PR
- [ ] El output de `terraform plan` aparece como comentario en el PR
- [ ] No se usan access keys ni secrets de AWS en el repositorio
- [ ] El apply se ejecuta solo al hacer merge a main
- [ ] Cada entorno tiene su propio job con el IAM role correcto
- [ ] El OIDC provider restringe acceso solo al repositorio autorizado
