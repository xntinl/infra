# 17.09 Pipeline de Rollback para Applies Fallidos

## Metadata
| Campo | Valor |
|-------|-------|
| **Complejidad** | Experto |
| **Modalidad** | Terraform + CI/CD |
| **Features TF** | State versioning, `terraform plan -target`, `-refresh-only` |

## Descripcion

Implementar una estrategia de rollback: guardar la version anterior del state antes del apply; si el apply falla o los health checks post-apply fallan, revertir automaticamente al state anterior y re-aplicar. Manejar applies parciales de forma elegante.

## Objetivos de aprendizaje

- Disenar estrategias de rollback para infraestructura gestionada con Terraform
- Recuperarse de applies parciales de forma controlada
- Implementar versionamiento de state y restauracion
- Integrar health checks post-deploy en el pipeline

## Requisitos previos

- Conocimientos: Terraform state management avanzado, CI/CD pipelines, health checks, S3 versioning
- Herramientas: Terraform CLI, AWS CLI (para S3 state), GitHub Actions o similar
- Features: S3 backend con versionamiento, `terraform state`, `-refresh-only`, `-target`

## Arquitectura

El pipeline implementa un patron de "checkpoint and rollback": antes de cada apply, se crea un snapshot del state actual. Si el apply falla o los health checks post-deploy no pasan, se restaura el snapshot y se ejecuta un apply con el state anterior para revertir los cambios.

```
Pre-Apply
  |
  +-- Descargar state actual (checkpoint)
  +-- Almacenar como snapshot versionado
  |
  v
Apply
  |
  +-- Exito -> Health Checks
  |              |
  |              +-- Pasan -> Fin (exito)
  |              +-- Fallan -> Rollback
  |
  +-- Fallo -> Rollback
  |
  v
Rollback
  |
  +-- Restaurar state snapshot
  +-- terraform apply (con state anterior)
  +-- Notificar equipo
```

## Pasos sugeridos

1. Configurar el backend S3 con versionamiento habilitado
2. Crear un step pre-apply que descargue y almacene el state actual como artefacto
3. Ejecutar `terraform apply` capturando el exit code
4. Si el apply es exitoso, ejecutar health checks (HTTP endpoints, CloudWatch alarms, etc.)
5. Si el apply falla o los health checks no pasan, iniciar el flujo de rollback
6. En el rollback: restaurar el state snapshot al backend S3
7. Ejecutar `terraform plan -refresh-only` para verificar el state restaurado
8. Ejecutar `terraform apply` con el state restaurado para revertir los recursos
9. Manejar el caso de apply parcial: usar `-target` para los recursos que si se crearon
10. Enviar notificaciones detalladas del rollback con los recursos afectados

## Validacion

- [ ] Se crea un snapshot del state antes de cada apply
- [ ] Un apply fallido dispara automaticamente el rollback
- [ ] Health checks fallidos post-apply disparan el rollback
- [ ] El state se restaura correctamente desde el snapshot
- [ ] El re-apply con state anterior revierte los cambios exitosamente
- [ ] Los applies parciales se manejan sin dejar recursos huerfanos
- [ ] Las notificaciones incluyen detalles del fallo y del rollback
- [ ] El historial de snapshots se mantiene para auditoria
