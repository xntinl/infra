# 17.07 Matrix Strategy para Despliegue Multi-Cuenta

## Metadata
| Campo | Valor |
|-------|-------|
| **Complejidad** | Avanzado |
| **Modalidad** | Terraform + GitHub Actions |
| **Features TF** | Backend config, variable files, OIDC |

## Descripcion

Usar la estrategia de matrix de GitHub Actions para desplegar la misma configuracion de Terraform en multiples cuentas de AWS en paralelo. Configurar roles OIDC por cuenta, tfvars especificos por entorno y agregar resultados. Manejar fallos parciales de forma elegante.

## Objetivos de aprendizaje

- Dominar deployments basados en matrix para multiples cuentas
- Ejecutar applies de Terraform en paralelo de forma segura
- Disenar pipelines CI/CD multi-cuenta con aislamiento adecuado
- Manejar fallos parciales sin afectar otros despliegues

## Requisitos previos

- Conocimientos: GitHub Actions (matrix strategy), Terraform workspaces/backends, AWS Organizations o multiples cuentas
- Herramientas: Terraform CLI, GitHub Actions, AWS CLI
- Features: OIDC provider por cuenta, backend configuration parcial, tfvars por entorno

## Arquitectura

Un unico workflow utiliza matrix strategy para generar jobs paralelos, uno por cada cuenta AWS. Cada job se autentica via OIDC con el rol correspondiente, usa el backend y tfvars de su cuenta, y reporta su resultado independientemente.

```
GitHub Actions Workflow
  |
  +-- Matrix: [dev, staging, prod]
  |
  +-- Job (dev)     -- OIDC Role Dev     -- dev.tfvars
  +-- Job (staging) -- OIDC Role Staging -- staging.tfvars
  +-- Job (prod)    -- OIDC Role Prod    -- prod.tfvars
  |
  +-- Aggregation Job (recopila resultados)
```

## Pasos sugeridos

1. Definir la matrix con las cuentas/entornos y sus parametros (account ID, region, role ARN)
2. Configurar un OIDC provider en cada cuenta AWS con trust policy para el repositorio
3. Crear archivos tfvars especificos por cuenta (dev.tfvars, staging.tfvars, prod.tfvars)
4. Configurar el backend con `-backend-config` para que cada cuenta use su propio state
5. Implementar el workflow con `strategy.matrix` que genere un job por cuenta
6. En cada job: autenticar via OIDC, init con backend correcto, plan y apply
7. Configurar `fail-fast: false` para que un fallo no cancele los demas despliegues
8. Agregar un job de agregacion que recopile los resultados de todos los jobs de matrix
9. Implementar notificaciones con el resumen de exito/fallo por cuenta
10. Probar el manejo de fallos parciales: simular un fallo en una cuenta y verificar que las demas continuan

## Validacion

- [ ] Los jobs se ejecutan en paralelo (uno por cuenta)
- [ ] Cada job usa el OIDC role correcto para su cuenta
- [ ] Los tfvars especificos por cuenta se aplican correctamente
- [ ] Cada cuenta tiene su propio state file aislado
- [ ] Un fallo en una cuenta no cancela los despliegues en otras
- [ ] El job de agregacion reporta correctamente el estado de cada cuenta
- [ ] Las notificaciones incluyen resumen de todos los despliegues
