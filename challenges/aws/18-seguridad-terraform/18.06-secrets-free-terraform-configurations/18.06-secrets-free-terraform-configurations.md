# 18.06 Configuraciones de Terraform Libres de Secretos

## Metadata
| Campo | Valor |
|-------|-------|
| **Complejidad** | Avanzado |
| **Modalidad** | Terraform |
| **Features TF** | `sensitive`, data sources, SOPS provider |

## Descripcion

Refactorizar un codebase de Terraform para eliminar todos los secretos hardcodeados. Usar: data sources para Secrets Manager/SSM, `sensitive = true` para outputs, archivos encriptados con SOPS, y credenciales dinamicas de providers via OIDC. Verificar que no haya secretos en el state ni en el plan.

## Objetivos de aprendizaje

- Aplicar higiene de secretos en proyectos de Terraform
- Usar valores sensibles y protegerlos en outputs y logs
- Comprender la seguridad del state y como proteger datos sensibles
- Integrar SOPS para gestion de secretos encriptados en Git

## Requisitos previos

- Conocimientos: Terraform intermedio, Secrets Manager/SSM Parameter Store, encriptacion KMS, SOPS
- Herramientas: Terraform CLI, SOPS, AWS CLI, KMS
- Features: `sensitive = true`, data sources, SOPS provider para Terraform

## Arquitectura

Los secretos nunca se almacenan en texto plano en el codigo. Se obtienen en runtime via data sources (Secrets Manager, SSM) o se gestionan como archivos encriptados con SOPS. Los outputs sensibles se marcan para evitar su exposicion en logs.

```
Fuentes de secretos:
  |
  +-- AWS Secrets Manager (data source)
  |     data "aws_secretsmanager_secret_version" "db_password" {}
  |
  +-- SSM Parameter Store (data source)
  |     data "aws_ssm_parameter" "api_key" {}
  |
  +-- SOPS (archivos encriptados en Git)
  |     data "sops_file" "secrets" { source_file = "secrets.enc.yaml" }
  |
  +-- OIDC (credenciales dinamicas del provider)
        provider "aws" { # sin access_key/secret_key }

Protecciones:
  +-- sensitive = true (en variables y outputs)
  +-- State encriptado (S3 con KMS)
  +-- Plan output filtrado
```

## Pasos sugeridos

1. Auditar el codebase actual: buscar secretos hardcodeados (passwords, API keys, tokens)
2. Buscar en tfvars, variables con defaults, y locals que contengan valores sensibles
3. Migrar passwords/tokens a AWS Secrets Manager y usar data sources
4. Migrar parametros de configuracion sensibles a SSM Parameter Store
5. Configurar SOPS con KMS key para archivos que necesiten estar en Git encriptados
6. Marcar todas las variables sensibles con `sensitive = true`
7. Marcar todos los outputs que expongan datos sensibles con `sensitive = true`
8. Verificar que el provider AWS no use access keys hardcodeadas (usar OIDC o instance profile)
9. Ejecutar `terraform plan` y verificar que los valores sensibles aparecen como `(sensitive value)`
10. Revisar el state file para confirmar que los secretos estan protegidos (state encriptado en S3)

## Validacion

- [ ] No existen secretos hardcodeados en ningun archivo `.tf` o `.tfvars`
- [ ] Los secretos se obtienen via data sources (Secrets Manager, SSM, SOPS)
- [ ] Todas las variables sensibles tienen `sensitive = true`
- [ ] Todos los outputs sensibles tienen `sensitive = true`
- [ ] `terraform plan` muestra `(sensitive value)` en lugar de los secretos
- [ ] El provider AWS no usa access keys hardcodeadas
- [ ] El state se almacena encriptado (S3 con KMS)
- [ ] Los archivos SOPS se pueden desencriptar solo con la KMS key autorizada
- [ ] `git log` no contiene secretos en texto plano
