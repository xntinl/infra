# 18.05 IAM de Minimo Privilegio para Ejecucion de Terraform

## Metadata
| Campo | Valor |
|-------|-------|
| **Complejidad** | Avanzado |
| **Modalidad** | Terraform |
| **Servicios AWS** | IAM, IAM Access Analyzer |

## Descripcion

Determinar los permisos IAM minimos necesarios para la configuracion de Terraform. Usar IAM Access Analyzer para identificar los permisos requeridos. Crear una IAM policy acotada en lugar de AdministratorAccess. Probar que plan y apply funcionan con la policy restringida.

## Objetivos de aprendizaje

- Aplicar el principio de minimo privilegio a la ejecucion de IaC
- Usar IAM Access Analyzer para analisis de permisos
- Crear roles de ejecucion acotados para Terraform
- Iterar sobre permisos hasta lograr el set minimo funcional

## Requisitos previos

- Conocimientos: IAM policies en profundidad (actions, resources, conditions), Terraform state operations, Access Analyzer
- Herramientas: Terraform CLI, AWS CLI, IAM Access Analyzer
- Servicios: AWS IAM, IAM Access Analyzer, CloudTrail

## Arquitectura

En lugar de usar AdministratorAccess para ejecutar Terraform (practica comun pero insegura), se crea un IAM role con exactamente los permisos necesarios. Se usa IAM Access Analyzer y CloudTrail para identificar las acciones API reales que Terraform invoca.

```
Enfoque actual (inseguro):
  Terraform -> AdministratorAccess -> Todos los servicios

Enfoque objetivo (minimo privilegio):
  Terraform -> IAM Role con policy acotada
                  |
                  +-- S3: GetObject, PutObject (state)
                  +-- DynamoDB: GetItem, PutItem (lock)
                  +-- EC2: RunInstances, DescribeInstances, ...
                  +-- Lambda: CreateFunction, UpdateFunctionCode, ...
                  +-- (solo las acciones necesarias)

Proceso de descubrimiento:
  CloudTrail logs -> Access Analyzer -> Policy generada -> Refinamiento manual
```

## Pasos sugeridos

1. Habilitar CloudTrail si no esta activo (para capturar las llamadas API)
2. Ejecutar `terraform plan` y `terraform apply` con AdministratorAccess (baseline)
3. Usar IAM Access Analyzer policy generation para analizar los logs de CloudTrail
4. Generar una policy basada en las acciones API realmente usadas
5. Refinar la policy: agregar permisos para el backend (S3, DynamoDB) y state operations
6. Crear el IAM role con la policy acotada
7. Probar `terraform plan` con el role restringido — corregir errores de acceso denegado
8. Probar `terraform apply` con el role restringido — agregar permisos faltantes
9. Probar `terraform destroy` y operaciones de state (import, mv)
10. Documentar la policy final y el proceso de mantenimiento cuando se agregan nuevos recursos

## Validacion

- [ ] No se usa AdministratorAccess para la ejecucion de Terraform
- [ ] La policy IAM contiene solo los permisos necesarios
- [ ] `terraform plan` funciona correctamente con la policy restringida
- [ ] `terraform apply` crea todos los recursos sin errores de permisos
- [ ] `terraform destroy` funciona correctamente
- [ ] Las operaciones de state (backend S3, lock DynamoDB) funcionan
- [ ] IAM Access Analyzer confirma que no hay permisos excesivos
- [ ] La policy esta documentada y es mantenible
