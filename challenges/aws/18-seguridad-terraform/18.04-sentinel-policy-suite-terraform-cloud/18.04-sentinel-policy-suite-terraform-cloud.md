# 18.04 Suite de Politicas Sentinel para Terraform Cloud

## Metadata
| Campo | Valor |
|-------|-------|
| **Complejidad** | Avanzado |
| **Modalidad** | Terraform + Sentinel |
| **Features TF** | Sentinel, tfplan/v2, policy testing |

## Descripcion

Escribir una suite completa de politicas Sentinel: limites de costo por workspace, tags obligatorios, solo providers aprobados, sin S3 buckets publicos, volumenes EBS encriptados, Lambda solo en VPC. Probar las politicas con mocks del Sentinel CLI.

## Objetivos de aprendizaje

- Dominar el lenguaje de politicas Sentinel
- Inspeccionar planes de Terraform programaticamente con tfplan/v2
- Escribir y ejecutar tests con mocks para politicas Sentinel
- Implementar enforcement de politicas a diferentes niveles

## Requisitos previos

- Conocimientos: Terraform Cloud/HCP Terraform, estructura del plan de Terraform, conceptos de policy-as-code
- Herramientas: Sentinel CLI, HCP Terraform (cuenta con Sentinel habilitado), Git
- Features: Sentinel policies, tfplan/v2 import, mock data, policy testing

## Arquitectura

Las politicas Sentinel se organizan en una suite dentro de un repositorio dedicado. Cada politica tiene su test con mock data. El policy set se vincula a los workspaces de HCP Terraform y se evalua automaticamente en cada run.

```
sentinel-policies/
  |
  +-- policies/
  |     +-- mandatory-tags.sentinel
  |     +-- approved-providers.sentinel
  |     +-- no-public-s3.sentinel
  |     +-- encrypted-ebs.sentinel
  |     +-- lambda-vpc-only.sentinel
  |     +-- cost-limits.sentinel
  |
  +-- test/
  |     +-- mandatory-tags/
  |     |     +-- pass.hcl  (mock: recursos con tags)
  |     |     +-- fail.hcl  (mock: recursos sin tags)
  |     +-- ...
  |
  +-- sentinel.hcl (configuracion de politicas y niveles)

HCP Terraform Run:
  Plan -> Sentinel Evaluation -> Apply (si pasa)
```

## Pasos sugeridos

1. Instalar Sentinel CLI: `brew install hashicorp/tap/sentinel`
2. Crear la estructura del repositorio de politicas
3. Escribir `mandatory-tags.sentinel`: validar que todos los recursos tengan tags especificos
4. Escribir `approved-providers.sentinel`: restringir a providers autorizados (aws, random, null)
5. Escribir `no-public-s3.sentinel`: prohibir S3 buckets con acceso publico
6. Escribir `encrypted-ebs.sentinel`: requerir encriptacion en volumenes EBS
7. Escribir `lambda-vpc-only.sentinel`: requerir configuracion VPC en funciones Lambda
8. Escribir `cost-limits.sentinel`: establecer limites de costo usando cost estimation data
9. Crear mock data y tests para cada politica (escenarios pass y fail)
10. Ejecutar tests localmente: `sentinel test`
11. Configurar el policy set en HCP Terraform con los niveles apropiados

## Validacion

- [ ] Cada politica tiene al menos un test de pass y un test de fail
- [ ] `sentinel test` pasa exitosamente para todas las politicas
- [ ] La politica de tags bloquea recursos sin tags obligatorios
- [ ] La politica de providers rechaza providers no autorizados
- [ ] La politica de S3 bloquea buckets con acceso publico
- [ ] La politica de EBS requiere encriptacion
- [ ] La politica de Lambda requiere configuracion VPC
- [ ] Los limites de costo funcionan correctamente con cost estimation
- [ ] Los niveles de enforcement (advisory, soft-mandatory, hard-mandatory) estan bien asignados
