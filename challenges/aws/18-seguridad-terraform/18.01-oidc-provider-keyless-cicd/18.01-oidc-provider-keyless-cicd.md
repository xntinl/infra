# 18.01 OIDC Provider para CI/CD sin Credenciales

## Metadata
| Campo | Valor |
|-------|-------|
| **Complejidad** | Intermedio |
| **Modalidad** | Terraform |
| **Servicios AWS** | IAM OIDC Provider, IAM Roles, STS |

## Descripcion

Configurar un AWS IAM OIDC provider para GitHub Actions. Crear un IAM role con una trust policy que restrinja a repositorios y branches especificos. Eliminar todas las credenciales de larga duracion de AWS en CI. Probar con workflows de plan y apply.

## Objetivos de aprendizaje

- Comprender la federacion OIDC y como funciona entre GitHub y AWS
- Implementar autenticacion keyless (sin credenciales persistentes)
- Disenar trust policies granulares para controlar acceso por repo y branch
- Configurar CI/CD libre de credenciales

## Requisitos previos

- Conocimientos: IAM roles y policies, STS AssumeRole, JWT tokens, GitHub Actions basico
- Herramientas: Terraform CLI, AWS CLI, GitHub Actions
- Servicios: AWS IAM, AWS STS

## Arquitectura

GitHub Actions emite un JWT token OIDC en cada workflow run. AWS IAM OIDC provider valida este token y permite que el workflow asuma un IAM role especifico via STS. La trust policy del role define exactamente que repositorios y branches pueden asumir el role.

```
GitHub Actions Workflow
  |
  +-- Solicita OIDC token a GitHub
  |
  v
AWS IAM OIDC Provider
  |
  +-- Valida JWT (issuer, audience, claims)
  |
  v
STS AssumeRoleWithWebIdentity
  |
  +-- Trust Policy evalua claims (repo, branch, environment)
  |
  v
Credenciales temporales (15 min - 1 hora)
  |
  v
Terraform Plan / Apply
```

## Pasos sugeridos

1. Crear el recurso `aws_iam_openid_connect_provider` con el thumbprint de GitHub
2. Crear un IAM role con trust policy que referencie al OIDC provider
3. Configurar la condicion `sub` en la trust policy para restringir por repositorio
4. Agregar condiciones adicionales para restringir por branch (ej: solo `main` puede hacer apply)
5. Adjuntar las policies de permisos necesarias al role (principio de minimo privilegio)
6. Crear un workflow de GitHub Actions que use `aws-actions/configure-aws-credentials` con OIDC
7. Probar `terraform plan` desde un PR (branch feature)
8. Probar `terraform apply` desde main
9. Verificar que un repositorio no autorizado no puede asumir el role
10. Eliminar cualquier AWS access key almacenada como secreto en el repositorio

## Validacion

- [ ] El OIDC provider esta configurado correctamente en IAM
- [ ] El IAM role solo puede ser asumido por el repositorio autorizado
- [ ] La restriccion por branch funciona (ej: apply solo desde main)
- [ ] No existen AWS access keys como secretos en el repositorio
- [ ] `terraform plan` funciona correctamente con credenciales OIDC
- [ ] `terraform apply` funciona correctamente con credenciales OIDC
- [ ] Un intento desde un repo no autorizado es rechazado por STS
- [ ] Las credenciales temporales tienen una duracion apropiada
