# 18.02 Escaneo y Remediacion con tfsec/Trivy

## Metadata
| Campo | Valor |
|-------|-------|
| **Complejidad** | Intermedio |
| **Modalidad** | Terraform + Trivy |
| **Features TF** | HCL scanning, custom rules |

## Descripcion

Ejecutar tfsec (ahora parte de Trivy) contra el codebase de Terraform. Corregir todos los hallazgos de severidad critica y alta. Implementar reglas personalizadas para requisitos especificos de la organizacion. Integrar en CI con umbrales de fallo por severidad.

## Objetivos de aprendizaje

- Dominar el analisis estatico para Infrastructure as Code
- Identificar y corregir misconfiguraciones de seguridad comunes
- Crear reglas personalizadas para requisitos organizacionales
- Integrar escaneo de seguridad en el flujo de CI/CD

## Requisitos previos

- Conocimientos: Terraform intermedio, conceptos de seguridad en la nube (encriptacion, acceso publico, permisos)
- Herramientas: Trivy (incluye tfsec), Terraform CLI, CI/CD pipeline
- Features: Escaneo HCL, reglas personalizadas en Rego o YAML

## Arquitectura

Trivy analiza los archivos HCL de Terraform de forma estatica (sin necesidad de plan o apply) y detecta misconfiguraciones de seguridad. Los hallazgos se clasifican por severidad y se pueden filtrar para fallar el CI solo en niveles criticos.

```
Codigo Terraform (HCL)
  |
  v
Trivy Scanner
  |
  +-- Reglas built-in (cientos de checks)
  +-- Reglas personalizadas (Rego/YAML)
  |
  v
Reporte de Hallazgos
  |
  +-- CRITICAL -> Fallo inmediato
  +-- HIGH     -> Fallo (configurable)
  +-- MEDIUM   -> Warning
  +-- LOW      -> Informativo
  |
  v
CI Gate (pass/fail basado en umbral)
```

## Pasos sugeridos

1. Instalar Trivy y ejecutar un escaneo inicial: `trivy config .`
2. Revisar el reporte completo de hallazgos, agrupados por severidad
3. Priorizar y corregir todos los hallazgos CRITICAL (ej: S3 publico, security groups abiertos)
4. Corregir hallazgos HIGH (ej: encriptacion faltante, logging deshabilitado)
5. Para hallazgos que sean falsos positivos, agregar `#trivy:ignore` con justificacion documentada
6. Crear reglas personalizadas en Rego para requisitos de la organizacion (ej: naming conventions)
7. Configurar un archivo `.trivyignore` para exclusiones permanentes justificadas
8. Integrar en CI: `trivy config --severity CRITICAL,HIGH --exit-code 1 .`
9. Generar reportes en formato SARIF para integracion con GitHub Security tab

## Validacion

- [ ] El escaneo inicial se ejecuta sin errores
- [ ] Todos los hallazgos CRITICAL estan corregidos o justificadamente ignorados
- [ ] Todos los hallazgos HIGH estan corregidos o justificadamente ignorados
- [ ] Las reglas personalizadas detectan violaciones especificas de la organizacion
- [ ] El CI falla cuando se introduce una misconfiguracion critica o alta
- [ ] Las exclusiones (`#trivy:ignore`) tienen justificacion documentada
- [ ] El reporte SARIF se genera correctamente para GitHub Security
