# 18.10 Seguridad de la Cadena de Suministro de Modulos

## Metadata
| Campo | Valor |
|-------|-------|
| **Complejidad** | Avanzado |
| **Modalidad** | Terraform + CI/CD |
| **Features TF** | `.terraform.lock.hcl`, version constraints, registries |

## Descripcion

Implementar practicas de seguridad para el consumo de modulos: fijar versiones de modulos, verificar checksums de modulos, usar registries privados, escanear modulos de terceros antes de adopcion, implementar un proceso de aprobacion de modulos, y auditar dependencias de modulos.

## Objetivos de aprendizaje

- Comprender la seguridad de la cadena de suministro aplicada a IaC
- Gestionar lock files y checksums para integridad de dependencias
- Configurar y usar registries privados de modulos
- Implementar un proceso de evaluacion y aprobacion de modulos externos

## Requisitos previos

- Conocimientos: Terraform modules (sources, versioning), registries, conceptos de supply chain security
- Herramientas: Terraform CLI, registry privado (HCP Terraform, S3, Artifactory), CI/CD pipeline
- Features: `.terraform.lock.hcl`, version constraints, module sources

## Arquitectura

La seguridad de la cadena de suministro se implementa en multiples capas: version pinning, verificacion de integridad via lock files, registry privado como unica fuente autorizada, y un proceso de aprobacion antes de adoptar modulos externos.

```
Modulo Externo (Terraform Registry / GitHub)
  |
  v
Proceso de Evaluacion
  +-- Revision de codigo
  +-- Escaneo de seguridad (trivy, checkov)
  +-- Evaluacion de mantenimiento (actividad, issues)
  |
  v
Aprobacion -> Fork / Mirror en Registry Privado
  |
  v
Registry Privado (HCP Terraform / S3 / Artifactory)
  |
  v
Consumo en Proyectos
  +-- Version pinning: version = "= 2.1.0" (exacta)
  +-- Lock file: .terraform.lock.hcl (checksums)
  +-- CI verificacion: terraform init -lock (verificar integridad)
  |
  v
Auditoria Periodica
  +-- Dependencias desactualizadas
  +-- CVEs conocidos
  +-- Modulos abandonados
```

## Pasos sugeridos

1. Auditar las dependencias actuales: listar todos los modulos externos usados y sus versiones
2. Implementar version pinning estricto: cambiar `~>` por `=` para versiones exactas en produccion
3. Generar y commitear el archivo `.terraform.lock.hcl` con `terraform init`
4. Configurar CI para verificar integridad: `terraform init` con lock file existente
5. Configurar un registry privado (HCP Terraform private registry, S3 backend, o Artifactory)
6. Crear un proceso documentado de evaluacion de modulos externos (checklist de seguridad)
7. Migrar modulos externos criticos al registry privado (fork o mirror)
8. Escanear modulos de terceros con trivy/checkov antes de adoption
9. Implementar alertas para modulos desactualizados o con CVEs conocidos
10. Crear un workflow de CI que detecte cambios en dependencias y requiera aprobacion

## Validacion

- [ ] Todos los modulos tienen version pinning estricto
- [ ] El archivo `.terraform.lock.hcl` esta commiteado y verificado en CI
- [ ] El registry privado esta configurado y contiene los modulos aprobados
- [ ] Existe un proceso documentado de evaluacion de modulos externos
- [ ] Los modulos de terceros se escanean antes de su adopcion
- [ ] Los cambios en dependencias requieren aprobacion explicita
- [ ] `terraform init` falla si los checksums no coinciden con el lock file
- [ ] Se monitorean actualizaciones de seguridad para modulos usados
