## Learning Objectives

By the end of this exercise you will be able to:

1. **Design** a permission boundary that constrains delegated administrators from escalating privileges
2. **Implement** IAM condition keys (`iam:PermissionsBoundary`, `aws:RequestedRegion`) to enforce governance guardrails
3. **Construct** a delegated admin role that can create child roles only within a defined boundary
4. **Justify** why permission boundaries must explicitly deny `iam:*` write actions to prevent privilege escalation
5. **Evaluate** the effective permissions of a role using the IAM policy evaluation logic

## Why This Matters

In organizations with multiple development teams, a central security team cannot review every IAM role request. Permission boundaries solve this by letting you delegate role creation while capping the maximum permissions any created role can have. Without boundaries, a developer who can call `iam:CreateRole` and `iam:AttachRolePolicy` could grant themselves `AdministratorAccess` -- a classic privilege escalation. The boundary acts as a ceiling: even if the identity policy grants `s3:*`, the effective permissions are the intersection of the identity policy and the boundary. Combined with condition keys like `iam:PermissionsBoundary`, you can require that every role a delegated admin creates must carry the boundary, closing the escalation loop entirely.

This pattern is the foundation of AWS multi-team governance. AWS Organizations SCPs work at the account level, but permission boundaries work at the role level -- giving you fine-grained control within a single account.

```hcl
# Requirements:

#   modifying IAM, which would let it remove its own boundary.

# TODO 2 -- Delegated Admin Role (Trust Policy)
# =======================================================

#   - Create a data "aws_iam_policy_document" named "admin_trust"
#     that allows the current account root to assume the role

#     with Action = "sts:AssumeRole"

#   - Tag with Name = "${var.project_name}-delegated-admin"
#
# Docs:
#   https://registry.terraform.io/providers/hashicorp/aws/latest/docs/resources/iam_role


# =======================================================
# TODO 3 -- Delegated Admin Permissions
# =======================================================
# Requirements:
#   - Create a data "aws_iam_policy_document" named "admin_permissions"
#   - Statement 1 (Allow): permit iam:CreateRole, iam:DeleteRole,
#     iam:AttachRolePolicy, iam:DetachRolePolicy, iam:PutRolePolicy,
#     iam:DeleteRolePolicy, iam:TagRole, iam:PassRole,
#     iam:PutRolePermissionsBoundary
#     on "arn:${local.partition}:iam::${local.account_id}:role/${var.project_name}-*"
#     with Condition: StringEquals on iam:PermissionsBoundary
#     equal to aws_iam_policy.boundary.arn
#     (this forces every created role to carry the boundary)
#   - Statement 2 (Allow): permit iam:CreatePolicy, iam:DeletePolicy,
#     iam:GetPolicy, iam:GetPolicyVersion, iam:ListPolicyVersions,
#     iam:CreatePolicyVersion, iam:DeletePolicyVersion
#     on "arn:${local.partition}:iam::${local.account_id}:policy/${var.project_name}-*"
#   - Statement 3 (Allow): permit iam:GetRole, iam:ListRoles,
#     iam:ListRolePolicies, iam:ListAttachedRolePolicies,
#     iam:GetRolePolicy on all resources (read-only IAM)
#   - Statement 4 (Deny): deny iam:DeleteRolePermissionsBoundary
#     on all resources (prevent removing the boundary)
#   - Create aws_iam_role_policy "delegated_admin" attaching
#     the document to the delegated_admin role
#
# Docs:
#   https://docs.aws.amazon.com/IAM/latest/UserGuide/reference_policies_iam-condition-keys.html#ck_PermissionsBoundary
#
# Hint: the condition on iam:PermissionsBoundary ensures that
#   CreateRole and AttachRolePolicy only succeed when the
#   target role has the boundary attached.
```

> **Best Practice:** Permission boundaries must explicitly deny `iam:*` write actions. Without this deny, a bounded role could call `iam:PutRolePolicy` to grant itself broader permissions or `iam:DeleteRolePermissionsBoundary` to remove the boundary entirely. The explicit deny in the boundary closes both escalation paths because deny always wins in IAM policy evaluation.

## Spot the Bug

A colleague wrote this delegated admin policy. It passes `terraform validate` but allows privilege escalation. **What is wrong and why does it matter?**

```hcl
data "aws_iam_policy_document" "admin_permissions" {
  statement {
    effect = "Allow"
    actions = [
      "iam:CreateRole",
      "iam:AttachRolePolicy",
      "iam:PutRolePolicy",
      "iam:PassRole",
    ]
    resources = ["arn:aws:iam::*:role/dev-*"]
    # No condition on iam:PermissionsBoundary   <-- BUG
  }
}
```

<details>
<summary>Explain the bug</summary>

The policy allows `iam:CreateRole` and `iam:AttachRolePolicy` without requiring the `iam:PermissionsBoundary` condition. This means the delegated admin can:

1. **Create a role without any boundary** -- the new role has no ceiling on its permissions.
2. **Attach `AdministratorAccess`** to that unbounded role -- giving it full account access.
3. **Assume the new role** (via `iam:PassRole`) -- escalating from delegated admin to full admin.

The fix is to add a `StringEquals` condition requiring the boundary:

```hcl
data "aws_iam_policy_document" "admin_permissions" {
  statement {
    effect = "Allow"
    actions = [
      "iam:CreateRole",
      "iam:AttachRolePolicy",
      "iam:PutRolePolicy",
      "iam:PassRole",
    ]
    resources = ["arn:aws:iam::*:role/dev-*"]

    condition {
      test     = "StringEquals"
      variable = "iam:PermissionsBoundary"
      values   = [aws_iam_policy.boundary.arn]
    }
  }
}
```

</details>

## Solutions

<details>
<summary>TODO 1 -- Permission Boundary Policy (boundary.tf)</summary>

```hcl
data "aws_iam_policy_document" "boundary" {
  # Allow application services
  statement {
    sid    = "AllowApplicationServices"
    effect = "Allow"
    actions = [
      "s3:*",
      "dynamodb:*",
      "lambda:*",
      "logs:*",
      "cloudwatch:*",
    ]
    resources = ["*"]
  }

  # Allow assuming roles
  statement {
    sid       = "AllowAssumeRole"
    effect    = "Allow"
    actions   = ["sts:AssumeRole"]
    resources = ["*"]
  }

  # Deny all IAM write actions to prevent privilege escalation
  statement {
    sid    = "DenyIAMWrite"
    effect = "Deny"
    actions = [
      "iam:*",
    ]
    resources = ["*"]
  }

  # Deny actions outside allowed regions
  statement {
    sid       = "DenyOutsideAllowedRegions"
    effect    = "Deny"
    actions   = ["*"]
    resources = ["*"]

    condition {
      test     = "StringNotEquals"
      variable = "aws:RequestedRegion"
      values   = var.allowed_regions
    }
  }
}

resource "aws_iam_policy" "boundary" {
  name   = "${var.project_name}-boundary"
  policy = data.aws_iam_policy_document.boundary.json

  tags = { Name = "${var.project_name}-boundary" }
}
```

## Summary

- **Permission boundaries** set the maximum permissions a role can have, regardless of its identity policies
- The effective permissions are the **intersection** of identity policy, boundary, and (if applicable) SCP
- The `iam:PermissionsBoundary` condition key forces delegated admins to attach the boundary to every role they create
- Explicitly **denying `iam:*`** in the boundary prevents bounded roles from modifying IAM and escalating privileges
- `aws:RequestedRegion` conditions restrict operations to approved regions, adding a geographic guardrail
- Always deny `iam:DeleteRolePermissionsBoundary` in the admin policy to prevent boundary removal
