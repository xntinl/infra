# 4.09 CloudTrail Lake y Queries Avanzados

## Metadata
| Campo | Valor |
|-------|-------|
| **Complejidad** | Avanzado |
| **Modalidad** | Manual (CloudTrail Lake es basado en consola/API) |
| **Servicios AWS** | CloudTrail, CloudTrail Lake, Organizations, GuardDuty |

## Descripcion

Configurar CloudTrail con un organization trail que registre management events y data events. Configurar CloudTrail Lake, crear un event data store, y escribir queries SQL para investigar: quien accedio a que recursos, patrones de llamadas API desde IPs especificas.

## Objetivos de aprendizaje

- Comprender la auditoria de seguridad con CloudTrail y los tipos de eventos
- Usar CloudTrail Lake como herramienta de analisis de eventos basada en SQL
- Desarrollar tecnicas de investigacion de incidentes de seguridad
- Escribir queries SQL avanzados para analisis forense de logs

## Requisitos previos

- **Conocimientos**: Fundamentos de CloudTrail (management events vs data events), SQL basico, conceptos de auditoria de seguridad, estructura de eventos CloudTrail
- **Herramientas**: AWS CLI v2, AWS Management Console (CloudTrail Lake), conocimiento de SQL
- **Servicios AWS**: Cuenta con Organizations (para organization trail), permisos para CloudTrail y CloudTrail Lake

## Arquitectura

Un organization trail configurado en la cuenta de management que registra management events de todas las cuentas miembro y data events para servicios clave (S3 GetObject/PutObject, Lambda Invoke). Los logs se almacenan en un bucket S3 centralizado. CloudTrail Lake tiene un event data store que ingesta eventos del trail, permitiendo queries SQL directos sin necesidad de configurar Athena. Las queries investigan patrones como: todas las acciones realizadas por un usuario especifico, llamadas API desde una IP sospechosa, accesos a recursos sensibles, cambios de configuracion de seguridad (IAM, security groups), y correlacion con findings de GuardDuty.

## Pasos sugeridos

1. Crear un organization trail que registre management events de todas las cuentas miembro
2. Habilitar data events en el trail para S3 (operaciones de objetos) y Lambda (invocaciones)
3. Configurar el bucket S3 centralizado con lifecycle policies para los logs
4. Crear un event data store en CloudTrail Lake con retencion de 90 dias
5. Esperar a que CloudTrail Lake ingeste eventos (al menos 1 hora para tener datos suficientes)
6. Escribir queries para investigar: todas las acciones de un usuario especifico en un rango de tiempo
7. Escribir queries para detectar: llamadas API desde IPs no reconocidas
8. Escribir queries para auditar: cambios en IAM (CreateUser, AttachRolePolicy, CreateAccessKey)
9. Escribir queries para analizar: accesos a un bucket S3 sensible (quien, cuando, que operacion)
10. Crear queries de deteccion de anomalias: API calls fuera de horario laboral, acciones desde regiones inusuales

## Validacion

- [ ] El organization trail esta configurado y registra management events de todas las cuentas
- [ ] Los data events para S3 y Lambda estan habilitados y registrandose
- [ ] El event data store de CloudTrail Lake esta creado y recibiendo eventos
- [ ] Se puede consultar todas las acciones de un usuario especifico con SQL
- [ ] Se puede filtrar eventos por IP de origen y detectar IPs sospechosas
- [ ] Se pueden auditar cambios de IAM (creacion de usuarios, roles, policies)
- [ ] Se puede rastrear accesos a recursos S3 especificos
- [ ] Las queries de deteccion de anomalias funcionan (horario, regiones inusuales)
