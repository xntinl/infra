# 4.03 GuardDuty, Security Hub y Remediacion Automatizada

## Metadata
| Campo | Valor |
|-------|-------|
| **Complejidad** | Avanzado |
| **Modalidad** | Ambos |
| **Servicios AWS** | GuardDuty, Security Hub, EventBridge, Lambda, SNS, IAM |

## Descripcion

Habilitar GuardDuty en multiples cuentas (via Organizations). Agregar findings en Security Hub. Construir una regla de EventBridge que dispare una funcion Lambda para remediar automaticamente findings comunes (aislar EC2 comprometidas, deshabilitar access keys expuestas).

## Objetivos de aprendizaje

- Comprender la deteccion de amenazas con GuardDuty y los tipos de findings
- Agregar y correlacionar findings de seguridad en Security Hub
- Implementar respuesta automatizada a incidentes con EventBridge y Lambda
- Disenar flujos de correlacion de eventos de seguridad

## Requisitos previos

- **Conocimientos**: Conceptos de seguridad en la nube, IAM en profundidad, VPC security groups, EventBridge patterns
- **Herramientas**: AWS CLI v2, Terraform >= 1.5, Python/Node.js para Lambda
- **Servicios AWS**: Cuenta de management con Organizations (idealmente), permisos para GuardDuty, Security Hub, EventBridge, Lambda

## Arquitectura

GuardDuty habilitado como detector en multiples cuentas miembro, con la cuenta de seguridad como administrador delegado. Los findings de GuardDuty se envian automaticamente a Security Hub, que tambien agrega findings de otros servicios (IAM Access Analyzer, Config). Una regla de EventBridge captura findings de severidad HIGH y CRITICAL y dispara una funcion Lambda de remediacion. La Lambda implementa acciones segun el tipo de finding: para EC2 comprometida, modifica el security group para aislar la instancia; para access keys expuestas, deshabilita la key y notifica via SNS. Todas las acciones de remediacion se registran en CloudWatch Logs y se notifican al equipo de seguridad via SNS.

## Pasos sugeridos

1. Habilitar GuardDuty en la cuenta principal y configurar como administrador delegado en Organizations
2. Habilitar GuardDuty en las cuentas miembro desde el administrador delegado
3. Habilitar Security Hub y configurar los estandares de seguridad (AWS Foundational Security Best Practices)
4. Verificar que los findings de GuardDuty aparecen en Security Hub
5. Crear la regla de EventBridge que captura findings de GuardDuty con severidad HIGH/CRITICAL
6. Desarrollar la funcion Lambda de remediacion con logica condicional segun el tipo de finding
7. Implementar la accion de aislamiento de EC2 (crear security group restrictivo, reemplazar el existente)
8. Implementar la accion de deshabilitacion de access keys (IAM UpdateAccessKey con Status=Inactive)
9. Configurar SNS para notificar al equipo de seguridad sobre cada remediacion ejecutada
10. Generar findings de prueba usando `CreateSampleFindings` de GuardDuty y verificar el flujo completo

## Validacion

- [ ] GuardDuty esta habilitado en todas las cuentas configuradas
- [ ] Los findings de GuardDuty se agregan correctamente en Security Hub
- [ ] La regla de EventBridge se activa para findings HIGH y CRITICAL
- [ ] La Lambda aisla correctamente una instancia EC2 ante un finding de tipo EC2
- [ ] La Lambda deshabilita access keys ante un finding de tipo IAM
- [ ] Las notificaciones SNS se envian al equipo de seguridad con detalles del finding y la accion tomada
- [ ] Todas las acciones de remediacion se registran en CloudWatch Logs
- [ ] Los findings de prueba (`CreateSampleFindings`) activan correctamente el pipeline
