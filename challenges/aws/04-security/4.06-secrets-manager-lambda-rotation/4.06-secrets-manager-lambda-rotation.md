- **Verify** that rotation succeeds by checking version stages and CloudWatch logs

## Why Secrets Matter

Hardcoded credentials in application code or environment variables are the number one cause of security breaches. Secrets Manager solves this by storing credentials centrally, encrypting them with KMS, and rotating them automatically on a schedule. The rotation process uses staging labels -- `AWSCURRENT` marks the active version that applications retrieve by default, while `AWSPENDING` marks the new version being tested. Only after the Lambda rotation function successfully verifies the new credential does it promote `AWSPENDING` to `AWSCURRENT`. This two-label system ensures applications always get a working credential, even during rotation.

The four-step rotation model (create, set, test, finish) is the same pattern whether you are rotating database passwords, API keys, or OAuth tokens. Understanding it once lets you implement rotation for any credential type.

> **Best Practice:** Secrets Manager rotation uses staging labels to manage credential versions. `AWSCURRENT` always points to the active credential that applications retrieve by default. If any step fails, `AWSCURRENT` remains unchanged and applications are never disrupted.

        SecretId=arn,
        VersionStage="AWSCURRENT",

        RemoveFromVersionId=current_version,

## Common Mistakes

### 1. Forgetting the Lambda permission for Secrets Manager

If you create the rotation configuration without granting Secrets Manager permission to invoke the Lambda, rotation will fail silently.

**Wrong -- missing `aws_lambda_permission`:**

```hcl
resource "aws_secretsmanager_secret_rotation" "this" {
  secret_id           = aws_secretsmanager_secret.this.id
  rotation_lambda_arn = aws_lambda_function.rotation.arn

  rotation_rules {
    automatically_after_days = 30
  }
  # No aws_lambda_permission -- Secrets Manager cannot invoke Lambda
}
```

**What happens:** The rotation configuration is created but when triggered, you see this error in CloudTrail:

```
"errorCode": "AccessDeniedException",
"errorMessage": "Secrets Manager cannot invoke the specified Lambda function."
```

**Fix -- add the Lambda permission:**

```hcl
resource "aws_lambda_permission" "secrets_manager" {
  statement_id  = "AllowSecretsManagerInvocation"
  action        = "lambda:InvokeFunction"
  function_name = aws_lambda_function.rotation.function_name
  principal     = "secretsmanager.amazonaws.com"
  source_arn    = aws_secretsmanager_secret.this.arn
}
```

### 2. Not handling idempotency in createSecret

The rotation Lambda may be invoked multiple times for the same step (retries). If `createSecret` generates a new value on every invocation instead of checking whether `AWSPENDING` already exists, you end up with mismatched credentials.

**Wrong -- always generates a new value:**

```python
def create_secret(arn, token):
    random = secrets_client.get_random_password(PasswordLength=64)
    secrets_client.put_secret_value(
        SecretId=arn,
        ClientRequestToken=token,
        SecretString=random["RandomPassword"],
        VersionStages=["AWSPENDING"],
    )
```

**What happens:** If `createSecret` is retried, a new random value overwrites the one that `setSecret` already applied to the target system. The credential in Secrets Manager no longer matches the target.

**Fix -- check for existing AWSPENDING first:**

```python
def create_secret(arn, token):
    try:
        secrets_client.get_secret_value(
            SecretId=arn, VersionId=token, VersionStage="AWSPENDING"
        )
        return  # Already exists, skip
    except secrets_client.exceptions.ResourceNotFoundException:
        pass
    # Generate and store new value only if AWSPENDING does not exist
```

### 3. Missing the testSecret validation

Skipping the test step means a broken credential gets promoted to `AWSCURRENT`, breaking all applications that depend on it.

**Wrong -- empty testSecret:**

```python
def test_secret(arn, token):
    pass  # "It probably works"
```

**Fix -- always verify the credential against the target system:**

```python
def test_secret(arn, token):
    pending = secrets_client.get_secret_value(
        SecretId=arn, VersionId=token, VersionStage="AWSPENDING"
    )
    # Use the credential to connect to the target system
    # Raise an exception if it fails
```

## Summary

- **Secrets Manager** stores credentials centrally with KMS encryption and automatic rotation
- The **four-step rotation** process (createSecret, setSecret, testSecret, finishSecret) ensures credentials are validated before promotion
- **Staging labels** (`AWSCURRENT`, `AWSPENDING`) provide zero-downtime rotation -- applications always get a working credential
- The rotation Lambda must handle **idempotency** because any step may be retried
- Always **test the new credential** in `testSecret` before promoting it -- skipping validation risks breaking all dependent applications
- The Lambda needs explicit `aws_lambda_permission` for `secretsmanager.amazonaws.com` to invoke it
