# 4.08 Data Perimeter con VPC Endpoints y Conditions

## Metadata
| Campo | Valor |
|-------|-------|
| **Complejidad** | Experto |
| **Modalidad** | Terraform recomendado |
| **Servicios AWS** | VPC Endpoints, S3, IAM, SCPs, Organizations |

## Descripcion

Implementar una estrategia de data perimeter asegurando que el acceso a S3 solo sea permitido desde dentro de la VPC via un VPC Gateway Endpoint. Usar S3 bucket policies con condiciones `aws:SourceVpce`, e implementar controles a nivel de SCP con `aws:PrincipalOrgID` para prevenir exfiltracion de datos.

## Objetivos de aprendizaje

- Comprender los patrones de data perimeter (network perimeter vs identity perimeter)
- Implementar VPC Endpoints como control de perimetro de red
- Dominar el uso de condition keys para seguridad en profundidad
- Disenar estrategias de defense in depth combinando multiples capas de control

## Requisitos previos

- **Conocimientos**: VPC avanzado (endpoints, route tables, NACLs), S3 bucket policies, IAM conditions, SCPs, conceptos de seguridad perimetral
- **Herramientas**: Terraform >= 1.5, AWS CLI v2
- **Servicios AWS**: VPC, S3, IAM, AWS Organizations con SCPs habilitadas

## Arquitectura

Una VPC con subnets privadas (sin Internet Gateway para las subnets de datos). Un VPC Gateway Endpoint para S3 con una endpoint policy que restringe acceso solo a buckets de la organizacion. Las route tables de las subnets privadas incluyen la ruta al endpoint. Los S3 bucket policies usan la condicion `aws:SourceVpce` para permitir acceso solo desde el VPC endpoint especifico, y `aws:PrincipalOrgID` para permitir solo principals de la organizacion. A nivel de SCP, una policy con `aws:PrincipalOrgID` previene que cualquier cuenta miembro acceda a buckets S3 fuera de la organizacion. El resultado es un data perimeter de tres capas: (1) red (VPC endpoint), (2) recurso (bucket policy), (3) identidad (SCP).

## Pasos sugeridos

1. Crear la VPC con subnets privadas (sin ruta a Internet Gateway) y subnets publicas separadas
2. Crear el VPC Gateway Endpoint para S3 y asociarlo a las route tables de las subnets privadas
3. Configurar la endpoint policy del VPC endpoint para permitir acceso solo a buckets de la organizacion
4. Crear un bucket S3 "sensible" con una bucket policy que restrinja acceso usando `aws:SourceVpce`
5. Agregar la condicion `aws:PrincipalOrgID` a la bucket policy para permitir solo principals de la organizacion
6. Crear una SCP que deniegue `s3:PutObject` y `s3:GetObject` a buckets que no sean de la organizacion (usando `aws:ResourceOrgID`)
7. Desplegar una instancia EC2 o Lambda en la subnet privada para probar el acceso
8. Verificar que el acceso al bucket funciona desde la VPC via el endpoint
9. Verificar que el acceso al bucket falla desde fuera de la VPC (por ejemplo, desde la consola AWS o CLI local)
10. Verificar que no se puede acceder a buckets de otras cuentas fuera de la organizacion (exfiltracion prevenida)

## Validacion

- [ ] El VPC Gateway Endpoint para S3 esta creado y asociado a las route tables correctas
- [ ] La endpoint policy restringe acceso solo a buckets de la organizacion
- [ ] El acceso a S3 desde la subnet privada funciona via el VPC endpoint
- [ ] El acceso al bucket "sensible" desde fuera de la VPC es denegado por la bucket policy
- [ ] Solo principals de la organizacion pueden acceder al bucket (`aws:PrincipalOrgID`)
- [ ] La SCP previene que las cuentas miembro accedan a buckets externos a la organizacion
- [ ] Un intento de exfiltracion de datos (copiar a un bucket externo) es bloqueado
- [ ] Las tres capas de control (red, recurso, identidad) funcionan independientemente
