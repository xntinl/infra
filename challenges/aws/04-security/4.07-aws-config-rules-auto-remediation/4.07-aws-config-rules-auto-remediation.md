> **AWS Cost Warning:** This exercise enables AWS Config (~$0.003 per configuration item recorded), creates an S3 bucket for Config history, and uses SSM Automation. Estimated cost: ~$0.02/hr. Remember to run `terraform destroy` when finished to avoid unexpected charges.

## Prerequisites

- Terraform >= 1.7 installed
- AWS CLI configured with a sandbox account
- Basic understanding of S3 bucket policies and security groups

## Learning Objectives

After completing this exercise, you will be able to:

- **Describe** how AWS Config continuously records resource configurations and evaluates compliance
- **Identify** the difference between detective controls (Config Rules) and corrective controls (SSM Automation)
- **Configure** managed Config Rules for common security baselines (S3 encryption, restricted SSH)
- **Apply** SSM Automation documents as auto-remediation actions for non-compliant resources

## Why Compliance Monitoring Matters

Security misconfigurations are the most common cause of cloud breaches. A developer creates an S3 bucket without encryption, another opens SSH to the world in a security group -- and nobody notices until a breach occurs. AWS Config solves this by continuously recording every resource configuration change and evaluating it against rules you define. When a resource is non-compliant, Config can trigger SSM Automation to fix it automatically -- turning a detective control into a corrective one.

This exercise sets up two foundational compliance rules that every AWS account should have: S3 buckets must be encrypted, and security groups must not allow unrestricted SSH access. These are the two most common findings in AWS security audits and the easiest to automate.

### `config-recorder.tf`

```hcl
# S3 bucket to store AWS Config configuration history.
# Config records every resource change as a configuration item
# and delivers snapshots to this bucket periodically.
resource "aws_s3_bucket" "config" {
  bucket = "${var.project_name}-config-${local.account_id}"

  tags = { Name = "${var.project_name}-config-bucket" }
}

resource "aws_s3_bucket_public_access_block" "config" {
  bucket = aws_s3_bucket.config.id

  block_public_acls       = true
  block_public_policy     = true
  ignore_public_acls      = true
  restrict_public_buckets = true
}

# Bucket policy allowing AWS Config service to write to the bucket.
data "aws_iam_policy_document" "config_bucket" {
  statement {
    sid    = "AllowConfigBucketAccess"
    effect = "Allow"
    actions = [
      "s3:GetBucketAcl",
      "s3:ListBucket",
    ]
    resources = [aws_s3_bucket.config.arn]

    principals {
      type        = "Service"
      identifiers = ["config.amazonaws.com"]
    }
  }

  statement {
    sid       = "AllowConfigBucketDelivery"
    effect    = "Allow"
    actions   = ["s3:PutObject"]
    resources = ["${aws_s3_bucket.config.arn}/AWSLogs/${local.account_id}/Config/*"]

    principals {
      type        = "Service"
      identifiers = ["config.amazonaws.com"]
    }

    condition {
      test     = "StringEquals"
      variable = "s3:x-amz-acl"
      values   = ["bucket-owner-full-control"]
    }
  }
}

resource "aws_s3_bucket_policy" "config" {
  bucket = aws_s3_bucket.config.id
  policy = data.aws_iam_policy_document.config_bucket.json
}

# IAM role for the Config recorder.
# Config assumes this role to read resource configurations
# and write to the S3 bucket.
data "aws_iam_policy_document" "config_trust" {
  statement {
    effect  = "Allow"
    actions = ["sts:AssumeRole"]

    principals {
      type        = "Service"
      identifiers = ["config.amazonaws.com"]
    }
  }
}

resource "aws_iam_role" "config" {
  name               = "${var.project_name}-config-role"
  assume_role_policy = data.aws_iam_policy_document.config_trust.json

  tags = { Name = "${var.project_name}-config-role" }
}

resource "aws_iam_role_policy_attachment" "config" {
  role       = aws_iam_role.config.name
  policy_arn = "arn:${local.partition}:iam::aws:policy/service-role/AWS_ConfigRole"
}

# The Config recorder captures configuration changes for all
# supported resource types in the region.
resource "aws_config_configuration_recorder" "this" {
  name     = "${var.project_name}-recorder"
  role_arn = aws_iam_role.config.arn

  recording_group {
    all_supported = true
  }
}

# The delivery channel tells Config where to send snapshots
# and how often.
resource "aws_config_delivery_channel" "this" {
  name           = "${var.project_name}-channel"
  s3_bucket_name = aws_s3_bucket.config.id

  depends_on = [aws_config_configuration_recorder.this]
}

# Start the recorder. Without this, Config is configured
# but not actually recording.
resource "aws_config_configuration_recorder_status" "this" {
  name       = aws_config_configuration_recorder.this.name
  is_enabled = true

  depends_on = [aws_config_delivery_channel.this]
}
```

> **Best Practice:** AWS Config Rules evaluate compliance continuously as resources change. When combined with SSM Automation for remediation, they create a closed-loop system: detect the misconfiguration, fix it automatically, and re-evaluate to confirm compliance. This is far more reliable than periodic manual audits.

### `config-rules.tf`

```hcl
# Rule 1: S3 buckets must have server-side encryption enabled.
# This is an AWS managed rule that checks the bucket encryption
# configuration. Buckets without default encryption are marked
# NON_COMPLIANT.
resource "aws_config_config_rule" "s3_encryption" {
  name = "${var.project_name}-s3-encryption"

  source {
    owner             = "AWS"
    source_identifier = "S3_BUCKET_SERVER_SIDE_ENCRYPTION_ENABLED"
  }

  depends_on = [aws_config_configuration_recorder_status.this]

  tags = { Name = "${var.project_name}-s3-encryption-rule" }
}

# Rule 2: Security groups must not allow unrestricted SSH.
# This managed rule checks whether any security group has an
# ingress rule allowing 0.0.0.0/0 on port 22. Groups with
# open SSH are marked NON_COMPLIANT.
resource "aws_config_config_rule" "restricted_ssh" {
  name = "${var.project_name}-restricted-ssh"

  source {
    owner             = "AWS"
    source_identifier = "INCOMING_SSH_DISABLED"
  }

  depends_on = [aws_config_configuration_recorder_status.this]

  tags = { Name = "${var.project_name}-restricted-ssh-rule" }
}
```

### `remediation.tf`

```hcl
# IAM role for SSM Automation remediation.
# SSM Automation assumes this role to perform the fix actions
# (e.g., enabling S3 encryption, revoking SG rules).
data "aws_iam_policy_document" "remediation_trust" {
  statement {
    effect  = "Allow"
    actions = ["sts:AssumeRole"]

    principals {
      type        = "Service"
      identifiers = ["ssm.amazonaws.com"]
    }
  }
}

resource "aws_iam_role" "remediation" {
  name               = "${var.project_name}-remediation-role"
  assume_role_policy = data.aws_iam_policy_document.remediation_trust.json

  tags = { Name = "${var.project_name}-remediation-role" }
}

# Permissions for the remediation role
data "aws_iam_policy_document" "remediation_permissions" {
  # S3 encryption remediation
  statement {
    sid    = "AllowS3EncryptionRemediation"
    effect = "Allow"
    actions = [
      "s3:PutEncryptionConfiguration",
      "s3:GetEncryptionConfiguration",
      "s3:GetBucketAcl",
    ]
    resources = ["arn:${local.partition}:s3:::*"]
  }

  # Security group remediation
  statement {
    sid    = "AllowSecurityGroupRemediation"
    effect = "Allow"
    actions = [
      "ec2:DescribeSecurityGroups",
      "ec2:RevokeSecurityGroupIngress",
    ]
    resources = ["*"]
  }
}

resource "aws_iam_role_policy" "remediation" {
  name   = "${var.project_name}-remediation-policy"
  role   = aws_iam_role.remediation.id
  policy = data.aws_iam_policy_document.remediation_permissions.json
}

# Auto-remediation for S3 encryption rule.
# When Config detects a bucket without encryption, this triggers
# the AWS-EnableS3BucketEncryption SSM document to enable
# AES-256 server-side encryption.
resource "aws_config_remediation_configuration" "s3_encryption" {
  config_rule_name = aws_config_config_rule.s3_encryption.name
  target_type      = "SSM_DOCUMENT"
  target_id        = "AWS-EnableS3BucketEncryption"

  parameter {
    name           = "BucketName"
    resource_value = "RESOURCE_ID"
  }

  parameter {
    name         = "SSEAlgorithm"
    static_value = "AES256"
  }

  parameter {
    name         = "AutomationAssumeRole"
    static_value = aws_iam_role.remediation.arn
  }

  automatic                  = true
  maximum_automatic_attempts = 3
  retry_attempt_seconds      = 60
}

# Auto-remediation for restricted SSH rule.
# When Config detects a security group with SSH open to 0.0.0.0/0,
# this triggers the AWS-DisablePublicAccessForSecurityGroup SSM
# document to revoke the offending ingress rule.
resource "aws_config_remediation_configuration" "restricted_ssh" {
  config_rule_name = aws_config_config_rule.restricted_ssh.name
  target_type      = "SSM_DOCUMENT"
  target_id        = "AWS-DisablePublicAccessForSecurityGroup"

  parameter {
    name           = "GroupId"
    resource_value = "RESOURCE_ID"
  }

  parameter {
    name         = "AutomationAssumeRole"
    static_value = aws_iam_role.remediation.arn
  }

  automatic                  = true
  maximum_automatic_attempts = 3
  retry_attempt_seconds      = 60
}
```

## Step 3 -- Wait for Config Evaluation

AWS Config evaluates rules asynchronously. Initial evaluation can take 5-10 minutes. You can trigger an immediate evaluation:

```bash
aws configservice start-config-rules-evaluation \
  --config-rule-names \
    "$(terraform output -raw s3_encryption_rule_name)" \
    "$(terraform output -raw restricted_ssh_rule_name)"
```

Wait 2-3 minutes, then check compliance:

```bash
aws configservice get-compliance-details-by-config-rule \
  --config-rule-name "$(terraform output -raw s3_encryption_rule_name)" \
  --compliance-types NON_COMPLIANT \
  --query "EvaluationResults[].{Resource:EvaluationResultIdentifier.EvaluationResultQualifier.ResourceId,Compliance:ComplianceType}" \
  --output table
```

Expected: the non-compliant test bucket appears as `NON_COMPLIANT`.

## Common Mistakes

### 1. Not starting the Config recorder

Creating the Config recorder and delivery channel is not enough -- you must also enable the recorder. Without `aws_config_configuration_recorder_status`, Config is configured but not recording.

**Wrong -- recorder created but not started:**

```hcl
resource "aws_config_configuration_recorder" "this" {
  name     = "my-recorder"
  role_arn = aws_iam_role.config.arn
}

resource "aws_config_delivery_channel" "this" {
  name           = "my-channel"
  s3_bucket_name = aws_s3_bucket.config.id
}

# Missing aws_config_configuration_recorder_status!
```

**What happens:** Config rules show no evaluation results. The Config dashboard says "Recording is off."

**Fix -- add the recorder status:**

```hcl
resource "aws_config_configuration_recorder_status" "this" {
  name       = aws_config_configuration_recorder.this.name
  is_enabled = true

  depends_on = [aws_config_delivery_channel.this]
}
```

### 2. Missing the Config bucket policy

The Config service needs explicit permission to write to the S3 bucket. Without the bucket policy, Config delivery fails silently and configuration history is not stored.

**Wrong -- bucket without policy:**

```hcl
resource "aws_s3_bucket" "config" {
  bucket = "my-config-bucket"
}
# No aws_s3_bucket_policy allowing config.amazonaws.com
```

**What happens:** The delivery channel creation succeeds, but Config cannot deliver configuration snapshots. The bucket stays empty.

**Fix -- add a bucket policy for the Config service:**

```hcl
data "aws_iam_policy_document" "config_bucket" {
  statement {
    effect    = "Allow"
    actions   = ["s3:PutObject"]
    resources = ["${aws_s3_bucket.config.arn}/AWSLogs/*/Config/*"]

    principals {
      type        = "Service"
      identifiers = ["config.amazonaws.com"]
    }
  }
}
```

## Summary

- **AWS Config** continuously records resource configurations and evaluates them against rules
- **Config Rules** are detective controls that mark resources as `COMPLIANT` or `NON_COMPLIANT`
- **Managed rules** (like `S3_BUCKET_SERVER_SIDE_ENCRYPTION_ENABLED` and `INCOMING_SSH_DISABLED`) cover common security baselines with zero custom code
- **SSM Automation documents** act as remediation handlers, fixing non-compliant resources automatically
- The Config recorder must be explicitly **started** with `aws_config_configuration_recorder_status` or no recording occurs
- Auto-remediation closes the loop: detect, fix, re-evaluate -- turning detective controls into corrective ones
