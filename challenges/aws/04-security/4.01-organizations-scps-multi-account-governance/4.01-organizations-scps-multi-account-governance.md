# 4.01 Organizations, SCPs y Gobernanza Multi-Cuenta

## Metadata
| Campo | Valor |
|-------|-------|
| **Complejidad** | Avanzado |
| **Modalidad** | Terraform recomendado |
| **Servicios AWS** | AWS Organizations, SCPs, IAM, CloudTrail |

## Descripcion

Configurar AWS Organizations con una jerarquia de OUs (Security, Workloads/Dev, Workloads/Prod, Sandbox). Implementar SCPs para: denegar la deshabilitacion de CloudTrail, restringir regiones, prevenir acceso publico a S3, requerir MFA para acciones destructivas, y prevenir que las cuentas abandonen la organizacion.

## Objetivos de aprendizaje

- Disenar una estrategia multi-cuenta con AWS Organizations y OUs
- Comprender la logica de evaluacion de SCPs (deny siempre gana)
- Disenar jerarquias de OUs que reflejen la estructura organizacional
- Implementar patrones de guardrails preventivos con SCPs

## Requisitos previos

- **Conocimientos**: IAM policies en profundidad (Effect, Action, Resource, Condition), conceptos de multi-cuenta, JSON policy syntax
- **Herramientas**: Terraform >= 1.5, AWS CLI v2
- **Servicios AWS**: Cuenta de management de AWS Organizations (o capacidad de crear una), al menos 2-3 cuentas miembro

## Arquitectura

Una organizacion AWS con la siguiente jerarquia de OUs: Root -> Security OU (cuenta de security/audit), Root -> Workloads OU -> Dev OU (cuentas de desarrollo), Root -> Workloads OU -> Prod OU (cuentas de produccion), Root -> Sandbox OU (cuentas de experimentacion). SCPs se aplican a diferentes niveles: a nivel de Root (prevenir salir de la org, denegar deshabilitar CloudTrail), a nivel de Workloads OU (restriccion de regiones), a nivel de Prod OU (requerir MFA para acciones destructivas, prevenir S3 public access). La cuenta de management queda fuera de las SCPs por diseno.

## Pasos sugeridos

1. Crear la organizacion AWS (si no existe) y habilitar todas las features
2. Disenar y crear la jerarquia de OUs: Security, Workloads/Dev, Workloads/Prod, Sandbox
3. Mover o crear cuentas miembro en las OUs correspondientes
4. Crear la SCP "deny-leave-org" y adjuntarla al Root (prevenir `organizations:LeaveOrganization`)
5. Crear la SCP "protect-cloudtrail" y adjuntarla al Root (denegar `cloudtrail:StopLogging`, `cloudtrail:DeleteTrail`)
6. Crear la SCP "region-restriction" y adjuntarla a Workloads OU (permitir solo regiones autorizadas)
7. Crear la SCP "deny-s3-public" y adjuntarla a Prod OU (denegar `s3:PutBucketPublicAccessBlock` con condicion)
8. Crear la SCP "require-mfa-destructive" y adjuntarla a Prod OU (requerir MFA para `ec2:TerminateInstances`, `rds:DeleteDBInstance`, etc.)
9. Probar cada SCP desde una cuenta miembro intentando las acciones restringidas
10. Verificar que la cuenta de management no esta afectada por las SCPs

## Validacion

- [ ] La jerarquia de OUs esta creada correctamente (Security, Workloads/Dev, Workloads/Prod, Sandbox)
- [ ] Las cuentas miembro estan asignadas a las OUs correctas
- [ ] Ninguna cuenta miembro puede deshabilitar CloudTrail
- [ ] Ninguna cuenta puede abandonar la organizacion
- [ ] Las cuentas en Workloads OU solo pueden operar en las regiones permitidas
- [ ] Las cuentas en Prod OU no pueden hacer S3 buckets publicos
- [ ] Las acciones destructivas en Prod OU requieren MFA
- [ ] La cuenta de management no esta restringida por las SCPs
