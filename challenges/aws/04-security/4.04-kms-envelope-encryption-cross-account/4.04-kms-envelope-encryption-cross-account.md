## Why This Matters

Every production workload needs encryption at rest, and AWS Key Management Service (KMS) is the control plane for encryption keys across AWS. When you enable S3 server-side encryption with a CMK, every `PutObject` and `GetObject` calls KMS to encrypt or decrypt the data key -- at $0.03 per 10,000 requests. For a bucket handling 100 million objects per month, that is $300 in KMS fees alone. Envelope encryption solves the performance problem (you encrypt data locally with a data key instead of sending it all to KMS), and S3 bucket keys solve the cost problem (they cache a bucket-level data key, reducing KMS calls by up to 99%).

Understanding the KMS key hierarchy -- CMK encrypts data keys, data keys encrypt data -- is essential for designing secure, cost-effective architectures. Key policies are the primary access control for KMS keys and override IAM policies, making them a critical piece of your security architecture that must be designed intentionally.

> **Best Practice:** KMS key rotation should be enabled at minimum on an annual basis. When rotation occurs, AWS creates new backing key material but keeps the old material available for decryption. Existing ciphertexts remain decryptable -- only new encrypt operations use the new key material. S3 bucket keys further reduce cost by caching a derived key for a short period, cutting KMS `GenerateDataKey` calls by up to 99% on high-throughput buckets.

## Spot the Bug

A colleague wrote this KMS key policy. It passes validation but creates a security risk and an operational problem. **What is wrong?**

```hcl
data "aws_iam_policy_document" "key_policy" {
  statement {
    sid    = "AllowLambdaFullAccess"
    effect = "Allow"
    actions = [
      "kms:*",                       # <-- BUG 1
    ]
    resources = ["*"]

    principals {
      type        = "AWS"
      identifiers = [aws_iam_role.lambda.arn]
    }
  }
  # No root account statement                 # <-- BUG 2
}
```

<details>
<summary>Explain the bug</summary>

There are two problems:

1. **`kms:*` grants too much access** -- the Lambda role can now call `kms:ScheduleKeyDeletion`, `kms:DisableKey`, `kms:PutKeyPolicy`, and `kms:CreateGrant`. A compromised Lambda could delete the key (destroying all encrypted data) or modify the key policy to grant access to an attacker's account. The fix is to grant only the actions the Lambda needs: `kms:Decrypt`, `kms:GenerateDataKey`, and `kms:DescribeKey`.

2. **Missing root account statement** -- without `"arn:aws:iam::ACCOUNT:root"` having `kms:*`, you cannot manage the key through IAM policies. If the Lambda role is deleted, no one can access or manage the key. AWS requires at least one key policy statement granting root access as a safety net.

The fix:

```hcl
data "aws_iam_policy_document" "key_policy" {
  statement {
    sid       = "AllowRootFullAccess"
    effect    = "Allow"
    actions   = ["kms:*"]
    resources = ["*"]

    principals {
      type        = "AWS"
      identifiers = ["arn:aws:iam::${local.account_id}:root"]
    }
  }

  statement {
    sid    = "AllowLambdaUseOnly"
    effect = "Allow"
    actions = [
      "kms:Decrypt",
      "kms:GenerateDataKey",
      "kms:DescribeKey",
    ]
    resources = ["*"]

    principals {
      type        = "AWS"
      identifiers = [aws_iam_role.lambda.arn]
    }
  }
}
```

```hcl
resource "aws_kms_key" "this" {

data "aws_iam_policy_document" "key_policy" {
  # Root account full access (safety net)
  statement {
    sid       = "AllowRootFullAccess"
    effect    = "Allow"
    actions   = ["kms:*"]
    resources = ["*"]

    principals {
      type        = "AWS"
      identifiers = ["arn:${local.partition}:iam::${local.account_id}:root"]
    }
  }

  # Allow S3 service to use the key for SSE
  statement {
    sid    = "AllowS3ServiceUse"
    effect = "Allow"
    actions = [
      "kms:GenerateDataKey",
      "kms:Decrypt",
    ]
    resources = ["*"]

    principals {
      type        = "Service"
      identifiers = ["s3.amazonaws.com"]
    }
  }

  # Allow Lambda execution role to use the key
  statement {
    sid    = "AllowLambdaUse"
    effect = "Allow"
    actions = [
      "kms:Decrypt",
      "kms:GenerateDataKey",
      "kms:DescribeKey",
    ]
    resources = ["*"]

    principals {
      type        = "AWS"
      identifiers = [aws_iam_role.lambda.arn]
    }
  }
}
```

<summary>TODO 4 -- Lambda Function (lambda.tf)</summary>

```hcl
data "archive_file" "lambda" {
  type        = "zip"
  output_path = "${path.module}/lambda.zip"

  source {
    content  = <<-PYTHON
import json
import os
import boto3

s3 = boto3.client("s3")
kms = boto3.client("kms")

def handler(event, context):
    bucket = os.environ["BUCKET_NAME"]
    key_id = os.environ["KMS_KEY_ID"]

    # Write an encrypted object
    test_data = json.dumps({"message": "Hello from envelope encryption!"})
    s3.put_object(
        Bucket=bucket,
        Key="test/encrypted-payload.json",
        Body=test_data.encode("utf-8"),
        ServerSideEncryption="aws:kms",
        SSEKMSKeyId=key_id,
    )

    # Read it back (S3 decrypts transparently using KMS)
    response = s3.get_object(Bucket=bucket, Key="test/encrypted-payload.json")
    body = json.loads(response["Body"].read().decode("utf-8"))

    # Verify encryption metadata
    head = s3.head_object(Bucket=bucket, Key="test/encrypted-payload.json")

    return {
        "statusCode": 200,
        "body": json.dumps({
            "decrypted_message": body["message"],
            "encryption": head.get("ServerSideEncryption"),
            "kms_key": head.get("SSEKMSKeyId"),
            "bucket_key_enabled": head.get("BucketKeyEnabled"),
        }),
    }
    PYTHON
    filename = "index.py"
  }
}

resource "aws_lambda_function" "this" {
  function_name    = "${var.project_name}-encrypt-test"
  role             = aws_iam_role.lambda.arn
  handler          = "index.handler"
  runtime          = "python3.12"
  timeout          = 30
  filename         = data.archive_file.lambda.output_path
  source_code_hash = data.archive_file.lambda.output_base64sha256

  environment {
    variables = {
      BUCKET_NAME = aws_s3_bucket.encrypted.id
      KMS_KEY_ID  = aws_kms_key.this.id
    }
  }

  tags = { Name = "${var.project_name}-encrypt-test" }
}

resource "aws_cloudwatch_log_group" "lambda" {
  name              = "/aws/lambda/${var.project_name}-encrypt-test"
  retention_in_days = 7
```

## Summary

- **Envelope encryption** encrypts data with a data key, then encrypts the data key with the CMK -- avoiding sending bulk data to KMS
- **Key policies** are the primary access control for KMS keys and must always include root account access as a safety net
- **S3 bucket keys** cache a derived key from the CMK, reducing `kms:GenerateDataKey` calls by up to 99% on high-throughput buckets
- **Key rotation** creates new backing material annually while keeping old material available for decryption of existing ciphertexts
- The Lambda execution role needs both S3 permissions AND KMS `Decrypt`/`GenerateDataKey` to work with encrypted objects
- Always use `data "aws_iam_policy_document"` for key policies -- never `jsonencode()` -- to benefit from Terraform's validation and interpolation
