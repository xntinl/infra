# 4.10 IAM Identity Center (SSO) con IdP Externo

## Metadata
| Campo | Valor |
|-------|-------|
| **Complejidad** | Avanzado |
| **Modalidad** | Ambos (configuracion manual del IdP + Terraform para el lado AWS) |
| **Servicios AWS** | IAM Identity Center, Organizations, IAM, SAML |

## Descripcion

Configurar IAM Identity Center con un identity provider externo (por ejemplo, Okta o simulado con una aplicacion SAML). Configurar permission sets mapeados a asignaciones basadas en OUs, implementar session policies, y configurar ABAC usando atributos del identity store.

## Objetivos de aprendizaje

- Comprender la identidad federada y como funciona SAML 2.0 con AWS
- Disenar la arquitectura de SSO con IAM Identity Center
- Implementar permission sets y entender como se traducen a IAM roles
- Configurar ABAC con atributos de identidad externa para control de acceso dinamico

## Requisitos previos

- **Conocimientos**: Conceptos de federacion de identidad (SAML, IdP, SP), IAM roles y trust policies, AWS Organizations, conceptos de SSO
- **Herramientas**: Terraform >= 1.5, AWS CLI v2, un IdP externo (Okta dev account gratuito, o JumpCloud, o simulacion SAML)
- **Servicios AWS**: AWS Organizations habilitado, IAM Identity Center habilitado, permisos administrativos

## Arquitectura

IAM Identity Center configurado como Service Provider (SP) conectado a un Identity Provider (IdP) externo via SAML 2.0. El IdP gestiona usuarios y grupos con atributos personalizados (Department, Project, CostCenter). En IAM Identity Center: permission sets definen los permisos (ReadOnlyAccess, PowerUserAccess, AdministratorAccess). Las asignaciones mapean grupos del IdP a permission sets en cuentas/OUs especificos: el grupo "Developers" obtiene PowerUserAccess en cuentas de Dev OU, el grupo "Admins" obtiene AdministratorAccess en todas las cuentas, el grupo "Auditors" obtiene ReadOnlyAccess en todas las cuentas. ABAC usa atributos del IdP (Department) como session tags para que los usuarios solo accedan a recursos con tags coincidentes.

## Pasos sugeridos

1. Crear una cuenta en un IdP externo (Okta dev, JumpCloud, o simular con una SAML test app)
2. Crear usuarios y grupos en el IdP con atributos personalizados (Department, Project)
3. Habilitar IAM Identity Center en la cuenta de management de Organizations
4. Configurar la conexion SAML entre el IdP externo y IAM Identity Center (metadata XML, ACS URL)
5. Configurar el mapeo de atributos SAML para pasar Department y Project como session tags
6. Crear permission sets: ReadOnlyAccess, PowerUserAccess, AdminAccess con sus IAM policies
7. Agregar inline policies a los permission sets para implementar ABAC (condiciones basadas en `aws:PrincipalTag`)
8. Crear asignaciones: grupo Developers -> PowerUserAccess en cuentas de Dev OU, grupo Admins -> AdminAccess en todas las cuentas
9. Probar el login SSO desde el IdP y verificar que el usuario llega a la cuenta correcta con los permisos correctos
10. Probar ABAC: verificar que un usuario del Department "Engineering" solo puede acceder a recursos con tag `Department=Engineering`

## Validacion

- [ ] IAM Identity Center esta conectado al IdP externo via SAML 2.0
- [ ] Los usuarios del IdP pueden hacer login SSO al portal de AWS
- [ ] Los permission sets estan creados con las policies correctas
- [ ] Las asignaciones mapean correctamente grupos a permission sets en las cuentas/OUs apropiadas
- [ ] Un usuario del grupo "Developers" solo tiene acceso a cuentas de Dev con PowerUserAccess
- [ ] Un usuario del grupo "Admins" tiene acceso a todas las cuentas con AdminAccess
- [ ] ABAC funciona: los atributos del IdP se pasan como session tags y condicionan el acceso a recursos
- [ ] La sesion expira segun la duracion configurada en el permission set
