4. **Design** custom rules combining IP sets, geo-blocking, and string match conditions
5. **Evaluate** WAF metrics and sampled requests to tune rules before switching from Count to Block

## Why This Matters

Web applications face a constant stream of automated attacks: credential stuffing, SQL injection, cross-site scripting, and volumetric abuse. AWS WAF inspects every HTTP request before it reaches your application and can block malicious traffic at the edge. Rate-based rules throttle abusive IPs automatically (no Lambda or custom logic needed), while AWS Managed Rule Groups provide battle-tested protection against OWASP Top 10 vulnerabilities without writing regex yourself.

The key to effective WAF deployment is starting in Count mode -- logging matches without blocking -- and analyzing the results before switching to Block. This prevents false positives from breaking legitimate traffic. Combined with IP reputation lists and geo-blocking, WAF gives you layered defense that stops most attacks before they consume application resources.

> **Best Practice:** Always deploy WAF managed rules in Count mode first. Analyze the `SampledRequests` and CloudWatch metrics for at least 24-48 hours of production traffic before switching to Block. The Core Rule Set (CRS) can generate false positives on legitimate requests that contain SQL-like syntax in form fields or URL-encoded payloads. Use rule group `excluded_rules` to disable specific rules within a managed group that cause false positives, rather than disabling the entire group.

## Spot the Bug

A colleague wrote this WAF WebACL. It deploys successfully but the rate limiting never triggers. **What is wrong?**

```hcl
resource "aws_wafv2_web_acl" "this" {
  name  = "my-webacl"
  scope = "REGIONAL"

  default_action { allow {} }

  rule {
    name     = "rate-limit"
    priority = 100            # <-- BUG

    action { block {} }

    statement {
      rate_based_statement {
        limit              = 2000
        aggregate_key_type = "IP"
      }
    }

    visibility_config {
      cloudwatch_metrics_enabled = false   # <-- BUG
      sampled_requests_enabled   = false   # <-- BUG
      metric_name                = "rate-limit"
    }
  }

  rule {
    name     = "aws-managed-core"
    priority = 1

    override_action { none {} }   # <-- BUG

    statement {
      managed_rule_group_statement {
        vendor_name = "AWS"
        name        = "AWSManagedRulesCommonRuleSet"
      }
    }

    visibility_config {
      cloudwatch_metrics_enabled = true
      sampled_requests_enabled   = true
      metric_name                = "aws-managed-core"
    }
  }

  visibility_config {
    cloudwatch_metrics_enabled = true
    sampled_requests_enabled   = true
    metric_name                = "my-webacl"
  }
}
```

<details>
<summary>Explain the bug</summary>

There are three problems:

1. **Rate limit rule at priority 100, managed rules at priority 1** -- WAF evaluates rules in priority order (lowest first). The Core Rule Set at priority 1 runs before the rate limiter at priority 100. If the CRS blocks a request, the rate limiter never sees it and never counts it. Rate-based rules should have a lower priority number than managed rules so every request is counted.

2. **`override_action { none {} }` on managed rules** -- `none {}` means "use the actions defined inside the rule group as-is." For the Core Rule Set, that means Block on match. In production, you should start with `count {}` to log matches without blocking, then analyze before switching to `none {}`.

3. **Metrics and sampling disabled on rate limit** -- with `cloudwatch_metrics_enabled = false` and `sampled_requests_enabled = false`, you have no visibility into rate limit activity. You cannot see which IPs are being throttled or debug why the rule appears to not work.

The fix:

```hcl
rule {
  name     = "rate-limit"
  priority = 2                   # Before managed rules

  action { block {} }

  statement {
    rate_based_statement {
      limit              = 2000
      aggregate_key_type = "IP"
    }
  }

  visibility_config {
    cloudwatch_metrics_enabled = true
    sampled_requests_enabled   = true
    metric_name                = "rate-limit"
  }
}

rule {
  name     = "aws-managed-core"
  priority = 10                   # After custom rules

  override_action { count {} }    # Count mode for tuning

  # ...
}
```

</details>

## Summary

- **WAF WebACLs** inspect every HTTP request before it reaches your application, applying rules in priority order
- **Rate-based rules** automatically throttle IPs exceeding a request threshold -- essential for defending against credential stuffing
- **Managed Rule Groups** provide OWASP Top 10 protection without writing custom regex; always deploy in Count mode first
- **IP sets** and **geo-match** rules block known-bad sources at the edge before they consume application resources
- Custom rules should have **lower priority numbers** (evaluated first) than managed rules for proper counting
- WAF logging to CloudWatch (prefix `aws-waf-logs-`) enables analysis and tuning before switching from Count to Block
